### A59-4 — LOW — UnitAssignmentBean: Undocumented @Builder private constructor with non-trivial logic

**File:** `bean/UnitAssignmentBean.java`, lines 21–27
**Severity:** LOW (constructor is private; no public surface impact, but the boolean-to-string conversion is a non-obvious side effect)

The private `@Builder` constructor at line 21 performs a non-trivial transformation: the `boolean isCurrent` parameter is converted to the `String` values `"Yes"` or `"No"` and stored in the `current` field (line 26). This design decision — storing a boolean input as a localised English string — has no documentation explaining why a `String` field is used rather than a `boolean`, or that callers reading `getCurrent()` will receive `"Yes"`/`"No"` rather than a `boolean`. Because the constructor is private and all public access is via Lombok-generated accessors, this is recorded as LOW; however, a field-level or class-level comment explaining the `current` field encoding would significantly improve maintainability.

### A59-6 — LOW — UnitBean: Typo in field name `fule_type_name` (should be `fuel_type_name`)

**File:** `bean/UnitBean.java`, line 25
**Severity:** LOW (documentation audit dimension: the field name itself is a misspelling that no comment corrects or flags)

The field `fule_type_name` (line 25) is a misspelling of `fuel_type_name`. The paired field `fuel_type_id` (line 24) is spelled correctly. Because `@Data` generates a public getter `getFule_type_name()`, this typo is now part of the public API. There is no Javadoc or inline comment on this field to flag the misspelling or provide an alias. Any future developer reading the code or generated accessors will encounter an inconsistent name without guidance.

### A59-7 — LOW — UnitBean: Undocumented public nested enum `KeypadReaderModel`

**File:** `bean/UnitBean.java`, lines 66–71
**Severity:** LOW

The nested `public enum KeypadReaderModel` (line 66) is part of the public API surface of `UnitBean` and has no Javadoc. The four constants (`ROSLARE`, `KERI`, `SMART`, `HID_ICLASS`) represent specific hardware keypad/reader models used in access-control scenarios. None of the constants is documented to explain what hardware vendor or device each constant corresponds to (e.g., whether `HID_ICLASS` refers to HID Global's iCLASS reader series). This makes it difficult to extend or maintain the enum without external knowledge.

---

### A59-8 — LOW — UnitBean: Fields `acchours`, `exp_mod`, `access_id` lack any documentation

**File:** `bean/UnitBean.java`, lines 28, 30, 35
**Severity:** LOW

The fields `acchours` (line 28), `exp_mod` (line 30), and `access_id` (line 35) use opaque abbreviated names with no inline or Javadoc comment explaining their meaning. Examples of ambiguity:
- `acchours`: "accumulated hours"? "access hours"? "account hours"?
- `exp_mod`: "expansion module"? "expiry mode"? "expected model"?
- `access_id`: the `access_type` and `accessible` fields are present separately; it is unclear what identifier `access_id` stores (e.g., an external system ID for the access-control record).

Given the size of the bean (24 fields) and the domain complexity, these abbreviations without documentation represent a meaningful maintenance burden.
