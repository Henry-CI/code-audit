### A71-2 [LOW] — No class-level Javadoc on `UserUpdateResponse`

**File:** `UserUpdateResponse.java`, line 11

`UserUpdateResponse` carries no class-level Javadoc. There is no description distinguishing its dual role: it carries updated user attributes on success, but also carries `code`, `message`, and `detail` fields that appear to represent error/status information. This dual-use design is undocumented and is worth calling out explicitly.

### A71-5 [LOW] — Private `@Builder` constructor undocumented on `UserUpdateRequest`

**File:** `UserUpdateRequest.java`, line 20

The all-args constructor is `private` and annotated `@Builder`, making it part of the Lombok Builder pattern. Because it is `private` it is not itself a public API concern, but its use via the generated `UserUpdateRequest.builder()` static method (which is public) is entirely undocumented. There is no Javadoc or inline comment explaining the Builder usage or any constraints on fields (e.g. whether `password` and `accessToken` may both be supplied simultaneously).

---

### A71-6 [LOW] — No field-level documentation for semantically ambiguous fields in `UserUpdateResponse`

**File:** `UserUpdateResponse.java`, lines 18-20

The fields `code` (`Integer`), `message` (`String`), and `detail` (`String`) have no comments. Their presence alongside standard user-attribute fields (`username`, `email`, etc.) implies a mixed response/error envelope pattern, but nothing documents:
- What values `code` takes (HTTP status codes? custom error codes?)
- Whether `message` and `detail` are only populated on error, or also on success
- The `@JsonInclude(JsonInclude.Include.NON_NULL)` annotation suppresses null fields in serialization but is itself unexplained

While this is filed as LOW (undocumented fields are generally not rated higher than the accessor severity), the ambiguity across the class's dual concern could be upgraded if callers are observed misusing the response object.
