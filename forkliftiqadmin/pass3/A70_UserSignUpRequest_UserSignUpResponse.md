### A70-3 [LOW] — `UserSignUpRequest`: `@Builder` all-args constructor is `private` but undocumented

**File:** `UserSignUpRequest.java`, lines 27–38
**Rule:** Undocumented non-trivial method (builder constructor).
**Detail:** The `@Builder`-annotated private constructor is the mechanism through which Lombok generates the public `UserSignUpRequest.builder()` static factory. There is no Javadoc explaining the expected values for `username`, `password`, `email`, `name`, `lastname`, `phoneNumber`, or `givenName` — for instance, whether `password` must meet Cognito strength requirements, or what format `phoneNumber` must follow (E.164, etc.).
Severity is LOW rather than MEDIUM because the constructor itself is `private`; however, the absence of any builder-level guidance is notable for a security-sensitive sign-up payload.

### A70-8 [MEDIUM] — `UserSignUpResponse`: `password` field present in response DTO is a design concern with no explanatory comment

**File:** `UserSignUpResponse.java`, line 23
**Rule:** Inaccurate / misleading documentation by omission on a security-sensitive element.
**Detail:** The response bean contains a `password` field (`String password`). Returning a password in a sign-up response is highly unusual and potentially dangerous if this value is ever serialized into an API response. The class is annotated with `@JsonInclude(JsonInclude.Include.NON_NULL)`, so a non-null password would be included in JSON output. There is no comment or Javadoc explaining why this field exists in a response object, whether it is intentionally populated, or whether it is a carry-over from request/response conflation. The absence of any explanatory documentation for this field elevates severity to MEDIUM.

---

### A70-9 [LOW] — `UserSignUpResponse`: does not implement `Serializable` despite having `serialVersionUID`

**File:** `UserSignUpResponse.java`, lines 12–17
**Rule:** Inaccurate / misleading (structural inconsistency, no documentation to clarify intent).
**Detail:** The class declares `private static final long serialVersionUID = 4896099777087206031L` (line 17) but does not declare `implements Serializable` in the class signature (line 12). This makes `serialVersionUID` meaningless (the JVM only consults it for `Serializable` classes). The field was likely copy-pasted from `UserSignUpRequest`. No comment explains the intent. Severity is LOW because it is a harmless dead field rather than dangerous, but it indicates lack of review.
