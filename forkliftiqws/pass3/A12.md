## Findings

### A12-1
**Severity:** HIGH
**File:** `Company.java`
**Location:** Class declaration (line 13)
**Description:** No class-level Javadoc. The `Company` class has no documentation explaining what business entity it represents, its lifecycle, its relationship to `Driver` (contact person), or the semantics of its fields such as `arrRoles`, `permission`, and `maxSessionLength`. A reader cannot determine the purpose of this model without reading the code of consuming classes.

---

### A12-2
**Severity:** CRITICAL
**File:** `Company.java`
**Location:** `@Builder` private constructor, lines 44–49
**Description:** The builder constructor contains a non-trivial, silent field-overwrite behaviour: when `contactDriver` is non-null, the constructed `Company` object's `name`, `email`, and `password` are replaced with values from the `Driver` object. This side-effect is completely undocumented. There is no class-level, constructor-level, or field-level Javadoc warning callers of this behaviour. A caller using the Lombok-generated builder has no way to discover this mutation from the API surface alone. The inline comment `// Override user information with the driver` (line 45) is inside a private method and never reaches generated documentation.

---

### A12-3
**Severity:** HIGH
**File:** `Company.java`
**Location:** All Lombok-generated public methods (25 methods)
**Description:** All public methods generated by `@Data` and `@NoArgsConstructor` — including getters, setters, `equals`, `hashCode`, `toString`, and `canEqual` — have no Javadoc. While trivial accessors may not warrant individual documentation, the absence of any documentation is a systemic gap. Of particular concern: `getPassword()` / `setPassword(String)` return/accept a credential field with no documentation of expected encoding, hashing state, or security constraints.

---

### A12-4
**Severity:** LOW
**File:** `Company.java`
**Location:** `serialVersionUID` (line 15–18)
**Description:** The existing Javadoc on `serialVersionUID` is an empty auto-generated stub containing only whitespace. It conveys no information and should either be removed or replaced with meaningful content. This pattern is repeated across all three files.

### A12-5
**Severity:** HIGH
**File:** `Driver.java`
**Location:** Class declaration (line 15)
**Description:** No class-level Javadoc. The `Driver` class has no documentation explaining the entity it models. The class contains 19 fields including security-sensitive ones (`password`, `securityno`, `licno`) and operationally significant ones (`contactperson`, `driver_based`, `gps_frequency`, `compliance_date`). Without class-level documentation, the semantics and constraints of these fields are opaque.

---

### A12-6
**Severity:** HIGH
**File:** `Driver.java`
**Location:** Field `drivers` (line 41), `contactperson` (line 32)
**Description:** The field `drivers` carries a meaningful inline comment — `// List of company drivers when this driver's contactperson is true` — but this comment is not a Javadoc comment and does not appear in generated API documentation. The relationship between `contactperson` and the population of the `drivers` list is a non-obvious business rule that requires documentation. The `contactperson` field itself has no documentation of what it means when `true` or `false`.

---

### A12-7
**Severity:** HIGH
**File:** `Driver.java`
**Location:** All Lombok-generated public methods (46 methods)
**Description:** All 46 public methods generated by `@Data`, `@NoArgsConstructor`, and `@EqualsAndHashCode` have no Javadoc. Notably:
- `equals(Object)` and `hashCode()` are governed by `@EqualsAndHashCode(onlyExplicitlyIncluded = true)`, meaning equality is based solely on `email`. This is a significant, non-default contract that is completely undocumented at the public API level.
- `getPassword()` / `setPassword(String)` expose a credential with no documentation of encoding or hashing state.
- `getSecurityno()` / `setSecurityno(String)` expose what appears to be a government-issued security number with no documentation of format, masking, or handling requirements.

---

### A12-8
**Severity:** CRITICAL
**File:** `Driver.java`
**Location:** `@EqualsAndHashCode(onlyExplicitlyIncluded = true)` annotation (line 14), `equals` and `hashCode` methods
**Description:** The `equals` and `hashCode` contract is silently restricted to the `email` field only. This means two `Driver` objects with the same email but different IDs, names, or any other field are considered equal. This non-standard equality contract is not documented anywhere — not on the class, not on the `email` field, and not on the generated `equals`/`hashCode` methods. Callers relying on standard object equality semantics will be misled without this documentation.

---

### A12-9
**Severity:** MEDIUM
**File:** `Driver.java`
**Location:** Field declarations (lines 19–40)
**Description:** Multiple fields use snake_case naming (`first_name`, `last_name`, `comp_id`, `driver_based`, `date_format`, `max_session_length`, `compliance_date`, `gps_frequency`, `photo_url`, `licno`, `expirydt`, `securityno`) in violation of Java naming conventions. No Javadoc or inline comment explains whether this convention is intentional (e.g., for direct ORM column mapping) or is a legacy convention. Without documentation, maintainers cannot determine if renaming would be safe.

---

### A12-10
**Severity:** HIGH
**File:** `DriverEmails.java`
**Location:** Class declaration (line 5)
**Description:** No class-level Javadoc. The class name `DriverEmails` and its structure — four separate `email_addr` fields rather than a list — require explanation. It is undocumented whether these represent alert email addresses, CC recipients, notification targets, or some other purpose. The relationship between `driver_id` and `Driver.id` is not documented.

### A12-11
**Severity:** HIGH
**File:** `DriverEmails.java`
**Location:** All 12 explicit public methods (lines 20–55)
**Description:** All 12 hand-written public getter and setter methods have no Javadoc. Because this class does not use Lombok and explicitly defines its accessors, the absence of Javadoc is a deliberate omission rather than a Lombok convention issue. Particularly notable: the purpose and distinction between `email_addr1`, `email_addr2`, `email_addr3`, and `email_addr4` is not documented anywhere, leaving readers to infer whether ordering is significant and whether null values are permitted.

### A12-13
**Severity:** MEDIUM
**File:** `Company.java`
**Location:** Fields `password` (line 23), `Driver.password` (line 25), `Driver.securityno` (line 30)
**Description:** Sensitive fields (`password`, `securityno`) have no Javadoc indicating whether values are plaintext, hashed, or encrypted. The `toString()` method generated by `@Data` on both `Company` and `Driver` will include these fields in its output, potentially exposing credentials and security numbers in log files. This documentation gap means developers cannot determine from the API alone whether it is safe to log instances of these classes.
