**Severity:** HIGH
**File:** `Manufacturer.java`
**Location:** Class declaration (line 5)
**Description:** The `Manufacturer` class has no class-level Javadoc comment. There is no description of the class's purpose, its role in the domain model, or any notes about serialisation. A developer encountering this class for the first time has no contextual documentation to rely upon.

### A19-2

There are no `@param`, `@return`, or descriptive sentences on any method.

### A19-7
**Severity:** HIGH
**File:** `PackageEntry.java`
**Location:** Class declaration (line 15)
**Description:** The `PackageEntry` class has no class-level Javadoc. The class represents a versioned software package with structured version parsing (major/minor/patch/env) and implements `Comparable`. Its purpose, the version string format it expects, and the semantics of its fields are entirely undocumented at the class level.

---

### A19-8
**Severity:** HIGH
**File:** `PackageEntry.java`
**Location:** Line 57 (`compareTo`)
**Description:** The `compareTo(PackageEntry o)` method is public, overrides `Comparable`, and contains non-trivial behaviour: (1) it returns `1` when the argument is `null`, which deviates from the `Comparable` contract that requires throwing `NullPointerException` for null arguments; (2) it compares only `major`, `minor`, and `patch` — the `env` field is intentionally excluded from comparison. Neither behaviour is documented. The null-tolerance in particular is a contract violation that could silently mask bugs in consuming code and must be described in a Javadoc `@throws` (or explicit note explaining the deviation).

---

### A19-9
**Severity:** MEDIUM
**File:** `PackageEntry.java`
**Location:** Lines 17–27 (fields `name`, `fileName`, `url`, `major`, `minor`, `patch`, `env`, `pattern`)
**Description:** All fields that back the Lombok-generated public accessors/mutators have no field-level Javadoc. The semantics of `env` (environment tag parsed from the version string), `pattern` (the compiled regex used for version parsing), and the version integer fields are not described anywhere. Because `@Data` generates the public API from field declarations, field-level Javadoc is the appropriate documentation point for those generated methods.

---

### A19-10
**Severity:** MEDIUM
**File:** `PackageEntry.java`
**Location:** Lines 28–34 (private `@Builder` constructor)
**Description:** The `@Builder` annotation on a private constructor generates a static `PackageEntry.builder()` factory method and a `PackageEntryBuilder` inner class. Because the constructor is `private`, the generated builder is the only intended construction path for parameterised instances. The expected format of the `version` parameter (required by `initVersion`) is not documented anywhere — neither on the constructor, nor on the builder method, nor at the class level. A caller has no guidance on what version string formats are accepted.

---

### A19-11
**Severity:** INFO
**File:** `PackageEntry.java`
**Location:** Lines 36–53 (private `initVersion`)
**Description:** `initVersion` is private and therefore outside the strict scope of public-API documentation requirements. However, the regex `(\d*)\.?(\d*)?\.?(\d*)?-(\w*)` and the group-index semantics are non-obvious. The absence of any inline or Javadoc comment makes this method difficult to maintain. Noted for completeness.
