## 3. Findings

---

### A11-1
**Severity:** HIGH
**File:** `src/main/java/com/journaldev/spring/jdbc/model/AuthenticationRequest.java`
**Location:** Class declaration (line 11)
**Description:** No class-level Javadoc. `AuthenticationRequest` carries security-sensitive fields (`password`, `newPassword`, `accessToken`). The absence of any class-level documentation means there is no description of purpose, lifecycle (e.g., whether `newPassword` and `accessToken` are used only during credential-change or token-refresh flows), or security contract (e.g., that password fields must not be logged or serialized to persistent storage). For a class that forms part of the authentication API surface, this is a meaningful documentation gap.

---

### A11-2
**Severity:** MEDIUM
**File:** `src/main/java/com/journaldev/spring/jdbc/model/AuthenticationRequest.java`
**Location:** Fields `username`, `password`, `newPassword`, `accessToken` (lines 17–20)
**Description:** No field-level Javadoc on any field. The semantics of `newPassword` (used only during a password-change flow?) and `accessToken` (used for token refresh, or passed in for re-authentication?) are not documented. Callers building the object via the Lombok-generated `builder()` cannot determine which fields are required vs. optional for a given operation.

---

### A11-3
**Severity:** LOW
**File:** `src/main/java/com/journaldev/spring\jdbc\model\AuthenticationRequest.java`
**Location:** `serialVersionUID` Javadoc stub (lines 13–15)
**Description:** The Javadoc comment block on `serialVersionUID` is a generated empty stub (`/** \n * \n */`) with no content. While `serialVersionUID` itself rarely warrants prose documentation, the presence of an empty stub constitutes inaccurate / misleading documentation (a comment block that appears to document something but says nothing).

---

### A11-4
**Severity:** HIGH
**File:** `src/main/java/com/journaldev/spring/jdbc/model/AuthenticationResponse.java`
**Location:** Class declaration (line 13)
**Description:** No class-level Javadoc. `AuthenticationResponse` is the primary object returned by the authentication endpoint and contains the JWT access token, session token, expiry information, and user profile data. The absence of a class-level description leaves callers without documentation of: the meaning of `actualDate` vs. `expirationDate`, whether `sessionToken` is distinct from `accessToken` and in what flows it is populated, when `message` is populated vs. when the `errors` list from the parent `ResponseWrapper` is used, and the semantics of `expiresIn` (is it seconds? ISO 8601?).

---

### A11-5
**Severity:** HIGH
**File:** `src/main/java/com/journaldev/spring/jdbc/model/AuthenticationResponse.java`
**Location:** Fields `expiresIn` (line 21), `actualDate` (line 22), `expirationDate` (line 23), `message` (line 26)
**Description:** Four fields with non-obvious semantics carry no Javadoc. `expiresIn` is typed as `String` but conventionally represents a duration; the unit (seconds, milliseconds, ISO 8601) is undocumented. `actualDate` and `expirationDate` have ambiguous names with no documentation of format or timezone. `message` is undocumented as to when it is populated (success message? error message? is it redundant with `errors` from `ResponseWrapper`?). This is a documentation deficit for the primary outbound model of the authentication subsystem.

### A11-7
**Severity:** HIGH
**File:** `src/main/java/com/journaldev/spring/jdbc/model/Charts.java`
**Location:** Class declaration (line 9)
**Description:** No class-level Javadoc. The purpose and usage context of `Charts` is entirely undocumented. The class aggregates a unit identifier, a unit label, a total value (BigDecimal), and a list of `Usage` objects. Without documentation it is unclear what "unit" refers to in a forklift-IQ context (a physical forklift unit? a measurement unit?), what `total` aggregates, and what the relationship between `Charts` and `Usage` is intended to be.

### A11-8
**Severity:** HIGH
**File:** `src/main/java/com/journaldev/spring/jdbc/model/Charts.java`
**Location:** `addUsageList(Usage usage)` (line 40)
**Description:** The method `addUsageList(Usage usage)` has no Javadoc. This is the only non-trivial (non-simple-getter/setter) method in the class. Its name is misleading — `addUsageList` suggests adding a list but the implementation adds a single `Usage` item to the internal list. Without Javadoc, callers cannot distinguish intended behavior from a naming defect, and there is no `@param` tag describing the `usage` argument. This is the most significant individual method documentation gap in the file.

**Location:** `getUnit_id()` (line 43), `setUnit_id(int)` (line 46)

There is no Javadoc clarifying the intent or the naming anomaly.
