## Findings

---

### A08-1
**Severity:** HIGH
**File:** `LocationController.java`
**Line:** 24-26
**Finding:** Inaccurate class-level Javadoc.
The only Javadoc present on `LocationController` is `/** Not in Use */`. This is factually inaccurate: the class defines three active `@RequestMapping` endpoints (`GET_GPS_LOC`, `SAVE_GPS_LOC`, `SAVE_GPS_LOCS`) that are compiled into the application. The misleading comment creates a false understanding of the class's active role and may lead maintainers to delete or ignore this controller incorrectly.

---

### A08-2
**Severity:** MEDIUM
**File:** `LocationController.java`
**Lines:** 35, 53, 72
**Finding:** All three public controller methods lack Javadoc.
`getGPSLocation`, `saveGPSLocation`, and `saveGPSLocations` are public REST endpoint handlers with no Javadoc. None have `@param`, `@return`, or `@throws` tags. The HTTP method, URI, expected request body format, and response semantics are not documented.

Affected methods:
- `public List<GPS> getGPSLocation(int uid)` (line 35)
- `public ResponseEntity<Results> saveGPSLocation(GPS gps)` (line 53)
- `public ResponseEntity<Results> saveGPSLocations(GPSList gpsList)` (line 72)

---

### A08-3
**Severity:** HIGH
**File:** `ReportAPI.java`
**Line:** 1-6 (class declaration)
**Finding:** No class-level Javadoc on `ReportAPI`.
`ReportAPI` is a non-trivial public class with constructor, business method (`downloadPDF`), and 14 accessor/mutator methods. There is no class-level Javadoc describing the class's purpose, responsibilities, or usage pattern. The relationship with `HttpDownloadUtility` and `RuntimeConfig.PDF_FOLDER` is entirely undocumented at the class level.

---

### A08-4
**Severity:** HIGH
**File:** `ReportAPI.java`
**Line:** 24-25
**Finding:** `downloadPDF()` has a block comment instead of Javadoc, and the comment is incomplete.
The comment `/*Read the pdf file from API, store locally, send email */` uses a standard block comment (`/* */`) rather than a Javadoc comment (`/** */`). It will not appear in generated API documentation. Additionally, the comment claims the method sends email, but inspecting the method body reveals it only calls `HttpDownloadUtility.sendPost(...)` and returns a file path — there is no email-sending logic in this method. The method also declares `throws Exception` with no `@throws` documentation and has no `@return` tag.

### A08-6
**Severity:** LOW
**File:** `ReportAPI.java`
**Lines:** 47-102
**Finding:** All 14 public accessor/mutator methods lack Javadoc.
The following methods have no Javadoc of any kind: `getSubject`, `setSubject`, `getTitle`, `setTitle`, `getContent`, `setContent`, `getsEmail`, `setsEmail`, `getrEmail`, `setrEmail`, `getFileURL`, `setFileURL`, `getName`, `setName`. ### A08-10
**Severity:** MEDIUM
**File:** `RestURIConstants.java`
**Line:** 52
**Finding:** Misspelled path segment in `RESUME_EQUIPMENT` constant — `frequencey`.
The constant is declared as:
```java
public static final String RESUME_EQUIPMENT = "/rest/driverresume/get/{uid}/frequencey/{frequencey}";
```
The path variable name `frequencey` is a misspelling of `frequency`. This misspelling is baked into the public URI constant and is present in both the path segment name and the path variable placeholder. Any client coded against this endpoint must replicate the misspelling. There is no Javadoc or inline comment flagging this as an intentional compatibility decision.

---

### A08-11
**Severity:** MEDIUM
**File:** `RestURIConstants.java`
**Line:** 50
**Finding:** Misspelled path segment in `GET_SERVICEDTL` constant — `sericedetail`.
The constant is declared as:
```java
public static final String GET_SERVICEDTL = "/rest/sericedetail/get/{userid}";
```
The path segment `sericedetail` is missing the letter `v` (should be `servicedetail`). This is a permanent part of the public API URI surface and is undocumented.
