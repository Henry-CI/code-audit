## Findings

### A22-1
**Severity:** MEDIUM
**File:** `Results.java`
**Location:** Line 5
**Description:** The only class-level comment on `Results` is an inline `//` comment reading "for the web service return return". This is not a Javadoc comment and therefore does not appear in generated API documentation. Additionally, it contains a duplicated word ("return return"), indicating it was never reviewed. The class purpose — a serializable DTO carrying a message ID and an error string for web service responses — is entirely undocumented at the Javadoc level.

---

### A22-2
**Severity:** MEDIUM
**File:** `Results.java`
**Location:** Lines 16, 19, 23, 27
**Description:** All four public methods of `Results` — `getMessage_id()`, `setMessage_id(String)`, `getError()`, `setError(String)` — lack Javadoc. There is no documentation describing what `message_id` or `error` represent (e.g., correlation ID format, error code conventions), nor any `@param` or `@return` tags.

---

### A22-3
**Severity:** MEDIUM
**File:** `Roles.java`
**Location:** Line 11 (class declaration)
**Description:** The `Roles` class has no class-level Javadoc. The class models a role/authority entity (with fields `id`, `name`, `authority`, `description`) and uses Lombok `@Data` and `@NoArgsConstructor` to generate a substantial public API surface. The absence of class-level documentation leaves consumers with no guidance on the purpose of the class or the semantics of each field.

---

### A22-4
**Severity:** HIGH
**File:** `Roles.java`
**Location:** Lines 44–52 (`RoleId.fromId(String id)`)
**Description:** The static factory method `RoleId.fromId(String id)` is entirely undocumented. It performs a lookup across all `RoleId` enum values and throws `IllegalArgumentException` if no match is found. The absence of Javadoc means there is no `@param` describing the expected format of the `id` argument, no `@return` describing what is returned on success, and no `@throws` documenting the unchecked exception. The thrown exception message references "Name" rather than "ID" (`"Name " + id + " is not valid role identifier"`), which is itself misleading, but that is a separate concern; the missing `@throws` tag is a documentation defect.

### A22-6
**Severity:** MEDIUM
**File:** `Roles.java`
**Location:** Lines 9–10 (`@Data`, `@NoArgsConstructor`)
**Description:** Lombok annotations `@Data` and `@NoArgsConstructor` generate a significant public API surface (getters, setters, `equals`, `hashCode`, `toString`, no-arg constructor). None of this generated API is accompanied by field-level Javadoc that would carry through into generated accessor documentation. Fields `id`, `name`, `authority`, and `description` have no Javadoc comments, leaving the semantics of each (e.g., whether `name` is a display label or a machine-readable key distinct from `authority`) undocumented.

### A22-7
**Severity:** MEDIUM
**File:** `Services.java`
**Location:** Line 6 (class declaration)
**Description:** The `Services` class has no class-level Javadoc. The class models a forklift service record (accumulated hours, service due threshold, last and next service intervals, duration, associated unit and driver). Without class-level documentation, the domain meaning of the class and the units or conventions for its numeric fields (e.g., whether `acc_hours` is in decimal hours, whether `last_serv`/`next_serv`/`serv_duration` are in hours or some other unit) are entirely opaque to readers of the API.

---

### A22-8
**Severity:** HIGH
**File:** `Services.java`
**Location:** Lines 29–44 (`getAcc_hours()`, `setAcc_hours()`, `getService_due()`, `setService_due()`)
**Description:** The four public methods dealing with `BigDecimal` fields `acc_hours` and `service_due` lack any Javadoc. These are the semantically richest fields in the model: `acc_hours` presumably holds accumulated engine/operating hours and `service_due` presumably holds a threshold in the same unit at which service becomes due. Without documentation, the precision, scale, and unit conventions for these `BigDecimal` values are unknown to consumers. This is rated HIGH because misinterpretation of these fields could result in incorrect service scheduling logic.
