## Findings

---

### A09-1
**Severity:** HIGH
**File:** `ResumeController.java` line 26-28
**Finding:** Class-level Javadoc is inaccurate. The comment reads `"Handles requests for the Product JDBC Service."` The class is not a product service; it provides resume/reporting endpoints (driver summary, equipment charts, report generation, scheduled report dispatch). The copied/generic description is misleading to maintainers and API consumers.

---

### A09-2
**Severity:** HIGH
**File:** `SessionController.java` line 47-49
**Finding:** Class-level Javadoc is inaccurate. The comment reads `"Handles requests for the Employee JDBC Service."` The class manages forklift session lifecycle endpoints (start, end, abort, offline save), pre-start inspection result persistence, and image uploads. The description bears no relation to the actual responsibility of the class.

---

### A09-3
**Severity:** HIGH
**File:** `ResumeController.java`
**Finding:** All five public endpoint methods lack Javadoc entirely. None of the following methods have any `/** ... */` documentation block:
- `getResumeDriver(int uid)` — returns driver failure/incomplete percentages and service/usage summaries
- `getResumeEquipment(int uid, String frequencey)` — returns equipment usage charts by frequency (yearly/monthly/weekly)
- `getReportList(int uid)` — returns list of subscribed reports for a driver
- `generateReport(int uid, int rid)` — triggers PDF report generation and emails it to the driver's registered addresses
- `sendReportList(String frq)` — batch-sends scheduled reports to company email addresses by frequency

These are REST endpoints with non-trivial business logic, complex query strategies, and external service calls; absence of documentation creates significant maintenance risk.

---

### A09-4
**Severity:** HIGH
**File:** `SessionController.java`
**Finding:** All nine public endpoint methods lack Javadoc entirely. None of the following have any `/** ... */` documentation block:
- `driverAccess(int sid, int imageno, MultipartFile file)` — stores an equipment inspection photo (front/left/right/back) against a session
- `getQuestion(int eid)` — retrieves the pre-start inspection question set for a given equipment unit
- `getForm(int qid, String type)` — retrieves form detail entries for a question/type combination
- `searchForm(int uid, String type, String keyword)` — intended to search form entries (contains a broken SQL query)
- `saveResults(Result result)` — persists a completed pre-start inspection result and associated answers
- `startSessions(Sessions session)` — creates a new forklift session and optional pre-op result record
- `endSessions(Sessions session)` — closes a session and updates accumulated service hours on the unit
- `abortSessions(int sid)` — deletes a session record by ID
- `saveOffline(OfflineSessions offlineSession)` — persists a session and inspection result captured while offline

---

### A09-5
**Severity:** MEDIUM
**File:** `RuntimeConfig.java`
**Finding:** The class has no class-level Javadoc and no field-level Javadoc on any of its 19 public static fields. As a central configuration constants class that is referenced throughout the application, the purpose and valid values of each constant (particularly message IDs, error message strings, API URLs, and credential fields) are entirely undocumented. Consumers cannot determine intent, expected value ranges, or ownership without reading callsites.

---

### A09-6
**Severity:** MEDIUM
**File:** `RuntimeConfig.java` line 4
**Finding:** Inline comment `//email settings, unused` is factually inaccurate. The fields `RuntimeConfig.emailContent` and `RuntimeConfig.mailFrom` from this block are actively referenced in `ResumeController.generateReport` (lines 333, 338, 343, 348) and `ResumeController.sendReportList` (line 417). Marking them as unused will mislead developers into removing or ignoring them.

---

### A09-7
**Severity:** MEDIUM
**File:** `ResumeController.java` line 120
**Finding:** The parameter `frequencey` in `getResumeEquipment(int uid, String frequencey)` is a misspelled version of "frequency". This typo propagates through all internal variable names and log output (`logger.info("... frequency:"+frequencey)`). In the absence of Javadoc, the parameter name is the only documentation of the parameter's purpose and accepted values (`"y"`, `"m"`, or any other value treated as weekly). The misspelling reduces readability and discoverability.

---

### A09-8
**Severity:** LOW
**File:** `ResumeController.java` lines 116, 143, 152, 170, 187, 229
**Finding:** Numerous `// TODO inserted [15 Jun 2017, ...]` and `// TODO [19 Jun 2017, ...]` comments are present but contain no description of what is outstanding or what action is required. These are stale task markers (approximately 8.5 years old) that contribute noise rather than documentation. Where business logic relies on them as markers (e.g., query construction loops), their presence implies incompleteness without clarifying intent.

---

### A09-9
**Severity:** LOW
**File:** `SessionController.java` line 225-251
**Finding:** A block of commented-out code (`// No need ot check other driver uses it`) at lines 246-251 inside `startSessions` represents a removed business rule about preventing concurrent vehicle use by different drivers. The inline comment is brief and contains a spelling error ("ot" instead of "to"). The commented-out block is undocumented — there is no explanation of why the rule was removed, whether it is intentional, or whether it may be reinstated. This is an absence of documentation on a deliberate architectural decision.

---

### A09-10
**Severity:** LOW
**File:** `SessionController.java` line 53
**Finding:** The logger is instantiated using `ConfigurationController.class` as the logger name rather than `SessionController.class`:
```java
private static final Logger logger = LoggerFactory.getLogger(ConfigurationController.class);
```
This is not a Javadoc issue but it means all log output from `SessionController` is attributed to `ConfigurationController` in log aggregation systems. The mismatch is undocumented and constitutes misleading operational documentation (log attribution).
