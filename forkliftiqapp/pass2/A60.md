#### A60-1 — `onItemClick()` sets operator identity with no credential challenge
**Severity:** HIGH
**File:** `DriverListFragment.java`
**Lines:** 55–59

```java
public void onItemClick(View v, int position) {
    CurrentUser.setUser(this.adapter.getDatas().get(position));
    Intent intent = Intent.makeMainActivity(new ComponentName(Objects.requireNonNull(getContext()), DashboardActivity.class));
    startActivity(intent);
}
```

A tap on any row in the driver list immediately calls `CurrentUser.setUser(...)` and launches the full dashboard as that user, with no PIN, password, biometric prompt, or any other credential challenge. The driver list is displayed to the currently authenticated supervisor (contact person). Any supervisor who reaches this screen can impersonate any associated driver account with a single tap. There is no test verifying that authentication is required before identity changes, and no test verifying that the dashboard cannot be reached without a successful credential check.

This is the specific finding flagged in the audit brief. The severity is HIGH because it allows unauthorized identity switching within an authenticated session — a supervisor can silently assume any driver's identity, affecting audit trails, pre-start check attribution, session records, compliance data, and GPS-tracked activity.

#### A60-11 — `loadData()` cache bypass: re-entering the fragment after a failure does not reload
**Severity:** MEDIUM
**File:** `DriverStatsFragment1.java`
**Lines:** 59–63

```java
private void loadData() {
    if (driverStatsResultArray != null) {
        onData(driverStatsResultArray);
        return;
    }
```

`driverStatsResultArray` is only set inside `onData()`, which is only called on success. After a network failure, `driverStatsResultArray` remains `null`. However, if the fragment is added to the back stack and `initViews()` is called again (via `onActivityCreated()`), `loadData()` will issue a new network request as expected. The problem is the cache is never invalidated on failure, so stale successful data can persist indefinitely across navigation. No test verifies the behavior across the success-then-failure or fragment-recreation scenarios.

#### A60-14 — `onData()` uses `Double.valueOf()` without catching `NumberFormatException`
**Severity:** HIGH
**File:** `DriverStatsFragment2.java`
**Lines:** 136–150

```java
} else if (driverStatsItem.field.equals(STATS_KEY_PRE_START_FAILED)) {
    String s = driverStatsItem.value;
    Double d = Double.valueOf(s);
    if (d != null) {
        preStartFailed = d.floatValue();
    } else {
        preStartFailed = 0.0f;
    }
}
```

`Double.valueOf(String)` throws `NumberFormatException` (an unchecked exception) if `s` is not a valid double representation. The code attempts to handle the null case but `Double.valueOf()` never returns `null` — it either returns a boxed value or throws. If the server returns an unexpected non-numeric value for the `"failed"` or `"incompleted"` fields, this call crashes the fragment with an unhandled exception. The null check (`if (d != null)`) is dead code and shows a misunderstanding of the API — it will never be false. No test supplies a non-numeric or malformed `value` string to verify crash behavior. This is rated HIGH because it is triggered by server-supplied data, making it exploitable via a malicious or misconfigured backend response.
