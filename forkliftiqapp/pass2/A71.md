## Findings

### A71-1 — CRITICAL: Complete absence of test infrastructure in build configuration
**Files:** `app/build.gradle`, `LibCommon/build.gradle`

`app/build.gradle` declares exactly one test-scoped dependency: `testImplementation 'junit:junit:4.12'`. No test sources exist in `app/src/` to consume it. `LibCommon/build.gradle` declares zero test-scoped dependencies. Neither module declares `testInstrumentationRunner`. Neither module has a `test/` or `androidTest/` source directory. The single JUnit 4 declaration in `app/build.gradle` is a placeholder with no corresponding test code — it is not evidence of testing; it confirms intent that was never acted upon.

The complete picture from build configuration:
- No `testInstrumentationRunner` configured in either module
- No `androidTestImplementation` dependencies in either module
- No Espresso, Mockito, Robolectric, or any mocking/assertion library in either module
- No test source sets defined
- No test build variants defined
- `proguard-rules.pro` in both modules contains no `-keep` rules referencing test infrastructure
- `LibCommon/build.gradle` has `lintOptions { abortOnError false }`, suppressing lint failures that might otherwise surface code-quality issues that tests would catch

The build configuration provides zero infrastructure for testing. No unit tests, no instrumentation tests, and no integration tests can be run for the primary application code or the shared library.

### A71-2 — HIGH: Signing credentials committed in plaintext to version control
**Files:** `gradle.properties`, `.gitignore`

`gradle.properties` contains the keystore path, alias, and passwords for all three signing environments (DEV, UAT, RELEASE) in plaintext:

```
DEV_STORE_PASSWORD=ciadmin
DEV_KEY_PASSWORD=ciadmin
UAT_STORE_PASSWORD=ciadmin
UAT_KEY_PASSWORD=ciadmin
RELEASE_STORE_PASSWORD=ciadmin
RELEASE_KEY_PASSWORD=ciadmin
```

`.gitignore` does not exclude `gradle.properties`. Git history confirms `gradle.properties` is tracked. Additionally, both keystore files (`fleetiq.jks` and `app.jks`) are committed to the repository and tracked by git — confirmed via `git ls-files`. Any person with read access to the repository can extract the release signing key and produce APKs signed with the production certificate, enabling unauthorized app distribution that would appear legitimate to Android package verification.

The RELEASE environment references the same keystore file (`../fleetiq.jks`) and the same credentials (`ciadmin`) as DEV and UAT, meaning there is no credential separation between environments.

### A71-3 — HIGH: Keystore files committed to version control
**Files:** `fleetiq.jks`, `app.jks` (tracked by git), `.gitignore`

Both `fleetiq.jks` and `app.jks` are committed to the repository. `.gitignore` contains no pattern to exclude `*.jks` or `*.keystore` files. Combined with the plaintext credentials in `gradle.properties` (A71-2), an attacker with repository access has everything needed to sign arbitrary APKs with the production key. Revoking this exposure requires rotating the signing key and all credentials — it cannot be resolved by simply removing the files from HEAD, as the files remain in git history.

### A71-4 — HIGH: minifyEnabled false in release builds — code is not obfuscated
**Files:** `app/build.gradle`, `LibCommon/build.gradle`

Both modules set `minifyEnabled false` in their `release` build type. ProGuard/R8 is not invoked at release time. Consequences:
- Class, method, and field names are not obfuscated — the APK is trivially reversible
- Dead code is not removed, increasing APK size and attack surface
- The referenced `proguard-rules.pro` files in both modules contain no active rules anyway (see A71-5), so enabling minification would require rules to be written first
- Business logic, API endpoint construction, and authentication flows are visible in plain form to anyone who decompiles the APK

### A71-5 — HIGH: proguard-rules.pro files are empty — no application-specific keep rules
**Files:** `app/proguard-rules.pro`, `LibCommon/proguard-rules.pro`

Both ProGuard configuration files contain only the default comment template. No `-keep`, `-keepclassmembers`, `-dontwarn`, or any other active directive has ever been written. Even if `minifyEnabled` were set to `true`, the absence of keep rules would cause ProGuard to strip or rename classes and methods referenced by Realm (which uses reflection), Retrofit (which uses GSON and reflection for model deserialization), the BLE services, and the Realm model objects — resulting in runtime crashes. The empty rule files indicate that minification has never been tested and could not be safely enabled without significant rule authoring.

### A71-8 — HIGH: jcenter() repository used — repository is read-only and sunset
**Files:** `build.gradle` (root)

`jcenter()` is listed as a dependency repository in both `buildscript` and `allprojects` blocks. JCenter was shut down for writes in March 2021 and its read availability is not guaranteed. More critically: JCenter did not remove artifacts when shut down — any dependency resolved from JCenter could have been replaced by a malicious artifact uploaded before the shutdown, and the repository did not enforce artifact integrity. All dependencies should be resolved from `google()` or `mavenCentral()` with verified checksums. The app currently uses JCenter as a fallback for any dependency not found in `google()`.

### A71-9 — MEDIUM: v2SigningEnabled false for all signing configurations
**Files:** `app/build.gradle`

All three signing configurations (`dev`, `uat`, `release`) set `v2SigningEnabled false`. APK Signature Scheme v2 (introduced in Android 7.0 / API 24) provides stronger integrity guarantees — it signs the entire APK file, not just individual entries. Disabling it means the APK relies only on JAR signing (v1), which is vulnerable to ZIP-level manipulation attacks that can modify content without invalidating the signature. Since `minSdkVersion` is 21 (Android 5.0), v1 is required for compatibility with API 21–23, but v2 should be enabled in addition to v1, not instead. Setting `v2SigningEnabled false` explicitly opts out of v2 protection on all API 24+ devices.

### A71-10 — MEDIUM: defaultConfig is incorrectly nested inside buildTypes block in both build.gradle files
**Files:** `app/build.gradle`, `LibCommon/build.gradle`

In both `app/build.gradle` and `LibCommon/build.gradle`, the `defaultConfig {}` block is nested inside `buildTypes {}`:

```groovy
// app/build.gradle, lines 36–56
buildTypes {
    defaultConfig {          // WRONG — should be a sibling of buildTypes, not a child
        applicationId ...
        minSdkVersion ...
        ...
    }
    release {
        minifyEnabled false
        ...
    }
}
```

The standard Android DSL requires `defaultConfig {}` to be a direct child of the `android {}` block, not inside `buildTypes {}`. That this configuration compiles and produces APKs suggests that AGP 3.0.0 silently accepts `defaultConfig` in unexpected positions (possibly treating it as a build type named "defaultConfig"), but the actual behavior is ambiguous and the `applicationId`, `minSdkVersion`, and `targetSdkVersion` values may or may not be applied as intended. This is a latent correctness risk that would be caught immediately by test builds on a modern toolchain.

### A71-11 — MEDIUM: android:allowBackup="true" — application data can be extracted via ADB
**Files:** `app/src/main/AndroidManifest.xml`

The `<application>` element sets `android:allowBackup="true"`. On devices with USB debugging enabled (or on rooted devices), this allows any user with physical access and ADB connectivity to extract the full application data directory — including Realm database files, cached authentication tokens, session state, and any locally stored credentials — without root access, using `adb backup`. For an application managing fleet operations, driver data, and location information, this represents a significant data exfiltration risk.

### A71-12 — MEDIUM: No network security configuration — cleartext traffic controls are absent
**Files:** `app/src/main/AndroidManifest.xml`

The manifest does not declare `android:networkSecurityConfig`. At `targetSdkVersion 26`, Android enforces cleartext traffic restrictions only for API 28+ apps by default; at API 26 the default policy permits cleartext HTTP. The app makes HTTPS calls to known BASE_URLs, but without an explicit `networkSecurityConfig`, there is no enforced certificate pinning, no restriction on cleartext fallback, and no declaration of trusted anchors. A downgrade attack could redirect traffic to a plaintext endpoint. Certificate pinning for the known production endpoint (`pandora.fleetiq360.com`) is not configured.

### A71-16 — LOW: gradle-wrapper.jar committed to version control
**Files:** `gradle/wrapper/gradle-wrapper.jar` (tracked by git)

The binary `gradle-wrapper.jar` is committed to git. This is a widespread Android project practice but is a security concern: the JAR is executed as part of every build without verification of its integrity. The Gradle project recommends verifying the wrapper JAR checksum or using Gradle's `--verify-checksums` flag.

Standard practice is to use separate keystores per environment with distinct credentials, so that a UAT key compromise does not affect the release certificate.

### A71-21 — INFO: android:largeHeap="true" — indicates memory pressure, no profiling test coverage
**Files:** `app/src/main/AndroidManifest.xml`

`android:largeHeap="true"` requests an expanded heap from the Android runtime. This attribute is typically used as a workaround for OutOfMemoryError conditions rather than as a primary solution. Its presence indicates the application has experienced or anticipates significant memory pressure. With no tests in place, no automated memory regression testing can detect if a future change increases heap usage to the point where even the large heap is insufficient. There is no baseline to verify against.
