### Findings

#### A37-4: All public constructors in `NonViewAware` lack Javadoc [INFO]
- File: `NonViewAware.java`, Lines: 40, 44
- Description: Neither constructor has a Javadoc comment. The two-argument constructor `NonViewAware(ImageSize, ViewScaleType)` delegates to the three-argument form with a `null` imageUri, which is a non-obvious design choice worth documenting. The three-argument constructor throws `IllegalArgumentException` if `imageSize` or `scaleType` is null (lines 45-46); these preconditions are not documented anywhere.

#### A37-5: `getWidth()` and `getHeight()` have no Javadoc [INFO]
- File: `NonViewAware.java`, Lines: 54, 59
- Description: `getWidth()` and `getHeight()` are both undocumented. Unlike the `ViewAware` implementations which perform a multi-step size resolution algorithm, these implementations directly delegate to `imageSize.getWidth()` and `imageSize.getHeight()` respectively. The simple, fixed-size behavior (no fallback, always returns the constructed `ImageSize`) is not noted.

#### A37-6: `getScaleType()`, `getWrappedView()`, `isCollected()`, and `getId()` have no Javadoc [INFO]
- File: `NonViewAware.java`, Lines: 64, 69, 74, 79
- Description: Four overriding public methods have no Javadoc:
  - `getScaleType()` simply returns the `scaleType` field passed at construction.
  - `getWrappedView()` always returns `null` (there is no wrapped view). This is a notable behavioral contract — callers must not assume a non-null result — but it is undocumented at the method level (the class-level Javadoc only loosely implies it).
  - `isCollected()` always returns `false`. Since `NonViewAware` holds no weak reference, it is never "collected". This fixed behavior is important for callers to understand and is undocumented.
  - `getId()` uses `imageUri.hashCode()` when a URI is provided, or falls back to `super.hashCode()` (the `Object` identity hash). This ID uniqueness strategy differs fundamentally from the view-based implementations and has no documentation.

#### A37-7: `setImageDrawable()` and `setImageBitmap()` are no-ops that return `true` — behavior conflicts with interface-level contract implication [LOW]
- File: `NonViewAware.java`, Lines: 84, 89
- Description: Both methods do nothing (inline comments confirm: `// Do nothing`) yet unconditionally return `true`. The `ImageAware` interface Javadoc for both methods states the return value is `true` if the image "was set successfully." Returning `true` from a no-op misleads callers into believing the drawable or bitmap was actually displayed. The intent appears to be that `NonViewAware` is a display-free sink, but neither the method-level Javadoc nor the return value communicates this clearly. A caller checking the return value for success/failure cannot distinguish between a real success and a silent discard. At minimum, a Javadoc comment explaining the intentional no-op and the rationale for returning `true` is needed.
