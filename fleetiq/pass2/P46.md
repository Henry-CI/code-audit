### P46-F01: No test coverage for email functionality across all 10 JSP files
**Severity:** CRITICAL
**Description:** All 10 JSP files (xlsx_driver_license_report_mail.jsp, xlsx_dyn_driver_report_mail.jsp, xlsx_dyn_unit_report_mail.jsp, xlsx_impact_meter_report_mail.jsp, xlsx_impact_report_mail.jsp, xlsx_key_hour_util_report_mail.jsp, xlsx_preop_chk_fail_report_mail.jsp, xlsx_preop_chk_report_mail.jsp, xlsx_restricted_access_usage_report_mail.jsp, xlsx_seat_hour_util_report_mail.jsp) invoke MailExcelReports.sendExcelReport() without any automated tests. This creates a critical security and reliability gap as email injection vulnerabilities, attachment path traversal attacks, and mail sending failures cannot be detected before production deployment.
**Fix:** Create comprehensive integration tests for each JSP file that verify: (1) email is sent with correct subject and recipient, (2) attachment path validation prevents directory traversal, (3) email address validation prevents header injection, (4) error responses are returned when mail sending fails, (5) JSON response format is correct for both success and failure cases.
**File:** All 10 JSP files in pages/admin/reports/email/
**Line(s):** xlsx_driver_license_report_mail.jsp:70, xlsx_dyn_driver_report_mail.jsp:117, xlsx_dyn_unit_report_mail.jsp:115, xlsx_impact_meter_report_mail.jsp:105, xlsx_impact_report_mail.jsp:102, xlsx_key_hour_util_report_mail.jsp:119, xlsx_preop_chk_fail_report_mail.jsp:100, xlsx_preop_chk_report_mail.jsp:104, xlsx_restricted_access_usage_report_mail.jsp:81, xlsx_seat_hour_util_report_mail.jsp:117

### P46-F02: SQL injection vulnerability in CustomerDAO.getCustomerName() with no test coverage
**Severity:** CRITICAL
**Description:** CustomerDAO.getCustomerName() concatenates user input (cust_cd parameter) directly into SQL query: `"select \"USER_NAME\" from \"FMS_CUST_MST\" where \"USER_CD\" = " + cust_cd`. An attacker can inject malicious SQL (e.g., cust_cd="1 OR 1=1--") to bypass authentication or extract sensitive data. This method is invoked by filter beans in multiple JSP files but has zero test coverage, meaning SQL injection attacks cannot be detected.
**Fix:** (1) Create unit tests for CustomerDAO.getCustomerName() that verify SQL injection attempts are blocked or safely handled, (2) Refactor code to use PreparedStatement with parameterized queries instead of string concatenation, (3) Add input validation tests for various injection payloads including numeric, string-based, and union-based SQL injection.
**File:** WEB-INF/src/com/torrent/surat/fms6/excel/reports/dao/CustomerDAO.java
**Line(s):** 32
**Untested function:** getCustomerName(String cust_cd)
