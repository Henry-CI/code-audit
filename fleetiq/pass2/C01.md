### C01-F05: Dual Log4j Versions Without Migration Tests
**Severity:** MEDIUM

Log4j 1.x reached end-of-life in 2015 and has known security vulnerabilities (including CVE-2021-4104). Having both configurations suggests either an incomplete migration from Log4j 1 to Log4j 2, or uncertainty about which version is deployed. Without tests, it's impossible to verify which logging framework is active in production.
**Fix:** Determine which Log4j version is actually used by examining the application's dependencies (check WEB-INF/lib for log4j-1.x.jar vs log4j-core-2.x.jar). Remove the unused configuration file to eliminate confusion. If migrating from Log4j 1 to Log4j 2, implement integration tests that: (1) Verify Log4j 2 classes are loaded, (2) Confirm log output is generated correctly, (3) Validate that Log4j 1 libraries are not present in the deployment. If Log4j 1.x is still in use, prioritize upgrading to Log4j 2.x due to security concerns and add tests to validate the migration.
**File:** WEB-INF/src/log4j.properties, WEB-INF/src/log4j2.properties
**Line(s):** All
**Untested function:** Log4j version detection, configuration file precedence, logging framework initialization

### C01-F07: Log Rotation Settings Untested for Disk Space Exhaustion
**Severity:** MEDIUM
**Description:** The log4j.properties file configures log rotation with MaxFileSize=10MB and MaxBackupIndex=100 (lines 26-27). This allows up to 1.01GB of log files (10MB × 101 files) to accumulate, but there are no tests to verify this rotation works correctly or that disk space limits are enforced. Without testing, log rotation failures could lead to disk space exhaustion, which can cause application crashes, database corruption, or system-wide outages. Additionally, 100 backup files seems excessive for most applications and may indicate inadequate log retention policies.
**Fix:** Implement automated tests to verify log rotation behavior: (1) Simulate high-volume logging to trigger rotation, (2) Verify that old log files are deleted when MaxBackupIndex is exceeded, (3) Confirm total disk space used by logs does not exceed expected limits, (4) Test rotation behavior under concurrent logging from multiple threads, (5) Validate that rotation doesn't cause log message loss. Review and justify the MaxBackupIndex=100 setting—consider reducing to a more reasonable number (e.g., 10-20) unless long-term retention is required. Implement monitoring alerts for disk space usage in the log directory.
**File:** WEB-INF/src/log4j.properties
**Line(s):** 26-27
**Untested function:** Log file rotation, backup index enforcement, disk space limit validation
