### P41-F01: No Test Coverage for Impact Meter Calculation Logic
**Severity:** CRITICAL
**Description:** The impact_meter.jsp file contains critical calculation logic (lines 4-42) that determines impact percentage based on site configuration (AU vs UK) with different mathematical formulas. This calculation has NO test coverage. The parameter parsing on line 5 has no null checking, and Double.parseDouble() on line 6 can throw NumberFormatException with no error handling. Invalid input can crash the page or produce incorrect safety-critical impact calculations.
**Fix:** Create unit tests for the calculation logic covering: (1) AU site calculations for impact values: 0, 25, 50, 75, 100, 150; (2) UK site calculations for same values; (3) null/empty imp parameter; (4) non-numeric imp parameter; (5) negative impact values; (6) boundary conditions at 50 and 100. Add try-catch for NumberFormatException with appropriate error response.
**File:** pages/admin/reports/impact_meter.jsp
**Line(s):** 4-42
**Untested function:** Impact percentage calculation, parameter parsing

### P41-F02: Unsafe Substring Operations Without Bounds Checking
**Severity:** CRITICAL
**Description:** Multiple JSP files perform substring operations on access_cust, access_site, and access_dept session variables without checking string length (e.g., impact_meter_report.jsp lines 94-96, impact_report.jsp lines 94-96, impact_unlock_report.jsp lines 44-46, inspection_report.jsp lines 50-52, machine_unlock_report.jsp lines 44-46, message_status_report.jsp lines 62-64, pre-op_check_list_report_original_with_no_filters.jsp lines 40-42, pre-op_check_report.jsp lines 43-45). Code assumes strings are always length >= 2 with `substring(2, length())`. If session is corrupted or manipulated to contain shorter strings, this will throw IndexOutOfBoundsException, causing complete page failure. NO tests verify this critical security control path.
**Fix:** Add bounds checking before substring operations: `if(access_cust != null && access_cust.length() > 2) { acc_cus = access_cust.substring(2); } else { // handle error }`. Create integration tests that: (1) test normal access control strings; (2) test empty strings; (3) test single-character strings; (4) test null session attributes; (5) test maliciously crafted short strings.
**File:** Multiple files (8 affected)
**Line(s):** impact_meter_report.jsp:94-96, impact_report.jsp:94-96, impact_unlock_report.jsp:44-46, inspection_report.jsp:50-52, machine_unlock_report.jsp:44-46, message_status_report.jsp:62-64, pre-op_check_list_report_original_with_no_filters.jsp:40-42, pre-op_check_report.jsp:43-45
**Untested function:** Access control string parsing

### P41-F03: No Input Validation Testing for Request Parameters
**Severity:** HIGH
**Description:** All 10 JSP files parse multiple request parameters using `request.getParameter()` with minimal validation. Parameters like user_cd, loc_cd, dept_cd, form_cd, st_dt, to_dt are directly used in database queries and filter operations with no sanitization or validation. SQL injection, XSS, and other injection attacks are possible. Integer.parseInt() is called without try-catch (e.g., impact_meter_report.jsp line 87), which will throw NumberFormatException on invalid input. NO tests validate parameter handling or error conditions.
**Fix:** Implement input validation framework with: (1) whitelist validation for all parameters; (2) type checking before parsing; (3) try-catch blocks around Integer.parseInt() and Double.parseDouble(); (4) SQL injection prevention via parameterized queries; (5) XSS prevention via output encoding. Create comprehensive test suite covering: (1) valid inputs; (2) null/empty parameters; (3) SQL injection attempts; (4) XSS payloads; (5) invalid type conversions; (6) boundary values.
**File:** All 10 JSP files
**Line(s):** Various parameter parsing lines in each file
**Untested function:** Request parameter parsing and validation
