## Findings

### P19-F01: Complete absence of test coverage for AJAX duplicate check endpoints
**Severity:** CRITICAL
**Description:** The AJAX endpoints `chk_dup_acode.jsp` and `chk_dup_card.jsp` perform duplicate checking for authorization codes and card IDs respectively. These endpoints return XML responses used by client-side JavaScript to prevent duplicate entries. No tests exist to verify: (1) correct parameter extraction, (2) proper invocation of database duplicate check logic, (3) correct XML response formatting, (4) handling of null/empty parameters, (5) SQL injection prevention in input parameters that flow to database queries.
**Fix:** Create integration tests that: (1) Verify XML response format matches expected structure `<body><rec><code>X</code></rec></body>`, (2) Test duplicate detection with known duplicate and unique values, (3) Test with null, empty, and malicious input parameters (SQL injection attempts), (4) Verify proper content-type header (text/xml), (5) Mock Databean_getuser to test JSP logic in isolation.
**File:** master/chk_dup_acode.jsp, master/chk_dup_card.jsp
**Line(s):** chk_dup_acode.jsp: 8-18 (parameter handling and duplicate check), chk_dup_card.jsp: 8-31 (parameter handling and duplicate check)
**Untested functions:** getDup_acode(), getDup_card()

### P19-F02: No test coverage for session-based access control logic
**Severity:** CRITICAL
**Description:** Multiple JSP files (copy_site_hour_settings.jsp, existing_alert_lst_admin.jsp, existing_cust_vehicle_lst.jsp, existing_customer_lst.jsp) implement session-based access control by retrieving session attributes (access_level, access_cust, access_site, access_dept) and applying default values if empty. No tests verify: (1) unauthorized users cannot access restricted data, (2) access_level defaults to "5" when empty/null, (3) users with different access levels see appropriate data, (4) session hijacking/manipulation is prevented. This creates a critical security vulnerability where access control bypasses could go undetected.
**Fix:** Create security-focused integration tests that: (1) Verify users without valid session are redirected/denied, (2) Test each access level (1-5) receives appropriate data filtering, (3) Verify default access level "5" is applied correctly, (4) Test session attribute manipulation (setting access_level="1" when user should have "5"), (5) Verify cross-customer data leakage is prevented, (6) Test with null/missing session attributes.
**File:** master/copy_site_hour_settings.jsp, master/existing_alert_lst_admin.jsp, master/existing_cust_vehicle_lst.jsp, master/existing_customer_lst.jsp
**Line(s):** copy_site_hour_settings.jsp: 52-66, existing_alert_lst_admin.jsp: 300-314, existing_cust_vehicle_lst.jsp: 480-494, existing_customer_lst.jsp: 283-297
**Untested functions:** Session attribute retrieval and default value assignment

### P19-F03: No validation testing for user input parameters flowing to database queries
**Severity:** CRITICAL
**Description:** All 10 JSP files retrieve user input from request parameters (using `request.getParameter()`) and pass these values to backend beans without visible input validation or sanitization. Parameters include: acode, usr_cd, bt, cpre, cid, ccd, scd, hid_type, id_type, wiegand, cust_cd, loc_cd, dep_cd, model_cd, user_cd, location_cd, dept_cd, veh_typ_cd, st_dt, end_dt, search_crit, gp_cd, form_cd, message.
