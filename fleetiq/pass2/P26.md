### P26-F01: Complete absence of automated tests for critical AJAX endpoints
**Severity:** CRITICAL
**Description:** All 10 JSP AJAX endpoints that handle data retrieval operations have zero automated test coverage. These endpoints are invoked by client-side JavaScript and handle sensitive operations including customer data, driver information, vehicle data, department/site lookups, and access-controlled queries. The complete absence of tests means there is no automated verification that these endpoints return correct data, handle errors properly, validate inputs, or enforce access controls correctly. This represents a critical gap in quality assurance for production systems.
**Fix:** Implement comprehensive integration tests for all AJAX endpoints using a JSP/Servlet testing framework (e.g., Arquillian, JWebUnit, or Spring MVC Test). Each test suite should verify: (1) successful response with valid inputs, (2) proper XML structure, (3) empty result handling, (4) null/invalid parameter handling, (5) error responses, (6) access control enforcement where applicable. Prioritize testing endpoints with access control first (get_dept.jsp, get_mst_cd_like.jsp, get_site.jsp).
**File:** All reviewed files
**Line(s):** Entire files

For each operation, test: successful data retrieval, empty results, SQL exceptions, null parameters, access control filtering, and connection cleanup.

**Line(s):** 1-10000+ (entire class)

### P26-F03: Null parameter handling not tested in any endpoint
**Severity:** HIGH
**Description:** All 10 JSP files accept request parameters via `request.getParameter()` but only 1 file (get_cust_vehicle.jsp) uses null-coalescing with the ternary operator `?:`. The remaining 9 files directly assign potentially null values to variables without null checks. For example, get_customer.jsp line 8 does `String gp_cd=request.getParameter("q")` without null checking, then passes this to `filter.setSet_div_cd(gp_cd)` at line 11. If `q` parameter is missing from the request, `gp_cd` will be null, which may cause NullPointerException in the bean or database layer. No tests verify behavior when parameters are null, missing, or malformed.
**Fix:** Create negative test cases for each JSP endpoint that verify behavior when: (1) required parameters are null/missing, (2) parameters contain empty strings, (3) parameters contain special characters, (4) parameters contain SQL injection attempts. Tests should verify that the application either handles nulls gracefully or returns appropriate error responses. Consider adding null checks in JSP files or implementing request parameter validation before bean method calls.
**File:** master/get_customer.jsp, master/get_customergp.jsp, master/get_dept.jsp, master/get_driver.jsp, master/get_impact.jsp, master/get_mst_cd_exist.jsp, master/get_mst_cd_like.jsp, master/get_site.jsp, master/get_unassigned_vehicle.jsp
**Line(s):** Lines where request.getParameter() is called: get_customer.jsp:8, get_customergp.jsp:8, get_dept.jsp:8-9, get_driver.jsp:8-9, get_impact.jsp:8, get_mst_cd_exist.jsp:3-5, get_mst_cd_like.jsp:8-11, get_site.jsp:8, get_unassigned_vehicle.jsp:8

### P26-F04: Access control logic not tested in security-sensitive endpoints
**Severity:** CRITICAL
**Description:** Three JSP files (get_dept.jsp, get_mst_cd_like.jsp, get_site.jsp) implement access control by reading session attributes (access_level, access_cust, access_site, access_dept) and passing them to the Databean_getuser filter. These endpoints default access_level to "5" if the session attribute is empty (lines get_dept.jsp:18-21, get_mst_cd_like.jsp:22-25, get_site.jsp:17-20). This access control logic has zero test coverage. No tests verify: (1) that unauthorized users are blocked from accessing data outside their scope, (2) that access_level="5" is the correct default, (3) that the filter correctly applies access restrictions, (4) that session attributes are validated, (5) that session hijacking or manipulation is prevented. The access level is also parsed to int without try-catch (e.g., get_dept.jsp:22), which could throw NumberFormatException if session is corrupted.
**Fix:** Implement security-focused integration tests for access-controlled endpoints. For each of get_dept.jsp, get_mst_cd_like.jsp, and get_site.jsp: (1) Test with valid session attributes at different access levels (1-5), (2) Test with empty/null session attributes to verify default behavior, (3) Test with invalid access_level values (non-numeric, negative, > 5), (4) Test that users with access_level=5 cannot access data from access_level=1-4 scopes, (5) Test that access_cust/access_site/access_dept filters are correctly applied. Add try-catch for Integer.parseInt() to handle NumberFormatException.
**File:** master/get_dept.jsp, master/get_mst_cd_like.jsp, master/get_site.jsp
**Line(s):** get_dept.jsp:13-27, get_mst_cd_like.jsp:17-31, get_site.jsp:12-26

### P26-F05: XML output generation and escaping not tested
**Severity:** HIGH
**Description:** All 10 JSP files generate XML responses by manually concatenating strings (e.g., `resp=resp+"<rec>"+"<code>"+tmp+"</code>"`). Only get_cust_vehicle.jsp performs XML escaping by replacing `&` with `&amp;` (line 45). The other 9 files do not escape XML special characters (<, >, &, ', "). Additionally, get_mst_cd_exist.jsp generates a different XML structure with an XML declaration, but no tests verify the XML is well-formed or parseable by client code. No tests verify: (1) XML well-formedness, (2) XML special character escaping, (3) handling of null values in XML output, (4) handling of empty collections, (5) correct XML structure for client parsing.
**Fix:** Create output validation tests for each JSP endpoint that: (1) Parse the XML response using an XML parser (e.g., DocumentBuilder, SAXParser) to verify well-formedness, (2) Test with database values containing XML special characters (<, >, &, ', ") to verify escaping, (3) Test empty result scenarios to verify valid XML is still produced, (4) Test null value handling in XML elements. Implement consistent XML escaping across all files using a utility function (e.g., StringEscapeUtils.escapeXml() from Apache Commons Lang) or consider using a proper XML serialization library instead of string concatenation.
**File:** All reviewed files
**Line(s):** get_cust_vehicle.jsp:28-45, get_customer.jsp:17-30, get_customergp.jsp:17-30, get_dept.jsp:38-53, get_driver.jsp:21-37, get_impact.jsp:22-25, get_mst_cd_exist.jsp:18-40, get_mst_cd_like.jsp:43-61, get_site.jsp:36-52, get_unassigned_vehicle.jsp:19-34

### P26-F06: Database connection and exception handling not tested
