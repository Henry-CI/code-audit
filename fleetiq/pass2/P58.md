## Findings

### P58-F01: No test coverage for vehicle allocation functionality
**Severity:** CRITICAL
**Description:** The add-general.jsp file handles vehicle allocation to customers, locations, and departments. This is a critical business function that modifies vehicle ownership and access relationships in the database. There are no automated tests to verify that vehicle allocation works correctly, properly validates input, handles edge cases (null/empty parameters, invalid customer/location combinations), or prevents unauthorized allocations. The lack of tests means regressions can be introduced without detection.
**Fix:** Create integration tests for the Frm_customer servlet's allocate_vehicle method. Test cases should include: (1) valid allocation with all parameters, (2) allocation with missing required parameters, (3) allocation with invalid customer/location/department combinations, (4) allocation of already-allocated vehicle, (5) allocation with invalid GMTP ID, (6) authorization checks for different access levels, (7) SQL injection attempts in parameters.
**File:** pages/admin/vehicles/add-general.jsp
**Line(s):** 64-183
**Untested function:** allocate_vehicle (Frm_customer servlet)

### P58-F02: No input validation testing for vehicle model creation
**Severity:** HIGH
**Description:** The add-model.jsp and add-model_multi.jsp files allow creation and editing of vehicle models (types). These operations insert/update records in the vehicle type master table. There are no tests to verify input validation for model name (vtype), description (vdesc), picture filenames, or vehicle category types. The picture upload functionality (AU version) accepts user-provided filenames without any visible validation testing. Missing validation could allow XSS attacks via stored model names/descriptions, path traversal via picture filenames, or SQL injection via unvalidated inputs.
**Fix:** Create unit and integration tests for Frm_saveuser servlet methods: veh_typ_add_pop, veh_typ_add_pop_au, and veh_typ_add_multi. Test cases should include: (1) valid model creation with all fields, (2) model name with special characters and XSS payloads, (3) description with SQL injection attempts, (4) picture filename with path traversal characters (../, ..\), (5) duplicate model names, (6) empty/null required fields, (7) maximum length validation for text fields, (8) invalid vehicle category type IDs.
**File:** pages/admin/vehicles/add-model.jsp, pages/admin/vehicles/add-model_multi.jsp
**Line(s):** add-model.jsp: 53-95, 125-133; add-model_multi.jsp: 57-99, 130-161
**Untested function:** veh_typ_add_pop, veh_typ_add_pop_au, veh_typ_add_multi (Frm_saveuser servlet)

### P58-F03: No authorization testing for broadcast message functionality
**Severity:** CRITICAL
**Description:** The broadcast JSP files (broadcast.jsp, broadcast-veh.jsp, broadcast-multi-veh.jsp, broadcast-multi-driver.jsp) allow sending messages to vehicle units and drivers. These are security-critical functions as they enable real-time communication with fleet vehicles and personnel. There are no tests to verify that: (1) access control is properly enforced based on access_level, access_cust, access_site, and access_dept session attributes, (2) users cannot broadcast to vehicles/drivers outside their authorization scope, (3) broadcast permissions (permission array checks) are correctly validated, (4) message content is sanitized before transmission. The lack of authorization tests could allow privilege escalation or unauthorized message broadcasting.
**Fix:** Create comprehensive authorization tests for all broadcast methods in Frm_vehicle servlet. Test cases should include: (1) authenticated user broadcasts to authorized vehicle - should succeed, (2) authenticated user attempts broadcast to vehicle outside their customer/site/dept scope - should fail, (3) user without broadcast permission attempts broadcast - should fail, (4) session without access_level attribute attempts broadcast - should fail, (5) broadcast with empty access control attributes, (6) broadcast to non-existent vehicle/driver IDs, (7) verify permission array index out of bounds handling.
