## Findings

### P42-F01: Complete absence of test coverage for all JSP report pages
**Severity:** CRITICAL
**Description:** All 10 JSP report pages have zero test coverage. No unit tests, integration tests, or end-to-end tests exist to verify correct functionality, access control enforcement, data rendering, or error handling. The repository contains only one file with "Test" in its name (EncryptTest.java), which is not a test class but an encryption utility. This represents a complete gap in quality assurance for security-critical reporting functionality that handles sensitive business data.
**Fix:** Implement comprehensive test coverage including: (1) Integration tests using frameworks like Selenium or JSP unit testing tools to verify page rendering and data display; (2) Unit tests for all bean classes (Databean_report, UtilBean, ServiceDueFlagBean, etc.); (3) Security tests to verify access control enforcement; (4) Input validation tests for all request parameters; (5) Error handling tests for null values, invalid data, and edge cases.
**File:** All 10 JSP files listed above
**Line(s):** Entire files
**Untested function:** All functionality

### P42-F02: Unchecked string substring operations - potential StringIndexOutOfBoundsException
**Severity:** HIGH
**Description:** Multiple JSP pages perform substring operations on session-based access control variables (access_cust, access_site, access_dept) without validating string length. For example, `access_cust.substring(2, access_cust.length())` will throw StringIndexOutOfBoundsException if the string length is less than 2. These operations occur in security-critical sections that determine user access levels. No tests verify behavior when these session variables contain unexpected values.
**Fix:** Add validation before substring operations: check if string length >= 2 before calling substring(2). Add unit tests to verify behavior with empty strings, single-character strings, and null values. Add integration tests to ensure proper error handling when session variables are corrupted or manipulated.
**File:** pages/admin/reports/pre-op_check_report_-_incorrect_only.jsp, pre-op_check_report_detailed.jsp, question_unlock_report.jsp, report_subscription.jsp, report_subscription_au.jsp, report_subscription_uk.jsp, restricted_access_usage.jsp, seen_iris_pedestrian_report.jsp, service_report.jsp, site_utilisation_report.jsp
**Line(s):** 41-43, 42-44, N/A (same pattern across files)
**Untested function:** Access control substring extraction logic

Similarly, dealer_id parsing occurs without error handling in multiple files. No tests verify behavior with invalid numeric input.
**Fix:** Wrap parseInt calls in try-catch blocks with appropriate error handling and default values. Add unit tests to verify behavior with: null values, empty strings, non-numeric strings, extremely large numbers, negative numbers.
