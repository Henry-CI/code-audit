### J84-F01: No test coverage for SQL injection vulnerability in escapeSingleQuotes
**Severity:** CRITICAL
**Description:** The `replaceSingleQuotes` method is used for SQL escaping but has no tests verifying it works correctly. This is a critical security function that prevents SQL injection attacks. Without tests, there is no assurance that the method correctly escapes single quotes in all scenarios, including edge cases like already-escaped quotes, empty strings, or null inputs.
**Fix:** Create comprehensive unit tests including: (1) null input handling, (2) empty string, (3) single quote escaping, (4) multiple consecutive single quotes, (5) strings with no quotes, (6) strings starting/ending with quotes, (7) already-escaped quotes to prevent double-escaping bugs.

### J84-F04: SQL injection vulnerability in fix_dept not tested
**Severity:** CRITICAL
**Description:** The `fix_dept` method concatenates user-supplied parameters (user_cd, loc_cd, dept_cd) directly into SQL queries without using prepared statements (e.g., line 121: `"select count(\"DEPT_CD\") from \"FMS_CUST_DEPT_REL\" where \"DEPT_CD\"="+dept_cd`). **Fix:** Create security tests attempting SQL injection through user_cd, loc_cd, and dept_cd parameters. Tests should verify that malicious payloads like `"1 OR 1=1"`, `"1; DROP TABLE FMS_DEPT_MST"`, etc. are properly handled. Ideally, refactor code to use PreparedStatement instead of string concatenation.
