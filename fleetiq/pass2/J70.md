### J70-F01: Critical authentication function doPost() in Frm_login has no test coverage
**Severity:** CRITICAL
**Description:** The main authentication function doPost() (line 29-142) in Frm_login.java handles user login, password validation, and session management. This function has NO test coverage. The function contains multiple SQL injection vulnerabilities (lines 58-59, 72-73) using direct string concatenation, performs password comparison using plain equals() instead of secure comparison, stores passwords in plain text variables, and has inadequate error handling. No tests verify authentication success, authentication failure, SQL injection attempts, session hijacking protection, or brute force attack mitigation.
**Fix:** Create comprehensive unit and integration tests including: (1) Valid login scenarios with various user types, (2) Invalid login attempts with wrong credentials, (3) SQL injection attack vectors (e.g., username: `' OR '1'='1`, password variations), (4) Session security validation, (5) Account lockout after repeated failures, (6) Password strength validation, (7) Error message consistency to prevent username enumeration, (8) Concurrent login attempt handling.
**File:** WEB-INF/src/com/torrent/surat/fms6/security/Frm_login.java
**Line(s):** 29-142
**Untested function:** doPost(HttpServletRequest request, HttpServletResponse res)

### J70-F02: SQL injection vulnerability in login query has no security test coverage
**Severity:** CRITICAL
**Description:** Lines 58-59 and 72-73 construct SQL queries using direct string concatenation of user-supplied login parameter without any sanitization or prepared statements: `"select count(*) from \"FMS_USR_MST\" where \"USER_NAME\" = '" + login + "'"`. This allows trivial SQL injection attacks. No tests verify that injection attempts are blocked or that prepared statements are used. Attackers can bypass authentication, extract database contents, or modify data.
**Fix:** (1) Implement prepared statements with parameterized queries immediately, (2) Create security test suite that attempts common SQL injection patterns: `admin'--`, `' OR '1'='1'--`, `'; DROP TABLE "FMS_USR_MST";--`, union-based injections, blind SQL injection, time-based attacks, (3) Verify input validation rejects malicious input, (4) Test that error messages don't leak database structure information.
**File:** WEB-INF/src/com/torrent/surat/fms6/security/Frm_login.java
**Line(s):** 58-59, 72-73
**Untested function:** doPost(HttpServletRequest request, HttpServletResponse res)

### J70-F03: Password comparison uses insecure equals() method with no test coverage
**Severity:** HIGH
**Description:** Line 81 performs password validation using `password.equals(pass_word)`. This is vulnerable to timing attacks that can be used to extract password information character-by-character by measuring response times. No tests verify secure password comparison, constant-time comparison, or protection against timing attacks. The code also logs passwords in plain text (line 98).
**Fix:** (1) Replace with constant-time comparison method like `MessageDigest.isEqual()` or use BCrypt password hashing with built-in secure comparison, (2) Implement password hashing (BCrypt, Argon2, or PBKDF2) instead of storing/comparing plain text passwords, (3) Create tests that verify: passwords are hashed before storage, comparison is constant-time, failed comparisons don't leak timing information, password complexity requirements are enforced, (4) Remove password logging from all code paths.
**File:** WEB-INF/src/com/torrent/surat/fms6/security/Frm_login.java
**Line(s):** 81, 98
**Untested function:** doPost(HttpServletRequest request, HttpServletResponse res)
