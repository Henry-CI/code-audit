## Findings

### P54-F01: No test coverage for critical access control logic
**Severity:** CRITICAL
**Description:** The session-based access control logic in cric_issues_lst.jsp, hire_vehicle_lst.jsp, hire_vehicle_lst1.jsp, uncal_vehicle_lst.jsp, and unrep_vehicle_lst.jsp has zero test coverage. This code controls who can view sensitive fleet management data based on access_level, access_cust, access_site, and access_dept session attributes. Without tests, there is no validation that unauthorized users are prevented from accessing restricted data, or that privilege escalation attacks are blocked. The Databean_cdisp.init() method processes these access controls in SQL queries, but has no tests verifying correct implementation.
**Fix:** Create comprehensive integration tests that verify: (1) Users with access_level=1 can only see their own data, (2) Users cannot access other customers' data by manipulating session attributes, (3) Site and department restrictions are properly enforced, (4) Invalid/null session attributes are handled securely, (5) Edge cases like empty access_cust strings are handled, (6) SQL injection through session attributes is prevented.
**File:** pages/admin/summary/cric_issues_lst.jsp, pages/admin/summary/hire_vehicle_lst.jsp, pages/admin/summary/hire_vehicle_lst1.jsp, pages/admin/summary/uncal_vehicle_lst.jsp, pages/admin/summary/unrep_vehicle_lst.jsp
**Line(s):** 11-21 (access control setup pattern in all files)
**Untested function:** Session-based access control and Databean_cdisp.init()

### P54-F02: Missing access control in export pages has no tests
**Severity:** CRITICAL
**Description:** The files dislockout_vehicle_lst.jsp and nothire_vehicle_lst.jsp perform Excel data exports but completely omit access control setup. Lines 15-17 in dislockout_vehicle_lst.jsp call filter.setSet_opcode() and filter.init() without first calling setAccess_user(), setAccess_level(), setAccess_cust(), setAccess_site(), or setAccess_dept(). This means the filter has no access restrictions and may return ALL vehicles in the system regardless of the user's permissions. There are no tests to verify this vulnerability or confirm that access control is required for exports.
**Fix:** Create security tests that verify: (1) Attempting to access these export pages without proper access control returns an error or enforces default restrictions, (2) Users can only export data they have permission to view, (3) Add access control setup in lines 14-15 of both files matching the pattern in other JSPs, (4) Add integration tests verifying exports respect user access levels, (5) Test that unauthorized export attempts are logged and blocked.
**File:** pages/admin/summary/dislockout_vehicle_lst.jsp, pages/admin/summary/nothire_vehicle_lst.jsp
**Line(s):** dislockout_vehicle_lst.jsp:15-17, nothire_vehicle_lst.jsp:15-16
**Untested function:** Data export without access control validation

### P54-F03: SQL injection via request parameter has no tests
**Severity:** CRITICAL
**Description:** In hire_vehicle_lst1.jsp line 8, the cust_cd parameter is extracted from the HTTP request with only null-coalescing (null?"":"value). This unsanitized value is then passed directly to filter.setSet_cust_cd(cust_cd) on line 25, which is used in SQL queries within Databean_cdisp.init(). Reading Databean_cdisp.java lines 183-199 confirms the cust_cd value is concatenated into SQL: 'query = "select \"VEHICLE_CD\" from \"FMS_USR_VEHICLE_REL\" where \"USER_CD\" ='" + set_cust_cd + "'"'. An attacker can inject SQL by passing cust_cd=' OR '1'='1 to bypass restrictions or extract unauthorized data. There are zero tests validating input sanitization or SQL injection prevention.
**Fix:** Create security tests that: (1) Pass SQL injection payloads like ' OR '1'='1, '; DROP TABLE--, etc. as cust_cd parameter and verify they are rejected or safely escaped, (2) Verify Databean_cdisp uses parameterized queries instead of string concatenation, (3) Add input validation in hire_vehicle_lst1.jsp to ensure cust_cd matches expected format (e.g., numeric ID), (4) Test that invalid cust_cd values return proper error responses without leaking data, (5) Add unit tests for Databean_cdisp.hire_veh_data1() method verifying SQL injection resistance.
**File:** pages/admin/summary/hire_vehicle_lst1.jsp
**Line(s):** 8, 25
**Untested function:** request.getParameter("cust_cd") and filter.setSet_cust_cd()

### P54-F04: XSS vulnerabilities in output rendering have no tests
**Severity:** CRITICAL
**Description:** Multiple JSP files output user-controlled data without HTML encoding, creating XSS vulnerabilities. Examples: (1) hire_vehicle_lst.jsp line 55 outputs customer name directly in link text: '<%=Vcust_nm.get(i) %>', (2) All files output customer names, site names, vehicle names without encoding (e.g., cric_issues_lst.jsp line 70: '<%=Vcnm.get(i) %>'), (3) hire_vehicle_lst.jsp line 55 includes unencoded cust_cd in URL parameters. If an attacker creates a customer named '<script>alert(document.cookie)</script>', this script will execute when the page renders. There are no tests verifying XSS prevention or output encoding.

**File:** pages/admin/summary/hire_vehicle_lst.jsp, pages/admin/summary/cric_issues_lst.jsp, pages/admin/summary/dislockout_vehicle_lst.jsp, pages/admin/summary/hire_vehicle_lst1.jsp, pages/admin/summary/nothire_vehicle_lst.jsp, pages/admin/summary/uncal_vehicle_lst.jsp, pages/admin/summary/unrep_vehicle_lst.jsp

### P54-F08: Excel export with duplicate headers has no tests
**Severity:** MEDIUM
**Description:** Files dislockout_vehicle_lst.jsp (lines 10-11) and nothire_vehicle_lst.jsp (lines 9-10) set the Content-disposition header twice: once with 'inline' and once with 'attachment'.
