## Findings

### P82-F01: Complete lack of test coverage for all report JSP files
**Severity:** CRITICAL
**Description:** All 10 JSP files in this audit have zero test coverage. There are no unit tests, integration tests, or automated tests of any kind for these report generation endpoints. The repository lacks any test infrastructure (no test directories, no JUnit/TestNG dependencies, no test files). This creates significant risk as code changes, refactoring, or security fixes cannot be validated automatically. Regression bugs and security vulnerabilities may be introduced without detection.
**Fix:** Establish comprehensive test coverage:
1. Set up test infrastructure (JUnit 5, Mockito, JSP testing framework like HtmlUnit or Selenium)
2. Create unit tests for all backend beans (Databean_report, UtilBean, FleetCheckBean, UnitutilBean)
3. Create integration tests for JSP rendering and parameter handling
4. Create security tests for XSS, SQL injection, access control, and session validation
5. Aim for minimum 80% code coverage with focus on security-critical paths
**File:** All 10 JSP files
**Line(s):** All lines
**Untested function:** All functions

### P82-F02: Unsanitized request parameters create XSS vulnerabilities
**Severity:** CRITICAL
**Description:** Multiple JSP files directly use request parameters without validation or sanitization, then render them in HTML output or use them to construct URLs. In rpt_monthly_charts.jsp (line 20) and rpt_national_monthly_charts.jsp (line 10), the chartURL obtained from the bean is rendered directly in an img src attribute without HTML encoding. In rpt_monthly_online.jsp, parameters are used to construct URLs (lines 143, 175, 200, 225, 250) and rendered in form values without escaping. These create reflected XSS attack vectors where malicious scripts can be injected via URL parameters.
**Fix:**
1. Add input validation for all request parameters (format, length, character whitelist)
2. HTML-encode all output using JSTL <c:out> or escapeXml functions
3. Use parameterized URL construction instead of string concatenation
4. Implement Content Security Policy (CSP) headers
5. Add automated XSS scanning tests
**File:** reports/rpt_monthly_charts.jsp, reports/rpt_national_monthly_charts.jsp, reports/rpt_monthly_online.jsp
**Line(s):** rpt_monthly_charts.jsp:20, rpt_national_monthly_charts.jsp:10, rpt_monthly_online.jsp:143,175,200,225,250,461,467
**Untested function:** chartURL rendering, URL construction, parameter output

### P82-F03: Missing input validation on date parameters
**Severity:** HIGH
**Description:** Date parameters (st_dt, to_dt, month) are accepted from requests without format validation or range validation. The only validation is client-side JavaScript (rpt_monthly_online.jsp line 137-140) which checks for empty values but can be bypassed. Malformed dates could cause backend errors, SQL injection if concatenated into queries, or logic errors. The month parameter accepts arbitrary strings when it should be restricted to 1-12. Default date values from UtilBean are used without validation.
**Fix:**
1. Add server-side date format validation (regex or DateFormat parsing with try-catch)
2. Validate month parameter is numeric and in range 1-12
3. Validate date ranges are logical (start_date <= end_date)
4. Use PreparedStatement parameters for any database queries
5. Add unit tests for all date validation edge cases (null, empty, malformed, out-of-range)
**File:** reports/rpt_monthly.jsp, reports/rpt_monthly_charts.jsp, reports/rpt_monthly_online.jsp, reports/rpt_preop_chk.jsp, reports/rpt_preop_chk_fail.jsp, reports/rpt_preop_chk_inc.jsp, reports/rpt_seat_hour_usage.jsp
**Line(s):** rpt_monthly.jsp:9-12, rpt_monthly_charts.jsp:8-10, rpt_monthly_online.jsp:276-277, rpt_preop_chk.jsp:14-15
**Untested function:** Date parameter parsing and validation

### P82-F04: Insecure session access control with unsafe defaults
**Severity:** HIGH
**Description:** Multiple JSP files implement access control based on session attributes (access_level, access_cust, access_site, access_dept), but if access_level is empty or null, it defaults to "5" (lines like rpt_monthly_online.jsp:290-293). This default may grant unintended access. The security implications of level "5" are unclear without testing. Empty string checks use .equalsIgnoreCase("") instead of proper null/empty validation. No tests verify that unauthorized users cannot access reports. Session attributes could be manipulated if session fixation or hijacking occurs.
**Fix:**

2. Use proper null and empty validation: StringUtils.isBlank() or null check first
3. Add unit tests for all access level boundary conditions (null, empty, invalid, 1-5, out of range)
4. Add integration tests verifying access control enforcement
5. Implement session token validation and CSRF protection
6. Log access denials for security monitoring
**File:** reports/rpt_monthly_online.jsp, reports/rpt_override_code_lst.jsp, reports/rpt_preop_chk.jsp, reports/rpt_preop_chk_fail.jsp, reports/rpt_preop_chk_inc.jsp, reports/rpt_seat_hour_usage.jsp

### P82-F05: Master override codes displayed without encryption or access logging
**Severity:** HIGH
**Description:** The rpt_override_code_lst.jsp displays master override codes (security-critical access codes for vehicles) in plain text (lines 122-135). There is no encryption of displayed codes, no audit logging of who views override codes, and no additional authentication required to view this sensitive data. The only protection is the session-based access control which defaults to level "5" if empty. These override codes could be used for unauthorized vehicle access if intercepted or viewed by unauthorized personnel. No tests verify that only authorized roles can view these codes.
**Fix:**
1. Encrypt or mask override codes in display (show only last 4 digits, or require click-to-reveal with re-authentication)
2. Add audit logging for all override code views (who, when, which codes)
3. Require additional authentication (password re-entry) before displaying override codes
4. Implement role-based access control with explicit "view_override_codes" permission
5. Add automated tests verifying access control and audit logging
6. Consider removing override code display entirely and only allow secure retrieval via separate authenticated API
**File:** reports/rpt_override_code_lst.jsp
**Line(s):** 67-72, 106-138
**Untested function:** Override code retrieval, display, access control, audit logging

### P82-F06: AJAX endpoints called without CSRF protection
**Severity:** HIGH
**Description:** The rpt_monthly_online.jsp makes AJAX calls to get_site.jsp (line 13) and get_dept.jsp (line 68) without CSRF tokens. These endpoints could be exploited via CSRF attacks where a malicious site tricks an authenticated user's browser into making requests. The XMLHttpRequest objects are created without withCredentials or custom headers for CSRF protection. Parameters are passed in URL query strings (GET requests) which is less secure than POST with tokens.
**Fix:**
1. Implement CSRF token validation for all AJAX endpoints
2. Change to POST requests with CSRF token in request body or header
3. Add automated tests verifying CSRF protection (requests without valid tokens should be rejected)
4. Use framework-provided CSRF protection (e.g., Spring Security)
5. Validate referrer header as additional defense
**File:** reports/rpt_monthly_online.jsp
**Line(s):** 5-19, 53-75
**Untested function:** AJAX request security, CSRF protection

### P82-F07: XML response parsing vulnerable to XXE attacks
**Severity:** HIGH
**Description:** The JavaScript functions stateChanged() and stateChanged1() in rpt_monthly_online.jsp parse XML responses from AJAX endpoints (lines 25-26, 81-82) using xmlHttp.responseXML without validating or sanitizing the XML. If the server returns malicious XML with external entity declarations, it could enable XXE (XML External Entity) attacks leading to information disclosure, SSRF, or DoS. The XML parsing uses getElementsByTagName and childNodes navigation without validating structure, which could cause errors if unexpected XML is returned.
**Fix:**
1. Disable external entity processing in XML parser configuration
2. Validate XML structure before parsing
3. Switch to JSON instead of XML for AJAX responses (safer and more standard)
4. Add error handling for malformed XML responses
5. Add automated tests for XML parsing with malicious payloads
**File:** reports/rpt_monthly_online.jsp
**Line(s):** 21-51, 77-102
**Untested function:** XML response parsing, external entity handling

### P82-F08: No error handling for bean initialization failures
**Severity:** MEDIUM
**Description:** All JSP files call filter.init() without try-catch blocks or error handling. If init() throws an exception due to database errors, configuration issues, or invalid parameters, the JSP will fail with a 500 error potentially exposing stack traces to users. No tests verify graceful error handling. Users may see sensitive error information including database connection strings, internal file paths, or SQL queries.
**Fix:**

2. Return user-friendly error messages without sensitive details
3. Log detailed errors server-side for debugging
4. Add custom error pages (error.jsp) instead of default server error pages
5. Add unit tests for all error paths (database unavailable, invalid parameters, null data)
6. Add integration tests verifying error pages don't leak sensitive information
**File:** All 10 JSP files
**Line(s):** rpt_monthly.jsp:18, rpt_monthly_charts.jsp:16, rpt_monthly_online.jsp:312, rpt_national_monthly.jsp:6, rpt_national_monthly_charts.jsp:6, rpt_override_code_lst.jsp:59, rpt_preop_chk.jsp:50, rpt_preop_chk_fail.jsp:50, rpt_preop_chk_inc.jsp:51, rpt_seat_hour_usage.jsp:51

### P82-F09: Null pointer exceptions in data iteration loops
