## Findings

### P67-F01: No Unit Tests for Critical Email Report JSP Files
**Severity:** CRITICAL
**Description:** Eight JSP email report files (email_unit_unlock_question.jsp, email_users_login_report.jsp, email_util_summary.jsp, email_util_wow.jsp, email_util_wow_pivot.jsp, email_utilisation.jsp, email_vor_report.jsp, email_vor_unit_report.jsp) have zero test coverage. These files handle sensitive fleet management data including user login information, vehicle utilization, and vehicle off-road (VOR) reports. All files extract request parameters without validation and use hardcoded access_level = "1" bypassing proper session-based authentication.
**Fix:** Create JUnit integration tests using Mockito or similar framework to test: 1) request parameter extraction and validation, 2) filter bean configuration with various access levels, 3) data retrieval from ArrayList collections, 4) rendering logic with empty/null data, 5) access control enforcement.
**File:** reports/email_unit_unlock_question.jsp, reports/email_users_login_report.jsp, reports/email_util_summary.jsp, reports/email_util_wow.jsp, reports/email_util_wow_pivot.jsp, reports/email_utilisation.jsp, reports/email_vor_report.jsp, reports/email_vor_unit_report.jsp
**Line(s):** 1-113, 1-208, 1-89, 1-249, 1-229, 1-166, 1-292, 1-98
**Untested function:** All scriptlet blocks, all parameter extraction, all filter configuration, all data retrieval

### P67-F02: Hardcoded Access Control Bypasses Session Authentication
**Severity:** CRITICAL
**Description:** Eight email report JSP files use hardcoded `access_level = "1"` (lines 24, 19, 47, 51, 31, 22, 22 respectively) instead of retrieving access level from session attributes. This completely bypasses the session-based authentication system used by the Excel export reports (excel_battery.jsp line 28, excel_battery_change.jsp line 31). The hardcoded value grants maximum privileges regardless of the actual user's access level, creating a severe authorization bypass vulnerability.
**Fix:** Change all instances of `String access_level = "1";` to `String access_level = (String)session.getAttribute("access_level");` and add null checks: `if(access_level == null || access_level.equalsIgnoreCase("")) { response.sendRedirect("/login.jsp"); return; }`. Create tests to verify: 1) null session redirects to login, 2) different access levels filter data appropriately, 3) unauthorized access attempts are logged.
**File:** reports/email_unit_unlock_question.jsp, reports/email_users_login_report.jsp, reports/email_util_wow.jsp, reports/email_util_wow_pivot.jsp, reports/email_utilisation.jsp, reports/email_vor_report.jsp, reports/email_vor_unit_report.jsp
**Line(s):** 24, 19, 47, 51, 31, 22, 22
**Untested function:** Access level validation, session authentication, authorization enforcement

### P67-F03: No Input Validation on Request Parameters
**Severity:** HIGH
**Description:** All 10 JSP files extract request parameters using `request.getParameter()` with null coalescing to empty strings or default values, but perform ZERO validation on the parameter values before using them in database queries, string operations, or integer parsing. Parameters extracted include: gp_cd, user_cd, model_cd, st_dt, to_dt, loc_cd, dept_cd, form_cd, search_crit, do_list, undo_list, sort_by, sort_asc, vor, cust_cd, fld_cd, condf_value, date_format. This creates SQL injection, command injection, and data integrity vulnerabilities.
**Fix:** Implement parameter validation before use: 1) Whitelist validation for code parameters (gp_cd, user_cd, etc.) using regex `^[a-zA-Z0-9_-]{1,50}$`, 2) Date format validation for st_dt/to_dt before substring operations, 3) Integer bounds checking for condf_value before array access, 4) Sanitize search_crit and do_list before use in queries. Create tests for: 1) SQL injection attempts, 2) XSS payloads in parameters, 3) boundary value violations, 4) null/empty parameter handling.
**File:** reports/email_unit_unlock_question.jsp, reports/email_users_login_report.jsp, reports/email_util_summary.jsp, reports/email_util_wow.jsp, reports/email_util_wow_pivot.jsp, reports/email_utilisation.jsp, reports/email_vor_report.jsp, reports/email_vor_unit_report.jsp, reports/excel_battery.jsp, reports/excel_battery_change.jsp
**Line(s):** 11-18, 11-16, 11-18, 27-37, 21-39, 12-18, 8-21, 8-21, 15-19, 16-22
**Untested function:** Parameter extraction, input validation, sanitization
