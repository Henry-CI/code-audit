### P55-F01: No test infrastructure in repository
**Severity:** CRITICAL
**Description:** The entire FleetIQ repository has ZERO automated tests. No testing framework (JUnit, TestNG, etc.) is configured, no test directories exist, and no test classes are present. This means all 10 user management JSP files reviewed have no automated test coverage whatsoever. Critical user management operations including authentication, authorization, password management, and access control are completely untested.
**Fix:** Implement a comprehensive testing strategy: (1) Add JUnit 4 or 5 dependency to the project, (2) Create src/test/java directory structure, (3) Set up integration testing framework for JSP/Servlet testing (e.g., Selenium, Arquillian, or Spring Test), (4) Establish code coverage goals (minimum 70% for security-critical code), (5) Configure CI/CD pipeline to run tests automatically.
**File:** Repository-wide
**Line(s):** N/A

### P55-F02: Untested authentication credential handling in BMS access
**Severity:** CRITICAL
**Description:** edit-bms-access.jsp handles BMS username and password updates (lines 113-125) with zero test coverage. Password fields use autocomplete="off" but there is no server-side validation visible in the JSP. Password confirmation logic (comparing "pass" and "con_pass") has no tests. JavaScript password strength check (line 156) is the only validation, which can be bypassed. No tests verify password complexity requirements, confirmation matching, or secure transmission to Frm_customer servlet.

### P55-F03: Untested access level enforcement logic
**Severity:** CRITICAL
**Description:** All user management JSPs implement complex access level enforcement (e.g., add-alert.jsp lines 20-69) with zero test coverage. Access levels (1-5) control visibility of customer/location/department dropdowns and determine data filtering. String manipulation using substring() without bounds checking (add-alert.jsp lines 31-33) could cause IndexOutOfBoundsException if access_cust/access_site/access_dept are shorter than 2 characters. No tests verify that users cannot bypass access restrictions, modify data outside their scope, or exploit boundary conditions.
**Fix:** Create parameterized tests for all 5 access levels: (1) Level 1 (admin) - verify "all" options available and "9999" admin options present, (2) Level 2 (customer) - verify customer locked, location/department filterable, (3) Level 3 (site) - verify customer and location locked, (4) Level 4 (department) - verify all locked, (5) Level 5 (default) - verify behavior. Test substring() calls with empty strings, single characters, and null values. Verify unauthorized users cannot access higher-level data.
**File:** pages/admin/user/add-alert.jsp, add-alerts.jsp, add-general-b.jsp, add-general.jsp, edit-general.jsp
