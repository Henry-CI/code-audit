## Findings

### P71-F01: Critical Path Traversal Vulnerability in File Download - No Test Coverage
**Severity:** CRITICAL
**Description:** file_dl.jsp constructs file paths using unsanitized user input (fileName, customer, location, department, vehicle_cd parameters). An attacker can use "../" sequences in the fileName parameter to traverse directories and download arbitrary files from the server. There are no tests verifying that path traversal attacks are prevented, and no tests for input validation.
**Fix:** Add comprehensive integration tests covering: (1) path traversal attempts with "../", "..\", and encoded variants, (2) null/empty parameter handling, (3) valid file download scenarios, (4) unauthorized access attempts. Implement input validation to reject path traversal sequences and restrict filenames to alphanumeric characters. Add authentication/authorization checks.
**File:** reports/file_dl.jsp
**Line(s):** 4-16
**Untested function:** Entire file download logic (lines 1-23)

### P71-F02: Critical Path Traversal in License File Download - No Test Coverage
**Severity:** CRITICAL
**Description:** file_dl_url.jsp accepts a filename parameter without any validation and directly uses it to construct a file path. No tests exist to verify path traversal prevention, authentication, or authorization. An attacker can download any file readable by the application user.
**Fix:** Create integration tests for: (1) path traversal attacks ("../../../etc/passwd"), (2) authorization bypass attempts, (3) invalid filename handling, (4) valid license file downloads. Implement whitelist-based filename validation and proper authentication/authorization.
**File:** reports/file_dl_url.jsp
**Line(s):** 5-11
**Untested function:** Entire file download logic (lines 1-24)

### P71-F03: Critical SQL Injection and Unauthorized Database Modification - No Test Coverage
**Severity:** CRITICAL
**Description:** fix_department.jsp calls fix_department.java's fix_dept() method which performs database UPDATE and INSERT operations using unsanitized user_cd, loc_cd, and dept_cd parameters. The Java class (lines 121-262) concatenates these parameters directly into SQL queries without prepared statements, creating SQL injection vulnerabilities. No tests exist for this critical database modification function. No authentication checks are present.
**Fix:** Create comprehensive integration tests covering: (1) SQL injection attempts in all three parameters, (2) unauthorized access without session validation, (3) CSRF attack scenarios, (4) valid department fix operations with proper authorization, (5) rollback scenarios on failures. Refactor fix_department.java to use PreparedStatements, add authentication/authorization checks, implement CSRF protection, change to POST-only requests, and add audit logging.
**File:** reports/fix_department.jsp, WEB-INF/src/com/torrent/surat/fms6/util/fix_department.java
**Line(s):** fix_department.jsp: 30; fix_department.java: 121, 142-262
