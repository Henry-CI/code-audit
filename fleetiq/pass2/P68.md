## Findings

### P68-F01: No test coverage for Excel export access control logic
**Severity:** CRITICAL
**Description:** All 10 JSP files implement session-based access control logic (access_level, access_cust, access_site, access_dept) that determines which data users can export to Excel. This security-critical authorization logic has no automated test coverage. The access_level defaults to "5" when empty (most permissive), but there are no tests verifying that users cannot export data outside their authorized scope or that access restrictions are properly enforced.
**Fix:** Create integration tests for each JSP file that verify: (1) Users with different access levels can only export authorized data, (2) Cross-customer/site/department data leakage is prevented, (3) Empty or null access_level is properly handled, (4) Access control bypass attempts are blocked.
**File:** reports/excel_battery_charge.jsp, reports/excel_blacklist_driv.jsp, reports/excel_broadcastmsg.jsp, reports/excel_curr_driv_report.jsp, reports/excel_curr_unit_report.jsp, reports/excel_daily_driver_summary.jsp, reports/excel_daily_veh_summary.jsp, reports/excel_driver_access_abuse.jsp, reports/excel_driver_league.jsp, reports/excel_driver_licence_expiry_detail.jsp
**Line(s):** Lines 31-46 (typical pattern across all files)
**Untested function:** Session-based access control logic

### P68-F02: No test coverage for null/missing request parameter handling
**Severity:** HIGH
**Description:** All files use request.getParameter() with inline null checks and default assignments, but there is no test coverage verifying correct behavior when parameters are null, empty, or malformed. This could lead to NullPointerExceptions, incorrect default values being used, or SQL injection if parameters are not properly validated before being passed to database queries through the filter beans.
**Fix:** Create unit/integration tests that verify: (1) Each parameter can safely handle null values, (2) Default values are correctly applied, (3) Empty strings are handled appropriately, (4) Malformed input (SQL injection attempts, XSS payloads) is sanitized or rejected.
**File:** All 10 JSP files
**Line(s):** Lines 16-23 (excel_battery_charge.jsp), Lines 22-23 (excel_blacklist_driv.jsp), Lines 15-21 (excel_broadcastmsg.jsp), Lines 28-31 (excel_curr_driv_report.jsp), Lines 28-31 (excel_curr_unit_report.jsp), Lines 12-19 (excel_daily_driver_summary.jsp), Lines 12-19 (excel_daily_veh_summary.jsp), Lines 13-19 (excel_driver_access_abuse.jsp), Lines 21-48 (excel_driver_league.jsp), Lines 20-28 (excel_driver_licence_expiry_detail.jsp)
**Untested function:** request.getParameter() calls and null-coalescing logic

### P68-F03: No test coverage for Integer.parseInt() failure paths
**Severity:** HIGH
**Description:** All files parse access_level using Integer.parseInt() without try-catch blocks. If access_level contains non-numeric data, this will throw NumberFormatException and crash the page. There are no tests verifying this error path or validating that access_level is numeric before parsing.
**Fix:** Implement defensive parsing with try-catch blocks and add tests for: (1) Non-numeric access_level values, (2) Numeric overflow values, (3) Null access_level after session retrieval, (4) Empty string access_level.
**File:** All 10 JSP files except excel_driver_league.jsp
**Line(s):** Line 40 (excel_battery_charge.jsp), Line 41 (excel_blacklist_driv.jsp), Line 43 (excel_broadcastmsg.jsp), Line 49 (excel_curr_driv_report.jsp), Line 49 (excel_curr_unit_report.jsp), Line 35 (excel_daily_driver_summary.jsp), Line 35 (excel_daily_veh_summary.jsp), Line 35 (excel_driver_access_abuse.jsp), Line 35 (excel_driver_licence_expiry_detail.jsp)
**Untested function:** Integer.parseInt(access_level)

### P68-F04: No test coverage for filter.init() execution and error handling

The init() method is the primary security boundary preventing unauthorized data access.
**Fix:** Create integration tests that: (1) Verify init() correctly filters data based on access control settings, (2) Test init() with malicious SQL injection payloads in all parameters, (3) Verify exception handling when database is unavailable, (4) Test performance and result set size limits.
**File:** All 10 JSP files

### P68-F06: No test coverage for XSS vulnerabilities in data output
**Severity:** HIGH
**Description:** All files render bean data directly into HTML/Excel output using <%= %> expressions without HTML encoding. If database records contain malicious JavaScript or HTML, this could lead to stored XSS when users open the Excel file in a browser or when Excel interprets formulas. There are no tests verifying that data is sanitized before output.
**Fix:** Implement output encoding for all dynamic data and create tests that verify: (1) JavaScript payloads in driver names/vehicle names are encoded, (2) Excel formula injection (=cmd) is prevented, (3) HTML entities in text fields are properly escaped, (4) Unicode/UTF-8 handling is correct.
**File:** All 10 JSP files
**Line(s):** All data rendering lines (e.g., lines 89-107 in excel_battery_charge.jsp)
**Untested function:** Direct data output with <%= %>

### P68-F07: No test coverage for Content-Disposition header injection
**Severity:** MEDIUM
