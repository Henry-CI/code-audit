### J42-F01: Formula Injection via Database-Sourced Duration Values in Excel Output

**Description:** Both `EmailPreOpChkFailReport` and `EmailPreOpChkReport` embed the `dur` (duration) value retrieved from the database directly into an Apache POI `setCellFormula()` call without any sanitisation or validation. The pattern is `contentCell.setCellFormula("VALUE(\"" + tmp + "\")")` where `tmp = dur.get(i).toString()` from database query results. If any `dur` value stored in the database contains formula syntax (e.g., a value beginning with `=`, or containing `+`, `-`, or function names), it will be interpreted as a live Excel formula when the spreadsheet is opened. The `setTotalDuration` method in `Frm_excel` uses the same pattern for the total duration footer row. An attacker with write access to the underlying operational data, or who has exploited the SQL injection vulnerabilities in `Databean_report`, could store a malicious value in a duration field and cause arbitrary formula execution in the generated Excel file when opened by a report recipient.

**Severity:** HIGH

**Checklist item:** Section 3 — Injection Vulnerabilities (formula injection / data export injection)

**Evidence:**

### J42-F03: Customer Code ("0") Treated as No Filter — Potential Cross-Tenant Data Access

**Description:** All three report classes convert `cust_cd == "0"` to an empty string before passing it to `Databean_report`. An empty `cust_cd` means no customer filter is applied in `Databean_report.Fetch_opchk_data()` (line 8371: `query = "select \"VEHICLE_CD\" from \"FMS_USR_VEHICLE_REL\""` — no WHERE clause at all). The same pattern is applied for `loc_cd` and `dept_cd` (both converted to `"all"` when `"0"` is received). In the multi-tenant context of this branch (`multi-customer-sync-to-master`), a caller that passes `cust_cd = "0"` will receive vehicle and driver data across all customers. The caller JSPs read `user_cd` directly from the request parameter (not the session), so a user who manipulates the `user_cd` request parameter to `"0"` may be able to retrieve cross-tenant PreOp check data or Restricted Access Usage data. Whether session-based access control prevents this depends on how `Databean_report` handles the `access_level` and `access_cust` session fields internally — this was not fully verifiable from the files in scope, but the pattern is a confirmed risk.

**Severity:** HIGH

**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / IDOR)

### J42-F04: Data Export Without Visible Server-Side Authorisation Check in Report Classes

**Description:** None of the three report classes (`EmailPreOpChkFailReport`, `EmailPreOpChkReport`, `EmailRestictedAccessUsageReport`) perform any authentication or authorisation check. They are pure report-generation classes that accept caller-supplied `cust`, `loc`, `dep` parameters and produce an Excel file. The security boundary is entirely in the caller (JSP or servlet). The JSP callers reviewed (`xlsx_preop_chk_report_mail.jsp`, `xlsx_restricted_access_usage_report_mail.jsp`) do read session attributes (`access_level`, `access_cust`, `access_site`, `access_dept`) and set them on the `Databean_report` filter bean — but these are set on a separate `filter` bean used only for the in-page data display. The actual `ExcelPreOpCheckReport`/`ExcelRestrictedUsageReport` Excel generation calls (`xlsxPreOp.createExcel(...)` at line 100 of the mail JSP) receive the resolved customer name from the session-scoped filter, but the `user_cd` passed to Excel generation at line 100 is derived from session-filtered data (`filter.getCustName()`), not the raw request. However, `EmailRestictedAccessUsageReport` is also instantiatable via its second constructor `(String[] args, String dir)` which calls `setParameters(false)` — this path reads parameters from the `params` array with no session access-control at all, and its security posture is unknown without tracing all callers. The complete absence of authentication enforcement in the class hierarchy means a future caller or refactored usage could bypass controls entirely.

### J42-F05: Driver Names and IDs Included in Emailed Excel Reports Without Visible Minimisation or Redaction Controls

**Description:** Both `EmailPreOpChkFailReport` and `EmailPreOpChkReport` include driver full name (`driv_nm`) and driver ID (`driv_id`) in every row of the emailed Excel report. These values come directly from the database (`Databean_report.getPo_dnm()` and `getPo_did()`) and are written to the spreadsheet cells without any masking or filtering. An authenticated user who can invoke the email endpoint with an arbitrary `mail_id` can redirect fleet and driver PII to an external address.

**Section 4 — Session and CSRF:** Not directly applicable to these server-side Excel generation classes (no form rendering or state-changing endpoints in the reviewed files themselves). CSRF exposure is in the calling JSPs, which are outside the scope of this assignment.

**Section 6 — Transport Security:** Not determinable from these Java source files alone; no connector configuration is present in these files.

No version is determinable from source alone; JAR analysis is required.
