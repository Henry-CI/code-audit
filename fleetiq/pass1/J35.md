### J35-F01: SSRF via unvalidated URL in addImage2() and addImageUtilisationChart()
**Description:** `addImage2()` (line 453) and `addImageUtilisationChart()` (line 860) and `addDashboardChart()` (line 898) in `Frm_excel` each construct a `java.net.URL` object directly from the `image` parameter and immediately call `url.openStream()`. If the `image` value is derived — directly or indirectly — from user-supplied input (e.g. a URL parameter, a database value populated by the user, or a report parameter), an attacker can supply an arbitrary URL. This allows the server to be used to make outbound HTTP requests to internal infrastructure (metadata APIs, other internal services, the loopback address), constituting a Server-Side Request Forgery (SSRF) vulnerability. No allowlist, scheme check, or host validation is performed before the connection is opened.
**Severity:** HIGH
**Checklist item:** Section 3 — SSRF (Server-Side Request Forgery)

### J35-F03: Missing customer_id/tenant scoping on UnitDAO.getUnusedUnit() call

The class provides public setter methods `setCust_cd()` and `setLoc_cd()` (inherited from `Frm_excel`) that allow any caller to replace the customer code after construction. The class itself cannot enforce multi-tenancy isolation because it applies no independent check that the supplied `cust_cd` matches the session-authenticated customer. ## Checklist Items With No Issues

The following checklist items were reviewed against these three files and no issues were found:

1. **Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, passwords, or connection strings are present in any of the three files. No commented-out connection strings observed.

2. **Section 3 — SQL Injection:** None of these files execute SQL directly. SQL execution is delegated entirely to DAO classes (`UnitDAO`, `ImpactDAO`, `CustomerDAO`). No `Statement.executeQuery()`, `executeUpdate()`, or string-concatenated SQL appears in these files. SQL injection risk, if present, is in the DAO layer and is out of scope for this file assignment.

3. **Section 3 — XSS:** These are server-side Excel generation classes with no JSP output, no `<%= %>` expressions, and no HTML rendering. XSS is not applicable to this layer.

4. **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls are present in any of the three files.

5. **Section 4 — Session and CSRF:** These classes generate Excel files and have no interaction with HTTP sessions, cookies, or form submissions. CSRF is not applicable at this layer.

6. **Section 6 — Transport Security:** These files do not configure TLS, HTTP connectors, or cookie flags. Transport security is governed by Tomcat configuration files outside the scope of these files.

7. **Section 7 — Dependency Vulnerabilities:** Apache POI is used for Excel generation. The specific POI version cannot be determined from source code alone — JAR version must be checked separately. No Struts, Log4j, Commons Collections, or Spring framework imports are present in these three files.
