### P67-F01: Hardcoded Access Level Bypasses Authorization
**Description:** Eight of ten JSP report files hardcode access_level to "1" (highest privilege level) instead of retrieving it from the session. This completely bypasses multi-tenant access controls, allowing any authenticated user to view data from all customers regardless of their actual authorization level. The hardcoded values are never retrieved from session authentication context, making the access level checks ineffective.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization - "Authorization and multi-tenancy" (lines 98-103)

**Evidence:**
```jsp
// email_unit_unlock_question.jsp line 24
```

**Description:** All JSP files output data retrieved from database ArrayLists directly using `<%= variable %>` expressions without any HTML escaping. This affects vehicle names, driver names, user names, location names, department names, serial numbers, and other fields that could contain attacker-controlled content. ### P67-F05: HTTP URLs Leak Session Tokens and Data Over Unencrypted Connections
**Description:** Multiple JSP files contain hardcoded HTTP (non-HTTPS) URLs for images and CSS resources hosted at fms.fleetiq360.com. If these reports are served over HTTPS but load resources over HTTP, browsers will send session cookies in cleartext HTTP requests, exposing JSESSIONID tokens to network eavesdroppers. ## Summary

**Critical Findings:** 2 (hardcoded access level, hardcoded empty customer ID)

**High Findings:** 2 (inconsistent access control, widespread XSS)

**Medium Findings:** 1 (HTTP URLs)

**Low Findings:** 3 (parameter validation issues)

**Total Findings:** 8

The most severe issues are the complete bypass of multi-tenant access controls in 8 out of 10 report files. These files hardcode access_level="1" and empty customer/site/department scopes, allowing any authenticated user to view data from all customers. This is a critical multi-tenant data isolation failure. The two Excel export files correctly retrieve access controls from session, indicating the email reports are incomplete/broken implementations.

All files also suffer from widespread XSS vulnerabilities due to lack of output escaping. Vehicle names, driver names, and other database fields are output directly to HTML without using c:out or escapeXml, allowing stored XSS attacks.
