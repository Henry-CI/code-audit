## Findings

### J17-F01: Plain-text password stored and exposed via public getter in VehNetworkSettingsBean
**Description:** `VehNetworkSettingsBean` holds a `password` field (line 31) representing a network/Wi-Fi credential (paired with `ssid` and `country`). The password is stored as a plain `String` and exposed via a public `getPassword()` method with no masking, encryption, or protection. Strings in Java are immutable and interned, meaning once created they remain in the JVM heap until garbage collected and cannot be zeroed out by the application. If this bean is serialised to JSON/XML for an API response, logged, or written to a JSP page without filtering, the credential is exposed in full. The field also has only package-private access control, meaning any class in the same package can read `vehNetworkSettingsBean.password` directly without going through the getter.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials; Section 5 — Data Exposure and Privacy
**Evidence:**
```java

24:        return password;

```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/VehNetworkSettingsBean.java`

### J17-F02: No customer/tenant scoping field in VehDiagnostic bean — IDOR risk
**Description:** `VehDiagnostic` carries hardware diagnostic data for a vehicle (firmware version, SIM CCID, APN, thresholds, sync states) identified only by `vehicleCd` (an integer code, line 5). There is no `customerId`, `organisationId`, or equivalent tenant identifier anywhere in the bean. On the `multi-customer-sync-to-master` branch — which explicitly implements multi-customer functionality — every data-bearing bean that represents vehicle or device data must carry or be paired with a tenant identifier so that the application layer can enforce that the requesting user belongs to the same customer as the vehicle. Without this scoping field, any code that queries and returns a `VehDiagnostic` object by `vehicleCd` alone cannot perform a customer membership check within the bean itself. The risk is an IDOR vulnerability: a user at Customer A supplying a `vehicleCd` belonging to Customer B would receive diagnostic data for that vehicle if the calling code does not independently enforce tenant scoping. This bean provides no structural guardrail to prevent that.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and Multi-tenancy (IDOR); specifically: "The `multi-customer-sync-to-master` branch name indicates multi-customer (multi-tenant) functionality. This is the highest-risk area in the application."
**Evidence:**

### J17-F03: SIM CCID and APN exposed as plain public getters in VehDiagnostic
**Description:** `VehDiagnostic` exposes `getCcid()` (line 45) and `getApn()` (line 69) as public getters. `ccid` is the SIM card ICCID — a globally unique identifier for the SIM module installed in the vehicle telematics unit. CCID disclosure enables enumeration of which SIM provider and SIM card is used per vehicle, which can aid targeted attacks against the cellular connectivity of the fleet (e.g., SIM swap, provider social engineering). `apn` is the Access Point Name for the cellular data connection. These fields are not credentials in the traditional sense, but they are sensitive device-identifying and configuration data. If this bean is rendered into a JSP response or API payload accessible to non-admin users, it constitutes unnecessary exposure of device infrastructure details. The absence of any `@JsonIgnore`, serialisation exclusion, or role-based field filtering in the bean itself means both values will be included in any automatic serialisation.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**
```java

14:    String apn;
...
45:    public String getCcid() {

```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/VehNetworkSettingsBean.java`

### J17-F04: All fields in UserFormBean (except id) have package-private access — missing private encapsulation
**Description:** In `UserFormBean`, only `id` is declared `private` (line 7). All other fields — `userFomrCd`, `userFomrName`, `userFormMenuView`, `userFormMenuEdit`, `userFormDelete`, `userFormMenuPrint` — are package-private (no access modifier, lines 9–14). This means any class in the `com.torrent.surat.fms6.bean` package can access and mutate these fields directly, bypassing the getter/setter methods. For a bean representing user form-level permissions (`menuView`, `menuEdit`, `delete`, `print`), direct field access from within the package circumvents any validation or auditing logic that might be placed in the setter methods. The same pattern applies to `VehDiagnostic` and `VehNetworkSettingsBean` — all fields in those classes are also package-private.
**Severity:** LOW

## Checklist Items With No Issues

The following checklist items were reviewed against the three files in scope. These files are plain JavaBeans (no SQL, no HTTP calls, no JSP output, no authentication logic) so many checklist items are not applicable at the bean layer. Items are listed explicitly as required.

- **Section 1 — Secrets and Credentials (hardcoded credentials):** No hardcoded database credentials, API keys, SMTP passwords, or encryption keys are present in any of the three files. The `password` field in `VehNetworkSettingsBean` is a runtime value holder, not a hardcoded credential. (The storage and exposure pattern is flagged separately as J17-F01.)
- **Section 3 — SQL Injection:** No JDBC calls, no SQL string construction, no `Statement` or `PreparedStatement` usage in any of the three files. Not applicable at the bean layer.
- **Section 3 — XSS:** No JSP expression output, no HTML rendering, no `<%= %>` expressions. Not applicable — these are server-side Java beans with no rendering logic.
- **Section 3 — File Inclusion:** No file inclusion mechanisms present. Not applicable.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()`, no `ProcessBuilder` usage. Not applicable.
- **Section 3 — SSRF:** No HTTP client calls, no URL construction. Not applicable.
- **Section 4 — Session and CSRF:** No session manipulation, no form output. Not applicable at the bean layer.
- **Section 6 — Transport Security:** No connector or TLS configuration present in these files. Not applicable.
- **Section 7 — Dependency Vulnerabilities:** No third-party library usage in any of the three files. `java.util.ArrayList` is imported but unused in `UserFormBean`; it is a JDK standard library with no CVE relevance.
