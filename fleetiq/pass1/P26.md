### P26-F01: Missing Authentication Check — All Ten Endpoints
**Description:** None of the ten JSP files check whether the requesting user has an active, authenticated session before processing the request and returning data. There is no `session.getAttribute("user")` null check, no session validity test, and no redirect to a login page. Any unauthenticated HTTP request to these endpoints will be processed and data returned. This exposes fleet master data (customers, customer groups, sites, departments, drivers, vehicles, impact parameters) to unauthenticated callers.
**Severity:** CRITICAL
**Checklist item:** Section 2 — Authentication and Authorization
**Evidence:**
```
get_customer.jsp lines 6–31: entire scriptlet executes with no session check
get_driver.jsp lines 6–37: entire scriptlet executes with no session check
get_unassigned_vehicle.jsp lines 6–34: entire scriptlet executes with no session check

get_customergp.jsp lines 6–31: entire scriptlet executes with no session check

```

### P26-F08: NullPointerException Risk — session.getAttribute() Without Null Guard in get_dept.jsp, get_mst_cd_like.jsp, get_site.jsp
**Description:** In three files, `session.getAttribute("access_level")` is cast directly to `String` and then `.equalsIgnoreCase("")` is called on it. If the session attribute is not set (e.g., session has expired, or user has no session), the cast returns `null` and `.equalsIgnoreCase()` throws a `NullPointerException`. This NPE will propagate to the error page and may expose internal stack trace information. Additionally, if `access_level` contains a non-integer value, `Integer.parseInt()` will throw a `NumberFormatException`.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (error responses / stack traces); Section 2 — Session management
**Evidence:**
```jsp
// get_dept.jsp lines 13–22:
String access_level = (String)session.getAttribute("access_level");
// ...
if(access_level.equalsIgnoreCase(""))  // NPE if access_level is null
{
    access_level = "5";
}
int al = Integer.parseInt(access_level);  // NumberFormatException if non-integer

// Same pattern in get_mst_cd_like.jsp lines 17–26
// Same pattern in get_site.jsp lines 12–21
```
**Files:** master/get_dept.jsp, master/get_mst_cd_like.jsp, master/get_site.jsp

### P26-F10: Unescaped Database Values Written Into XML Response — Potential XML Injection
**Description:** All ten files build XML responses by directly concatenating values retrieved from the database (vehicle names, customer names, driver names, site names, department names) into the XML string without any XML encoding. If any database value contains XML special characters (`<`, `>`, `"`, `'`, `&`), the response will be malformed XML. The only mitigation present is in `get_cust_vehicle.jsp` which replaces `&` with `&amp;` — but `<` and `>` are not handled even there. An attacker who can influence database values (e.g., via another endpoint) could inject XML content into these responses, potentially manipulating the behaviour of the consuming JavaScript on the client side (stored XSS via XML injection).
**Severity:** MEDIUM
**Checklist item:** Section 3 — XSS; XML injection
**Evidence:**
```jsp
// get_customer.jsp line 25:
resp=resp+"<rec>"+"<code>"+tmp+"</code><name>"+tmp1+"</name></rec>";
// tmp1 is customer name from DB — no XML/HTML encoding

// get_driver.jsp line 29:
resp=resp+"<rec>"+"<code>"+tmp+"</code><name>"+tmp1+"</name></rec>";
// tmp1 is driver name from DB — no XML/HTML encoding

// get_cust_vehicle.jsp line 37 — only & is handled:
resp = resp.replace("&","&amp;");  // but < > " ' not handled
```
**Files:** All ten files
**Line(s):** XML-building loops in all ten files; get_cust_vehicle.jsp:45

### P26-F11: Null Parameter Values Not Guarded in Multiple Files — NullPointerException Risk
**Description:** Several files read request parameters without null checks and immediately use them. In `get_customer.jsp` (line 8), `get_customergp.jsp` (line 8), `get_driver.jsp` (lines 8–9), `get_impact.jsp` (line 8), `get_site.jsp` (line 8), `get_mst_cd_exist.jsp` (lines 3–5), and `get_mst_cd_like.jsp` (lines 8–11), parameters are assigned but no null guard is applied before passing to the bean. If a required parameter is absent, `null` is passed into the bean, which may cause NPEs or unexpected query behaviour (e.g., `WHERE cust_cd = null` matching nothing or causing an error depending on the backend). Compare with `get_cust_vehicle.jsp` which uses the ternary null-coalescing pattern (`== null ? "" : ...`) as the correct approach.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (unexpected null may affect query construction)
**Evidence:**
```jsp
// get_customer.jsp line 8 — no null guard:
String gp_cd=request.getParameter("q") ;

// get_driver.jsp lines 8–9 — no null guard:
String gp_cd=request.getParameter("r") ;
String s_cd=request.getParameter("q") ;

// Contrast with get_cust_vehicle.jsp line 8 — correct pattern:
```
