## Findings

### P28-F01: Unescaped XSS — Database-sourced module names output directly into HTML
**Description:** In both menu files the module name (`mdn` / `mnm`) is retrieved from the database via `Menu_Bean` / `Menu_Bean1` and written into HTML anchor text, CSS class attribute values, and URL query-string parameters using raw `<%= %>` expressions with no HTML encoding. If an administrator or database record contains HTML meta-characters (`<`, `>`, `"`, `&`), or if the underlying data store can be tampered with, this produces a stored XSS vector that executes in every session that renders the menu. This affects every user of the application, not just the attacker's own session.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```
menu-new.jsp line 90:
<a href="<%=path %>"><span class="custom-icon icon-<%=mdn.toLowerCase() %>"></span><span class="menulabel"><%=mdn %></span></a>

menu-new.jsp line 102:
<li><a href="<%=path %>&sub=reports/<%=(String)mform_name1.get(i).toString().toLowerCase().replace(" ", "_") %>"><%=(String)mform_name1.get(i) %></a></li>

menu.jsp lines 89 and 92:
<li><a href="<%=path %>">...<font color="#002037"><%=mdn %></font></a></li>
<li><a href="<%=path %>">...<font color="#135483"><%=mdn %></font></a></li>
```
**File:** menu/menu-new.jsp, menu/menu.jsp
**Line(s):** menu-new.jsp: 90, 102; menu.jsp: 89, 92

### P28-F02: Unescaped XSS — `home_link_menu` URL written into href without encoding
**Description:** In both files `home_link_menu` is constructed by appending the raw session attribute `user_cd` into a URL string and then output directly into an `href` attribute via `<%= %>`. If `user_cd` contains a quote character (`"`) or `javascript:` prefix (unlikely but possible depending on how user codes are created), the href attribute can be broken or hijacked. More importantly, there is no HTML-attribute encoding applied, which is the correct output context here.

### P28-F03: `user_cd` exposed in client-facing URL (IDOR / information exposure)
**Description:** The authenticated user's `user_cd` is appended as a plain-text query parameter in the Home link URL in both menu files (`?user_cd=<value>`). This exposes the internal user identifier in browser history, server-side access logs, and referrer headers. More critically, if the target pages (`index.jsp`, `index4.jsp`) use this query parameter to scope data rather than — or in addition to — the session, this constitutes an IDOR vulnerability: an attacker who knows or guesses another user's `user_cd` could access that user's home page data by manipulating the URL parameter. This is particularly concerning on the `multi-customer-sync-to-master` branch where multi-tenant isolation is the highest-risk area.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and multi-tenancy / IDOR
**Evidence:**
```
menu-new.jsp line 38:
String home_link_menu = "../pages/index.jsp?user_cd="+u_cd_menu+"";

menu.jsp line 56:
String home_link_menu = "../home/index4.jsp?user_cd="+u_cd_menu+"";
```
**File:** menu/menu-new.jsp, menu/menu.jsp
**Line(s):** menu-new.jsp: 36–38, 55; menu.jsp: 54–56, 75, 77

### P28-F04: Flawed authentication check — string comparison with `==` instead of `.equals()`
**Description:** Both files check whether the session `username` attribute is an empty string using the Java `==` operator: `session.getAttribute("username")==""`. In Java, `==` on `String` objects tests reference equality, not value equality. A non-null empty `String` object that is not the interned literal `""` will pass this check as `false` (i.e., the condition will not recognise it as empty), so `sId` will be set to `"1"` (authenticated) even though the username is effectively blank. The correct check is `"".equals(session.getAttribute("username"))`. Although `sId` itself is only used inside the now-commented-out `changePage()` logic in `menu.jsp`, the pattern demonstrates fragile authentication guard logic that could cause incorrect authorisation decisions if the `sId` variable is re-activated or copied elsewhere.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization
**Evidence:**
```
menu-new.jsp lines 7–13:
if(session.getAttribute("username")==null||session.getAttribute("username")=="")
{
    sId="0";
}
else
{
    sId="1";
}

menu.jsp lines 28–35: (identical pattern)
```
**File:** menu/menu-new.jsp, menu/menu.jsp
**Line(s):** menu-new.jsp: 7–13; menu.jsp: 28–35

### P28-F05: Raw `ArrayList` usage without type safety — unchecked cast risk
**Description:** Both files use raw (non-generic) `ArrayList` throughout all bean result retrieval. While this is not a direct security vulnerability, it means that unexpected or maliciously injected data types returned from the bean methods will cause a `ClassCastException` at runtime rather than a compile-time or controlled error. In the context of a multi-tenant application, a data integrity failure in one customer's menu data could cause a runtime exception that disrupts the session state in unpredictable ways.
**Severity:** INFO
**Checklist item:** Section 3 — Injection Vulnerabilities (general robustness)

### P28-F07: No CSRF protection on menu navigation links
**Description:** Both files render menu links that navigate the user to administrative and functional pages. No CSRF tokens are present in these links. While menu links are typically GET requests and therefore less critical than form submissions, the logout link in `menu.jsp` (`../home/index1.jsp?val=logout`) is a state-changing action performed via a plain hyperlink (GET). A Cross-Site Request Forgery attack targeting this endpoint could force a user to log out involuntarily (logout CSRF). If the logout mechanism also clears or modifies server-side session state in a way that could be exploited (e.g., session fixation after logout), the risk is elevated.
**Severity:** LOW
**Checklist item:** Section 4 — Session and CSRF
**Evidence:**
```
menu.jsp line 101:
<li><a href="../home/index1.jsp?val=logout"><img src="../images/linde/menu/menu.png" height="33" border="0"><font color="#135483">Logout</font></a></li>
```
**File:** menu/menu.jsp
**Line(s):** 101

### P28-F08: `menu2` module hardcoded to "Reports" — other modules' submenus not rendered
**Description:** In `menu-new.jsp`, `menu2` is always initialised with `menu2.setModule("Reports")` inside the loop over all modules. This means sub-menu items are only ever fetched for the "Reports" module, regardless of which module is currently being iterated. This appears to be a functional defect rather than a direct security vulnerability, but it is notable from a security perspective: if submenu-level authorization checks exist in the bean's `init()` path and are being bypassed because the wrong module is always passed, there is a risk that report sub-items are rendered or accessible to users who should only see other modules' items, or vice versa.
**Severity:** INFO
**Checklist item:** Section 2 — Authorization and multi-tenancy
**Evidence:**
```
menu-new.jsp lines 73–76:
menu2.setOption("SUBMOD");
menu2.setModule("Reports");
menu2.setSet_user_cd(user_cd);
menu2.init();
```
**File:** menu/menu-new.jsp
**Line(s):** 73–76
