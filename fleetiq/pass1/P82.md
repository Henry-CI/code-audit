### P82-F12: Session Attribute Access Without Null Check Could Cause NullPointerException
**Description:** Multiple JSP files retrieve session attributes (access_level, access_cust, access_site, access_dept) and use them without proper null checking before calling equalsIgnoreCase() or other methods. While a check exists for access_level being an empty string, if the session attribute is null, a NullPointerException will occur. This could be exploited to cause denial of service or reveal information about application state.
**Severity:** LOW
**Checklist item:** 2. Authentication and Authorization - Session management
**Evidence:**
```jsp
rpt_monthly_online.jsp, lines 285-292:
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");

if(access_level.equalsIgnoreCase(""))  // NPE if access_level is null
{
    access_level = "5";
}
```
This pattern repeats in: rpt_override_code_lst.jsp (lines 35-43), rpt_preop_chk.jsp (lines 23-31), rpt_preop_chk_fail.jsp (lines 23-31), rpt_preop_chk_inc.jsp (lines 23-31), rpt_seat_hour_usage.jsp (lines 23-31)
**File:** reports/rpt_monthly_online.jsp, reports/rpt_override_code_lst.jsp, reports/rpt_preop_chk.jsp, reports/rpt_preop_chk_fail.jsp, reports/rpt_preop_chk_inc.jsp, reports/rpt_seat_hour_usage.jsp
**Line(s):** Multiple files (see evidence)

## Summary

This audit reviewed 10 JSP report files and identified **15 security findings**:
- **4 HIGH severity findings:** XSS vulnerabilities, missing authentication checks, potential multi-tenant IDOR issues
- **8 MEDIUM severity findings:** XSS in database-derived outputs, missing CSRF protection
- **2 LOW severity findings:** Session attribute null handling
- **1 INFO severity finding:** Wildcard imports

The most critical issues are:
1. **Widespread XSS vulnerabilities** due to lack of output escaping throughout all files
2. **Missing authentication checks** in 4 report files that could expose sensitive data
3. **Potential multi-tenant IDOR vulnerability** where customer/site/dept parameters are not visibly validated against session access levels
4. **No CSRF protection** on forms

All findings require remediation. The XSS issues should be addressed by implementing consistent output escaping using JSTL `<c:out>` tags or `${fn:escapeXml()}`. Authentication must be enforced on all report pages. Multi-tenant data access requires validation that user-supplied IDs match the authenticated user's access scope.
