### J48-F06: Formula Injection (CSV/Excel Injection) in Excel cell formulas — ExcelDriverAccessAbuseReport
**Description:** In `ExcelDriverAccessAbuseReport.createDriverAccessAbuseReport()`, time values retrieved from `DriverAccessAbuseBean` (`a_st_tm`, `a_end_tm`, `a_ol_start_list`) are concatenated directly into Apache POI `setCellFormula()` calls. If any of these string values contain Excel formula syntax (e.g., a value beginning with `=`, `+`, `-`, or `@`, or containing `"` to escape the surrounding quotes), the workbook will contain an attacker-controlled Excel formula. When the exported `.xlsx` file is opened in Excel or LibreOffice Calc, the formula executes. This is a formula injection (also known as CSV injection or Excel injection) vulnerability. Because these values likely originate from database records that were themselves populated from user-supplied data (driver event records), the attack chain is: user-controlled data stored in DB → rendered into Excel formula → formula executes on recipient's machine.

### J48-F08: Missing customer-ID scoping in report data — no tenant isolation verification in Excel export layer
**Description:** None of the three Excel report classes (`ExcelDriverAccessAbuseReport`, `ExcelDriverImpactReport`, `ExcelDriverLicenceExpiry`) perform any verification that the data in the bean/list being rendered belongs to the authenticated user's customer. The `cust`, `loc`, `dep` parameters passed to `createExcel()` / `createDriverAccessAbuseReport()` / `createDrivImpReport()` / `createLicenceExpiryReport()` are used only for subtitle label lookups (via CustomerDAO) — they are not used to re-filter or re-validate the data rows that are written to the spreadsheet. ### J48-F10: Exception swallowing with stack trace to console — CustomerDAO
**Description:** All five methods in `CustomerDAO` catch the broad `Exception` type and call `e.printStackTrace()` then `e.getMessage()` (the latter discards the return value, so the message is ignored). The stack trace is printed to the server's standard output/log. This means SQL errors — including those triggered by injection attempts — will appear in server logs, potentially leaking table names, column names, and query structure. The exceptions are then silently swallowed and default values returned, which could mask injection attempts or connectivity failures from monitoring.
**Severity:** MEDIUM
**Checklist item:** 5. Data Exposure and Privacy — Error responses

### J48-F11: Raw (untyped) ArrayList fields used throughout — type-safety bypass risk
**Description:** All three report classes and `DriverAccessAbuseBean` declare raw `ArrayList` fields without generic type parameters (e.g., `ArrayList a_driv_nm` rather than `ArrayList<String> a_driv_nm`). This bypasses Java's compile-time type checking. Data is cast to `String` at use sites (e.g., `(String)vdriver_name.get(i)`) with no bounds or null check. If a non-String object is placed in the list by the data-population code, a `ClassCastException` will occur at export time, and the export will silently fail (exception is swallowed or propagates up). More importantly, raw types are a code-quality indicator of age and lack of review; they are consistent with a codebase that may have other unreviewed data-handling paths.
**Severity:** LOW
**Checklist item:** 5. Data Exposure and Privacy (code quality / defence in depth)
**Evidence:**
