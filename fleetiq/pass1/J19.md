## Findings

### J19-F01: SQL Injection via String Concatenation in get_inactunit_rpt_subscription()
**Description:** In `CustomReports.java`, the `get_inactunit_rpt_subscription()` method builds a SQL query using direct string concatenation of the `type` parameter. The `type` value originates from `set_opcode`, which is set externally via the public setter `setSet_opcode()`. If a caller (a JSP or servlet) assigns this value from a request parameter without sanitization, the concatenated SQL is executed without parameterization through a raw `Statement.executeQuery()` call. Even if `type` is currently only set from the two known string literals (`"summary"` and `"summary_apac"`) within this class, the public setter means any caller can supply an arbitrary value, and the SQL path has no defense.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL injection)
**Evidence:**
```java
// CustomReports.java lines 123–132
query = "select \"EMAIL_ADDR\" from inactiveunit_maillst as e"
        + " left outer join \"FMS_USR_MST\" as u on u.\"USER_CD\" = e.user_cd  "
        + " where \"EMAIL_ADDR\" is not null and \"EMAIL_ADDR\" != ''";

if (!type.equalsIgnoreCase(""))
{
    query += " and report_type = '" + type + "'";
}

rset = stmt.executeQuery(query);
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/businessinsight/CustomReports.java`
**Line(s):** 123–132

### J19-F03: Exception Stack Traces and SQL Query Printed to stdout/logs
**Description:** In the `init()` catch block, `exception.printStackTrace()` and a `System.out.println()` that includes the live SQL query string and method name are both called. If stdout is captured in Tomcat logs (the default), a SQL error caused by injection or malformed input will print the full query — including any injected payload — to the application log. This aids an attacker performing blind SQL injection by confirming syntax errors. More broadly, exposing SQL query text in logs is a data exposure finding.

### J19-F05: No Customer/Tenant Scoping on Email Subscription Query
**Description:** The `get_inactunit_rpt_subscription()` query selects email addresses from `inactiveunit_maillst` joined to `FMS_USR_MST` with no customer ID filter. The query returns all email addresses where `EMAIL_ADDR` is not null and matches a `report_type`. There is no `customer_cd`, `cust_cd`, or tenant identifier in the WHERE clause. This means report subscription emails for all customers are returned as a single list — there is no tenant isolation. If distinct customers have subscriptions to the same report type, a cross-tenant email dispatch or data exposure could occur. The branch name `multi-customer-sync-to-master` makes this a critical design concern.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / IDOR); Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// CustomReports.java lines 123–131
query = "select \"EMAIL_ADDR\" from inactiveunit_maillst as e"

if (!type.equalsIgnoreCase(""))

```
No `customer_cd` or tenant scoping condition is present anywhere in this method.

**Description:** `BarChartCategory.java` uses a wildcard import `com.googlecode.charts4j.*`. ### J19-F08: Use of Deprecated Google Image Charts API (SSRF / Data Leakage Risk)
**Description:** `BarChartCategory.createChart()` uses the charts4j library to construct a Google Image Charts URL at line 124 (`String url = chart.toURLString()`). The Google Image Charts API was shut down on March 14, 2019. Any URL generated is sent to a third-party external endpoint (`chart.googleapis.com` or potentially a third-party mirror). Chart data — including axis labels (`yAxisLabel`) derived from fleet/vehicle data — is encoded in the URL query string and transmitted to this external endpoint when the URL is fetched. This is a data leakage risk: internal fleet operational data (usage figures, unit counts) is sent to an external service. If axis labels or data values include customer-identifiable information, this is a PII / business data leakage finding. There is also a latent SSRF risk if the base URL of the charts API is configurable.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities (SSRF); Section 5 — Data Exposure and Privacy

## Checklist Items With No Issues

The following checklist items were reviewed against the three files in scope and no issues were found:

- **Section 1 — Secrets and Credentials:** No hardcoded database passwords, API keys, SMTP passwords, or encryption keys are present in these three files. The hardcoded email address is flagged separately as LOW (J19-F04) but is not a credential.
- **Section 3 — Command Injection:** No use of `Runtime.getRuntime().exec()` or `ProcessBuilder` in any of the three files.
- **Section 3 — File Inclusion:** No dynamic JSP include paths. These are pure Java files with no JSP inclusion logic.
- **Section 3 — XSS:** These are Java utility/bean classes, not JSP output classes. No `<%= %>` expressions or HTML rendering.
- **Section 4 — Session and CSRF:** These files are not servlets and do not handle HTTP request/response directly in a way that would involve session or CSRF token handling. `CustomReports.init()` receives a request but only uses it for `getSession().getServletContext()` to resolve a path — no session manipulation.
- **Section 6 — Transport Security:** No TLS configuration in these files. No HTTP client calls that configure transport in these files.
- **Section 7 — Dependency Vulnerabilities (JAR specifics):** Cannot be confirmed from source alone; the charts4j library concern is flagged as INFO (J19-F07/F08). No Apache Struts, Commons Collections, Log4Shell, or Spring usage visible in these three files. Log4j 2 (`org.apache.logging.log4j`) is imported in `CustomReports.java` — version cannot be determined from source; a separate inventory check of the committed JARs is required to assess Log4Shell risk.
