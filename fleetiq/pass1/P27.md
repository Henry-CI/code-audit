### P27-F14: Client-Side-Only File Type Validation in pic_upload_logo.jsp
**Description:** The file upload form in `pic_upload_logo.jsp` validates the file extension in JavaScript only (lines 27–33). The JavaScript checks that the extension is `jpg`, `JPG`, `gif`, `GIF`, `bmp`, or `BMP` before submitting. This check can be trivially bypassed by disabling JavaScript or submitting the form programmatically. The server-side servlet `Frm_upload` must also validate file type, but no server-side validation is visible in this JSP. If `Frm_upload` does not independently validate content type, an attacker can upload arbitrary files including JSP/HTML files that could be served as web-accessible content.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (File Inclusion); Section 5 — Data Exposure
**Evidence:**
```javascript

var ext = pnm.substr(pnm.lastIndexOf('.')+1);

// No server-side validation present in this file
```

### P27-F16: Potential Tenant Bypass via User-Controlled cust_cd in rpt_preop_unsync_qustion.jsp
**Description:** While `rpt_preop_unsync_qustion.jsp` does read `access_level`, `access_cust`, `access_site`, and `access_dept` from the session (lines 244–247) and passes them to the bean, the `cust_cd` filter used to display the report also comes from the request parameter (line 235). The session-based access controls at lines 244–258 are necessary but not sufficient without confirming the bean enforces that the requested `cust_cd` is within the session user's `access_cust` scope.

filter.setSet_cust_cd(cust_cd);   // request value passed to bean — may override session-based access_cust
