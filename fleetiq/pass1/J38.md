### J38-F01: SQL Injection via String Concatenation of cust_cd, loc_cd, and dept_cd
**Description:** All three report classes set `cust_cd`, `loc_cd`, and `dept_cd` directly from their constructor parameters (which ultimately originate from caller-supplied report parameters such as scheduler or email-report request values). These values are passed unmodified into `Databean_report` and `Databean_dyn_reports`, where they are concatenated directly into SQL query strings executed via `Statement.executeQuery()`. No `PreparedStatement` with `?` placeholders is used for any of these values. A caller controlling any of `cust`, `loc`, or `dep` (the constructor arguments) can inject arbitrary SQL.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — SQL injection

### J38-F03: Excel Formula Injection via Unvalidated Database Value in setCellFormula
**Description:** In `EmailCimplicityUtilReport.createCimplicityUtilisationReport()`, the variable `temp` is built from raw data returned by the DAO (`temp_usage.get(l)`, which comes from database query results). This value is then embedded directly into an Excel formula string: `contentCell.setCellFormula("VALUE(\""+temp+"\")")`. Additionally, the parent class `Frm_excel.setTotalDuration()` uses the same pattern. The risk is realised when a recipient opens the emailed Excel report.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — XSS (analogous output-encoding failure in a non-HTML context; formula injection)
