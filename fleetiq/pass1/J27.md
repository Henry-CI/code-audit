## Findings

### J27-F01: Driver Licence Expiry Data Transmitted to Third-Party Google Charts API
**Description:** Both `DriverActivityChart.createChart()` and `ExpiryChart.createChart()` use the `charts4j` library (version 1.3, JAR at `WEB-INF/lib/charts4j-1.3.jar`) which wraps the Google Image Charts API. The method `chart.toURLString()` (lines 112 in both files) encodes the chart data — including numeric counts of driver licence expiries per department and driver activity metrics (duration, hydraulics, seat, traction) — into a URL that must be fetched from an external Google server to render the chart image. This means fleet operational data (driver licence expiry counts by department, driver activity measurements) is sent to Google's servers every time a chart is rendered. The Google Image Charts API was officially deprecated by Google in 2012 and shut down in March 2019. The library and the service it calls are both obsolete. Any caller that attempts to fetch the generated URL will receive no image. Beyond the data exposure, the feature is non-functional.

**Severity:** HIGH

**Checklist item:** Section 5 — Data Exposure and Privacy; Section 6 — Transport Security (data sent to an external party over a URL that the application constructs)

**Evidence:**
```java
// DriverActivityChart.java, line 112
String url = chart.toURLString();
// EXAMPLE CODE END. Use this url string in your web or Internet application.
return url;

// ExpiryChart.java, line 112
String url = chart.toURLString();
// EXAMPLE CODE END. Use this url string in your web or Internet application.
return url;
```
The comment `// EXAMPLE CODE START` / `// EXAMPLE CODE END` (lines 69 and 113 in both files) further suggests this code was copied from library documentation examples and never hardened for production use.

**File:** `WEB-INF/src/com/torrent/surat/fms6/chart/excel/DriverActivityChart.java` and `WEB-INF/src/com/torrent/surat/fms6/chart/excel/ExpiryChart.java`

**Line(s):** DriverActivityChart.java: 69, 112–113; ExpiryChart.java: 69, 112–113

### J27-F02: Use of Deprecated and Non-Functional Third-Party Chart Library (charts4j 1.3)
**Description:** The library `charts4j-1.3.jar` (committed at `WEB-INF/lib/charts4j-1.3.jar`) is a Java wrapper for the Google Image Charts API, which Google deprecated in 2012 and shut down entirely in March 2019. Version 1.3 is the final release of the library; it has received no updates since approximately 2011. Any chart rendered via `createChart()` in either class will return a URL that resolves to a dead endpoint — the charts will not display. Beyond functional breakage, committing a binary JAR with no `pom.xml` reference means the dependency cannot be audited for known CVEs through standard tooling. The library version and its transitive dependencies are opaque.

**Severity:** MEDIUM

**Checklist item:** Section 7 — Dependency Vulnerabilities (JAR committed directly with no build system reference, obsolete with unknown CVE status)

### J27-F03: ExpiryBean Exposes Driver Licence Expiry Counts Without Tenant Scoping at the DTO Level
**Description:** `ExpiryBean` carries `deptName` (a string) along with four integer counts of driver licence expiry states. There is no `customerId`, `tenantId`, or equivalent field in the bean. If the objects populating this bean do not rigorously scope the underlying SQL query to the authenticated customer, a cross-tenant data leak is structurally invisible at the DTO level: the bean itself provides no mechanism to verify which customer's data it holds. This is not a defect in `ExpiryBean` itself but is a structural gap — the DTO provides no audit trail of ownership. Callers of `ExpiryChart` that pass in data derived from cross-customer queries would produce a chart with no indication of the data's scope. The finding is raised as an informational architectural concern that must be verified in the data access layer that populates `ExpiryBean` instances.

**Severity:** INFO

**Checklist item:** Section 2 — Authorization and Multi-Tenancy (IDOR risk, multi-tenant data isolation)

**Evidence:**

### J27-F04: Production Infrastructure Path Hardcoded in Utility Class
**Description:** While reviewing `DataUtil.java` (read to resolve the `DataUtil.maxArrayValue` reference in both chart classes), two hardcoded absolute server filesystem paths were observed:
- `uploadLicenceFile` (line 827): `String base="/home/gmtp/fms_files/licence/";`
- `uploadDocumentFile` (line 845): `String base="/home/gmtp/fms_files/CFTS/" + cust_loc + "/";`

These paths disclose the Linux username (`gmtp`) and the production server directory structure. The username `gmtp` is likely a service account. This information assists an attacker in targeting the server if any other vulnerability allows command or path traversal. Additionally, `uploadDocumentFile` constructs a filesystem path by appending the caller-supplied `cust_loc` string without sanitisation, which could allow path traversal if `cust_loc` contains `../` sequences. This finding is in `DataUtil.java` (out of scope for this agent's assigned files) but is recorded because it was read to resolve a dependency; the path traversal concern in `uploadDocumentFile` warrants a dedicated finding in whichever agent reviews `DataUtil.java`.

**Severity:** MEDIUM (hardcoded path / infrastructure disclosure); note that path traversal in `uploadDocumentFile` may be HIGH — flagged for the agent assigned to `DataUtil.java`.
