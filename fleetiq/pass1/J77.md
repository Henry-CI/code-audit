## Findings

### J77-F01: Path Traversal ‚Äî Arbitrary File Read via `GetFile` Servlet
**Description:** The `GetFile` servlet accepts `filename`, `customer`, `location`, `department`, and `vehicle_cd` directly from HTTP query parameters and concatenates them into a filesystem path with no sanitisation, no canonicalisation, and no path-traversal prevention. An unauthenticated attacker can supply `../` sequences in any of these parameters to read arbitrary files accessible to the Tomcat process on the server filesystem (e.g., `/etc/passwd`, Tomcat configuration files, application property files containing credentials).

In `doGet`, when `method=inspection`, the full path is:
```
/home/gmtp/fms_files/CFTS/<customer>/<location>/<department>/<vehicle_cd>/<filename>
```
Every segment is attacker-controlled. For the default (non-inspection) path:
```
/home/gmtp/fms_files/licence/<filename>
```
`filename` alone is attacker-controlled with no restriction.

**Severity:** CRITICAL

**Checklist item:** Section 3 ‚Äî Injection Vulnerabilities (File Inclusion / Path Traversal)

**Evidence:**
```java
// GetFile.java, lines 19‚Äì31
String filename = request.getParameter("filename") == null ? "" : request.getParameter("filename");
String filepath = "/home/gmtp/fms_files/licence/";

String vehCd = request.getParameter("vehicle_cd") == null ? "" : request.getParameter("vehicle_cd");
String cust = request.getParameter("customer") == null ? "" : request.getParameter("customer");
String loc = request.getParameter("location") == null ? "" : request.getParameter("location");
String dept = request.getParameter("department") == null ? "" : request.getParameter("department");

if (method.trim().equals("inspection")) {
    filepath = "/home/gmtp/fms_files/CFTS/" + cust + "/" + loc + "/" + dept + "/"+ vehCd + "/";
}

```

**Description:** The `GetFile` servlet (`doGet` and `getFile` methods) performs no session check, no authentication verification, and no authorisation check before serving files. Any unauthenticated request to this servlet endpoint can download files from the server. Combined with the path traversal finding (J77-F01), this allows completely unauthenticated arbitrary file read. Even without path traversal, unauthenticated users can access any legitimately served file (vehicle inspection records, licence files) for any customer.

2. **Line 98:** `gmtp_id` (from prior result set, originally from `ftp_outgoing.gmtp_id`) concatenated into a multi-join SELECT.

### J77-F05: FTP Password Written in Plaintext to Database
**Description:** At line 373, an FTP command string is constructed that embeds the plaintext FTP password (`RuntimeConf.firmwarepass`) directly. This string is then inserted verbatim into the `outgoing` database table (line 374‚Äì375). The password is therefore stored in plaintext in a database table that other application code reads and processes. The plaintext password also appears in SELECT queries (line 384) used to retrieve the record just inserted. Any user or process with read access to the `outgoing` table can recover the FTP server password.

A masked version (`RuntimeConf.firmwarepassmask`) is used for the `outgoing_stat` table insert (line 397‚Äì400), confirming the developers were aware of the sensitivity but did not apply the same protection to the primary `outgoing` table.

**Severity:** HIGH

**Checklist item:** Section 1 ‚Äî Secrets and Credentials

**Evidence:**

## Checklist Items With No Issues

The following checklist items were reviewed against the three files and no issues were found:

- **Section 1 ‚Äî Secrets and Credentials: hardcoded JDBC credentials in source.** No `DriverManager.getConnection()` calls with inline credentials. Database access in `FleetCheckFTP` uses JNDI lookup (`RuntimeConf.invoice_database`), which is the correct pattern. FTP credentials are referenced via `RuntimeConf` and `LindeConfig` constants rather than inline literals; the concern raised in J77-F05 is about persistence to the database, not about literal strings in source.
- **Section 3 ‚Äî XSS.** None of the three files are JSP files or produce HTML output. `FleetCheckFTP` writes binary files to disk. `GetFile` serves raw file bytes. `GForceCalibration` performs only arithmetic. No XSS surface in these files.
- **Section 3 ‚Äî Command Injection.** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls in any of the three files.
- **Section 3 ‚Äî SSRF.** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) in any of the three files.
- **Section 4 ‚Äî CSRF.** None of the three files render HTML forms or process form submissions. `GetFile` is a GET-only download servlet with no state-changing effect. `FleetCheckFTP` is an internal batch utility with no HTTP entry point. CSRF is not applicable to these specific files (must be assessed at the form/JSP layer separately).
- **Section 6 ‚Äî Transport Security.** No TLS configuration in these files. Transport configuration must be assessed in `server.xml` and `context.xml`.
- **Section 7 ‚Äî Dependency Vulnerabilities.** No dependencies are declared in these files; library dependency review must be done against `pom.xml` or `build.xml` and the committed JARs.
- **Section 2 ‚Äî Session fixation.** These files do not handle login or session creation.
- **GForceCalibration.java ‚Äî all sections.** This class contains only arithmetic computation (mean and standard deviation). It has no I/O, no SQL, no HTTP, no authentication surface, and no security-relevant behaviour. The only notes are the two unused imports flagged in J77-F10.
