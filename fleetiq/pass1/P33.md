### P33-F01: BMS Credentials Transmitted in Plaintext Over HTTP via iframe URL
**Description:** `battery_management_au.jsp` constructs an `<iframe>` that loads `http://www.itagworld.com/api_/login.php` — an external third-party endpoint over plain HTTP — and passes the BMS username and password (or hash) as cleartext query parameters in the URL. This transmits credentials over an unencrypted connection, making them visible to any network observer (ISP, corporate proxy, on-path attacker). In addition to network interception, the URL is recorded in browser history, server access logs, proxy logs, and HTTP Referer headers at the destination server — permanently exposing the credential values. The parameter name `hash` suggests the password value may be pre-hashed, but a reusable hash is functionally equivalent to a password for authentication purposes.
**Severity:** CRITICAL
**Checklist item:** Section 6 — Transport Security (plain HTTP for credential-bearing traffic); Section 1 — Secrets and Credentials (credentials exposed in transmitted URL)
**Evidence:**
```jsp
// Line 28
<iframe src="http://www.itagworld.com/api_/login.php?user=<%=bms_user%>&hash=<%=bms_pass%>&dt=<%=timeStamp %>" height="465px" width="1050px" ></iframe>
```
**File:** pages/admin/battery/battery_management_au.jsp
**Line(s):** 28

### P33-F02: BMS Password Exposed as a URL Query Parameter (Credential Leakage Independent of Transport)
**Description:** Passing a secret credential (password or password hash) as a URL query parameter is a vulnerability independent of whether HTTP or HTTPS is used. Query parameters are logged by every HTTP server in the request path as part of the access log URI field, recorded in browser history, stored in referrer headers sent to third-party resources loaded by the destination page, and visible to any analytics or monitoring tooling. Even over HTTPS, a credential in a query parameter will appear in the itagworld.com server access log in plaintext. This is a separate finding from the HTTP transport issue in F01.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials; Section 5 — Data Exposure and Privacy
**Evidence:**
```jsp
// Line 28
<iframe src="http://www.itagworld.com/api_/login.php?user=<%=bms_user%>&hash=<%=bms_pass%>&dt=<%=timeStamp %>" height="465px" width="1050px" ></iframe>
```
**File:** pages/admin/battery/battery_management_au.jsp
**Line(s):** 28

### P33-F06: Unescaped Output of Database Values into HTML — Stored XSS
**Description:** `battery_management.jsp` renders multiple database-sourced values directly into HTML output using raw `<%= ... %>` expressions without escaping. The values `batt_id.get(i)`, `fleet_no.get(i)`, `Fms_Usr_Mst_Name.get(i)`, `Fms_Loc_Mst_Name.get(i)`, and `Fms_Dept_Mst_Name.get(i)` are all written directly to the response. If any of these database columns contain HTML special characters (`<`, `>`, `"`, `&`, `'`) — whether from malicious input via another endpoint or from legitimate data containing special characters — the browser will interpret them as HTML markup. An attacker who can write a payload into any of these columns via any other endpoint achieves stored XSS on this page. The correct escaping method for JSP is `<c:out value="${variable}" />` or `${fn:escapeXml(variable)}`.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (unescaped `<%= ... %>` rendering database values)
**Evidence:**
```jsp

// Line 240 — fleet number from database rendered into input value attribute

// Line 180 — customer name and code from database rendered into option elements
<option value="<%=Fms_Usr_Mst_Cd.get(i)%>" ...><%=Fms_Usr_Mst_Name.get(i)%></option>

// Line 192 — location name and code from database rendered into option elements
```

### P33-F10: user_cd Placed in Hidden Form Field — Potential IDOR on BMS Password Change
**Description:** In `change_bms_pass.jsp`, `user_cd` is retrieved from session and placed into a hidden form field that is submitted to `../servlet/Frm_security`. Hidden form fields are fully visible and modifiable by any user with browser developer tools. If the `Frm_security` servlet uses the submitted `user_cd` parameter to identify whose BMS password to change — rather than always reading `user_cd` from the server-side session — then any authenticated user can change another user's BMS credentials by modifying this field before submission. This is a potential IDOR (Insecure Direct Object Reference) vulnerability. The servlet backend (`Frm_security`) must be audited separately to confirm whether it trusts this submitted value; the JSP-side pattern is flagged as inherently suspect.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (IDOR — object reference submitted via user-modifiable form field)
**Evidence:**
```jsp
// Line 110
<input type="hidden" name="user_cd" value="<%=user_cd%>">
```
This value is submitted to `../servlet/Frm_security` alongside `op_code=pass_bms_chg`. The `bmsuser` field (line 84) is also editable and submitted, compounding the risk.
**File:** pages/admin/battery/change_bms_pass.jsp
**Line(s):** 84, 110

### P33-F11: External Third-Party Service Embedded via iframe — Supply Chain and Integrity Risk
**Description:** `battery_management_au.jsp` embeds an iframe pointing to `http://www.itagworld.com`, an external third-party domain outside the application's control. The application passes live authentication credentials to this external domain. **Severity:** HIGH

**File:** pages/admin/battery/battery_management_au.jsp

### P33-F12: Integer.parseInt on User-Controlled form_cd Without Validation — Exception Risk and Permission Bypass
**Description:** In `battery_management.jsp` (line 243) and `change_bms_pass.jsp` (line 62), `form_cd` (a raw request parameter) is passed to `Integer.parseInt()` and then used as an index into the session `permision` array. ## Checklist Items With No Issues

- **Section 1 (Secrets and Credentials) — hardcoded credentials in source code:** No hardcoded database passwords, JDBC connection strings with embedded credentials, API keys, or encryption keys are present as string literals in any of the three JSP files. The BMS credential issues (F01, F02) concern runtime URL transmission, not source-hardcoded secrets.

- **Section 2 (Authentication and Authorization) — multi-tenancy scoping in battery_management.jsp:** The access-level scoping logic at lines 64–115 of `battery_management.jsp` correctly overrides user-supplied `customer`, `location`, and `department` request parameters with session-sourced values for users with `access_level > 1`, preventing lower-privilege users from querying other tenants' battery data through this filter.
