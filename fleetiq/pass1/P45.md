## Findings

### P45-F01: Unvalidated mail_id Parameter Reflected in JSON Response (Stored/Reflected Content Injection)
**Description:** In all 10 files, the `mail_id` request parameter is read directly from the HTTP request and interpolated into the JSON response string without any sanitisation, escaping, or validation. The response is written with `response.getWriter().println(message)` with Content-Type `text/html`. Although the JSON is likely consumed by JavaScript rather than rendered as HTML, the combination of `text/html` content type and unescaped user-controlled content in the response creates a reflected XSS vector. An attacker who can cause a victim to submit a crafted `mail_id` value receives a response containing their injected payload.
**Severity:** HIGH
**Checklist item:** Section 3 â XSS (Cross-Site Scripting)
**Evidence:**
```java
// xlsx_battery_report_mail.jsp, lines 96-99:
String mail_id=request.getParameter("mail_id");
...
if(mailReport.sendExcelReport(subject, mail_id, excelLoc, attachmentName))
{
    message = "{ \"status\":\"success_close\", \"message\":\"Mail has been sent to " + mail_id + ".\" }";
}
response.getWriter().println(message);
```
Identical pattern present in all 10 files. The JSP page declares `contentType="text/html"` (line 11-12 of each file) while writing JSON containing an unescaped user-supplied value.
**File:** pages/admin/reports/email/xlsx_battery_report_mail.jsp (and all 9 other files in this set)
**Line(s):** xlsx_battery_report_mail.jsp:83,98; xlsx_broadcastmsg_report_mail.jsp:79,92; xlsx_cimplicity_shock_report_mail.jsp:88,95; xlsx_cimplicity_util_report_mail.jsp:98,105; xlsx_curr_driv_report_mail.jsp:86,100; xlsx_curr_unit_report_mail.jsp:85,98; xlsx_daily_driver_summary_report_mail.jsp:104,120; xlsx_daily_veh_summary_report_mail.jsp:107,122; xlsx_driver_access_abuse_report_mail.jsp:97,109; xlsx_driver_impact_report_mail.jsp:90,101

### P45-F02: Unvalidated subject Parameter Passed to Mail Function
**Description:** In all 10 files, the `subject` parameter is taken directly from the HTTP request without validation and passed to `mailReport.sendExcelReport()`. There is no check on the content or length of the subject line. Depending on the SMTP library implementation in `MailExcelReports`, a crafted subject containing newline characters (`\r\n`) could constitute an email header injection attack, allowing an attacker to inject arbitrary headers (e.g., `Bcc:`, `To:`, `From:`) into the outgoing email, potentially redirecting the report to unintended recipients or using the server as a spam relay.
**Severity:** HIGH
**Checklist item:** Section 3 â Injection Vulnerabilities

### P45-F07: Filesystem Path Construction Using getProtectionDomain â Traversal-Adjacent Pattern
