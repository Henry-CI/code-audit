## Findings

### J09-F01: WiFi password stored and exposed in plaintext via serializable bean
**Description:** `NetworkSettingBean` holds a `password` field representing a WiFi network credential. The field is stored as a plain `String` with a public getter `getPassword()` that returns the raw value. The class is `Serializable`, meaning the password can be written to session storage, persisted to disk, transmitted over the wire in serialized form, or appear in heap dumps. No hashing, encryption, or masking is applied. If this bean is ever logged, serialized to a cookie, or stored in the database as a string, the WiFi credential is fully exposed. In a fleet management context these credentials likely belong to in-vehicle GPS hardware networks.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials
**Evidence:**
```java
// NetworkSettingBean.java lines 14, 34–38
private String password = "";

public String getPassword() {
    return password;
}
public void setPassword(String password) {
    this.password = password;
}
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/NetworkSettingBean.java`
**Line(s):** 14, 34–38

### J09-F02: MymessagesUsersBean fields are package-private, not private
**Description:** All eight fields in `MymessagesUsersBean` (`cust_cd`, `loc_cd`, `dept_cd`, `threshold`, `user_id`, `user_email`, `descrption`, `id`) are declared without an access modifier, making them package-private by default. Any class in the `com.torrent.surat.fms6.bean` package can read or write these fields directly, bypassing the public getter/setter methods entirely. This is a data encapsulation failure. In a multi-tenant context, `cust_cd` controlling which customer's data is accessed could be modified directly by any bean in the same package, undermining tenant isolation at the Java object level.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy)
**Evidence:**
```java
// MymessagesUsersBean.java lines 13–19
String cust_cd = "";
String loc_cd = "";
String dept_cd = "";
String threshold = "";
String user_id = "";
String user_email = "";

String id = "";
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/MymessagesUsersBean.java`
**Line(s):** 13–19

### J09-F03: MymessagesUsersBean carries PII (user_id, user_email) in a serializable object with no access controls
**Description:** `MymessagesUsersBean` implements `Serializable` and contains `user_id` and `user_email` — personal identifiable information. Because the class is serializable, its contents can be written to session state, persisted, or transmitted. There is no field-level protection (fields are package-private), no `transient` declaration on the PII fields to exclude them from serialization, and no evidence of encryption at rest. If the serialized form of this bean is ever stored or transmitted, user PII travels with it in plaintext.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy

### J09-F04: MenuBean exposes Form_Path and ReskinPath — potential path traversal or XSS surface in consuming code
**Description:** `MenuBean` carries two path-valued fields: `Form_Path` (line 11) and `ReskinPath` (line 12). These are set by the application and presumably rendered in JSP menu navigation to build URLs or include paths. If these values are ever rendered in JSP without escaping (e.g., `<%= menuBean.getForm_Path() %>`), or used as arguments to `<jsp:include>` or `<%@ include file="..." %>`, they represent XSS and/or local file inclusion surfaces. This bean file itself does not contain the exploit, but the field design creates the precondition. The checklist requires flagging path-valued fields in beans used by JSPs. Additionally, `ArrayList` is imported at line 3 but never used, indicating dead code.

### J09-F05: MymessagesUsersBean.id is a generic unqualified identifier — IDOR risk in consuming code
**Description:** The field `id` (line 19, `MymessagesUsersBean`) is a plain `String` with no qualification indicating what entity it identifies. Public getter `getId()` returns it without constraint. If this `id` is set from a request parameter and used to look up a message record without verifying that the record belongs to the authenticated user's `cust_cd`, it is a direct IDOR vector. The bean itself does not implement the lookup, but the field design — an unscoped generic `id` sitting alongside `cust_cd` — is a known pattern preceding IDOR bugs in this codebase style. This finding flags the structural risk; consuming servlets/JSPs must be verified separately.
