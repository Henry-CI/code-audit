## Findings

### J23-F01: Unvalidated caller-supplied filesystem path passed directly to file write operation
**Description:** In `StackedBarChart.saveChart()`, the `chartDir` parameter is accepted from the caller and used to construct a filesystem path for writing a PNG file. The only transformation applied is a URL-decode of `%20` to a space. If `chartDir` is derived from user input (e.g., a request parameter passed through a calling servlet), an attacker could supply a path traversal string such as `../../webapps/ROOT/malicious` to write arbitrary files anywhere the application server process has write access, including outside the intended `chart_img/` directory. The returned value is the absolute filesystem path of the written file, which is also an information disclosure risk if surfaced to the user.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — File inclusion / path traversal; also 5. Data Exposure and Privacy (filesystem path returned to caller)
**Evidence:**
```java

// Line 298
```

### J23-F02: Swallowed exceptions expose internal stack traces to server logs without proper error handling
**Description:** Both `StackedBarChart.createDataset()` (line 253) and `StackedBarChart.createChart()` (line 180) catch broad `Exception` and respond only with `e.printStackTrace()`. This means application errors are silently dropped from the user-visible flow while full stack traces (including class names, method names, and line numbers) are written to the server log. If server logs are accessible or exfiltrated, they provide an attacker with detailed implementation intelligence. Additionally, the silent swallowing means chart generation can fail without any signal to the caller, potentially returning null or incomplete data downstream.
**Severity:** LOW
**Checklist item:** 5. Data Exposure and Privacy — Error responses leaking implementation detail
**Evidence:**

### J23-F04: Chart title passed unsanitised to Google Charts API URL in PieChartR
**Description:** In `PieChartR.createChart(String title)`, the `title` parameter is passed directly to `chart.setTitle(title, Color.BLACK, 16)` (line 61), which the charts4j library encodes into a Google Charts API URL returned by `chart.toURLString()` (line 65). If the `title` originates from user-supplied input or database content (e.g., a customer-supplied fleet name or report label) and the charts4j library does not fully URL-encode all characters, this could result in URL injection into the Google Charts endpoint. The risk is moderate and depends on the library's encoding behaviour and how callers supply the title.
**Severity:** LOW
**Checklist item:** 3. Injection Vulnerabilities — XSS / injection via unvalidated output
**Evidence:**
```java
// Line 61
chart.setTitle(title, Color.BLACK, 16);
// Line 65

return url;
```

### J23-F05: No customer/tenant scoping in chart data processing
**Description:** Neither `PieChartR` nor `StackedBarChart` performs any check that the data being rendered belongs to the authenticated user's customer/tenant. Both classes are purely data-display utilities that render whatever data is supplied by their callers. This is not a vulnerability in these classes themselves, but it means the security of multi-tenant data isolation is entirely dependent on callers correctly scoping data before passing it in. Given the branch is `multi-customer-sync-to-master` (explicitly multi-tenant), this delegation of all tenant-scoping responsibility to callers is a design risk that must be verified in every servlet/JSP that instantiates these chart classes. This finding flags the gap for follow-up in calling code.
**Severity:** INFO
**Checklist item:** 2. Authentication and Authorization — multi-tenancy, IDOR; the chart classes themselves contain no scoping; callers must be audited
**Evidence:**
