## Findings

### J14-F01: SuperMasterAuthBean does not implement Serializable
**Description:** `SuperMasterAuthBean` is used as a data-transfer / bean object alongside sibling beans that all implement `Serializable`. It does not itself implement `Serializable`. If the application stores instances of this bean in the HTTP session (a common pattern in JSP/Tomcat applications for "super master" / privileged session data), session serialization will fail at runtime with a `NotSerializableException`, causing session loss. More importantly from a security perspective, the class holds `superMasterCode`, `authStart`, `authEnd`, and `custName` — privileged credential-adjacent data. The absence of `Serializable` suggests this object's lifecycle and session-storage behaviour is poorly considered, which is a risk indicator that it may be handled inconsistently across the codebase.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (session management)
**Evidence:**
```java
// SuperMasterAuthBean.java, line 3
public class SuperMasterAuthBean {
    // no "implements Serializable"

    private String fleetNo,serialNo,authStart,authEnd,servHourFrom,superMasterCode,custName;
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/SuperMasterAuthBean.java`
**Line(s):** 3, 5

### J14-F02: SubscriptionBean and UnitBean fields are package-private, not private
**Description:** In both `SubscriptionBean` and `UnitBean`, all domain fields are declared without an access modifier, making them package-private. In `SubscriptionBean` this affects `id`, `cust_cd`, `loc_cd`, `dept_cd`, `month`, and `email` (lines 14–19). In `UnitBean` this affects all 20 domain fields (lines 13–33). Any class within the same package (`com.torrent.surat.fms6.bean`) can read and write these fields directly, bypassing the getters and setters entirely. This means no validation, sanitization, or access-control logic placed in the setters can be enforced within-package — a caller can set `cust_cd` or `veh_cd` directly without going through `setCust_cd()` / `setVeh_cd()`. In a multi-tenant application where `cust_cd` is a tenant discriminator, bypassing the setter is a design-level risk. The intent is clearly for these to be `private` with access only through the accessors.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy; Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// SubscriptionBean.java, lines 14–19 — no "private" modifier
String id = "";
String cust_cd = "";
String loc_cd = "";
String dept_cd = "";
String month = "";
String email = "";

// UnitBean.java, lines 13–33 — no "private" modifier on any field
String veh_cd = "";
String gmtp_id = "";
String cust_nm = "";
// ... (all fields through line 33 are package-private)
int threshold;
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/SubscriptionBean.java` (lines 14–19); `WEB-INF/src/com/torrent/surat/fms6/bean/UnitBean.java` (lines 13–33)
**Line(s):** SubscriptionBean 14–19; UnitBean 13–33

### J14-F03: SubscriptionBean.id field has no getter or setter — silently inaccessible
**Description:** `SubscriptionBean` declares a field `id` at line 14 with a default value of `""`. There is no `getId()` getter and no `setId()` setter. Because the field is also package-private (see J14-F02), it is directly accessible within the package but completely inaccessible to callers outside the package via the standard bean pattern. If `id` is a primary key or subscription identifier used to scope data queries, the lack of an accessor means callers outside the package cannot read it via the bean interface. This could lead to the field being populated by direct field access within the package (bypassing any future validation), or the field being permanently left at its default empty string value in practice, which would be a silent data loss. Either outcome is a correctness and potential security concern in a multi-tenant system where subscription identifiers may be used to scope data.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy
**Evidence:**
```java
// SubscriptionBean.java, line 14
String id = "";
// No getId() or setId() method exists anywhere in the file.
// Getters exist for cust_cd, loc_cd, dept_cd, month, email only.
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/SubscriptionBean.java`
**Line(s):** 14, 21–50 (no getId/setId in the entire file)

### J14-F04: UnitBean exposes CCID (SIM card identifier) and SIM provider without any access control
**Description:** `UnitBean` contains fields `ccid`, `old_ccid`, `ccid_rpt_time`, and `sim_provider` (lines 26–28). CCID (Chip Card ID / ICCID) is the unique serial number of a SIM card installed in a GPS tracking device. Exposure of CCID values for all vehicles in a customer's fleet (or across customers if multi-tenancy is broken) constitutes a data privacy risk: CCIDs can be used to identify SIM cards, correlate device identities, and potentially facilitate SIM-related attacks. The bean provides public getters for all these fields with no protection. The scope of actual exposure depends on how the bean is populated and rendered — this finding flags the design risk. The `old_ccid` field additionally retains historical SIM identifiers, widening the exposure surface.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J14-F05: SuperMasterAuthBean exposes superMasterCode via public getter with no protection
**Description:** `SuperMasterAuthBean` holds a field `superMasterCode` with a fully public getter `getSuperMasterCode()`. The field name implies a privileged authentication code — a "super master" credential used to authorize devices or sessions (`authStart`, `authEnd` fields suggest an authorization window). Exposing a privileged credential-like value via an unrestricted public getter on a plain bean (with no `Serializable`, no access control, no annotation) means any code that obtains a reference to this bean can extract the code. In a JSP application, if this bean is placed in session or request scope, a JSP expression `<%= bean.getSuperMasterCode() %>` would render the value directly to the browser. The design provides no mechanism to restrict access to this sensitive field.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials; Section 2 — Authentication and Authorization
**Evidence:**
```java
// SuperMasterAuthBean.java, line 5

// Line 31 — fully public, no access restriction:
public String getSuperMasterCode() {

// Line 55 — fully public setter:

    this.superMasterCode = superMasterCode;
}
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/SuperMasterAuthBean.java`
**Line(s):** 5, 31–33, 55–57

### J14-F06: UnitBean contains no customer identifier field — multi-tenancy isolation risk
**Description:** `UnitBean` represents a vehicle/unit record and contains fields for `cust_nm` (customer name, a display string) but no `cust_cd`, `cust_id`, or equivalent customer code/identifier field. The field `cust_nm` is a display name only, not a primary key or discriminator. In a multi-tenant system, data isolation queries are performed on a customer identifier (typically a code or numeric ID), not a display name. A display name is mutable, non-unique, and unsuitable as a tenant discriminator. By contrast, the sibling bean `SubscriptionBean` correctly includes `cust_cd` as a tenant discriminator field. If `UnitBean` instances are used in queries or rendered in JSPs without a reliable customer discriminator, there is a risk that vehicle records are not properly scoped to the authenticated tenant. This is a design-level finding that requires verification against the DAOs and servlets that populate and use `UnitBean`.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and multi-tenancy (IDOR / multi-tenant data isolation)
**Evidence:**
```java
// UnitBean.java, lines 15–16 — only customer name present, no customer code/ID
String cust_nm = "";   // display name only — not a discriminator
// No cust_cd, no cust_id, no customer_key field anywhere in the class.

// Compare SubscriptionBean.java, line 15:
String cust_cd = "";   // code-based discriminator present
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/UnitBean.java`
**Line(s):** 15 (absence of a discriminator field throughout lines 13–33)

## Checklist Items With No Issues

The following checklist items were reviewed against the content of these three bean files and no issues were found. Note that these are plain JavaBeans (data-transfer objects) — they contain no SQL, no HTTP handling, no JSP output, no session management code, and no dependency references. Many checklist items are therefore not applicable at this file level and must be checked in the servlets, DAOs, and JSP files that use these beans.

- **Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, SMTP passwords, or connection strings present in any of the three files. No calls to `DriverManager.getConnection()` or any JDBC API. No credentials in comments.
- **Section 3 — SQL Injection:** No SQL statements, no `Statement`, no `PreparedStatement`, no JDBC calls of any kind in any of the three files.
- **Section 3 — XSS:** No JSP output expressions, no HTML rendering, no `PrintWriter` usage in any of the three files.
- **Section 3 — File Inclusion:** No file I/O, no `include` directives, no `Runtime.exec()` or `ProcessBuilder` in any of the three files.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage.
- **Section 3 — SSRF:** No HTTP client calls, no URL construction in any of the three files.
- **Section 4 — Session and CSRF:** No session manipulation, no CSRF token handling — not applicable to plain bean classes.
- **Section 6 — Transport Security:** No connector configuration, no TLS settings — not applicable to bean classes.
- **Section 7 — Dependency Vulnerabilities:** No external library usage beyond `java.io.Serializable`, `java.util.ArrayList` (unused), and `java.util.Date` — all standard JDK types.
- **Wildcard imports:** No wildcard imports (`import java.sql.*` or similar) in any of the three files.
