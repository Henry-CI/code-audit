### J53-F01: Customer/tenant parameters passed directly to DAO lookup without validation ‚Äî potential for cross-tenant data exposure
**Description:** In `ExcelSuperMasterAuthReport.createSuperMasterAuthReport()`, the `cust`, `loc`, and `dep` string parameters are passed directly to `CustomerDAO.getCustomerName()`, `CustomerDAO.getLocationName()`, and `CustomerDAO.getDepartmentName()` at lines 66-68 with no validation that these IDs belong to the authenticated user's tenant. This class is an Excel report generator; if the calling servlet constructs these parameter values from user-supplied request parameters and passes them through without scoping them to the authenticated user's customer, an attacker could supply an arbitrary customer/location/department ID and retrieve data belonging to a different tenant. The vulnerability is an IDOR risk at the report-generation layer. The `CustomerDAO` implementation is not visible in this file, so it is not possible to confirm whether the DAO itself enforces tenant scoping, but no such check is visible at this layer.
**Severity:** HIGH
**Checklist item:** Section 2 ‚Äî Authorization and multi-tenancy; IDOR
**Evidence:**

### J53-F03: No authorisation check in any report class ‚Äî report generation proceeds regardless of caller identity
**Description:** None of the three report classes (`ExcelSuperMasterAuthReport`, `ExcelUnitUnlockReport`, `ExcelUtilWOWReport`) contain any check that the requesting user is authenticated or authorised to receive the data being generated. There are no session checks, no role checks, and no customer-scoping assertions. All three classes accept their data via constructor-injected beans and caller-supplied string parameters and produce the report unconditionally. The security of these reports is entirely dependent on the calling servlet performing adequate authentication and authorisation checks before constructing and invoking these classes. If any calling servlet is missing those checks, the full report content is produced and written to disk with no safeguard at the report layer. Given that `ExcelSuperMasterAuthReport` generates a "SuperMaster Authorisation Report" ‚Äî which contains fleet numbers, serial numbers, authorisation start/end times, and super master codes ‚Äî the absence of any in-class authorisation check is particularly sensitive.
**Severity:** HIGH
**Checklist item:** Section 2 ‚Äî Authentication and Authorization; Section 5 ‚Äî Data Exposure and Privacy (data export)
**Evidence:**

## Checklist Items With No Issues

The following checklist items were reviewed against the three files and no issues were found within the scope of these files:

- **Section 1 ‚Äî Secrets and Credentials:** No hardcoded database credentials, passwords, API keys, or encryption keys are present in any of the three files. No `DriverManager.getConnection()` calls with embedded credentials. No credentials in comments.
- **Section 3 ‚Äî SQL Injection:** None of the three files contain any SQL statements, JDBC calls, `Statement.executeQuery()`, `PreparedStatement`, or string concatenation used to build SQL queries. All database access is delegated to DAO classes (`CustomerDAO`, `UnitDAO`) whose implementations are not in scope for this agent. No direct SQL execution occurs in these files.
- **Section 3 ‚Äî XSS (Cross-Site Scripting):** These are Excel report generator classes, not JSP files. There are no JSP expressions, no HTML rendering, and no `<%= %>` output. XSS is not applicable to these files.
- **Section 3 ‚Äî File Inclusion:** No dynamic file inclusion patterns (`jsp:include`, `<%@ include %>` with user-controlled paths) are present. Not applicable to these Java classes.
- **Section 3 ‚Äî Command Injection:** No calls to `Runtime.getRuntime().exec()` or `ProcessBuilder` in any of the three files.
- **Section 4 ‚Äî Session and CSRF:** These are Excel report generator classes invoked server-side. They do not handle HTTP requests, session objects, or form submissions directly. CSRF token management is the responsibility of the calling servlet/JSP layer, not these report classes.
- **Section 6 ‚Äî Transport Security:** These files do not configure or reference TLS settings, HTTP connectors, or cookie attributes. Transport security is not applicable to these classes.
- **Section 7 ‚Äî Dependency Vulnerabilities:** The only third-party dependency directly visible in these files is Apache POI (`org.apache.poi.ss.usermodel.*`). The POI version cannot be determined from these source files alone ‚Äî it must be assessed via the JAR inventory (see ff-new-inventory.md). No version information is embedded in the import statements.
