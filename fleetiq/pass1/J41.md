## Findings

### J41-F01: Hardcoded Production Hostname in Image URL Construction
**Description:** Both `EmailImpactMeterReport.java` and `EmailImpactReport.java` construct an image URL by concatenating a hardcoded production hostname (`http://fleetfocus.lindemh.com.au`) with a path component from `RuntimeConf.impactDir` and a database-sourced filename. The hostname is baked into the source code. Any change to the production domain requires a code change and redeployment. More critically, the URL uses plain HTTP (not HTTPS), meaning the image fetch is transmitted unencrypted.
**Severity:** MEDIUM
**Checklist item:** Section 6 — Transport Security; Section 1 — Secrets and Credentials (hardcoded infrastructure values)
**Evidence:**
```java
// EmailImpactMeterReport.java line 192
// EmailImpactReport.java line 192
String tmp = "http://fleetfocus.lindemh.com.au/fms/"+ RuntimeConf.impactDir+"/"+img_link.get(j);
addImage2(sheet, tmp, 0, true, m);
```

**Line(s):** 192–193

**File (second instance):** WEB-INF/src/com/torrent/surat/fms6/excel/reports/EmailImpactReport.java

### J41-F02: Plain HTTP Used for Server-Side Image Fetch (SSRF-Adjacent / Transport Security)
**Description:** The URL constructed at line 192 in both impact report classes uses the `http://` scheme. When `addImage2()` fetches this URL server-side to embed the impact photo in the Excel workbook, it does so over unencrypted HTTP. GPS and fleet impact images may be sensitive. Additionally, if `RuntimeConf.impactDir` or `img_link` values can be influenced by an attacker (e.g., via crafted database content or parameter manipulation upstream), the path component of the URL becomes partially attacker-controlled, making this an SSRF risk against the production server's own HTTP interface.
**Severity:** MEDIUM
**Checklist item:** Section 6 — Transport Security; Section 3 — SSRF
**Evidence:**
```java
// EmailImpactMeterReport.java line 192
String tmp = "http://fleetfocus.lindemh.com.au/fms/"+ RuntimeConf.impactDir+"/"+img_link.get(j);
```

### J41-F03: Database-Sourced Values Injected Directly into Excel Formula Strings (Formula Injection / CSV Injection)
**Description:** In `EmailKeyHourUtilReport.java`, utilisation time values retrieved from the database are directly interpolated into Excel `VALUE()` formula strings using string concatenation, without any sanitisation or validation. If a database value contains characters meaningful to Excel formula evaluation (such as `"`, `+`, `-`, `=`, `@`, or a closing parenthesis), the formula could be malformed or, depending on Excel's formula engine, execute unintended operations. This is a formula injection (also called CSV injection) vulnerability: a malicious or corrupted database value could break out of the `VALUE("...")` context and inject arbitrary Excel formula logic.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**
```java
// EmailKeyHourUtilReport.java lines 151-152
contentCell.setCellFormula("VALUE(\""+vutil1.get(i).toString()+"\")");
contentCell.setCellValue(vutil1.get(i).toString());

// and again in the "Un Utilised Units" section (lines 280-317)
```
**File:** WEB-INF/src/com/torrent/surat/fms6/excel/reports/EmailKeyHourUtilReport.java

### J41-F04: Report Subtitle Labels Swapped — Customer, Site, and Dept Data Displayed Under Wrong Headers

### J41-F05: Customer Scoping Relies on Caller-Supplied cust_cd with No Visible Authorisation Verification
**Description:** All three report classes accept `cust_cd` (customer code) as a constructor parameter (inherited from `Frm_excel`), normalise it, and pass it directly to `Databean_report` for database query scoping. There is no visible check in these classes that the supplied `cust_cd` belongs to the authenticated user's session. The tenant isolation relies entirely on the caller (the email dispatch or report generation servlet) supplying the correct `cust_cd`. If the caller can be manipulated — for example by a parameter in an HTTP request that is passed through without server-side session validation — an attacker could request a report scoped to a different customer's data. This is an IDOR / multi-tenant isolation risk. The `Databean_report` layer and the calling code would need to be audited to confirm whether session-level enforcement exists upstream.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and Multi-Tenancy; IDOR
**Evidence:**

### J41-F06: cust_cd Value "0" Resolves to Empty String — Potential All-Customer Data Query
**Description:** In all three report classes, when `cust_cd` equals `"0"` it is replaced with an empty string before being passed to `Databean_report`. Depending on how `Databean_report.init()` handles an empty `cust_cd`, this may result in a query that returns data for ALL customers rather than a specific one. In a multi-tenant fleet management system, a report query with no customer filter is a critical data isolation failure. The actual SQL executed by `Databean_report` is not visible in these files and must be audited, but the pattern of converting `"0"` to `""` before scoping a data query is a known multi-tenancy anti-pattern.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and Multi-Tenancy; Section 3 — SQL injection (parameterisation)

### J41-F08: Unused UnitDAO Instantiation in Both Impact Report Classes
**Description:** In both `EmailImpactMeterReport.java` and `EmailImpactReport.java`, a `UnitDAO` object is instantiated at line 106 (`UnitDAO unitDAO = new UnitDAO()`) but is never referenced again. If `UnitDAO` opens a database connection in its constructor and the connection is not closed (because the object is never used and likely has no finally/close block called on it), this could cause a connection leak on every report generation. Connection leaks in a reporting system that generates many reports can exhaust the database connection pool over time, causing denial of service. The actual `UnitDAO` constructor would need to be audited to confirm whether a connection is opened.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (resource management); Section 3 — general code quality risk
**Evidence:**
```java
// EmailImpactMeterReport.java line 106
// EmailImpactReport.java line 106
UnitDAO unitDAO = new UnitDAO();
// unitDAO is never referenced again in either file
```
**File:** WEB-INF/src/com/torrent/surat/fms6/excel/reports/EmailImpactMeterReport.java
