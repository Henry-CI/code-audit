An attacker can craft a URL with a value like `');alert(1)//` in any of these parameters to execute arbitrary JavaScript in the victim's browser.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî XSS (Cross-Site Scripting)
**Evidence:**

### P11-F03: Unescaped Output of Database-Sourced Values Across All Report JSPs (Stored XSS Risk)
**Description:** All eight JSPs output database-sourced values ‚Äî vehicle names, driver names, vehicle IDs, model names, field names, sensor values, and date strings ‚Äî directly to HTML using raw `<%= ... %>` expressions with no HTML escaping. The correct pattern would be `<c:out value="${variable}" />` (JSTL) or `${fn:escapeXml(variable)}`. If any of these values contain HTML special characters or JavaScript payloads (e.g. stored in vehicle names, driver names, or dynamic field names via a data-entry screen), the payload will execute in any user's browser that views the report. This constitutes a stored XSS risk as these values originate from the database and could have been injected by a user with data-entry privileges.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî XSS (Cross-Site Scripting)
**Evidence (representative samples across files):**
