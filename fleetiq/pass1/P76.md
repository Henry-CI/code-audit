### P76-F01: Cross-Site Scripting (XSS) - Unescaped Output of User-Controlled Data
**Description:** All 10 JSP files output user-controlled data directly to the HTML page without proper escaping or sanitization. Request parameters (dates, customer codes, location codes, department codes, form codes, search criteria) and database-retrieved values (customer names, site names, department names, driver names, vehicle names, comments, questions) are rendered using `<%= ... %>` expressions without HTML entity encoding. This allows attackers to inject malicious JavaScript that will execute in the context of other users' browsers when they view these reports. Given the multi-tenant nature indicated by the `multi-customer-sync-to-master` branch, XSS in report outputs could allow one customer to attack another customer's users or administrators.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**

### P76-F02: Missing CSRF Protection on State-Changing Operations
**Description:** None of the reviewed JSP files implement CSRF (Cross-Site Request Forgery) protection. While these files are primarily report generation/display pages triggered by GET requests with print() JavaScript, they retrieve and display sensitive multi-tenant fleet data based on request parameters. The underlying report filter beans are initialized with user-supplied parameters (customer_cd, location_cd, dept_cd, etc.) without any CSRF token validation. If any of these report endpoints or their upstream form handlers perform state changes or if they can be used to exfiltrate data through carefully crafted URLs, attackers could force authenticated users to generate reports for unauthorized customers or expose sensitive information. This is particularly concerning in a multi-tenant environment where cross-customer data access would be a critical security breach.

**Severity:** MEDIUM

**Checklist item:** 4. Session and CSRF

**Evidence:**

### P76-F04: Information Disclosure via Verbose Error Messages
**Description:** Multiple JSP files contain code paths that could trigger runtime exceptions without proper error handling. For example, if session attributes are null, casting operations like `Integer.parseInt(access_level)` on lines where `access_level` is retrieved from the session could throw `NumberFormatException`. Additionally, the report JSP files have auto-print JavaScript (`onload="print('document');"`) and some have window close behavior, but do not include visible error handling for cases where the filter bean initialization fails or data retrieval throws exceptions. If exceptions occur and the application is configured to display stack traces (common in legacy JSP applications without proper `<error-page>` configuration in web.xml), sensitive information about the application's internal structure, database schema, file paths, and Java class names could be exposed to users. This information aids attackers in crafting more targeted attacks.

**Severity:** LOW

**Checklist item:** 5. Data Exposure and Privacy - Error responses

### P76-F05: Hardcoded HTTP URLs in Report Output
**Description:** Several JSP files contain hardcoded HTTP (non-HTTPS) URLs for loading image resources. These URLs reference `http://fms.fleetiq360.com//images/linde/` paths. Hardcoded HTTP URLs in production code suggest the application may not be consistently enforcing TLS, potentially exposing sensitive data in transit.

### P76-F06: Dynamic Report Body Generation with Unvalidated Parameters
**Description:** The file `print_report.jsp` uses a generic dynamic report generation mechanism where it accepts `rpt_name` and `params` from request parameters and passes them directly to `ExcelUtil.getPrintBody(rpt_name, params)`. The method then outputs the returned body directly to the page without any visible validation or escaping. This pattern is highly risky as it creates a potential injection point where attackers could manipulate `rpt_name` to load arbitrary reports or manipulate `params` to inject malicious content. Without examining the `ExcelUtil` class implementation, it is unclear whether proper validation, authorization, and output encoding are performed. If `getPrintBody()` constructs SQL queries, file paths, or HTML based on the `params` input, this could lead to SQL injection, path traversal, or XSS vulnerabilities. The generic nature of this endpoint makes it a prime target for exploitation.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - SQL injection, XSS, File inclusion

**Evidence:**

print_report.jsp:

### P76-F08: Inconsistent Conditional Rendering Based on Customer Name
**Description:** The file `print_preop_chk_detailed.jsp` contains conditional logic that checks if the customer name equals "BMW" to determine whether to display certain columns (comment column). While not a direct security vulnerability, this pattern could lead to information disclosure if the conditional rendering logic is bypassed or if customer names are not properly validated.
