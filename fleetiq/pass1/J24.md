### J24-F01: SQL Injection — String Concatenation in `getEmailList()` (ChartsExcelDao)
**Description:** The `getEmailList()` method in `ChartsExcelDao` builds a SQL query by directly concatenating the `custCd` and `locCd` parameters into a `Statement.executeQuery()` call with no parameterisation. These values originate from `cust_cd` and `loc_cd` fields in `ChartDashboardUtil`, which are set by callers of this reporting system. Any path that allows user or external input to influence these values is a SQL injection vector.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — SQL injection
**Evidence:**

### J24-F06: Missing Customer Isolation Check — `cust_cd` and `loc_cd` Not Validated Against Authenticated Session
**Description:** `ChartDashboardUtil` accepts `cust_cd` and `loc_cd` as fields set on the object before `createExcel()` or `createTestChart()` is called. The class does not perform any internal check that these codes belong to the authenticated user. All data access in `ChartsExcelDao` methods is parameterised by whatever `cust_cd`/`loc_cd` values are set by the caller. In the multi-tenant context of this branch (`multi-customer-sync-to-master`), this is a critical concern. The class itself provides no tenant isolation enforcement.

### J24-F07: Exception Stack Traces Swallowed and Printed to stdout — No Structured Error Handling
**Description:** All exception handlers throughout `ChartsExcelDao` and `ChartDashboardUtil` call `e.printStackTrace()` and then silently continue (returning empty results or null). This means: (1) SQL exceptions are printed to `catalina.out` in plain text, potentially including SQL query text with embedded customer codes; (2) callers receive empty or partial results with no indication of failure, making the application silently unreliable; (3) no audit trail of errors is written to a structured log. On line 683–685 of `ChartDashboardUtil`, a `SQLException` from DAO calls is caught and swallowed entirely after `e.printStackTrace()`.

### J24-F09: External HTTP Call to Google Charts API Embeds Customer Fleet Data in URLs
**Description:** `BatteryChargeChart` (and by extension all chart classes in this package) generate chart URLs via the `charts4j` library, which calls the Google Charts API (`chart.toURLString()`). All chart data — battery charge counts per department, impact counts, driver activity metrics — is encoded as query parameters in the URL and transmitted to Google's servers as an outbound HTTP GET request initiated by the server. This constitutes SSRF-adjacent data leakage: internal customer fleet data (battery charge counts per department) is sent to an external third-party service. Depending on the environment's network configuration, this may also violate data residency or privacy obligations.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — SSRF / 5. Data Exposure and Privacy
**Evidence:**
