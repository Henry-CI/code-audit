## Findings

### J06-F01: ImpactBean fields are package-private — customer code field is accessible within the package without access control
**Description:** All fields in `ImpactBean` (and also `ImpactDeptBean` and `FleetCheckBean`) are declared without any access modifier, making them package-private. In particular, `cust_cd` in `ImpactBean` is the customer-scoping field for this multi-tenant application. Package-private access means any class in the same package (`com.torrent.surat.fms6.bean`) can read or write `cust_cd` directly, bypassing the getter/setter boundary entirely. This is a structural weakness: there is no enforcement that `cust_cd` is set to the authenticated user's customer code before the bean is populated and rendered. If a caller in the same package modifies or omits `cust_cd`, data belonging to a different customer could be returned without any guard. This risk is elevated because the branch under review (`multi-customer-sync-to-master`) explicitly targets multi-tenant use.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy (IDOR / multi-tenant data isolation)
**Evidence:**
```java
// ImpactBean.java lines 11-20
String cust_cd = "";
String loc_cd = "";
String dept_cd = "";
String dept_name = "";
String model_cd = "";
String model_name = "";
String driver_cd = "";
String driver_name = "";
String unit_cd = "";
String unit_name = "";
```
All fields are package-private (no `private` modifier). `cust_cd` at line 11 is the tenant-scoping key.
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/ImpactBean.java`
**Line(s):** 11–25

### J06-F02: FleetCheckBean does not implement Serializable — deserialization inconsistency in a multi-tenant session context
**Description:** `ImpactBean` and `ImpactDeptBean` both implement `java.io.Serializable` (with explicit `serialVersionUID` values), indicating these beans are serialized — most likely stored in the HTTP session or passed between tiers. `FleetCheckBean` does not implement `Serializable` and has no `serialVersionUID`. If `FleetCheckBean` is stored in a session alongside the other beans, or placed inside a serializable collection that is session-scoped, this will cause a `NotSerializableException` at runtime. More importantly from a security perspective, the inconsistency suggests `FleetCheckBean` was added or modified without the same design discipline as the other beans, which raises a concern that its session-lifecycle and customer-scoping may not have been reviewed with the same rigour as `ImpactBean`.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (session management); Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// FleetCheckBean.java lines 1-10
package com.torrent.surat.fms6.bean;

import java.util.ArrayList;

public class FleetCheckBean {

    String unitName = "";
    String avg_completion_time = "N/A";
    ArrayList<String>  frequent_failed_question = new ArrayList<String>();
```
No `implements Serializable`, no `serialVersionUID`. Compare with `ImpactBean.java` line 6: `public class ImpactBean implements Serializable` and line 10: `private static final long serialVersionUID = -3410929526546378632L;`
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/FleetCheckBean.java`
**Line(s):** 5–10

### J06-F03: ImpactBean carries driver PII fields with no access restriction or masking
**Description:** `ImpactBean` stores `driver_cd` and `driver_name` (lines 17–18) as plain package-private strings. Driver name is personally identifiable information (PII). The bean is serializable and is designed to be passed across layers of the application. There is no masking, redaction, or role-based field-level control: any consumer of the bean receives the full driver name unconditionally. In a multi-tenant deployment, if a bean is populated from a shared query that is not tightly scoped to `cust_cd`, driver PII from one customer's fleet could be exposed to another. The structural absence of access modifiers means this field can also be set directly by any class in the same package.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (driver data / PII exposure)
**Evidence:**
```java
// ImpactBean.java lines 17-18
String driver_cd = "";
String driver_name = "";
```
Both fields are package-private and carry no annotation or documentation indicating how they are scoped to a customer.
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/ImpactBean.java`
**Line(s):** 17–18

**Description:** The public method at line 42 is named `addImactMap` (missing the 'p' in "Impact"). This is a minor code quality issue on its own, but it is a marker of insufficient code review. In a multi-tenant security context, low code-review rigour raises the likelihood that security-critical logic (customer ID scoping, authorization checks in callers) was also not carefully reviewed. This is flagged as an INFO item to document the pattern, not as a standalone vulnerability.

## Scope Notes

These three files are data-transfer bean (POJO) classes. They hold no business logic, execute no queries, and make no authorization decisions themselves. The security findings above are structural and design-level. The critical authorization and multi-tenancy questions — whether `cust_cd` is correctly set from the authenticated session before these beans are populated, whether the beans are passed to JSP output without escaping, and whether callers scope their queries to the correct customer — cannot be confirmed from these files alone. The servlet and DAO classes that populate `ImpactBean` and the JSP files that render `driver_name`, `unit_name`, and related fields must be reviewed to confirm or rule out IDOR and XSS vulnerabilities.
