## Findings

### P77-F01: Cross-Site Scripting (XSS) - Unescaped User-Controlled Output
**Description:** All 10 JSP files output user-controlled or database-derived values using raw `<%= %>` expressions without HTML escaping. This allows attackers to inject JavaScript code through vehicle names, driver names, customer names, site names, or other data fields that are later rendered in these reports. For example, a malicious user with access to create/modify vehicle records could inject a vehicle name containing `<script>alert(document.cookie)</script>`, which would execute when the report is viewed by other users.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**

### P77-F03: Session Management - Missing Session Timeout Enforcement
**Description:** All files rely on session attributes (`access_level`, `access_cust`, `access_site`, `access_dept`, `user_cd`) for access control but do not explicitly verify that the session is still valid or has not expired. While session timeout may be configured in `web.xml`, these JSP files do not perform any session validity checks before processing sensitive fleet/driver data.

**Severity:** MEDIUM

**Checklist item:** 2. Authentication and Authorization - Session management

**Evidence:**
```jsp
// print_serv_maintenance.jsp, lines 26-29
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");

// No validation that session is valid or that these attributes are not null before use
```

**File:** All 10 files
**Line(s):** Various lines where session attributes are retrieved without validation

### P77-F04: Authorization - Insufficient Multi-Tenant Data Isolation Verification
**Description:** While all files retrieve access control session variables (`access_cust`, `access_site`, `access_dept`) and pass them to the filter bean, the JSP code does not verify that the returned data actually respects these boundaries. The files blindly trust that the `Databean_report.init()` method properly enforces customer isolation. There is no defensive check in the presentation layer to ensure cross-customer data leakage cannot occur. Given the branch name `multi-customer-sync-to-master`, this is a critical concern.

**Severity:** HIGH

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
```jsp
// print_serv_maintenance.jsp, lines 37-40
filter.setAccess_level(access_level);
filter.setAccess_cust(access_cust);
filter.setAccess_site(access_site);
filter.setAccess_dept(access_dept);

// Lines 95-107: Data retrieval - no verification that returned data belongs to access_cust
veh_cd = filter.getVmachine_cd();
veh_nm = filter.getVmachine_nm();
```

**File:** All 10 files
**Line(s):** Lines where access attributes are set and data is retrieved without verification

### P77-F05: Error Handling - Null Pointer Risk from Session Attributes
**Description:** Multiple files perform string operations on session attributes that could be null. While there are default assignments for some attributes (e.g., `access_level` defaults to "5" if empty), null values are not consistently handled. For example, `access_level.equalsIgnoreCase("")` on line 31 will throw a NullPointerException if `access_level` is null rather than empty string.

**Severity:** MEDIUM

**Checklist item:** 5. Data Exposure and Privacy - Error responses

**Line(s):** Lines 26-35 and similar patterns in all files

### P77-F06: Input Validation - Unvalidated Request Parameters Used in Business Logic
