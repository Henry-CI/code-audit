### J02-F01: BroadcastmsgBean lacks customer/tenant identifier — IDOR risk in multi-tenant context
**Description:** `BroadcastmsgBean` carries message data including `veh_id`, `driver`, `unit`, `text`, and timestamps but contains no `customer_id` or equivalent tenant-scoping field. In a multi-tenant fleet management system, broadcast messages are customer-specific. Without a customer identifier embedded in the bean, any code that populates lists of these beans from a query must enforce tenant scoping at the query level alone. If any query or method that populates this bean fails to filter by customer, messages from other tenants will be silently returned. The branch name `multi-customer-sync-to-master` confirms multi-tenancy is active.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy (IDOR / multi-tenant data isolation)

### J02-F02: CanruleBean access-scope fields are plain mutable Strings with no enforcement
**Description:** `CanruleBean` carries four access-control fields: `access_level`, `access_cust`, `access_site`, and `access_dept` (lines 14–17). These fields are plain `String` types with unrestricted public setters. There is no validation, no enum type, no immutability, and no indication that these values are verified server-side after being set. If any code path populates these fields from a request parameter (URL query string, form POST) rather than from a trusted server-side session, an attacker could supply arbitrary values and escalate their access scope across customers, sites, or departments. The concern is amplified because the branch represents multi-customer functionality and `custCd` (line 13) and `access_cust` (line 15) both exist, suggesting access filtering is implemented through these string fields at runtime.

### J02-F03: DailyUsageDeptDataBean lacks customer/tenant identifier — cross-tenant data leakage risk
**Description:** `DailyUsageDeptDataBean` holds daily usage statistics per department (`dept_id`, `dept_name`, `modelName`, `data`, `loc`, `week`). It contains no `customer_id` or equivalent field. In a multi-tenant deployment, departments belong to specific customers and their usage data is confidential. Without a customer identifier embedded in the bean, the only protection against returning one customer's departmental data to another customer is that the populating query must be correctly scoped. There is no defence-in-depth: if a bug or SQL injection bypasses query-level scoping, the application has no secondary check at the bean or presentation layer to detect the wrong customer's data is being returned.

### J02-F04: DailyUsageDeptDataBean.getData() returns mutable internal collection without defensive copy
**Description:** The `getData()` method at line 55 returns the internal `ArrayList<double[]>` directly. Any caller can mutate the list (add, remove, or replace elements) without going through the bean's setter, bypassing any future validation that might be added to `setData()`. While this is primarily a data integrity issue, in a multi-tenant context mutation of a shared or cached bean could cause usage data from one tenant's request to bleed into another tenant's response if bean instances are reused across requests (e.g., stored in application scope or a shared cache).

### J02-F05: DailyUsageDeptDataBean implements Serializable without serialVersionUID
**Description:** `DailyUsageDeptDataBean` implements `java.io.Serializable` but declares no `serialVersionUID` field. The JVM will auto-generate one based on class structure, meaning any change to the class (adding a field, changing method signatures) will produce a different implicit `serialVersionUID` and cause `InvalidClassException` when deserializing previously serialized instances — for example from HTTP session persistence, distributed caches, or JVM restarts with persisted sessions. This is not itself a direct security vulnerability, but if serialized bean instances from one version of the class are accepted by a different version, unexpected field handling could occur. More importantly, if the application deserializes this bean from an untrusted source (e.g., a cookie or request parameter), the absence of strict type control compounds deserialization risk.
**Severity:** INFO
**Checklist item:** Section 7 — Dependency Vulnerabilities (deserialization hygiene)
**Evidence:**
```java

// No serialVersionUID declared anywhere in the file
```

**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/DailyUsageDeptDataBean.java`
**Line(s):** 6

### J02-F06: BroadcastmsgBean driver field carries PII with no access-control metadata
**Description:** The `driver` field (line 17) in `BroadcastmsgBean` carries driver identity information. Driver names or identifiers are personally identifiable information (PII) in a fleet management context. The bean carries no role or access-level field to indicate which user roles are permitted to see this value. If this bean is serialized to JSON or XML in an API response, the driver field will be included unconditionally. The concern is the absence of any field-level access annotation or documentation indicating that driver data should be filtered for lower-privileged roles.
