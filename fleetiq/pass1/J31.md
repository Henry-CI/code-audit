### J31-F04: Race condition / shared mutable state on servlet instance fields in BatteryCharge
**Description:** `BatteryCharge.java` declares `rsList`, `dateList`, and `cusList` as instance-level (non-static) fields on the servlet (lines 34–36). Tomcat creates only one servlet instance per `@WebServlet` registration and serves all concurrent requests through it. Two simultaneous requests will therefore read and write the same `rsList` and `dateList` objects concurrently — a `getTable()` call for user A may return data from a `getDates()` call belonging to user B. This is a data-leakage race condition between sessions.

Note: `CriticalBattery.java` corrects exactly this flaw by using `ConcurrentHashMap` keyed by session ID (lines 40–82), confirming the developers were aware of the problem. The fix was applied to `CriticalBattery` but not backported to `BatteryCharge`.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization — multi-tenancy / data isolation
**Evidence:**
```java
// BatteryCharge.java lines 34-36: instance fields, shared across all concurrent requests
LinkedList<HashMap<String, String>> cusList = new LinkedList<HashMap<String, String>>();
LinkedList<HashMap<String, String>> rsList = new LinkedList<HashMap<String, String>>();
LinkedList<HashMap<String, String>> dateList = new LinkedList<HashMap<String, String>>();

// getDates() writes to rsList (line 136): rsList = Config.saveasList(query);
// getTable() reads from rsList (line 251): for (Map<String, String> entry : rsList)
// No synchronisation exists between these operations.
```
Compare with CriticalBattery.java lines 40-42 (the corrected approach):
```java
private static final ConcurrentHashMap<String, LinkedList<...>> rsListMap = new ConcurrentHashMap<>();
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/dashboard/BatteryCharge.java`
**Line(s):** 34–36, 135–136, 249–261

### J31-F06: No authentication check in any servlet doGet() handler
**Description:** Neither `BatteryCharge.doGet()` nor `CriticalBattery.doGet()` performs any authentication check before processing requests. The code reads session attributes (e.g., `dealer_id`, `access_level`) but does not verify that a session exists or that the user is authenticated before using them. A request with no valid session will cause a `NullPointerException` rather than an `HTTP 401` or redirect to login. If session attributes can be null (unauthenticated request), the code proceeds to execute SQL queries using those null values. There is no `isUserLoggedIn()` guard, no check for a non-null session attribute, and no call to `request.getSession(false)` to detect a missing session safely.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization
**Evidence:**

### J31-F07: Insufficient multi-tenant scoping — cust/site/dept filter bypass at access_level 0 or 1
**Description:** In both `BatteryCharge.getDates()` and `CriticalBattery.getDates()`, the vehicle filter SQL appends customer, site, and department conditions only when `level > 1`, `level > 2`, or `level > 3` respectively, OR when the request parameter is not "all". This means: if a user has `access_level = 1` and `sessionCust != 0`, and they send `cust_cd=all` in the request, the customer condition is not added (lines 94-96 in BatteryCharge, lines 140-142 in CriticalBattery). A user at level 1 with `sessionCust != 0` should not have access to all-customer data.

## Checklist Items With No Issues

The following checklist items were reviewed against the three assigned files and no findings were identified:

- **1. Secrets and Credentials:** No hardcoded database passwords, API keys, JDBC connection strings with embedded credentials, or `.env`-style secrets found in the source of these three files. Database access is via JNDI `DataSource` lookup using `RuntimeConf.penom_database`, which is an externally configured reference. The commented-out values (J31-F11) are access-control defaults, not credentials.

- **3. Injection — File inclusion:** No `<jsp:include>` with user-controlled paths (these are Java servlet files, not JSPs). No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage found.

- **3. Injection — Command injection:** No command execution of any kind found in any of the three files.

- **3. Injection — SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) found in these files. No external URL construction.

- **6. Transport Security:** No Tomcat connector configuration present in these files. TLS configuration is out of scope for these source files specifically.

- **7. Dependency Vulnerabilities:** Cannot be assessed from these three source files alone. No `pom.xml` or `build.xml` was assigned to this agent.
