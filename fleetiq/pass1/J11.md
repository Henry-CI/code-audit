## Findings

### J11-F01: PreCheckSummaryBean fields are package-private (missing access modifiers)
**Description:** All five fields in `PreCheckSummaryBean` (`loc_cd`, `loc_name`, `arrPrecheck`, `index`, `total`) are declared without an explicit access modifier, making them package-private. This means any class within the same package (`com.torrent.surat.fms6.bean`) can read and write these fields directly, bypassing the public getter/setter interface. In a multi-tenant application this creates a risk that a bug or future code in the same package could set fields — including the location code `loc_cd` — without going through validated setter paths. While not a direct exploitation vector by itself, it violates encapsulation in a data-carrying object that participates in pre-check (vehicle safety check) report assembly.
**Severity:** LOW
**Checklist item:** Section 2 — Authorization and multi-tenancy (encapsulation of tenant-scoped data)
**Evidence:**
```java
// Lines 8–12 — no access modifiers
String loc_cd = "";
String loc_name = "";
ArrayList<PreCheckBean> arrPrecheck = new ArrayList<PreCheckBean>();
int index = 0;
int total = 0;
```

**Line(s):** 8–12

### J11-F02: QuestionBean exposes access-control fields (access_level, access_cust, access_site, access_dept) as plain mutable Strings with no validation
**Description:** `QuestionBean` carries four fields that appear to encode a multi-tenant access-control context: `access_level`, `access_cust`, `access_site`, and `access_dept`. These fields are plain nullable `String` values with no type safety, no enumeration of valid values, and no validation in the setters — any caller can set any value. The field `custCd` (customer code) similarly defaults to `null`. If these fields are populated from request parameters or other user-controlled input before being passed to a data-access layer, they could be manipulated to escalate access across tenant boundaries. The risk depends on how the calling servlet or DAO uses these fields; the bean itself provides no protection. This is a design-level finding: access-control attributes should not be plain mutable strings on a data-transfer bean.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy (customer ID scoping; IDOR risk)
**Evidence:**

### J11-F03: RestrictedAccessUsageBean name implies restricted access but the bean carries no tenant/customer identifier
**Description:** The class is named `RestrictedAccessUsageBean`, which suggests it is used in a context requiring restricted (likely customer-scoped) access. However, the bean contains only operational data (`fleetNo`, `serialNo`, `hourStart`, `hourFinish`, `servHoursFrom`, `totalUsage`) and no `customerId`, `custCd`, `tenantId`, or equivalent tenant-scoping field. If this bean is serialized to a response or rendered directly in a JSP without the caller enforcing customer-level scoping externally, there is a risk of cross-tenant data leakage — one customer's fleet usage data could be returned to another customer's session. The bean design provides no mechanism to carry or enforce the tenant boundary. This must be verified against the servlets and JSPs that populate and consume this bean.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy (verify that every data access is scoped to the authenticated user's customer/organisation); Section 5 — Data Exposure and Privacy

### J11-F05: QuestionBean carries question text and expected/critical answer values as unvalidated Strings
**Description:** `QuestionBean` holds `question` (the question text), `exp_ans` (expected answer), and `crit_ans` (critical answer) as plain mutable Strings. If these values are written to JSP output without escaping (e.g., via `<%= bean.getQuestion() %>`), they are XSS vectors. The concern is amplified because these values are likely sourced from a database and could include content entered by administrators or other users. The bean itself has no sanitization. The actual XSS risk depends on how JSPs render these fields, but the absence of any escaping contract in the bean is a design-level gap that should be confirmed in the consuming JSPs.
**Severity:** MEDIUM
**Checklist item:** Section 3 — XSS (Cross-Site Scripting); check every `<%= variable %>` that could originate from user input or database values
**Evidence:**
