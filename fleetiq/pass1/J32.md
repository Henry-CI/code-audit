### J32-F01: SQL Injection via String Concatenation of User-Supplied Parameters — Impacts.java

No `PreparedStatement` with `?` placeholders is used anywhere in the file. An attacker controlling any of these parameters can inject arbitrary SQL, including data exfiltration across all customers, DROP TABLE, or execution of database-specific system commands.

**Checklist item:** Section 3 — SQL Injection

### J32-F04: Race Condition / Cross-Tenant Data Leakage via Shared Servlet Instance Fields — Pct_utilisation.java
**Description:** `Pct_utilisation` stores query results and request-derived state as instance fields on the servlet object: `rsList`, `dateList`, `cusList`, `vehicles`, `vehNum`, `drivers`, `oneDay`, `dealer`, `level`, `sessionCust`, `thresholdLow`, `thresholdHigh`. Servlets are singletons: Tomcat creates one instance and services all concurrent requests through it. A request from Customer A calling `procedure()` can overwrite `rsList` mid-flight, causing a subsequent `getTable()`, `getPie()`, or `updateTable()` call from Customer B to read Customer A's data. This is a cross-tenant data exposure vulnerability. By contrast, `Impacts.java` and `Licence.java` correctly use per-session `ConcurrentHashMap` to isolate data.

**Severity:** CRITICAL

### J32-F05: No Authentication Check in Any of the Three Servlets
**Description:** None of the three servlets (`Impacts`, `Licence`, `Pct_utilisation`) perform an explicit authentication check before processing requests. They read session attributes (`dealer_id`, `access_level`, `access_cust`) and use them to scope queries, but they do not first verify that an authenticated session exists. If `req.getSession().getAttribute("dealer_id")` returns null (unauthenticated or expired session), the call to `.toString()` at line 129 (Impacts), 94 (Licence), or 49 (Pct_utilisation) will throw a NullPointerException, which is caught by the generic `catch (Exception e)` block and swallowed silently with a `System.out.println`. This means an unauthenticated request results in an empty or malformed JSON response rather than an HTTP 401/403 redirect. Whether this is exploitable depends on what container-level security constraints are configured in `web.xml`, but no defence-in-depth check exists at the servlet level. The reliance on session attributes as the sole access control mechanism without null-checking is a pattern inconsistent with secure design.

### J32-F06: Incomplete Multi-Tenant Scoping in Licence.java — getChart() Does Not Check Access Level
**Description:** `Impacts.java` and `Pct_utilisation.java` read the session attributes `access_level` and `access_cust` and use them to conditionally restrict queries to a specific customer, site, and department based on the user's permission tier. `Licence.java`'s `getChart()` method reads `dealer_id` from the session but does not read `access_level` or `access_cust` at all. It relies solely on the URL parameters `cust_cd`, `loc_cd`, and `dept_cd` to scope the driver query. If the UI passes "all" for these parameters (or if a user crafts a request omitting them), the query falls through to the defaults of "all", and no level-based scoping is applied. A lower-privilege user who knows the URL can bypass their access level restriction and retrieve licence expiry data for all drivers under the dealer — across all customers.

**Severity:** HIGH

**Checklist item:** Section 2 — Authorization and Multi-Tenancy; IDOR

**Evidence:**
```java
// Licence.java, getChart(), lines 93–118
```

Compare this to `Impacts.getVehicle()` lines 129–133, which reads `access_level` and `access_cust` and adjusts the effective level to enforce stricter scoping automatically. `Licence.getChart()` has no equivalent logic.

### J32-F07: Driver PII (Full Names, Licence Data) Written to Stdout via System.out.println — Licence.java

### J32-F08: Stack Traces Printed to stdout — Potential Information Disclosure
**Description:** All three servlets call `e.printStackTrace()` in catch blocks throughout the codebase. `printStackTrace()` writes the full Java stack trace to `System.err`, which in a default Tomcat deployment is merged into `catalina.out`. Stack traces expose internal package names, class names, method call chains, and the SQL engine's error messages (which often include the fragment of SQL that caused the error — including injected content). If an attacker is probing for SQL injection, the error messages in `catalina.out` confirm whether their injection syntax was parsed. While the stack traces are not returned in HTTP responses (the catch blocks do not write to `res`), they are present in server logs. This is an information disclosure risk in environments where log access is broader than it should be.

**Severity:** LOW

**Checklist item:** Section 5 — Data Exposure and Privacy (Error Responses)

**Evidence:**
```java
// Representative example — Impacts.java, lines 258–260
} catch (Exception e) {
    System.out.println(servlet + ": " + e);
    e.printStackTrace();
}
```

The same `e.printStackTrace()` pattern is present in every catch block in all three files: Impacts.java lines 259–260, 265–266, 271–272, 581, 916–917; Licence.java lines 175–176, 297; Pct_utilisation.java lines 165, 950–951.

### J32-F09: No CSRF Protection in Any of the Three Servlets
**Description:** All three servlets handle state-reading operations via HTTP GET, which is generally acceptable for read-only JSON data endpoints. However, the `updateTable` operations (Impacts line 103, Licence part="update") and the threshold-setting operations in Pct_utilisation (`ct_model`, `ct_date`, `ct_location`) also use GET. These are state-changing in the sense that they alter the session-cached result set used for subsequent table renders (`rsList`, `dateList`). There is no CSRF token mechanism visible in any of these servlets. If any of these endpoints trigger consequential server-side state changes (e.g., in conjunction with other servlets not reviewed here), they would be CSRF-vulnerable. The threshold update in `updateDateChart()` (line 626) writes to instance fields `thresholdLow` and `thresholdHigh`, which due to F04 are shared across all sessions — a GET request from an attacker-controlled page could manipulate thresholds for all users simultaneously.

**Severity:** MEDIUM

### J32-F10: Pct_utilisation Access Level Scoping Is Conditional on Exact Parameter Value — Potential Bypass
**Description:** In `Pct_utilisation.getVehicleData()` and `getDriverData()`, customer/site/department scoping is applied only when the parameter is not the string "all". The check `!cust.equals("all")` (not `equalsIgnoreCase`) means the comparison is case-sensitive. `Impacts.java` uses `equalsIgnoreCase("all")`. If an attacker submits `cust_cd=All` or `cust_cd=ALL`, the condition `!cust.equals("all")` evaluates to true (not equal to "all"), causing the customer filter to be applied as `and "USER_CD" = All` — which in PostgreSQL would be interpreted as a column reference or cause a syntax error, not necessarily a security bypass. However, the inconsistency between case-sensitive and case-insensitive "all" checks across the codebase is a code quality and potential correctness risk that could be exploited if the database or query context handles the unrecognised value in an unexpected way. More critically, it means the scoping logic is not uniformly applied across all data endpoints.
