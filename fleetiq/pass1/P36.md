### P36-F01: Reflected XSS via unescaped `message` request parameter
**Description:** In both `site_hours.jsp` and `update_site_hours.jsp`, the `message` variable is initialised from `request.getParameter("message")` and later rendered directly into the HTML page with `<%=message%>` inside a `<p>` tag. Although `site_hours.jsp` overwrites the variable with `filter.getSiteHourMessage()` (line 200), `update_site_hours.jsp` does not perform this overwrite and uses the raw parameter value at line 189. If an attacker can construct a URL with a malicious `message` parameter, arbitrary HTML and JavaScript is injected into the page. Even where the overwrite occurs, the initial assignment still accepts the raw input.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// update_site_hours.jsp, line 73:
String message = request.getParameter("message") == null ? "" : request.getParameter("message");

// update_site_hours.jsp, line 189:
<p style="color: red; font-weight: bold; font-size: 16px" align="center"><%=message%></p>

// site_hours.jsp, line 106:
String message = request.getParameter("message") == null ? "" : request.getParameter("message");
// site_hours.jsp, line 227:
<p style="color: red; font-weight: bold; font-size: 16px" align="center"><%=message%></p>
```
**File:** pages/admin/customer/update_site_hours.jsp, pages/admin/customer/site_hours.jsp
**Line(s):** update_site_hours.jsp:73,189; site_hours.jsp:106,227

### P36-F07: Absence of CSRF protection on all state-changing forms
**Description:** All three JSP files that contain forms posting to servlets (`site_hours.jsp`, `site_setup_config.jsp`, `update_site_hours.jsp`) lack any CSRF token.
