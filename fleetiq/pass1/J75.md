## Findings

### J75-F01: SQL Injection via string concatenation in `getDriverOrSupervisorName`
**Description:** The method `getDriverOrSupervisorName` in `DriverUtil` builds a SQL query by directly concatenating the `userCd` parameter into the query string using `Statement.executeQuery()`. If `userCd` is derived from any user-supplied or externally-supplied input (e.g., a request parameter, a parsed message field, or a value from another table), an attacker can inject arbitrary SQL. The method is `public static` and callable from any servlet or JSP in the application.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — SQL Injection
**Evidence:**
```java
// DriverUtil.java lines 91–99

        + "   ELSE ''  "
        + " END ||  "
        + " COALESCE(\"CONTACT_LAST_NAME\", '') "
      + " from \"FMS_USR_MST\" WHERE \"USER_CD\" = '" + userCd + "'";
rset = stmt.executeQuery(queryString);
```

### J75-F02: SQL Injection via dynamic WHERE clause construction in `checkInterval`
**Description:** In `DriverMedicalAlert.checkInterval()`, the variables `cust_cd`, `loc_cd`, and `dept_cd` are retrieved from the database (`mymessages_users` table via `getAlertlist()`) and appended as raw string literals into a SQL WHERE clause without parameterisation. While these values come from the database rather than directly from a web request, they are not sanitised or parameterised. A value stored in the database containing SQL metacharacters (e.g., a single quote, comment sequence, or UNION payload) would be interpreted as SQL syntax when the query is executed. This is a second-order SQL injection vulnerability. The same `stmt` object is also reused across queries within the method, which is an additional code quality concern.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — SQL Injection
**Evidence:**

### J75-F03: SQL Injection in email INSERT statements — `email` and `body` values concatenated directly
**Description:** In `DriverMedicalAlert.checkInterval()`, the `email` address (from the database) and the constructed `body` string (which includes `driver_name` and `medical_date` from the database, as well as `subject`, `paragraph1`, `paragraph2`, and `signature` values also from the database) are concatenated directly into SQL INSERT statements. The `body` contains driver names read from `FMS_USR_MST` — if a driver name contains a single quote, the INSERT will fail or, if exploited, execute injected SQL. ### J75-F05: Missing multi-tenant scoping — `getAlertlist()` retrieves records across all customers
**Description:** The `getAlertlist()` method queries `mymessages_users` with no customer-scoping predicate. The query joins to `FMS_USR_MST` but does not filter by any customer identifier. On a multi-tenant deployment (as indicated by the branch name `multi-customer-sync-to-master`), this means the alert job will process notification configurations belonging to every customer in the database in a single pass. Combined with the dynamic WHERE clause in `checkInterval()` (F02), the cust_cd values from the result set are the only tenant boundary — if these values are empty or zero, the driver query runs unscoped against all tenants' driver records. There is no explicit assertion that a `cust_cd` of `""` or `"0"` correctly restricts cross-customer data access; the guard conditions merely skip appending the WHERE clause term, causing the query to return drivers across all customers.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy
**Evidence:**

### J75-F06: Stack traces printed to stdout — information leakage via `e.printStackTrace()`
**Description:** Both `checkInterval()` and `getAlertlist()` catch all `Exception` types and call `e.printStackTrace()`. In a Tomcat deployment, `printStackTrace()` writes to `catalina.out` or the console, which may be accessible via log aggregation, shared hosting, or misconfigured logging infrastructure. Stack traces expose internal class names, method names, line numbers, and SQL error messages (which may include table names, column names, or partial query text). Similarly, `getDriverOrSupervisorName` in `DriverUtil` calls `e.printStackTrace()` in its catch and finally blocks. The comment `// TODO Auto-generated catch block` confirms these were IDE-generated stubs never properly reviewed.
**Severity:** MEDIUM
**Checklist item:** 5. Data Exposure and Privacy — Error responses
**Evidence:**

### J75-F07: `SimpleDateFormat` instance used as a static field — thread safety issue
**Description:** `DriverMedicalAlert` declares `sdf` as a `private static SimpleDateFormat`. `SimpleDateFormat` is not thread-safe. If `checkInterval()` is invoked concurrently (e.g., by a scheduler firing overlapping executions, or in a multi-threaded context), concurrent calls to `getDate()` via the shared `sdf` instance can produce incorrect date strings or throw exceptions. Incorrect date comparisons could cause medical alert emails to be suppressed or sent to wrong recipients, which is a data integrity and privacy concern in the context of driver medical certification expiry notifications.
**Severity:** MEDIUM
**Checklist item:** 5. Data Exposure and Privacy
**Evidence:**
```java
// DriverMedicalAlert.java line 26
private static SimpleDateFormat sdf = new SimpleDateFormat("dd/MM/yyyy");

// line 190

    return sdf.format(cal.getTime());
}
```

**File:** `WEB-INF/src/com/torrent/surat/fms6/util/DriverMedicalAlert.java`

## Checklist Items With No Issues

The following checklist items were reviewed against the three files in scope and no issues were identified:

1. **Secrets and Credentials (Section 1):** No hardcoded credentials, passwords, API keys, or connection strings are present in any of the three files. Database connections are obtained via `DBUtil.getConnection()` with no visible credential embedding.

2. **XSS / Cross-Site Scripting (Section 3 — XSS):** No JSP output expressions are present in these utility classes. These are pure Java classes with no direct HTTP response output.

3. **File Inclusion (Section 3 — File inclusion):** No file path construction or JSP include directives are present in these files.

4. **Command Injection (Section 3 — Command injection):** No calls to `Runtime.getRuntime().exec()` or `ProcessBuilder` are present in any of the three files.

5. **SSRF (Section 3 — SSRF):** No HTTP client calls or URL construction from external input are present in any of the three files.

6. **Session and CSRF (Section 4):** These are utility/backend classes with no session management or form handling. CSRF does not apply to these files directly.

7. **Transport Security (Section 6):** No HTTP/TLS configuration is present in these files. Transport security must be assessed at the container/configuration level, which is outside the scope of these three files.

8. **Dependency Vulnerabilities (Section 7):** No third-party library usage beyond standard JDK and javax.mail is present. The javax.mail imports are unused (see F04). No version-identifiable dependencies are introduced by these files directly.

9. **`getDriverName` in DriverUtil — parameterised correctly:** This method uses `CallableStatement` with `?` placeholders for both input parameters and correctly handles null return values. No SQL injection risk identified in this method.
