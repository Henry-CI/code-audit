### J81-F01: Multiple Hardcoded Plaintext Passwords in RuntimeConf.java
**Description:** `RuntimeConf.java` contains at least three distinct sets of hardcoded plaintext passwords committed directly to source control. These include a firmware server password (`ciifirmware`), a secondary firmware/FTP password (`Sdh79HfkLq6`), and an application-level test credential (`testadmin`). All are stored as `public static` fields, making them globally accessible throughout the application at runtime with no access control whatsoever. The field `pass` is annotated in a comment as "old firmware password" and `user` as "old firmware user name," suggesting these may be deprecated — but they remain live in source code and are committed to the repository. There is also a separate `firmwarepass` field with a different value, indicating multiple credential sets exist simultaneously. Any developer with repository access, or any attacker who achieves source disclosure, obtains all credentials immediately.
**Severity:** CRITICAL
**Checklist item:** Section 1 — Secrets and Credentials
**Evidence:**
```java

public static String server="fms2.ciifm.com";//old firmware server
```

### J81-F03: SQL Injection via String Concatenation of User-Controlled Input — set_user_cd
**Description:** In `Menu_Bean1.fetchform_rights()`, the field `set_user_cd` is concatenated directly into a SQL query string with no parameterisation and no sanitisation. `set_user_cd` is a `public` field with a public setter (`setSet_user_cd()`), meaning any caller — including JSP pages handling HTTP requests — can supply an arbitrary value. The resulting query is executed on a `Statement` (not a `PreparedStatement`), making this a straightforward SQL injection point. An attacker can terminate the quoted string and append arbitrary SQL, up to and including database-level administrative commands depending on the database user's privileges.
**Severity:** CRITICAL
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL injection)
**Evidence:**
```java
// Menu_Bean1.java line 73
query="select \"GROUP_CD\" from \"FMS_USR_GRP_REL\" where \"USER_CD\" = '"+set_user_cd+"'";
rset=stmt.executeQuery(query);
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/Menu_Bean1.java`
**Line(s):** 73–74

**Description:** At the end of `fetchSubModule()`, the raw SQL query string `query1` (which contains the complete SQL statement including any injected content) is added directly into the `FormName` ArrayList that is returned to callers. ### J81-F06: No Customer/Tenant Scoping in Menu and Access-Rights Queries
**Description:** The `Menu_Bean1` class constructs menu items and access rights for a user based solely on `set_user_cd` (the user code) and the derived `gp_cd` (group code). None of the queries in `fetchform_rights()`, `fetchMenuAttr1()`, or `fetchSubModule()` include a customer ID or tenant ID predicate. In a multi-tenant system (as indicated by the branch name `multi-customer-sync-to-master`), this means that if two users from different customer organisations share the same group code, one customer's user could be presented with — and potentially gain access to — menu items and forms belonging to another customer. The `RuntimeConf.DEFAULTCUSTOMER` field is set to a hardcoded static value `"CI"` rather than being scoped per-session, further indicating that tenant isolation is not enforced at the query level.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy)
**Evidence:**

### J81-F08: SMS API and Logo Images Loaded over Plain HTTP (No TLS)
**Description:** The Clickatell SMS API endpoint is referenced as `http://api.clickatell.com` (line 89), and both logo image URLs use plain `http://` (lines 48–49). Calls to the SMS API over plain HTTP expose the API credentials (username, password, API ID) and message content to network interception. In a fleet management context where SMS may be used for driver alerts or authentication, this is a serious risk. The HTTP logo image URLs also represent mixed-content on any HTTPS-served page, which modern browsers will block, and they indicate the application was not designed with HTTPS-everywhere as a baseline.

### J81-F10: Exception Stack Traces Printed to System.out
**Description:** All exception handlers in `Menu_Bean1` print stack traces to `System.out` via `e.printStackTrace()` and `System.out.println()`. In a Tomcat deployment, `System.out` is typically redirected to `catalina.out`, which may be accessible to operations staff but should not surface to end users. However, the practice of printing full stack traces — including SQL query strings (which in this codebase contain user-supplied data) — to the application log risks exposing SQL structure, table names, column names, and potentially injected payload content in log files. This also applies to the exception handlers that swallow errors silently after printing to stdout, masking failures from any proper error monitoring system.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J81-F11: Hardcoded Firmware FTP Port Using Non-Standard Value
**Description:** The firmware FTP port is hardcoded to `"211"` (line 25) for the external firmware server. Standard FTP is port 21, and the internal port is correctly set to `"21"` (line 26).
