## Findings

### J10-F01: Package-Private Fields in Serializable Beans Allow Uncontrolled Direct Access
**Description:** In both `PreCheckBean` and `PreCheckDriverBean`, all data fields are declared without an access modifier, making them package-private. This means any class in the same package (`com.torrent.surat.fms6.bean`) can read or write these fields directly, bypassing the public accessor/mutator methods entirely. In combination with `Serializable`, this pattern allows unintended field mutation during deserialisation or via reflection-based frameworks without enforcing encapsulation. For a bean carrying driver names, driver codes, department IDs, and check completion counts, this undermines any future attempt to add validation in setters.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J10-F03: Driver PII Carried in Serializable Bean With No Field-Level Protection
**Description:** `PreCheckDriverBean` stores `driver_name` and `driver_cd` (driver code/identifier) as package-private String fields in a `Serializable` class. Because the class is `Serializable` and its fields are not `transient`, these values will be included in any Java serialisation output (e.g. written to session, cached to disk, or transmitted over object streams). If the serialised form of these objects is ever stored externally or transmitted, driver PII travels with it without any explicit marking or protection. The serialVersionUID values are explicitly declared, which is correct practice, but the absence of `transient` on PII fields means the data is unconditionally serialised.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**
```java

String driver_cd = "";
```

### J10-F04: NotificationSettingsBean Fields `header`, `title`, `content`, `signature` Are Free-Text Strings With No Sanitisation Enforcement at Bean Level
**Description:** `NotificationSettingsBean` carries four free-text String fields: `header`, `title`, `content`, and `signature`. These are notification template fields that, by their nature, are likely populated from user input or database-stored values and subsequently rendered in notification emails or on-screen. The bean itself applies no sanitisation, length constraints, or encoding. This is expected for a pure DTO; however, the absence of any validation layer at the bean level means security entirely depends on all callers correctly escaping output before rendering. If any JSP or template engine renders these fields without escaping (e.g. raw `<%= bean.getContent() %>`), the result is stored XSS. This finding flags the structural risk; the actual XSS surface must be confirmed in the consuming JSP/servlet files.
**Severity:** MEDIUM
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```java
// NotificationSettingsBean.java lines 5-10

private String header = "";
private String title = "";
private String content = "";
private String signature = "";
private boolean enabled = false;
```
Setters at lines 15, 22, 28, 34, 40 accept arbitrary String values without validation. No sanitisation method exists in the class.
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/NotificationSettingsBean.java`
**Line(s):** 5-10, 15, 22, 28, 34, 40

### J10-F05: No Customer/Tenant Scoping Field in Any of the Three Beans
**Description:** None of `NotificationSettingsBean`, `PreCheckBean`, or `PreCheckDriverBean` contains a `customer_id`, `tenant_id`, `org_id`, or equivalent field to bind the data record to a specific customer or organisation. This is a multi-tenant application (`multi-customer-sync-to-master` branch). Beans used in report generation (`PreCheckBean`) and notification configuration (`NotificationSettingsBean`) carry no inherent tenant scope. This means that customer isolation — ensuring one customer's pre-check reports and notification settings cannot be accessed by another customer — must be enforced entirely by the data access layer and the servlets/DAOs that populate these beans. There is no structural guarantee at the bean level. If any caller omits the customer filter when loading data into these beans, cross-tenant data leakage will occur silently with no indication from the bean itself.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and Multi-Tenancy
