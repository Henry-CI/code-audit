### P47-F01: Unvalidated email address injected into JSON response body
**Description:** In all five files, the `mail_id` request parameter is read directly from user input and inserted unsanitised into a JSON string literal via string concatenation, which is then written to the HTTP response. An attacker can supply a `mail_id` value containing double-quote characters, backslashes, or other JSON metacharacters to break out of the JSON string and inject arbitrary JSON structure. Because the response is consumed by the calling page's JavaScript to display a status message, this is also a stored-or-reflected XSS vector depending on how the client handles the JSON payload.
**Severity:** HIGH
**Checklist item:** Section 3 â XSS (Cross-Site Scripting)

### P47-F02: No authentication or session validity check before executing report and sending email
**Description:** None of the five JSP files perform any check that the caller is authenticated before executing the report pipeline and sending email. The files read `access_level` from the session, but they do not verify that the session is valid or that the user is logged in before proceeding. If the session attribute `access_level` is null, the call to `access_level.equalsIgnoreCase("")` will throw a `NullPointerException`, but there is no guard that redirects unauthenticated users. An unauthenticated request that happens to carry a session with `access_level` set (e.g. a session created by another page) will proceed to generate and email a report. There is no explicit authentication gate such as `if (session.getAttribute("user_cd") == null) { response.sendRedirect(...); return; }`.
**Severity:** HIGH
**Checklist item:** Section 2 â Authentication and Authorization

### P47-F06: Filesystem path traversal in Excel output directory construction
**Description:** All five files construct the target directory for Excel file output using a path derived from the class's code source location and a fixed relative traversal of seven levels up followed by `excelrpt/`. The string manipulation uses `replaceAll("/", "")` to count slashes and `DataUtil.nthOccurrence()` to find a position in the path string. This path is constructed without sanitisation and without verifying that the resolved path stays within an expected directory boundary. If the code source path itself contains unexpected characters or the traversal depth changes (e.g. due to redeployment into a different container directory structure), the resolved path could land outside the intended `excelrpt/` directory. Additionally, the `%20` replacement (`replace("%20", " ")`) is applied to handle URL-encoded spaces in the code source path, but other URL-encoded characters are not decoded, potentially causing inconsistencies. While the directory itself is not directly attacker-controlled, this is a fragile and unvalidated path construction pattern in proximity to file write operations.

### P47-F07: `access_level` null dereference â potential NullPointerException masking missing session check
**Description:** In all five files, `session.getAttribute("access_level")` is cast to `String` and immediately passed to `.equalsIgnoreCase("")`. If the session attribute is not set (e.g. the user is not logged in, or the session has expired), this returns `null` and the call to `null.equalsIgnoreCase("")` throws a `NullPointerException`. The container will return a 500 error. This indicates the application relies on a higher-level mechanism (such as a `web.xml` security constraint or a filter) to ensure these pages are never reached without a valid session. If that outer mechanism has any gap, the lack of a null check here means the error response is a 500 rather than a clean redirect to login, potentially leaking stack trace information. No such outer guard is verifiable from these JSP files alone.
**Severity:** MEDIUM
**Checklist item:** Section 2 â Authentication and Authorization; Section 5 â Data Exposure (error response leaking internal detail)
