**Attribute security analysis:**

- `<mail server="..." user="..." password="...">` — the `user` and `password` attributes both have `<rtexprvalue>yes</rtexprvalue>`, meaning they can be set from runtime expressions. This is necessary for dynamic SMTP configuration. However, it means values may originate from JSP expressions or request parameters — any JSP that passes user-controlled input to these attributes would be a critical credential injection vector. The TLD itself does not contain credentials.
- `<mail authenticate>` attribute exists (`<required>no</required>`). Authentication is optional — pages could send mail unauthenticated.
- The `<attach url="...">` attribute exists with `<rtexprvalue>no</rtexprvalue>` (static only), limiting SSRF risk from attachment URLs at the tag level.
- Library version is Jakarta Taglibs Mailer 1.1 — an **extremely old library** (circa 2002–2004, no longer maintained). Known CVE history should be checked; at minimum, this represents a dependency that receives no security patches.
- The inline XML example in the TLD `<info>` block (lines 29–49) uses `server="home.net"` and example addresses `foo@home.net`, `bar@home.net` — these are documentation examples, not real credentials.

| `queryString` | `org.apache.taglibs.request.QueryStringTag` | Query string access |

**Security-relevant observations:**

1. **`isSessionFromUrl` tag is present** — the existence of this tag in the TLD, and its deployment in the application, indicates the application was designed with awareness that sessions can be propagated via URLs (`jsessionid=` in URL). This is a Tomcat legacy behaviour. The presence of this tag does not confirm it is used, but it is a flag for JSP-layer session-in-URL usage.

2. **`log` tag** — JSPs using `<req:log>` will write request data to Tomcat logs. If driver names, GPS coordinates, card/pin IDs, or vehicle serial numbers are logged via this tag, that constitutes a data exposure risk.

3. **`parameter` and `queryString` tags** — these expose raw request parameters to JSP scope. Any JSP that passes these values to SQL queries or uses them in `<%= %>` expressions without escaping creates injection/XSS vulnerabilities. The TLD itself does not introduce these bugs, but its presence means JSP authors have easy access to raw user input.

4. Library version 1.0 — another unmaintained Jakarta Taglibs component of the same vintage as the mailer library.

### C02-F01: Remaining Login Attempt Count Disclosed to Unauthenticated Users

**Description:** The UI string bundle explicitly tells an unauthenticated user how many login attempts remain before account lockout. This assists brute-force attackers by confirming which accounts have not yet been locked, and by letting them pace attacks to stay within the attempt window.

**Severity:** MEDIUM

**Checklist item:** Section 2 — Authentication and Authorization (session management / lockout)

**Evidence:**
```
# WEB-INF/src/main/resource/Messages.properties, lines 613–614
login.user.retry = Incorrect Login or Password.Please Re-try \n (1/5) Remaining Attempt.
login.user.retry.count = Incorrect Login or Password. Please Re-try \n ({0}/5) Remaining Attempt.
```
The pattern `{0}/5` confirms the server computes and returns the remaining count, and this string is rendered to the browser. An attacker learns both the total attempt ceiling (5) and their current position in it.

### C02-F05: Duplicate Property Keys Causing Silent Value Override

**Description:** The `Messages.properties` file (and all locale variants) define `gen.error.dberror` three times with three different values (lines 37, 43, 44). Java's `Properties.load()` silently discards all but the last definition. The effective value is `Please Select a User` — a user-management-specific message — but this key is also referenced for generic database errors. Any code path that displays `gen.error.dberror` for a genuine database error will show the wrong message. More critically, this means error message content cannot be reliably controlled: developers may add new error messages using existing keys believing they have configured meaningful user feedback, when the properties loader is silently discarding all but the last definition. The same issue affects `gen.success.mail_grp_add` (lines 56–57). This is a code quality defect with secondary security consequence: error message confusion can mask application behaviour from both users and security reviewers.

**Severity:** LOW

**Checklist item:** Section 5 — Data Exposure and Privacy (error message quality); Section 2 — Authentication and Authorization (correctness of access-related error messages)

**Evidence:**
```

44: gen.error.dberror = Please Select a  User      ← this value wins (last definition)

56: gen.success.mail_grp_add = Mail Group Successfully added.
```

### C02-F09: Malformed Property Key `global.driverid` Will Not Load

**Description:** Line 992 of `Messages.properties` uses a hyphen/dash separator instead of `=` or `:`, which are the only valid separators for Java `.properties` files:

```
global.driverid - Driver Id
```

`Properties.load()` treats the entire string `global.driverid - Driver Id` as a key with an empty value, rather than creating a key `global.driverid` with value `Driver Id`. Any JSP that references `<fmt:message key="global.driverid" />` will render an empty string or the key literal, silently. This is a code quality defect and a latent UI display bug. It also suggests the file has not been validated by a linter or automated test.

**Severity:** INFO

**Checklist item:** Section 5 — Data Exposure and Privacy (error message quality)

**Evidence:**
```
# WEB-INF/src/main/resource/Messages.properties, line 992
global.driverid - Driver Id
```

## Checklist Items With No Issues

**Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, SMTP passwords, encryption keys, or connection strings found in any of the 14 files reviewed. The mailer TLD `password` attribute is runtime-evaluated (not hardcoded). The help.support message uses `{0}` and `{1}` parameter substitution, not literal contact details.

**Section 3 — Injection Vulnerabilities:** Properties files are static string resources; they do not contain SQL, JSP expressions, or include directives. The TLD files define tag contracts but do not contain executable code. No injection vulnerabilities are directly evidenced in these files. (SQL injection and XSS risks from use of these string values in JSPs require separate JSP-level review.)

**Section 3 — File Inclusion:** No `<jsp:include page="${param.*}">` or dynamic include directives found in TLD definitions.

**Section 3 — Command Injection:** No `Runtime.exec()` or `ProcessBuilder` references in these files.

**Section 3 — SSRF:** The `<mt:attach url="...">` attribute in taglibs-mailer.tld has `rtexprvalue=no` (static only). No dynamic URL construction from user input is present in these files.

**Section 4 — CSRF:** Properties files do not contain form markup or token handling logic. CSRF protection cannot be assessed from resource bundles alone; this requires JSP and servlet review.

**Section 6 — Transport Security:** No HTTP/HTTPS connector configuration in these files. TLS settings require `server.xml` review (out of scope for this agent).

**Section 8 — Cross-Branch Security Comparison:** Cannot be completed by this agent alone; requires comparison with the FleetFocus branch findings once the FleetFocus Pass 1 audit is complete.
