### J86-F01: SQL injection via unparameterised `from`/`to` date range in BatteryDAO
**Description:** All four public data-retrieval methods in `BatteryDAO` (`getBatteryCharge`, `getBatteryChange`, `calculateBatteryChange`, `calculateBatteryCharge`) concatenate the caller-supplied `from` and `to` string parameters directly into a SQL query executed via a plain `Statement`. If these values originate from HTTP request parameters (which is the typical pattern for date-range report filters), an attacker can inject arbitrary SQL. The PostgreSQL `::timestamp` cast does not prevent injection because the attacker can close the quote and append additional SQL before the cast.
**Severity:** HIGH
**Checklist item:** Section 3 — SQL injection
**Evidence:**
```java
// getBatteryCharge, line 42
" and d1.utc_time >= '"+from+"'::timestamp"+" and d1.utc_time<='"+to+"'::timestamp"

// calculateBatteryCharge, lines 275-276
```

### J86-F08: Absence of PreparedStatement throughout all three files — systemic finding
**Description:** Across all three reviewed files, `java.sql.PreparedStatement` is never used. Every database interaction uses `java.sql.Statement` with string-concatenated SQL. This is a systemic design deficiency: the codebase has no architectural barrier against SQL injection. Any parameter passed to any of these methods — whether from user input, session attributes, or downstream callers — is directly interpolated into SQL. The correct remediation is a wholesale replacement of `Statement` with `PreparedStatement` using `?` placeholders for all variable values.
**Severity:** HIGH
**Checklist item:** Section 3 — SQL injection (PreparedStatement not used consistently)
**Evidence:**

### J86-F13: Hardcoded date filter in `send_updatepreop` — dead or stale business logic
