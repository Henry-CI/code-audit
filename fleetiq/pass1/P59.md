### P59-F01: Cross-Site Scripting (XSS) - Unescaped User-Controlled Output in Multiple JSP Files

**Description:** Multiple JSP files output user-controlled values directly to HTML without escaping, allowing attackers to inject malicious JavaScript. The following categories of unescaped output were identified:

1. **Database values in HTML attributes and content**: Customer names, location names, department names, vehicle details (hire numbers, serial numbers, comments, firmware versions, vehicle type names, canbus model names)
2. **Request parameters in JavaScript**: vehicle_cd, module_cd, customer codes embedded directly in JavaScript variables and URL construction
3. **Dynamic header values**: `UtilBean.getHireNumHeader()` output without escaping
4. **CSS injection vectors**: Values embedded in CSS `content:` property and inline `style` attributes

An attacker who can control database values (e.g., by creating a customer with a malicious name like `<script>alert(document.cookie)</script>`) or manipulate URL parameters could execute arbitrary JavaScript in the context of other users' sessions, leading to session hijacking, data theft, or unauthorized actions.

### P59-F02: Missing CSRF Protection on All State-Changing Forms

**Description:** All forms that perform state-changing operations (vehicle edits, impact settings, file uploads) lack CSRF (Cross-Site Request Forgery) protection. No CSRF tokens are visible in any of the reviewed forms. This allows an attacker to craft malicious web pages or emails that, when visited by an authenticated user, will submit requests to the application on behalf of that user without their knowledge or consent.

For example, an attacker could create a page that automatically submits a form to change vehicle settings, upload malicious files, or modify critical configuration parameters like impact calibration settings or idle timers. Given this is a fleet management system handling vehicle safety settings, unauthorized changes could have safety implications.
