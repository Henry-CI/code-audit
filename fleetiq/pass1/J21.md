### J21-F03: Use of deprecated and shut-down Google Image Charts API (SSRF-adjacent / external dependency)
**Description:** Both `BarChartUtil.java` and `BarChartUtil_bak.java` use the `charts4j` library, which generates URLs for the Google Image Charts API (`https://chart.googleapis.com/chart?...`). Google officially shut down this service on 14 March 2019. Any chart rendering that still calls this URL will silently fail, return an error image, or — if the domain were ever re-registered by a third party — could be redirected to an attacker-controlled endpoint. Although the URL is constructed by the library rather than by user input, the application sends data (axis labels derived from database values including driver/unit names) to an external Google endpoint. This is a data-exfiltration concern for a fleet-management system handling potentially sensitive vehicle and driver names.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (driver data, vehicle names sent externally); Section 6 — Transport Security (external HTTP/HTTPS dependency on a deprecated service); Section 3 — SSRF (data routed to external URL constructed programmatically).
**Evidence:**

### J21-F04: Database-sourced entity names embedded in chart URL without sanitisation
**Description:** In `BarChartUtil_bak.java`, `datalist.getName()` — a string sourced from the database via `EntityBean` — is placed directly into the `unit[]` label array and then passed to the Google Charts URL builder. If a vehicle or driver name stored in the database contains characters that have special meaning in a URL query string (e.g., `&`, `=`, `%`, `+`), the resulting chart URL could be malformed or, in an unlikely but possible scenario where chart URLs are rendered back to JSP, contribute to injection. The name is truncated at 18 characters if over `RuntimeConf.maxUnitName`, but no character-level sanitisation is applied.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (XSS/injection via unescaped data-sourced values); Section 5 — Data Exposure and Privacy (driver/vehicle names sent to external endpoint).

## Checklist Items With No Issues

The following checklist sections were reviewed against these three files and no findings were identified:

1. **Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, SMTP passwords, connection strings, or embedded secrets found in any of the three files. No calls to `DriverManager.getConnection()` or equivalent JDBC setup.

2. **Section 2 — Authentication and Authorization:** These are utility/chart-generation classes with no HTTP request handling, session access, or authorisation logic. No `HttpServletRequest`, `HttpSession`, or role-check code is present. No IDOR risk within these classes themselves (they operate on data already passed in by the caller).

3. **Section 3 — SQL Injection:** No JDBC calls, no `Statement`, no `PreparedStatement`, no SQL string construction of any kind in these files.

4. **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage.

5. **Section 3 — File Inclusion:** No JSP includes; these are pure Java utility classes.

6. **Section 4 — Session and CSRF:** No session management, no form handling, no CSRF token logic — not applicable to these utility classes.

7. **Section 5 — Logging of sensitive data:** No logging statements (`log.info`, `System.out.println`, or similar) present in any of the three files.

8. **Section 6 — Transport Security (Tomcat/connector config):** Not applicable to these source files; no connector or TLS configuration present.

9. **Section 7 — Dependency Vulnerabilities (beyond the Google Charts API finding):** The `charts4j` library is referenced by specific class imports (not wildcard) in `BarChartUtil.java` and `Chart.java`. The library version cannot be determined from source alone — the JAR in `WEB-INF/lib/` should be inspected separately. No Struts, Commons Collections, Log4j, or Spring references are present in these files.

10. **Multi-tenancy / customer_id scoping:** These chart utility classes accept pre-processed data arrays and entity bean lists from their callers. They contain no data-access logic and therefore no customer scoping responsibility. The responsibility for ensuring only the correct customer's data is passed to these utilities lies with the calling code, which is out of scope for this agent's assigned files.
