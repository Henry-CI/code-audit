### P51-F01: Reflected XSS via unescaped `message` / `msg` request parameter
**Description:** In `driver_limiting_by_truck.jsp`, `driver_settings_to_unit.jsp`, `driver_limiting_by_truck_old.jsp`, and `driver_settings_to_unit_old.jsp`, the URL parameter `msg` (or `message`) is read from the request and echoed directly into an HTML table row using `<%= message %>` without any escaping. An attacker who can induce a victim to follow a crafted URL can inject arbitrary HTML and JavaScript into the page.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```
// driver_limiting_by_truck.jsp, line 662:
<tr class="heading2"><%=message %></tr>

// driver_limiting_by_truck_old.jsp, line 776:
<tr class="heading2"><%=message %>
</tr>

// driver_settings_to_unit.jsp, line 609:
<tr class="heading2"><%=message %></tr>

// driver_settings_to_unit_old.jsp, line 791:
<tr class="heading2"><%=message %>
</tr>
```
Where `message` is assigned at:
```
// driver_limiting_by_truck.jsp, line 381:
String message = request.getParameter("msg") == null ? "" : request.getParameter("msg");

// driver_settings_to_unit.jsp, line 338:
String message = request.getParameter("msg") == null ? "" : request.getParameter("msg");
```
**File:** pages/admin/setting/driver_limiting_by_truck.jsp, pages/admin/setting/driver_settings_to_unit.jsp, pages/admin/setting/driver_limiting_by_truck_old.jsp, pages/admin/setting/driver_settings_to_unit_old.jsp

### P51-F04: XSS — unescaped DB values (customer names, driver names, vehicle hire numbers, location/department names) throughout all files
**Description:** Across all reviewed JSP files, data retrieved from the database (customer names, driver first/last names, vehicle hire numbers, location names, department names) is rendered into HTML using raw `<%= ... %>` expressions without any HTML escaping. If any of these database values contains HTML special characters or JavaScript (e.g., through stored XSS from another injection vector, or from malicious customer/driver names entered in an admin interface), the content will be rendered as HTML in the browser. This is a stored XSS risk.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
