### J22-F01: Wildcard imports in chart classes (code quality / dependency risk)
**Description:** Both `LineChartR.java` and `LineChartR_au.java` use the wildcard import `import com.googlecode.charts4j.*`. Wildcard imports import the entire public API of a package, making it impossible to determine at a glance which specific classes are used, complicating dependency analysis and making it easier to accidentally shadow class names.

**Checklist item:** 7. Dependency Vulnerabilities

### J22-F02: charts4j library uses deprecated Google Charts API (SSRF-adjacent / data leakage)
**Description:** `LineChartR.createChart()` and `LineChartR_au.createChart()` both use the charts4j library (version 1.3, JAR confirmed at `WEB-INF/lib/charts4j-1.3.jar`) which generates URLs for the Google Chart API (`https://chart.googleapis.com/chart?...`). The chart title (`report` field) and all data points are encoded into this URL. When the URL is rendered in the browser as an `<img src="...">`, the data is sent to Google's servers. For a fleet management application, this means vehicle utilisation figures and report names are transmitted to a third-party (Google) on every page load. The Google Image Charts API was officially deprecated on March 14, 2019, and is no longer supported. Any endpoint relying on it may silently fail or return incorrect charts. Additionally, the URL length limit for the Google Chart API was 2,000 characters; longer requests were silently truncated. charts4j-1.3 was released circa 2010.
**Severity:** MEDIUM
**Checklist item:** 6. Transport Security; 5. Data Exposure and Privacy
**Evidence:**
```java
// LineChartR.java, lines 45-79
com.googlecode.charts4j.LineChart chart = GCharts.newLineChart(line);
chart.setSize(700, 400);
chart.setTitle(report, Color.BLACK, 14);  // report field used as chart title
// ...
String url = chart.toURLString();  // produces https://chart.googleapis.com/chart?...
return url;
```
```java
// LineChartR_au.java, lines 63-97
LineChart chart = GCharts.newLineChart(line, lineTotal);
chart.setTitle(report, Color.BLACK, 14);
// ...

```

### J22-F05: LineChartR constructor silently discards the yAxis parameter
**Description:** The `LineChartR(double yAxis, double[] data)` constructor accepts a `yAxis` parameter but never uses it. Instead, `setyAxis()` is called (line 17) which computes the Y-axis scale from `DataUtil.maxValue(this.data)`. The constructor parameter `yAxis` shadows the inherited field name and is silently discarded. This is a logic defect — callers that believe they are controlling the Y-axis maximum are not. If a caller provides a `yAxis` value for a specific display reason (e.g., to normalise charts across time periods), that value is silently ignored, which could cause misleading chart output. This is a data integrity / presentation correctness issue.

**Checklist item:** 5. Data Exposure and Privacy (chart data misrepresentation)

### J22-F06: Report name sourced from database and embedded in external Google Chart URL without sanitisation
**Description:** In `Databean_report.java`, the `report_name` field is set from `rset.getString(1)` (the `FORM_NAME` column from `form_table`) at line 13160. This database value is passed via `lineChart.setReport(report_name)` at line 16165 into `LineChartR_au.createChart()`, where it becomes the `report` field used as `chart.setTitle(report, ...)`. The charts4j library then URL-encodes this title into the Google Chart API URL. While the value originates from the database (not direct user input), the `FORM_NAME` column could contain arbitrary strings if the form table is not tightly controlled. If an attacker can inject content into `form_table.FORM_NAME` (e.g., via an admin form or SQL injection elsewhere), that content would be included in the Google Charts API URL sent to external servers. Additionally, the SQL query that fetches `report_name` uses string concatenation with `set_form_cd`, which is itself an injection risk (see J22 scope: this file is not a primary assigned file, so this is noted as context rather than a primary finding).
**Severity:** LOW

## Checklist Items With No Issues

The following checklist items were reviewed and no issues were found in the three assigned files:

1. **Secrets and Credentials (Section 1):** No hardcoded credentials, passwords, API keys, connection strings, or database URLs in any of the three files. No `.env` or secret-named files referenced.

2. **Authentication and Authorization (Section 2):** None of the three files handle HTTP requests, session management, or access control decisions. They are pure chart-building utilities with no authentication surface.

3. **SQL Injection (Section 3 — SQL):** None of the three files contain JDBC calls, `Statement.executeQuery()`, or any SQL. No database access.

4. **XSS in JSP expressions (Section 3 — XSS):** These are Java classes, not JSP files. No `<%= %>` expressions. The chart URL produced is a Google Charts API URL, not HTML output; however see F06 for the data-in-URL concern.

5. **File Inclusion (Section 3 — File inclusion):** No `jsp:include`, file path parameters, or file I/O in any of the three files (though `JfreeGroupStackChart.java` has dead imports for `File` and stream classes, these are unused).

6. **Command Injection (Section 3 — Command injection):** No `Runtime.getRuntime().exec()` or `ProcessBuilder` in any of the three files.

7. **SSRF (Section 3 — SSRF):** The Google Charts URL is constructed by the charts4j library and is a fixed endpoint (`chart.googleapis.com`). The URL is not derived from user input in these files. Noted as a data leakage concern in F02 but not a classic SSRF (attacker-controlled URL).

8. **Session and CSRF (Section 4):** No session handling, form processing, or state-changing operations in any of the three files.

9. **Transport Security (Section 6):** The charts4j library constructs HTTPS URLs to `chart.googleapis.com`. No HTTP downgrade visible in these files.

10. **Multi-tenancy / IDOR (Section 2 — Authorization):** These are chart rendering utilities. They do not directly query data by customer ID or accept object IDs as parameters. Tenant scoping must be enforced by the callers that supply the data arrays — that is out of scope for these three files.
