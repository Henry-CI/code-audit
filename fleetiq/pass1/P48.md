## Findings

### P48-F01: Unescaped user-controlled request parameters written into HTML attributes (XSS)
**Description:** In every file in this set, `request.getParameter("mnm")` and `request.getParameter("sub")` are read directly and written into HTML `<input type="hidden" value="...">` attributes using raw `<%= %>` expressions without any HTML encoding. An attacker who can craft a URL with malicious values for `mnm` or `sub` (e.g., containing `"><script>...`) can inject arbitrary HTML or JavaScript into the page. These are report header fragments included into all admin report pages, so the attack surface is broad.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// report_head.jsp lines 70–71
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">
```
Same pattern present in:
- report_head_a.jsp lines 56–57
- report_head_b.jsp lines 56–57
- report_head_c.jsp lines 57–58
- report_head_d.jsp lines 61–62
- report_head_keyword.jsp lines 68–69
- report_head_keyword_a.jsp lines 68–69
- report_head_nodate.jsp lines 50–51
- report_head_nodate_model.jsp lines 49–50
- report_head_todate.jsp lines 56–57
**File:** pages/admin/reports/inc/report_head.jsp (and all variants listed above)
**Line(s):** report_head.jsp:70–71; replicated across all 10 files

### P48-F02: Unescaped database-sourced values written into HTML option elements (Stored XSS)
**Description:** Customer names, site names, and department names sourced from the database (via the `filter` object) are written into HTML `<option value="...">` attributes and option text content using raw `<%= %>` without HTML encoding. If any of these values contain HTML special characters (e.g., a customer named `"><img src=x onerror=alert(1)>`), stored XSS would be triggered for any user viewing a report. Fleet management data such as customer names and site names is typically admin-entered but should still be escaped on output.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// report_head.jsp lines 82–83
<option value="<%=Vuser_cd.get(i) %>" <% if(user_cd.equals(Vuser_cd.get(i))) {%> selected="true"<%} %>><%=Vuser_nm.get(i) %></option>

```
Identical pattern in all 10 files throughout the option-rendering loops.
**File:** pages/admin/reports/inc/report_head.jsp (and all variants)
**Line(s):** report_head.jsp:82–83, 93, 104; same pattern in every variant file

### P48-F03: NullPointerException risk — request.getParameter() called with .toString() without null check
**Description:** In every file, `request.getParameter("mnm").toString()` and `request.getParameter("sub").toString()` are called without first checking whether the parameters are null. `request.getParameter()` returns `null` if the parameter is absent, and calling `.toString()` on null throws a `NullPointerException`. This will cause a 500 error for any request that omits `mnm` or `sub`. While primarily a reliability issue, it also means these include fragments will expose exception information unless proper error pages are configured — a data exposure concern per Section 5.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (error responses)
**Evidence:**
```jsp
// report_head.jsp lines 70–71
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">
```
Same pattern in all 10 files.
**File:** pages/admin/reports/inc/report_head.jsp (and all variants)
**Line(s):** report_head.jsp:70–71; replicated across all files

### P48-F08: No CSRF protection on report filter forms
**Description:** None of the 10 report header fragment files include a CSRF token hidden field in their forms. All forms submit via GET to `../pages/admin.jsp`. While GET forms triggering report queries are lower severity than state-changing POST forms, report generation can disclose sensitive fleet data. More critically, these fragments establish the form pattern used across all reports. If any report action (email report, subscribe, export) is also processed without CSRF tokens, state-changing operations (subscribing to scheduled reports, sending email) can be triggered via cross-site request forgery.
