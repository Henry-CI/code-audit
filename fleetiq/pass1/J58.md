## Findings

### J58-F01: No customer/tenant scoping field in any report bean
**Description:** All three bean classes are used to carry data for driver-related Excel reports. None of the three beans contains a customer ID, tenant ID, or organisation ID field. In a multi-tenant application (branch name `multi-customer-sync-to-master` explicitly indicates multi-tenancy), every data container used to populate a report must carry a tenant identifier so that the layer populating the bean and the layer rendering the report can verify or record which customer's data is being presented. The absence of such a field means the beans themselves provide no mechanism to enforce or audit per-tenant data isolation. The risk is realised in the code that populates these beans — if the SQL query or service layer that fills the lists does not filter by customer, nothing in the bean structure will catch or signal the omission.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy ("verify that every data access is scoped to the authenticated user's customer/organisation")

### J58-F02: Raw (unparameterised) ArrayList types used throughout all three beans
**Description:** The majority of fields across all three beans use the raw type `ArrayList` rather than a typed generic such as `ArrayList<String>` or `ArrayList<Date>`. Raw collection types suppress compiler type-checking. When the bean is populated by calling code, any type may be inserted without compile-time error. If a caller inserts an unexpected type and the rendering layer (the Excel report writer) iterates the list and casts elements, this will produce a `ClassCastException` at runtime. More relevantly to security, the absence of type constraints means there is no enforced contract on what data enters the bean from the database or request layer — a wider surface for unexpected data injection if calling code is later modified. This is also a data-integrity risk: driver names, vehicle serial numbers, and timestamps are all held in untyped lists with no validation layer enforced by the type system.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy ("driver names, licence numbers, or contact details are exposed... beyond what is necessary")

### J58-F03: Package-private field access on sensitive data fields in DriverImpactReportBean and DriverLicenceExpiryBean
**Description:** In `DriverImpactReportBean` all ten fields (lines 7-16) and in `DriverLicenceExpiryBean` both fields (lines 6-7) are declared without an explicit access modifier, making them package-private. Any class in the same package (`com.torrent.surat.fms6.excel.reports.beans`) can read or write these fields directly, bypassing the getter/setter methods entirely. The fields include driver names (`dir_driv_nm`, `driver_nm`) and licence review dates (`review_date`), which are personally identifiable information (PII). Package-private visibility on PII-carrying fields means there is no encapsulation boundary protecting this data within the package. Note: `DriverAccessAbuseBean` has the same issue for its eleven unlabelled fields (lines 8-18), though it also has two properly declared `private` fields and two `private` parameterised list fields.

- **Section 6 — Transport Security:** No network or transport layer code present.
- **Section 7 — Dependency Vulnerabilities:** The only import across all three files is `java.util.ArrayList` from the Java standard library — no third-party dependencies introduced.
