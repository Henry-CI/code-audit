## Findings

### P29-F01: Dynamic JSP File Inclusion via User-Controlled Parameters (Path Traversal / Local File Inclusion)

**Description:** In both `admin.jsp` and `index.jsp`, the `mnm` and `sub` request parameters are read directly from the URL query string and concatenated into a file path that is then passed to `<jsp:include page="<%= filepath %>">`. An attacker can manipulate these parameters to include arbitrary JSP files within the web application, potentially accessing restricted administrative pages or internal pages not intended to be directly accessible. While the code checks that the resolved real path file exists, there is no whitelist validation of `mnm`/`sub` values. In `index.jsp`, there is no permission check at all around the include (unlike `admin.jsp` which at least checks `form__cd` against permissions — but even that check is bypassed when `form__cd` is null, per line 70: `if( form__cd == null || ...)`).

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities — File inclusion

**Evidence:**
```java
// admin.jsp lines 57–71
String mnm = request.getParameter("mnm")==null? "Home" : request.getParameter("mnm");
String sub = request.getParameter("sub")==null? "" : request.getParameter("sub");
String filepath = "";
if( sub.isEmpty() ) {
  filepath = "/pages/admin/"+mnm+".jsp";
} else {
  filepath = "/pages/admin/"+sub+".jsp";
}
...
<jsp:include page="<%= filepath %>" flush="true" />

// index.jsp lines 34–49 (identical construction, NO permission check at all)
String mnm = request.getParameter("mnm")==null? "Home" : request.getParameter("mnm");
String sub = request.getParameter("sub")==null? "" : request.getParameter("sub");
...
<jsp:include page="<%= filepath %>" flush="true" />
```

**File:** pages/admin.jsp, pages/index.jsp
**Line(s):** admin.jsp 57–71; index.jsp 34–49

**Description:** The `message` request parameter is read without sanitisation and injected directly into a JavaScript `swal()` call's `text` property. ### P29-F05: Session Fixation — Session Not Invalidated Before Re-Use on Login (checker.jsp)

**Description:** `checker.jsp` is the post-authentication handler. On successful authentication it calls `request.getSession(true)` (line 44) but does NOT first call `request.getSession().invalidate()`. This means the session ID in use before login is preserved after login, enabling a session fixation attack: an attacker who can set the victim's JSESSIONID cookie before login can later use that same ID to access the authenticated session.

**Severity:** HIGH

**Checklist item:** 2. Authentication and Authorization — Session management (session fixation)

**Evidence:**
```java
// checker.jsp lines 24–44
session.setAttribute("user_cd",user_cd);
session.setAttribute("access_level",access_level);
session.setAttribute("access_cust",access_cust);

...

response.setStatus(response.SC_MOVED_TEMPORARILY);

### P29-F18: Permission Check in admin.jsp is Bypassable When `form_cd` is Null

**Description:** The permission guard in `admin.jsp` around the dynamic `<jsp:include>` uses the condition `form__cd == null || permision[...][0].equalsIgnoreCase("t")`. When `form_cd` is not supplied in the URL (i.e., the parameter is absent), `form__cd` is null and the condition short-circuits to true, bypassing the permission check entirely and including whatever file `filepath` resolves to. An attacker simply omits `form_cd` from the URL to bypass the permission system.

**Severity:** HIGH

**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy

**Evidence:**

} %>
```
When `form_cd` is absent: `form__cd == null` is true, the OR short-circuits, include proceeds regardless of permissions.

**File:** pages/admin.jsp
**Line(s):** 12, 70–74
