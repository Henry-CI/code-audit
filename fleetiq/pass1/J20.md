## Findings

### J20-F01: Wildcard import in BarChartR.java
**Description:** `BarChartR.java` uses a wildcard import `import com.googlecode.charts4j.*` on line 6. While not a direct runtime vulnerability, wildcard imports hide exactly which classes are in scope, making it harder to audit third-party library usage and increasing the risk that a class name collision or unexpected class is used silently. The checklist explicitly flags wildcard imports.
**Severity:** LOW
**Checklist item:** Section 1 (Secrets and Credentials) — Reading Evidence requirement; Checklist reading-evidence instruction: "flag wildcard imports: `import java.sql.*`"
**Evidence:**

### J20-F02: Google Chart Image API generates external HTTP URLs — data leakage to third party
**Description:** All three chart classes use the `charts4j` library to generate Google Chart API URLs via `chart.toURLString()`. The Google Chart Image API encodes chart data (fleet statistics, utilisation percentages, driver counts by time of day, impact category counts) as URL query parameters and sends them to Google's servers at `https://chart.apis.google.com` or `https://chart.googleapis.com` for rendering. This means that operational fleet data — vehicle utilisation figures, driver activity breakdowns, and impact/event counts — is transmitted to Google's infrastructure every time a chart is rendered. This constitutes a data exposure risk for a fleet management platform handling customer business data and potentially PII-adjacent information (driver activity patterns). The Google Chart Image API was deprecated in 2012 and was officially shut down. Depending on the current state of that API, chart rendering may also silently fail.
**Severity:** MEDIUM
**Checklist item:** Section 5 (Data Exposure and Privacy) — "fleet management data including vehicle locations (GPS coordinates), driver identities, customer business data"; Section 6 (Transport Security) — external data transmission path.
**Evidence:**
```java
// BarChartImpactCategory.java, line 118
String url = chart.toURLString();

// BarChartNational.java, line 99
String url = chart.toURLString();

// BarChartR.java, line 89
String url = chart.toURLString();
```
**File:** All three reviewed files.
**Line(s):** BarChartImpactCategory.java:118; BarChartNational.java:99; BarChartR.java:89

### J20-F03: Database-sourced entity names embedded directly in external Google Chart URL without sanitisation
**Description:** In `BarChartNational.java`, `datalist.getName()` (line 70–74) returns a driver or unit name retrieved from the database via `EntityBean`. This string is placed directly into the `unit[]` array which is then passed to `AxisLabelsFactory.newAxisLabels(unit)` (line 85) and encoded into the Google Chart URL returned by `chart.toURLString()` (line 99). There is no sanitisation or encoding of the name value before it is incorporated into the URL. If a unit or driver name contains characters that have special meaning in a URL or in the chart API's label encoding (e.g., `|`, `&`, `=`), this could corrupt the chart URL or, if the URL is later embedded in a JSP page without escaping, contribute to an XSS vector. Additionally, this constitutes unsanitised database data being sent to an external service.
**Severity:** MEDIUM
**Checklist item:** Section 3 (Injection Vulnerabilities — XSS): "Check for HTML rendered from database values (vehicle names, driver names, customer names, notes fields) — these must also be escaped on output." Section 5 (Data Exposure and Privacy).
**Evidence:**

### J20-F04: No customer/tenant scoping in any chart class — multi-tenancy isolation not enforced at chart layer
**Description:** None of the three chart classes — `BarChartImpactCategory`, `BarChartNational`, or `BarChartR` — accept or enforce a customer ID or tenant ID parameter. The data (`ArrayList<double[]>` or `List<EntityBean>`) is passed in pre-populated with no reference to which customer it belongs to. Scoping must be enforced entirely by the calling code. This is a structural pattern risk: if any caller passes the wrong data set (e.g., a query that does not include a `WHERE customer_id = ?` clause), the chart will render cross-tenant data without any safety net at the chart layer. Given that the branch is explicitly named `multi-customer-sync-to-master` — indicating multi-tenant functionality — this absence of any tenant check in the chart construction layer is a flag for reviewer attention. This finding does not confirm a vulnerability in these files alone but requires verification that all callers supply correctly-scoped data.
**Severity:** MEDIUM
**Checklist item:** Section 2 (Authentication and Authorization): "verify that every data access is scoped to the authenticated user's customer/organisation"; "The `multi-customer-sync-to-master` branch name indicates multi-customer (multi-tenant) functionality. This is the highest-risk area in the application."
**Evidence:**
```java
// BarChartImpactCategory.java, line 41 — constructor takes raw data, no customer_id
public BarChartImpactCategory(ArrayList<double[]> data, String title, ResourceBundle bundle)

// BarChartNational.java, line 33 — constructor takes raw EntityBean list, no customer_id
```

## Checklist Items With No Issues

The following checklist items were reviewed against the three assigned files and no issues were found within the scope of these files:

- **Section 1 — Secrets and Credentials:** No hardcoded credentials, API keys, or database passwords exist within `BarChartImpactCategory.java`, `BarChartNational.java`, or `BarChartR.java` themselves. (Note: RuntimeConf.java, a dependency, does contain credentials — see J20-F05.)
- **Section 2 — Container-managed authentication / Session management:** These are chart-generation utility classes; they do not handle HTTP sessions, authentication, or login. Not applicable at this layer.
- **Section 3 — SQL Injection:** No JDBC calls, `Statement`, `PreparedStatement`, or SQL strings of any kind appear in any of the three files. No SQL injection risk in these files.
- **Section 3 — File inclusion:** No `<%@ include %>` or `<jsp:include>` directives. Not JSP files. Not applicable.
- **Section 3 — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage in any of the three files.
- **Section 3 — SSRF:** The chart classes do not construct or invoke HTTP client calls directly. The URL produced by `chart.toURLString()` is returned as a string; the HTTP request is made by the browser (client-side image load), not the server. No server-side SSRF in these files.
- **Section 4 — Session and CSRF:** These classes generate chart URL strings only; they do not render HTML forms, handle POST requests, or manage sessions. CSRF is not applicable at this layer.
- **Section 6 — Transport Security:** No connector configuration, TLS settings, or HTTP client calls within these files.
- **Section 7 — Dependency Vulnerabilities:** The `charts4j` (com.googlecode.charts4j) library version cannot be confirmed from source alone — the JAR must be checked separately. No other external dependencies are introduced by these three files beyond what is already in the application.
