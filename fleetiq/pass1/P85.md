## Findings

### P85-F01: Cross-Site Scripting (XSS) via unescaped mail_id output
**Description:** The mail_id request parameter is output directly using `<%= %>` expression without HTML escaping. An attacker can inject malicious JavaScript by providing a crafted mail_id value such as `<script>alert(document.cookie)</script>`, which will execute in the user's browser context when the success message is displayed.

**Severity:** MEDIUM

**Checklist item:** 3. Injection Vulnerabilities — XSS (Cross-Site Scripting)

**Evidence:**
```jsp
<%=mail_id%>
```

**File:** reports/xlsx_driver_access_abuse_report_mail.jsp

**Line(s):** 123

**Description:** This JSP file generates and returns Excel reports but performs NO authentication or session validation checks. Any user (authenticated or not) can access reports by directly calling this JSP with rpt_name and params parameters. This completely bypasses the application's security model and exposes all report data to unauthenticated users.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization — Container-managed authentication

**Evidence:**


### P85-F12: NullPointerException risk from null session attribute
**Description:** The code retrieves access_level from the session and then checks if it equals empty string, but does not check for null first. If the session attribute is not set, session.getAttribute() returns null, and calling .equalsIgnoreCase("") on null will throw a NullPointerException. This can lead to application errors and denial of service.

### P85-F14: Missing customer scoping validation in multi-tenant report
**Description:** While xlsx_driver_access_abuse_report_mail.jsp retrieves access control attributes from the session (access_level, access_cust, access_site, access_dept) and sets them on the filter bean, there is no visible validation in this JSP to ensure that the requested report data (identified by parameters like user_cd, loc_cd, dept_cd, form_cd) actually belongs to the authenticated user's customer. The branch name "multi-customer-sync-to-master" indicates multi-tenant functionality, making this a critical concern. The actual enforcement may occur in the filter bean or ExcelDriverAccessAbuseReport class, but without seeing that code, we cannot verify proper isolation. If the backend does not enforce customer scoping, any authenticated user could potentially access other customers' driver abuse reports.

**Severity:** HIGH

**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy

**Evidence:**
```jsp
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");

String access_dept=(String)session.getAttribute("access_dept");
```
