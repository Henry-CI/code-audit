## Findings

### P54-F01: Unescaped database values output directly to HTML in all seven files (stored XSS)

**Description:** Every `<%= ... %>` expression across all seven JSP files outputs data retrieved from the database directly into the HTML response without any escaping. The correct pattern for a JSP/JSTL stack is `<c:out value="${variable}" />` or `${fn:escapeXml(variable)}`. If any database-stored string contains characters such as `<`, `>`, `"`, or `&` — either through legitimate entry or through a prior injection — it will be rendered as raw HTML in the browser, enabling stored Cross-Site Scripting. Fields rendered without escaping include customer names, site names, vehicle names, vehicle descriptions, hire numbers, model numbers, driver names, card numbers, impact times, severity labels, return times, and dates.

**Severity:** HIGH

**Checklist item:** Section 3 — XSS (Cross-Site Scripting)

### P54-F02: HTML attribute injection via unescaped database value in HTML attribute context

**Description:** In `hire_vehicle_lst.jsp` line 55, the customer name `Vcust_nm.get(i)` is injected unescaped into the `data-title` HTML attribute value. If a customer name stored in the database contains a double-quote (`"`), it will terminate the attribute and allow injection of arbitrary attributes or JavaScript event handlers (e.g., a customer name of `foo" onmouseover="alert(1)` would produce `data-title="Units on Hire for foo" onmouseover="alert(1)"`, executing JavaScript). This is a distinct stored XSS vector from P54-F01 because it targets an HTML attribute rather than element content.

**Severity:** HIGH

**Checklist item:** Section 3 — XSS (Cross-Site Scripting)

**Evidence:**

### P54-F03: Unvalidated customer code from request parameter passed to data filter (IDOR)

**Description:** In `hire_vehicle_lst1.jsp`, the `cust_cd` query parameter is read directly from the HTTP request at line 8 and passed to `filter.setSet_cust_cd(cust_cd)` at line 25. There is no visible verification that the supplied `cust_cd` belongs to or is permitted for the authenticated user. Although session-based access attributes (`access_cust`, `access_level`, etc.) are also set on the filter, whether the `Databean_cdisp` bean actually enforces that `cust_cd` is restricted to values permitted by `access_cust` cannot be confirmed from this file alone. If the bean does not perform this check, any authenticated user could enumerate arbitrary customer codes in the `cust_cd` parameter and view vehicle hire data for any customer. The parent page `hire_vehicle_lst.jsp` generates the link with `cust_cd` values drawn from the database, but those values are visible in the URL and trivially manipulable. This is a potential IDOR vulnerability across tenant boundaries.

**Severity:** HIGH

**Checklist item:** Section 2 — Authorization and multi-tenancy (IDOR)

**Evidence:**
```jsp
// hire_vehicle_lst1.jsp, lines 8 and 25:
String cust_cd = request.getParameter("cust_cd")==null?"":request.getParameter("cust_cd");
...
filter.setSet_cust_cd(cust_cd);
filter.init();
```
**File:** pages/admin/summary/hire_vehicle_lst1.jsp
**Line(s):** 8, 25, 26

### P54-F04: Missing tenant scoping on dislockout_vehicle_lst.jsp filter initialisation

**Description:** In `dislockout_vehicle_lst.jsp`, the filter bean is initialised with only the opcode (`filter.setSet_opcode("unlockout")`) and no session-derived access attributes (`access_user`, `access_level`, `access_cust`, `access_site`, `access_dept`) are set before `filter.init()` is called. This is in contrast to five of the other six files reviewed, which all pass the full set of session access attributes to the filter before calling `init()`. If `Databean_cdisp.init()` uses access attributes to scope its database query to the authenticated user's tenant, omitting these attributes may cause this page to return data from all tenants rather than only the user's own. The same omission is present in `nothire_vehicle_lst.jsp`.

### P54-F10: Raw HTML attribute injection via unescaped bgc variable in cric_issues_lst.jsp

**Description:** In `cric_issues_lst.jsp` line 123, the variable `bgc` is injected directly into an HTML tag attribute position: `<td width = 25% <%=bgc %>>`. The `bgc` string is set in scriptlet logic (lines 113–122) to one of two fixed string literals (`"bgcolor='red'"` or `"bgcolor='orange'"`) based on a case-insensitive comparison against the severity string `tmp`. However, `tmp` is itself derived from database data (`iene.get(j)`, line 111). If `tmp` is neither "Red Level" nor "Amber Level", `bgc` remains as its initial value of an empty string `""`, which is safe. But if an attacker can influence the impact energy severity labels stored in the database (e.g., through another injection point), they could set `tmp` to a string that bypasses the comparison and causes `bgc` to be empty — however the injection would be harmless in that path. The primary risk is that `bgc` as a raw HTML injection into tag attribute position (`<td ... <%=bgc %>>`) is an architectural antipattern — any future modification that sets `bgc` from un-validated data would be immediately exploitable. The existing logic uses fixed literals, so this is LOW/INFO as-coded, but the pattern is flagged.

**Severity:** LOW

**Evidence:**
