### J43-F01: SQL Injection via cust_cd, loc_cd, and dept_cd parameters

**Description:** All three report classes accept `cust_cd`, `loc_cd`, and `dept_cd` constructor parameters that originate directly from HTTP request parameters (confirmed in `service_report.jsp` line 17: `request.getParameter("cust_cd")`). These values are stored in `Frm_excel` fields and passed without sanitisation to `Databean_report.setSet_cust_cd()`, `setSet_loc_cd()`, `setSet_dept_cd()`. Inside `Databean_report.init()` and the `Fetch_*` methods, `java.sql.Statement` (not `PreparedStatement`) is used throughout, and the values are concatenated directly into SQL query strings. At line 958 of `Databean_report`, `set_cust_cd` is concatenated without surrounding single-quotes (`where cust_id = " + set_cust_cd`), which allows direct numeric SQL injection. ### J43-F02: HTML Injection / Stored XSS in Email Report Body

**Description:** `EmailServMaintenanceReport.createBody()` builds an HTML email body by directly appending database-sourced values into HTML `<td>` elements with no escaping. Fleet names, service dates, service types, and meter reading values are concatenated into raw HTML strings. If any of these database values contain HTML or JavaScript characters (e.g., fleet names entered by users with `<script>`, `"`, `<` characters), the resulting email body will contain injected HTML. This constitutes stored XSS delivered via email, and also HTML injection that could manipulate the email layout or embed malicious links. The affected values — fleet names, last service type carried out, hours values — are user-editable fleet management data.

**Severity:** HIGH

### J43-F03: Formula Injection (CSV/Excel Injection) in Excel Output

**Description:** All three report classes write data-sourced values into Excel cells using both `setCellValue()` and `setCellFormula()`. Specifically, `EmailSeatHourUtilReport` and `EmailServMaintenanceReportNew` call `setCellFormula("VALUE(\"" + vutil1.get(i).toString() + "\")")`, constructing Excel formula strings by concatenating values retrieved from the database. If any utilisation time value stored in the database contains Excel formula characters (e.g., `=`, `+`, `-`, `@`), the resulting `.xlsx` file may execute that formula when opened in Microsoft Excel or LibreOffice Calc. This is a formula injection vulnerability (also known as CSV injection when applied to spreadsheet exports). Although time values are expected to be formatted as `HH:MM:SS` strings, the application performs no validation or sanitisation before constructing the formula string.

### J43-F04: Multi-Tenancy Isolation Relies Solely on Caller-Supplied cust_cd

**Description:** None of the three report classes verify that the `cust_cd` supplied at construction time matches the authenticated user's customer. The constructors accept `cust_cd` as a plain parameter and use it directly as the data-scoping criterion when querying `Databean_report` and `ServiceDueFlagBean`. There is no server-side check that the requesting session's authenticated customer code equals the `cust_cd` in the report request. An authenticated user who belongs to customer A can supply `cust_cd=B` in the HTTP request and receive the service maintenance or utilisation report for customer B. This is a classic IDOR (Insecure Direct Object Reference) vulnerability in a multi-tenant context.

The calling JSP (`service_report.jsp` line 17) takes `cust_cd` directly from `request.getParameter("cust_cd")`. Although `session.getAttribute("access_cust")` is read (line 29) and passed to a `filter` object (line 40), this is used only to populate a UI dropdown — it is not enforced as the upper bound on what `cust_cd` value the report accepts.

**Severity:** CRITICAL

**Checklist item:** Section 2 — Authorization and Multi-Tenancy; IDOR
