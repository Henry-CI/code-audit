### J12-F09: Missing Multi-Tenant Isolation in Fetch_service_reminder
**Description:** `Fetch_service_reminder()` queries `service_reminder` using `set_cust_cd` as the only filter. If `set_cust_cd` is empty or blank, the condition block is skipped entirely and the query `select * from service_reminder` executes with no WHERE clause, returning service reminder records for all customers. There is no fallback to the authenticated session's customer code — if an empty string is passed, all tenant data is exposed.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization — multi-tenancy; also 5. Data Exposure and Privacy
**Evidence:**
```java
// Lines 885-893
query = "select * from service_reminder";
if (!set_cust_cd.equalsIgnoreCase("")) {
    query += " where cust_cd =" + set_cust_cd;
    rset = stmt.executeQuery(query);
    while (rset.next()) {
        service_id.add(rset.getString(1));
        service_hour.add(rset.getDouble(2) + "");
    }
}
```
Note: if `set_cust_cd` is empty, the query is built but never executed (the `executeQuery` is inside the `if` block) — however the base query `select * from service_reminder` with no scoping represents the intended fetch-all-if-no-customer pattern, which itself is a design-level data exposure risk, and the SQL injection (F06) means an attacker can bypass any value check.
