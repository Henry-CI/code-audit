### P70-F01: Cross-Site Scripting (XSS) via Unescaped User-Controlled Date Parameters
**Description:** Multiple JSP files output user-controlled date parameters (st_dt, to_dt) directly into HTML without escaping. These parameters are retrieved from the request and rendered using raw `<%= %>` expressions. An attacker can inject malicious JavaScript by providing crafted date values (e.g., `st_dt=<script>alert(document.cookie)</script>`), which will execute in the victim's browser when the report is viewed. This affects Excel export reports which render HTML content before being processed by the client.

**Severity:** HIGH

**Checklist item:** Section 3 - Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Description:** All reviewed JSP files render database-sourced content (driver names, vehicle names, customer names, failure descriptions, comments, etc.) directly using raw `<%= %>` expressions without any HTML escaping. ### P70-F03: Hardcoded External URL in Production Code
**Description:** The file excel_serv_maintenance.jsp contains a hardcoded external URL `http://fms.fleetiq360.com/css/alternate_rows.css` for a CSS stylesheet. This creates multiple security concerns: (1) mixed content warning if the page is served over HTTPS, (2) dependency on external infrastructure that could be compromised or become unavailable, (3) potential for CSS-based attacks if the external domain is compromised, and (4) information leakage about the application's structure. External resources should be hosted locally or loaded via HTTPS with Subresource Integrity (SRI) checks.

### P70-F05: File Path Traversal Risk in Excel Report Generation
**Description:** The file excel_preop_chk_detailed.jsp constructs a file system path using `getProtectionDomain().getCodeSource().getLocation()` and string manipulation to navigate the directory structure. The code performs path manipulation including counting forward slashes and using substring operations (lines 83-88). While the path appears to be derived from the application's deployment location rather than user input, the manual string manipulation approach is error-prone and could lead to path traversal vulnerabilities if the logic is modified or if the deployment path contains unexpected characters. The constructed path is then used for file operations (reading Excel files). A safer approach would use Path.normalize() or similar canonicalization methods.

### P70-F06: Insufficient Multi-Tenant Data Isolation Verification
**Description:** All reviewed JSP files retrieve session-based access control attributes (access_level, access_cust, access_site, access_dept) and pass them to the filter bean, but there is no visible enforcement or verification in the JSP layer that these values are correctly scoping the data. The actual SQL queries and data filtering happen in the Databean_report class which is not visible in this review. The JSP files trust that filter.init() and subsequent getter methods will return only customer-scoped data, but there's no defensive check in the JSP to verify that returned data belongs to the authenticated customer. If the Databean_report class has a SQL injection vulnerability or incorrect WHERE clause, these JSPs would display cross-tenant data without detection. This violates defense-in-depth principles for multi-tenant applications on the "multi-customer-sync-to-master" branch.

**Severity:** MEDIUM

**Checklist item:** Section 2 - Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**

### P70-F07: Error Information Disclosure via Stack Trace
**Description:** The file excel_preop_chk_detailed.jsp contains a try-catch block that catches all exceptions during file I/O operations and prints the full stack trace using `e.printStackTrace()`. ### P70-F08: Session Attribute Null Pointer Exception Risk
**Description:** All reviewed JSP files retrieve session attributes (access_level, access_cust, access_site, access_dept) and immediately use them in operations like .equalsIgnoreCase() or Integer.parseInt() without null checks first. While there is a null check for access_level at line 42 in most files, it checks if the value is an empty string, not if it's null. If the session attribute is null (e.g., session expired or user not authenticated), calling .equalsIgnoreCase("") on a null value will throw a NullPointerException, resulting in a 500 error and potential information disclosure. The session include file "../sess/Expire.jsp" is included but its contents are not visible in this review to verify if it handles null sessions.

### P70-F09: Inconsistent Response Header Setting for File Downloads

## Summary

This Pass 1 security audit reviewed 10 JSP report files in the reports/ directory. A total of **9 findings** were identified:

- **HIGH severity:** 2 findings (XSS vulnerabilities)
- **MEDIUM severity:** 4 findings (CSRF, external URL, path traversal risk, multi-tenant isolation)
- **LOW severity:** 3 findings (error disclosure, null pointer risk, inconsistent headers)

The most critical findings are the Cross-Site Scripting (XSS) vulnerabilities affecting all report outputs. Both user-controlled parameters and database-sourced content are rendered without HTML escaping, creating stored and reflected XSS attack vectors.

All reports implement session-based access control by retrieving and passing access control attributes to the filter bean, but the enforcement happens in Java classes not visible in this review. The JSP layer does not perform defensive verification of returned data.

No SQL injection vulnerabilities were found at the JSP layer, as all database access is properly abstracted through the Databean_report class. However, the actual SQL queries require separate review of the Java source code.

CSRF protection is absent from all report generation endpoints, and several files have minor implementation issues including error disclosure and inconsistent response header configuration.
