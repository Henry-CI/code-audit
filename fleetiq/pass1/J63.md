### J63-F01: SQL Injection — getCustomerName() concatenates cust_cd directly into SQL
**Description:** The method `getCustomerName()` constructs a SQL query by concatenating the caller-supplied parameter `cust_cd` directly into the query string using `java.sql.Statement`. No parameterisation (`PreparedStatement` with `?` placeholders) is used. Any value of `cust_cd` that originates from user input (e.g., an HTTP request parameter, a form field, or a report filter) is injected verbatim into the SQL sent to the database. An attacker can manipulate this value to alter query logic, extract data from other tables, or — depending on database permissions — execute administrative commands.
**Severity:** CRITICAL
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL Injection)
**Evidence:**
```java
// CustomerDAO.java lines 28–33
stmt=conn.createStatement(ResultSet.TYPE_SCROLL_SENSITIVE,ResultSet.CONCUR_READ_ONLY);

if(!cust_cd.equalsIgnoreCase("")&& !cust_cd.equalsIgnoreCase("0") && !cust_cd.equalsIgnoreCase("all"))
{
    String sql = "select \"USER_NAME\" from \"FMS_CUST_MST\" where \"USER_CD\" = " + cust_cd;
    rs=stmt.executeQuery(sql);
```

### J63-F05: SQL Injection — getFormName() concatenates form_cd and RuntimeConf.form_table into SQL
**Description:** The method `getFormName()` constructs a SQL query by concatenating two externally-derived values into the SQL string. First, `RuntimeConf.form_table` is a static application configuration field used as the table name — if this value is ever settable via a configuration interface or property file it represents a second-order injection point. Second, `form_cd` is concatenated directly into the WHERE clause without parameterisation. The combination means both the table name and the filter value are injection vectors. Unlike the other methods, the string version uses single-quote delimiters around `form_cd` (suggesting it is treated as a string type in the database), but this does not prevent SQL injection — a value such as `' OR '1'='1` would still escape the literal.
**Severity:** CRITICAL
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL Injection)

### J63-F06: Missing Customer (Tenant) Scoping in All CustomerDAO Queries
**Description:** None of the four lookup methods in `CustomerDAO` — `getCustomerName()`, `getLocationName()`, `getDepartmentName()`, `getModelName()`, or `getFormName()` — includes a predicate that constrains the query to the authenticated user's customer or tenant. In a multi-tenant application (which this branch, `multi-customer-sync-to-master`, explicitly is), every query against master data tables must be filtered to the rows belonging to the requesting user's customer. As written, a caller can supply any `cust_cd`, `loc_cd`, `dept_cd`, or `model_cd` value and retrieve the corresponding name from any customer's data. This is an IDOR (Insecure Direct Object Reference) vulnerability: knowledge of a valid ID belonging to a different customer is sufficient to leak that customer's data.

The tables affected are: `FMS_CUST_MST` (customer master), `FMS_LOC_MST` (location master), `FMS_DEPT_MST` (department master), `FMS_VEHICLE_TYPE_MST` (vehicle type master), and `RuntimeConf.form_table` (form/report master). None of these queries includes a `customer_id` or equivalent tenant-discriminating WHERE clause.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and Multi-tenancy (IDOR / cross-customer data leakage)
**Evidence:**
```java
// getCustomerName — CustomerDAO.java line 32: no tenant column in WHERE clause
String sql = "select \"USER_NAME\" from \"FMS_CUST_MST\" where \"USER_CD\" = " + cust_cd;

// getLocationName — CustomerDAO.java line 70: no tenant column in WHERE clause
```

### J63-F07: Stack Traces Printed to Server Output in All Exception Handlers
**Description:** Every `catch` block in `CustomerDAO` calls `e.printStackTrace()`. In a production Tomcat deployment this writes the full Java stack trace — including class names, method names, line numbers, and SQL error messages (which may include the malformed SQL constructed from injected input) — to `catalina.out` or the container's standard error stream. If logging is misconfigured or if error output is returned to the user in any way, this leaks internal implementation details (table names, column names, Java package structure) to an attacker. Even when not shown to the user, this pattern means SQL injection payloads will produce verbose diagnostic output in server logs that the attacker can potentially observe through timing differences or error responses, aiding blind SQL injection attacks.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (error responses leaking implementation details)
**Evidence:**
```java
// CustomerDAO.java lines 41–44 (same pattern at lines 78–81, 115–118, 152–155, 190–193)
```

### J63-F08: Logical Defect in getFormName() Guard Condition — form_cd Input Validation Ineffective
**Description:** The guard condition in `getFormName()` at lines 177–179 tests `formName` (the local variable initialised to `"unknown report"`) rather than `form_cd` (the method parameter) against empty-string, `"0"`, and `"all"`. Because `formName` is initialised to `"unknown report"` and never equals `""`, `"0"`, or `"all"`, the condition `!formName.equalsIgnoreCase("") && !formName.equalsIgnoreCase("0") && !formName.equalsIgnoreCase("all")` is always `true`. The only operative guard is `!form_cd.equalsIgnoreCase("")`. This means the empty-string, zero, and "all" suppression checks on `form_cd` are entirely non-functional. The query will execute for any non-empty `form_cd` value, including `"0"` and `"all"`, contrary to the apparent developer intent. While this is primarily a logic error, it widens the SQL injection surface by allowing inputs the developer intended to reject.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL Injection — widened attack surface due to defective input filtering)
**Evidence:**

### J63-F09: Raw (Unparameterised) ArrayList Types Used as Bean Fields
**Description:** Both `UtilWowReportBean` and `UtilisationReportBean` declare multiple `ArrayList` fields without generic type parameters (raw types). This is a Java type-safety issue rather than a direct security vulnerability on its own, but it contributes to security risk in this codebase because raw `ArrayList` instances populated from JDBC result sets may carry mixed types, making validation and sanitisation of values before output (e.g., in JSP) harder to enforce. It also means the compiler cannot detect type errors, increasing the likelihood that unescaped user-controlled data is passed through these lists to JSP output expressions.
**Severity:** INFO
**Checklist item:** Section 3 — Injection Vulnerabilities (XSS — facilitating unsafe output in JSP consumers)
