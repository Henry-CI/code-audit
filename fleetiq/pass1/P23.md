### P23-F01: Pervasive Unescaped `<%= %>` Output — Reflected and Stored XSS
**Description:** Every JSP file in this set outputs values directly using raw `<%= %>` expressions with no HTML escaping. This affects both database-sourced values (stored XSS if a user-controlled string reaches the database) and request-parameter values reflected back into the response (reflected XSS). No JSTL `<c:out>` or `${fn:escapeXml()}` is used anywhere. The pattern is universal across all 10 files. Particularly high-severity instances are documented individually below, but the root cause is the same: zero output encoding discipline throughout these files.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// frm_mastercode_step2.jsp line 305
<h2>Send Master Codes to Customer: <%=get_user%> & Site: <%=get_loc%> & Department: <%=get_dep%>.</h2>

// frm_new_customer.jsp line 278
<TEXTAREA NAME="addr" ROWS="3" cols="60"><%=usr_addr %></TEXTAREA>

// frm_new_driver.jsp line 907
<td align=center colspan = 4 style="color: red" id="message">
&nbsp; <%=message %>
</td>

// frm_new_group.jsp line 207
<%=message %>
```

### P23-F02: Reflected XSS via `message` Parameter
**Description:** All nine state-mutating JSP pages accept a `message` request parameter and reflect it directly into the HTML response body without any escaping. This is textbook reflected XSS: an attacker can craft a URL with `message=<script>...</script>` and trick a logged-in user into clicking it, executing arbitrary JavaScript in the context of the application. Affected files: frm_mastercode_step2.jsp (line 453), frm_new_branch.jsp (line 199), frm_new_branch_old.jsp (line 163), frm_new_customer.jsp (line 321), frm_new_department.jsp (line 143), frm_new_department_win.jsp (line 100), frm_new_division.jsp (line 143), frm_new_driver.jsp (line 907), frm_new_group.jsp (line 207).

### P23-F03: Database Values Injected as Unescaped JavaScript Arguments in Event Handlers
**Description:** Multiple files inject database-sourced strings (branch names, location names, department names, group names, addresses, descriptions) directly as arguments to inline JavaScript functions embedded in `onclick` attributes, with no escaping. If any of these database values contains a single quote, double quote, or JavaScript metacharacter, it breaks out of the string literal context and executes arbitrary JavaScript. This constitutes stored XSS — any user or admin who can create/edit these records can inject code that executes in any admin's browser when viewing management pages.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
