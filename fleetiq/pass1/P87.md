### P87-F01: No Authentication Check on Any of the Four JSP Pages
**Description:** None of the four JSP files — `get_mail_data.jsp`, `get_module_data.jsp`, `output.jsp`, or `threshold_par.jsp` — perform any session validation, role check, or authentication guard. Any unauthenticated user who can reach these URLs (directly or via path traversal) can invoke them. `output.jsp` and `threshold_par.jsp` perform database write operations (UPDATE, INSERT) with no authentication whatsoever.
**Severity:** CRITICAL
**Checklist item:** Section 2 — Authentication and Authorization
**Evidence:**

**Description:** `get_mail_data.jsp` and `get_module_data.jsp` accept a `q` parameter and pass it directly to the `Databean_security` bean without any check that the requested module or mail group belongs to the authenticated user's customer. `output.jsp` and `threshold_par.jsp` accept `cust_cd`, `loc_cd`, and `dept_cd` parameters but never use them in any query — the actual data operated on is determined solely by the content of a server-side file (`D:/reset.csv` / `D:/PAR_messages.log`), with no tenant scoping at all. ### P87-F03: SQL Injection via String Concatenation in output.jsp
**Description:** `output.jsp` uses `PreparedStatement` in name only. The SQL strings are built by direct string concatenation of `gmtp_id` (read from the file `D:/reset.csv`) and `msg` (derived from a database read result multiplied by 10) before being passed to `prepareStatement()`. Using `prepareStatement()` with a pre-concatenated string provides no protection against SQL injection — the parameterization only helps when `?` placeholders are used. Five separate queries in this file are affected: one SELECT, one UPDATE, and three involving the outgoing/outgoing_stat tables. If the content of `D:/reset.csv` can be influenced (e.g., it is a log file written by another process), an attacker controlling that file's content controls the SQL.
**Severity:** HIGH
**Checklist item:** Section 3 — SQL Injection
**Evidence:**

### P87-F05: XSS in get_mail_data.jsp — Unescaped Database Value Written to XML Response
**Description:** `get_mail_data.jsp` retrieves `mail_id` from the database via `filter.getMail_grp_lst()` and directly concatenates it into an XML response string without any escaping. If a mail group name stored in the database contains XML special characters or XML/HTML markup (e.g., injected by an admin through another form), the XML structure of the response is broken and the value is reflected verbatim. Since the `contentType` is set to `text/xml` but the outer page directive declares `text/html`, and this response is likely consumed by client-side JavaScript, unescaped content from the DB can lead to XSS in the consuming page.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```java

String mail_id = "";
for(int i=0;i<vmail_id.size();i++)
{

}
// ...
resp=resp+"<rec>"+"<name>"+mail_id+"</name></rec>";  // line 32 — mail_id not escaped
out.println(resp);  // line 36
```
**File:** security/get_mail_data.jsp
**Line(s):** 15–19, 32, 36
