### J44-F04: SSRF — Unvalidated URL Fetched by `addImageUtilisationChart` in `EmailUtilWowReport`
**Description:** `EmailUtilWowReport.createCurrReport()` retrieves `chartUrl` from `dbreport.getChartURL()` and passes it directly to `addImageUtilisationChart(sheet, chartUrl, 18, true, 0)`. The parent class `Frm_excel.addImageUtilisationChart()` (and the similar `addImage2()`) call `new URL(image).openStream()` unconditionally. Even without direct user control, the design pattern is inherently unsafe: any chart URL construction that incorporates user data becomes an SSRF vector.

### J44-F05: Missing Customer-Scoping Verification — `cust_cd = "0"` Bypasses Tenant Filter
**Description:** All three report classes normalise a `cust_cd` value of `"0"` to `""` (empty string) before passing it to `Databean_report`. In `Databean_report`, an empty `set_cust_cd` causes the customer filter condition (`cond_cust`) to be omitted entirely from queries, so a query is executed across all customers (e.g., line 5448: `"select \"VEHICLE_CD\" from \"FMS_USR_VEHICLE_REL\""` with no WHERE clause). The normalisation logic in all three files is identical:
```java
if (cust_cd.equalsIgnoreCase("0")) { cust_cd = ""; }
```
This means that supplying the string `"0"` as the customer parameter triggers a cross-tenant data dump — every vehicle, every auth record, or every unlock record in the database is returned. On the `multi-customer-sync-to-master` branch, multi-tenancy isolation is the critical requirement. A caller passing `cust_cd=0` to any of these report generators would receive data from all tenants.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and Multi-Tenancy: "verify that every data access is scoped to the authenticated user's customer/organisation."
**Evidence:**
```java
// EmailSuperMasterAuthReport.java lines 70-73
if (cust_cd.equalsIgnoreCase("0")) {
    cust_cd = "";
}

// EmailUnitUnlockReport.java lines 40-42
if (cust_cd.equalsIgnoreCase("0")) {
    cust_cd = "";
}

// EmailUtilWowReport.java lines 54-57
if (cust_cd.equalsIgnoreCase("0")) {
    cust_cd = "";
}

// Databean_report.java line 5446-5449 — consequence: no WHERE filter on customer

    query = "select \"VEHICLE_CD\" from \"FMS_USR_VEHICLE_REL\"";
}
```

## Checklist Items With No Issues

The following checklist sections were examined against the three reviewed files and no findings were raised:

- **Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, SMTP passwords, or encryption keys found in any of the three source files. Connection setup is delegated entirely to `Databean_report` and its underlying DAO layer; no `DriverManager.getConnection()` calls or connection strings appear in these files.
- **Section 3 — File Inclusion:** No `<jsp:include>` or equivalent dynamic file inclusion is present — these are Java classes, not JSP files.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls found in any of the three files or in `Frm_excel`.
- **Section 4 — Session and CSRF:** These are server-side Excel report generation classes; they do not handle HTTP requests, session management, or form processing directly. CSRF does not apply at this layer.
- **Section 6 — Transport Security:** These classes generate Excel files and do not configure TLS, connectors, or HTTP communication (the `addImageUtilisationChart` SSRF concern is reported under Section 3 above; transport security of that outbound call is a secondary aspect of that finding).
- **Section 7 — Dependency Vulnerabilities:** The files use Apache POI for Excel generation. No version is determinable from these source files alone — dependency version analysis requires inspection of `pom.xml` or committed JARs, which is out of scope for this agent's assigned files.
