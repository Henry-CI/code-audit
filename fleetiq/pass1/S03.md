## Findings

### S03-F01: Dynamic SQL with Unsanitised Table Name in sp_equipment_bypass_alert
**Description:** The function constructs a dynamic `SELECT` statement using `EXECUTE format(...)` where the `lang` variable is interpolated as a table name via `%I`. The `lang` variable is derived from the `emailUsersRecord.pref_lang` column (a database value), which is set conditionally: `lang := 'email_layout_' || emailUsersRecord.pref_lang;`. Although this is a database-sourced value rather than direct user input, the `pref_lang` column value could be manipulated if an attacker can write to the `mymessages_users` table or if the column is populated from user-controlled registration data. The `%I` quoting in `format()` reduces but does not eliminate risk — an attacker who can control `pref_lang` content can target any table in the schema.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — SQL injection
**Evidence:**
```sql
-- Lines 112-114 (first occurrence), lines 153-155 (second occurrence)
EXECUTE format('SELECT * FROM %I '
                   'WHERE email_type = ''%I''', lang,emailType)
    INTO email_layout_rec;
```
The `lang` variable is set at line 107:
```sql
lang := 'email_layout_' || emailUsersRecord.pref_lang;
```
**File:** `sql/sp_equipment_bypass_alert.sql`
**Line(s):** 107, 112–114, 153–155

### S03-F02: sp_equipment_bypass_alert Owned by postgres Superuser
**Description:** The function `sp_equipment_bypass_alert` is assigned to the `postgres` superuser (`OWNER TO postgres`). All other functions in this file set are owned by the `gmtp` application role. If this function is exploitable (e.g., via the dynamic SQL finding above), it executes with superuser-level database privileges, allowing an attacker to read from or write to any table, execute any function, or potentially use `COPY TO/FROM` for file system access.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization
**Evidence:**
```sql
-- Line 175-176
ALTER FUNCTION public.sp_equipment_bypass_alert(integer, character varying, integer, integer)
    OWNER TO postgres;
```
**File:** `sql/sp_equipment_bypass_alert.sql`
**Line(s):** 175–176

### S03-F03: GRANT EXECUTE TO PUBLIC on sp_get_driver_name
**Description:** The function `sp_get_driver_name` is granted `EXECUTE` to the `PUBLIC` role, meaning any database user — including minimally privileged application users and any user who gains a database connection — can invoke this function. This function returns driver names (PII) given a vehicle identifier and card ID. The `PUBLIC` grant is overly broad; only the application role (`gmtp`) should have execute permission.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization; 5. Data Exposure and Privacy
**Evidence:**
```sql
-- Line 107
GRANT EXECUTE ON FUNCTION public.sp_get_driver_name(character varying, character varying) TO PUBLIC;
```
**File:** `sql/sp_get_driver_name.sql`
**Line(s):** 107

### S03-F04: Weaker Multi-Tenant Driver Lookup in sp_store_mk3dbg
**Description:** The driver lookup in `sp_store_mk3dbg` at line 99 uses a different (less strict) tenant-scoping subquery compared to equivalent lookups in `sp_pstat_message` and `sp_opchks_response`. The latter two functions scope the technician fallback path through `dealer_user_rel` and `dealer_cust_rel`, correctly tying technicians to the vehicle's customer. The lookup in `sp_store_mk3dbg` instead uses `"is_technician" is true` without any customer or dealer scoping, meaning a technician belonging to a different customer's dealer could potentially be resolved as the driver for this vehicle's VOR event.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization — multi-tenancy; Injection Vulnerabilities — IDOR
**Evidence:**
```sql
-- sp_store_mk3dbg line 99 (weaker):
SELECT INTO driv_cd "USER_CD" from "FMS_USR_MST"
where "CARD_ID" ilike driv_wieg
  and "ACTIVE" is true
  and ("USER_CD" in (select "USER_CD" from "FMS_USER_DEPT_REL" where "CUST_CD"=c_cd and "LOC_CD" = s_cd)
    OR "USER_CD" in (select "USER_CD" from "FMS_USR_MST" where "is_technician" is true));

-- sp_pstat_message lines 155-157 (correct, with dealer scoping):
SELECT INTO diver_cd "USER_CD" from "FMS_USR_MST"
where "CARD_ID" ilike cardID and "ACTIVE" is true
  and ("USER_CD" in (select "USER_CD" from "FMS_USER_DEPT_REL" where "CUST_CD" = c_cd and "LOC_CD" = s_cd)
    OR "USER_CD" in (SELECT "FMS_USR_MST"."USER_CD" FROM "FMS_USR_MST", "FMS_USER_DEPT_REL"
        WHERE ... and "FMS_USR_MST"."USER_CD" in (select user_id from dealer_user_rel
            where dealer_id=(select dealer_id from "dealer_cust_rel" where cust_id = c_cd))
        and is_technician is true));
```
**File:** `sql/sp_store_mk3dbg.sql`
**Line(s):** 99

### S03-F05: time_zone_impl (varchar overload) Hardcodes NSW as Base Timezone
**Description:** The varchar/integer overload of `time_zone_impl` hardcodes `'%NSW%'` as the base/reference timezone for all timezone arithmetic. This is a deployment assumption baked into a stored procedure rather than a configurable parameter. If the server is deployed in a different region, or if the `FMS_STATE_MST` table does not contain an NSW entry, timezone conversions will be silently incorrect. More critically, this is also an information disclosure: the hardcoded string `'%NSW%'` reveals the application's assumed server location in the source code committed to the repository.
**Severity:** LOW
**Checklist item:** 5. Data Exposure and Privacy
**Evidence:**
```sql
-- Line 42
select into tzs1,dst1 "STATE_TM_ZONE","DST" from "FMS_STATE_MST" where "STATE_NM" ilike '%NSW%';
```
**File:** `sql/time_zone_impl.sql`
**Line(s):** 42

### S03-F06: time_zone_impl (timestamptz overload) Silently Discards Timezone Conversion Result
**Description:** The `timestamptz` overload of `time_zone_impl` (in `time_zone_impl_varchar_int.sql`) executes a timezone conversion expression (`SELECT ret_ts AT TIME ZONE 'Australia/Sydney' AT TIME ZONE vehTzone;`) but does not assign the result to any variable. The function then returns `ret_ts` unchanged (line 35). This means the function always returns the original timestamp with no timezone adjustment, making the function a no-op for all callers. Any feature relying on this overload for timezone correction silently receives uncorrected UTC data displayed as local time. This is a functional defect that also has a data-accuracy implication for fleet operators interpreting event timestamps.
**Severity:** MEDIUM
**Checklist item:** 5. Data Exposure and Privacy (incorrect timestamp data presented to users)
**Evidence:**
```sql
-- Lines 30-35
SELECT ret_ts AT TIME ZONE 'Australia/Sydney' AT TIME ZONE vehTzone;
-- result not assigned to ret_ts or any variable

-- ...
return ret_ts;  -- returns the original, unadjusted value
```
**File:** `sql/time_zone_impl_varchar_int.sql`
**Line(s):** 30, 35

### S03-F07: time_zone_impl Hardcodes 'Australia/Sydney' as Conversion Source
**Description:** The `timestamptz` overload of `time_zone_impl` hardcodes `'Australia/Sydney'` as the source timezone in the AT TIME ZONE conversion. This assumes all incoming timestamps are in AEST/AEDT regardless of actual origin. For a multi-customer, potentially multi-region deployment (as implied by the branch name `multi-customer-sync-to-master` and the EU timezone branch in the varchar overload), this assumption is incorrect for non-Australian deployments.
**Severity:** LOW
**Checklist item:** 5. Data Exposure and Privacy
**Evidence:**
```sql
-- Line 30
SELECT ret_ts AT TIME ZONE 'Australia/Sydney' AT TIME ZONE vehTzone;
```
**File:** `sql/time_zone_impl_varchar_int.sql`
**Line(s):** 30

### S03-F08: Verbose RAISE NOTICE Logging of Sensitive Operational Data
**Description:** Multiple stored procedures emit `RAISE NOTICE` messages containing sensitive operational data that would appear in the PostgreSQL server log. In `sp_pstat_message`, vehicle ID and vehicle code are logged on every call (lines 106–107). In `time_zone_impl` (varchar overload), every computed timestamp is logged (line 127). In `sp_get_driver_name`, the GMTP ID of any vehicle with a failed firmware check is logged (line 34). In production, PostgreSQL `log_min_messages` may be set to `NOTICE` or lower, causing this data (vehicle identifiers, timestamps) to appear in log files. Log files may be accessible to operations staff without a need-to-know, stored without encryption, or forwarded to centralised logging systems without appropriate access controls.
**Severity:** LOW
**Checklist item:** 5. Data Exposure and Privacy — Logging
**Evidence:**
```sql
-- sp_pstat_message lines 106-107
RAISE NOTICE 'VEHICLE_ID %', $3;
RAISE NOTICE 'VEHCD %', veh_cd;

-- time_zone_impl.sql line 127
RAISE NOTICE 'final-ret_ts %',ret_ts;

-- sp_get_driver_name line 34
RAISE NOTICE 'SHOW_NAME and/or firmware version check failed for vehicle: %', $1;
```
**Files:** `sql/sp_pstat_message.sql` (lines 106–107), `sql/time_zone_impl.sql` (line 127), `sql/sp_get_driver_name.sql` (line 34)
**Line(s):** Various

### S03-F09: op_chk_checklistanswer Inserts Not Scoped to Customer in sp_opchks_response
**Description:** When inserting into `op_chk_checklistanswer`, the function does not include `veh_cd` or `driver_cd` as filters in the duplicate check (line 559), only `checklist_result_id` and `question_id`. The unsync variant (line 544) does include `veh_cd` in the duplicate check. This inconsistency means that for synced checklist answers, the deduplication logic does not scope by vehicle, potentially allowing a checklist result from one vehicle to suppress an answer insertion intended for a different vehicle if sequence IDs collide.
**Severity:** LOW
**Checklist item:** 2. Authentication and Authorization — multi-tenancy
**Evidence:**
```sql
-- Line 559: missing veh_cd scope in synced answer dedup check
Select into answerCount COUNT(*) from op_chk_checklistanswer
where checklist_result_id = uniq_id::integer and question_id = $6 and answer = response;

-- Line 544: unsync variant correctly includes veh_cd
Select into answerCount COUNT(*) from op_chk_checklistanswer_unsync
where checklist_result_id = uniq_id::integer and question_id = $6 and answer = response and veh_cd = veh_cds;
```
**File:** `sql/sp_opchks_response.sql`
**Line(s):** 544, 559
