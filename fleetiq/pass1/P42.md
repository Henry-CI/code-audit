## Findings

### P42-F01: Reflected XSS via `message` request parameter in question_unlock_report.jsp
**Description:** The `message` request parameter is read from the URL without any sanitisation and output directly as raw HTML into the page. An attacker can craft a URL with a `message` parameter containing HTML or JavaScript that will execute in the victim's browser.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// Line 19 — parameter read without sanitisation:
String message = request.getParameter("message")==null?"":request.getParameter("message");

// Line 130 — raw output into HTML, no escaping:
<td align=center style="color: red" colspan="7" >  &nbsp; <%=message %></td>
```
**File:** pages/admin/reports/question_unlock_report.jsp
**Line(s):** 19, 130

---

### P42-F02: Reflected XSS via `mnm` and `sub` request parameters in hidden form inputs (multiple files)
**Description:** The request parameters `mnm` and `sub` are read directly and emitted without escaping into `value` attributes of hidden `<input>` elements. An attacker who can control these URL parameters can inject HTML that breaks out of the attribute context and executes JavaScript. This pattern appears in at least six files.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// report_subscription.jsp lines 169–170:
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">

// report_subscription_au.jsp lines 147–148 (identical pattern)
// report_subscription_uk.jsp lines 165–166 (identical pattern)
```
