## Findings

### J08-F01: `master_code` Security-Sensitive Field Stored and Exposed as Plain String
**Description:** `LockOutBean` contains a field named `master_code` (line 17). In a fleet management context, a "master code" for a vehicle lockout/unlock system is a highly security-sensitive credential — it is the override code that unlocks a vehicle. This value is stored as a plain `String` with no hashing, encryption, or other protection. It is exposed directly through the public getter `getMaster_code()` (line 49). Furthermore, because the field itself is package-private (no access modifier), any class within `com.torrent.surat.fms6.bean` can read it without even invoking the getter. If this bean is ever serialised to an HTTP session (the class implements `Serializable`) or included in a serialised response, the raw master code travels with it. Without reviewing the callers it cannot be confirmed how this value is used, but storing a vehicle override code as an unprotected plain string in a serialisable bean is a design-level security deficiency.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials; Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// LockOutBean.java, lines 17 and 49-51
String master_code = "";          // line 17 — package-private, no protection

public String getMaster_code() {  // line 49
    return master_code;           // line 50 — returns raw value
}                                 // line 51
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/LockOutBean.java`
**Line(s):** 17, 49–51

### J08-F02: Package-Private Fields on Serialisable Beans Bypass Encapsulation
**Description:** Both `LockOutBean` (lines 11–17) and `MaxHourUsageBean` (lines 13–15) declare their data fields with no access modifier, making them package-private. This means every class in `com.torrent.surat.fms6.bean` can read and write these fields directly, bypassing the public getter/setter pairs entirely. If validation or sanitisation logic is later added to setters, direct field access from within the package would silently bypass it. For `LockOutBean`, this is particularly significant because `master_code` is directly accessible to all package peers. Both classes implement `Serializable`, so these fields are included in serialisation output (e.g., to HTTP sessions or any object stream) without encapsulation protection.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization; Section 5 — Data Exposure and Privacy
**Evidence:**

### J08-F03: No Customer/Tenant Scoping in `LockOutBean` or `MaxHourUsageBean`
**Description:** Neither `LockOutBean` nor `MaxHourUsageBean` contains a customer identifier (`custCd`, `customerId`, `tenantId`, or equivalent). The application is on the `multi-customer-sync-to-master` branch, which indicates multi-tenant operation. `LicenseBlackListBean` does carry `custCd`, `access_level`, `access_cust`, `access_site`, and `access_dept` fields (lines 8–16), demonstrating the team is aware of the multi-tenant model. The absence of any customer identifier in `LockOutBean` and `MaxHourUsageBean` means tenant isolation for lockout records and vehicle utilisation data cannot be enforced at the bean layer. If the calling query or servlet fails to apply a customer filter, cross-tenant data leakage would be invisible at the bean level. This is a structural gap that increases the risk of IDOR and cross-tenant data exposure if any query in the data access layer omits a WHERE clause for customer scope.

### J08-F04: `LicenseBlackListBean` Does Not Implement `Serializable` Despite Carrying Access-Control State
**Description:** `LicenseBlackListBean` holds access-control fields (`access_level`, `access_cust`, `access_site`, `access_dept`) that appear to represent the multi-tenant scope of the authenticated session. The class does not implement `Serializable` (no `implements Serializable`, no `serialVersionUID`). If instances of this bean are stored in the HTTP session for access-control purposes, Tomcat session persistence or replication (where enabled) may silently skip serialising this object, which could result in access-control fields being lost on session restore — potentially causing a user to be treated as having no access restrictions, or causing a null-pointer error that exposes an error page. This is an architectural concern rather than a confirmed exploitable vulnerability at the bean level, but it warrants investigation of how this bean is used in the session.
**Severity:** LOW

### J08-F05: `model_img` Field May Carry Unsanitised Path or URL for Image Rendering
**Description:** `MaxHourUsageBean` contains a `model_img` field (line 14) with a public getter (line 23). The field name suggests it stores a path or URL to a vehicle model image. If this value is populated from a database column and rendered in a JSP page using a raw `<%= bean.getModel_img() %>` expression (without escaping), it is an XSS vector. If it is used in an `<img src="...">` attribute, a stored value containing a JavaScript URI (`javascript:...`) or a relative path pointing to sensitive files could be exploited. The risk cannot be confirmed without reviewing the JSP(s) and servlet(s) that consume this bean, but the field is flagged here as requiring follow-up in the presentation layer review.
**Severity:** LOW (flag for follow-up in JSP review)
