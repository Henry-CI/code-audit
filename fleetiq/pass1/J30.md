### J30-F02: Multi-Tenant Data Isolation Not Enforced at DAO Layer — UnitDAO.java
**Description:** The `cust_cd` (customer code) used to scope data to the authenticated tenant is accepted as a plain `String` parameter passed into each DAO method. The DAO performs no verification that the supplied `cust_cd` matches the authenticated user's session. Any caller that passes an arbitrary or attacker-controlled `cust_cd` will retrieve data belonging to a different customer. This is a structural IDOR: if the calling servlet or JSP forwards a request parameter as `cust_cd` without re-validating against the session, an authenticated user from customer A can query data for customer B by changing the `cust_cd` value. The branch is named `multi-customer-sync-to-master`, indicating multi-tenancy is a deliberate feature, making this isolation gap especially serious.

Additionally, `getUnitSummary(String type)` at line 1050 does not accept a `cust_cd` at all — it queries across all customers filtered only by a `type` string ("summary" or "summary_apac"), with the filter being a hardcoded dealer ID list. This method returns vehicle data including CCID (SIM card identifiers), firmware versions, customer names, location names, and department names for vehicles across multiple customers with no per-session customer check.

**Severity:** CRITICAL

**Checklist item:** Section 2 — Authorization and multi-tenancy (IDOR / multi-tenant data isolation)

**Evidence:**

### J30-F04: Stack Traces Printed to Standard Error in Every Catch Block
**Description:** Every `catch` block in UnitDAO.java calls `e.printStackTrace()` followed by the no-op `e.getMessage()`. `e.printStackTrace()` writes the full exception including SQL error text, table names, column names, and Java class/method stack frames to standard error. In a Tomcat deployment this typically writes to `catalina.out` or a similar server log. More critically, if any surrounding JSP or servlet forwards exception information to the response (a common legacy pattern), this leaks the full SQL query text (including concatenated parameters) and the internal class hierarchy to the end user. Even without forwarding, SQL error messages in logs confirm table structure to anyone with log access, aiding an attacker post-breach.

**Severity:** MEDIUM

**Checklist item:** Section 5 — Data Exposure and Privacy (error responses)

**Evidence** (pattern repeating in every method — example at lines 59–62):
```java
}catch(Exception e)
{
    e.printStackTrace();
    e.getMessage();
}
```
`e.getMessage()` on its own does nothing (return value discarded) — the call is a dead statement, but `e.printStackTrace()` always executes and outputs to stderr.

### J30-F08: `UserLoginChart` Uses Deprecated Google Charts Image API
**Description:** `UserLoginChart.java` uses the `charts4j` library which wraps the Google Chart Tools Image API (chart.apis.google.com). Google officially deprecated and shut down this API in 2012. Any deployment that still calls this API will receive errors or no chart. More importantly from a security perspective, the chart URL is constructed and returned as a String containing user-influenced data (the axis labels passed to `createChart`). Although URL-encoding is applied to axis labels (line 77), the URL is returned raw to the caller, which may embed it in an HTML page. If the caller does not properly handle or escape this URL, there is a secondary XSS risk in the output context.

**Severity:** INFO (functional / dependency concern; XSS risk is in the caller, not confirmed here)

## Checklist Items With No Issues

The following checklist items were examined for the three files reviewed and no issues were found within the scope of these files:

- **Section 1 — Secrets and Credentials:** No hardcoded database passwords, API keys, SMTP credentials, or JDBC connection strings with embedded passwords in any of the three files. `DBUtil.getConnection()` is called without inline credentials.
- **Section 3 — File inclusion:** No JSP file inclusion directives in these files (Java classes only).
- **Section 3 — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage in any of the three files.
- **Section 3 — SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`) in any of the three files. `UserLoginChart` constructs a Google Charts URL string but does not make an outbound HTTP connection itself.
- **Section 4 — Session and CSRF:** These are DAO and chart classes; they do not handle HTTP sessions or form submissions directly. No session or CSRF concerns are within scope for these files.
- **Section 6 — Transport Security:** These files contain no TLS configuration, connector setup, or cookie attribute settings.
- **Section 7 — Dependency Vulnerabilities:** Beyond the deprecated charts4j / Google Image Charts API noted in J30-F08, no other dependency issues are visible from imports in these three files. The `charts4j` import suggests a JAR is present in WEB-INF/lib — this JAR should be inventoried and checked for CVEs by the dependency audit agent.
- **Section 8 — Cross-branch comparison:** Cannot be completed by a single agent; deferred to orchestrating session after both FleetIQ and FleetFocus Pass 1 reports are available.
