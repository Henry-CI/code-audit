## Findings

### J16-F01: Package-private fields on UnusedUnitBean expose cross-customer data without access control
**Description:** All data-carrying fields in `UnusedUnitBean` are declared with package-private (default) visibility rather than `private`. This means any class in the same package (`com.torrent.surat.fms6.bean`) can read and write fields such as `cust_name`, `fleet_no`, `serial_no`, `gmtp_id`, `ccid`, `vcd`, and `last_report` directly, bypassing the getter/setter layer entirely. In a multi-tenant context this is particularly dangerous: if any in-package code constructs or populates these beans without going through the setters, there is no encapsulation boundary that could later be tightened with validation logic. The bean holds cross-customer identifiers (`cust_name`, `site_name`) that need to be treated as sensitive in a multi-tenant deployment.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy
**Evidence:**
```java

String site_name = "";

boolean onHire = true;
```

### J16-F02: Package-private fields on UserDriverBean expose driver PII without access control
**Description:** All fields in `UserDriverBean` are declared with package-private (default) visibility: `id`, `user_cd`, `weigand`, `veh_type`, `firstName`, `lastName`. Of particular concern is `weigand` (line 7), which is the Wiegand card/RFID credential number used for physical access control and driver identification. Driver first name, last name, and Wiegand credential are personally identifiable and sensitive data. Package-private visibility means any class in `com.torrent.surat.fms6.bean` can directly read or overwrite these fields, bypassing the setter methods entirely. There is no opportunity to add input validation in the setters later without refactoring callers that may be using direct field access.
**Severity:** MEDIUM

### J16-F03: UserBean contains no customer/tenant scoping field
**Description:** `UserBean` models an application user with fields `id`, `username`, `firstname`, `lastname`, `email`, and `lastUpdate`. There is no `customerId`, `tenantId`, `orgId`, or equivalent field present in the bean. In a multi-tenant application (branch `multi-customer-sync-to-master`), a user must be associated with exactly one customer/tenant, and this association must be enforced on every data access. The absence of a tenant-scoping field on the user bean means that wherever this bean is used to represent the authenticated principal, the consuming code cannot derive customer context from it and must obtain customer ID through a separate mechanism. This creates a structural risk: any code path that uses only a `UserBean` to authorise data access lacks the information necessary to scope the query to the correct tenant.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy
**Evidence:**
```java
// UserBean.java lines 3-9
public class UserBean {
    private int id;
    private String username;
    private String firstname;
    private String lastname;
    private String email;
    private String lastUpdate;
    // No customerId / tenantId / orgId field
```
**File:** WEB-INF/src/com/torrent/surat/fms6/bean/UserBean.java
**Line(s):** 3–9

### J16-F04: UnusedUnitBean contains cross-customer identifiers with no tenant-isolation enforcement
**Description:** `UnusedUnitBean` holds `cust_name` and `site_name` alongside device identifiers (`gmtp_id`, `ccid`, `serial_no`, `fleet_no`). The bean is a pure data carrier with no validation, no customer-scoping enforcement, and no relationship to an authenticated user's tenant. The name "UnusedUnit" together with the `inactive_days` and `last_report` fields indicates this bean surfaces in reports or dashboards listing vehicles that have not reported. In a multi-tenant deployment, if any query populating a list of these beans does not filter by the authenticated user's customer, units from other customers will be included. The bean itself provides no safeguard. This is a structural data-isolation risk specific to the multi-customer branch.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy (IDOR / multi-tenancy cross-customer data leakage)

### J16-F05: UserDriverBean has no customer/tenant scoping field — IDOR structural risk
**Description:** `UserDriverBean` models a driver/user relationship with fields `id`, `user_cd`, `weigand`, `veh_type`, `firstName`, `lastName`. Like `UserBean`, it contains no `customerId` or `tenantId` field. In a multi-tenant fleet system, drivers belong to specific customers. Any endpoint that retrieves a `UserDriverBean` by `id` alone (without enforcing customer membership) creates an IDOR vulnerability: an attacker could enumerate driver records across all tenants by varying the `id` parameter. The Wiegand credential value stored in this bean would be particularly sensitive if leaked across tenant boundaries.
**Severity:** HIGH

### J16-F06: Wiegand card credential stored as plain String with no masking or protection
**Description:** The `weigand` field in `UserDriverBean` stores what appears to be a Wiegand RFID card number — a physical access credential used for driver identification. It is stored as a plain unencrypted `String` with package-private visibility and a plain-text getter (`getWeigand()`) that returns the raw value. There is no indication of hashing, masking, or any access restriction. If this bean is serialised into a session, written to a log, or returned in an API response or JSP output, the raw Wiegand credential is exposed. Wiegand codes are used to authenticate drivers to vehicles and can be replayed for physical access fraud.
**Severity:** HIGH
**Checklist item:** 5. Data Exposure and Privacy — Driver data
**Evidence:**
```java

String weigand = "";

public String getWeigand() {
    return weigand;
```

**Line(s):** 7, 17–18
