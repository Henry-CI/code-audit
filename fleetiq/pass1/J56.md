## Findings

### J56-F01: Raw ArrayList fields used without generic type parameters — type safety absent
**Description:** All three bean classes declare their `ArrayList` fields without generic type parameters (e.g., `ArrayList` instead of `ArrayList<String>`). This is not itself a security vulnerability in isolation, but it means there is no compile-time type enforcement on what values are stored in these lists. Content from any source — including database queries or external input — can be placed in these lists without type verification. If downstream JSP or servlet code iterates over these lists and outputs values without type-checking, it contributes to the risk of unescaped output being rendered in reports (connecting to XSS risk at the consumer layer). This also makes auditing data flow significantly harder.
**Severity:** INFO
**Checklist item:** Section 3 — Injection Vulnerabilities (XSS) — raw collections reduce auditability of data flow to output
**Evidence:**

### J56-F02: Package-private field access on PII-containing bean fields — missing encapsulation
**Description:** All fields in all three beans are declared without an access modifier, which in Java means package-private (default) access. Any class in the same package (`com.torrent.surat.fms6.excel.reports.beans`) can directly read and write these fields without going through the getter/setter methods, bypassing any future validation logic added to those accessors. For `CurrDrivReportBean` and `CurrUnitReportBean`, which carry driver names (`Vrpt_veh_driv_nm`, `Vrpt_driv_nm`) and driver codes (`Vrpt_veh_driv_cd`, `Vrpt_driv_cd`) — which are PII — this means data can be injected or read from outside the intended access path. The fields should be `private`.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy — driver names and identifiers are PII; uncontrolled access paths increase risk of inadvertent exposure
**Evidence:**

### J56-F03: No customer/tenant scoping field present in any report bean — multi-tenancy isolation risk
**Description:** The branch under audit is named `multi-customer-sync-to-master`, indicating a multi-tenant deployment. None of the three report beans contain any field representing a customer ID, tenant ID, organisation ID, or equivalent scoping identifier. `CimplicityUtilReportBean` carries vehicle codes and names; `CurrDrivReportBean` and `CurrUnitReportBean` carry vehicle IDs, vehicle codes, vehicle names, driver codes, and driver names. Without a customer ID field in the bean, there is no structural mechanism within the bean to record or verify which customer's data it holds. If the population logic (servlet or DAO layer, not visible in these files) fails to scope the query to the authenticated customer, these beans will silently carry cross-tenant data with no in-bean indicator that anything is wrong. This is a data isolation concern that must be traced to the population layer. The absence of a tenant scoping field in a bean that holds fleet and driver data on a multi-customer branch is a significant design gap.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and Multi-tenancy — every data retrieval in a multi-tenant system must be scoped to the authenticated user's customer; the bean structure provides no scaffolding for this
**Evidence:**

### J56-F04: CurrDrivReportBean and CurrUnitReportBean are structurally identical — likely copy-paste duplication
**Description:** `CurrDrivReportBean` (119 lines) and `CurrUnitReportBean` (119 lines) are byte-for-byte identical in structure: same 15 fields with the same names and types, same 30 public getter/setter methods with the same signatures, same package, same single import. The only difference is the class name. This is a strong indicator of copy-paste development. Duplicate bean classes that carry driver names and vehicle IDs in a multi-tenant report context mean that any security fix or validation added to one class will not automatically propagate to the other. If the population logic differs between the two (e.g., one is populated with cross-tenant data due to a missing filter), the duplication makes it harder to detect the discrepancy. This is flagged as a structural risk factor rather than a standalone vulnerability.
**Severity:** INFO
**Checklist item:** Section 2 — Authorization and Multi-tenancy; Section 5 — Data Exposure and Privacy — duplicate data-carrying classes with PII fields increase the attack surface if one copy is fixed and the other is not

## Checklist Items With No Issues

The following checklist sections were reviewed against the three files and no issues were found within the scope of these files alone. Note that these are pure data-transfer bean classes with no SQL, no HTTP handling, no authentication logic, no output rendering, and no external connections; many checklist items are not directly applicable to this file type but are noted for completeness.

- **Section 1 — Secrets and Credentials:** No hardcoded credentials, passwords, API keys, JDBC connection strings, or comments containing secrets. No `.env` or `secrets.properties` references. No `DriverManager.getConnection()` or equivalent calls.
- **Section 3 — Injection Vulnerabilities (SQL injection):** No SQL statements, no `Statement` or `PreparedStatement` usage, no string concatenation in queries. These classes contain no database access logic whatsoever.
- **Section 3 — Injection Vulnerabilities (XSS):** No JSP output expressions, no `<%= %>` usage, no HTML rendering. These are pure Java bean classes. XSS risk, if present, is in the JSP consumer layer (not in scope for this agent).
- **Section 3 — Injection Vulnerabilities (File inclusion):** No file path construction, no `include` directives, no `File` or `Path` usage.
- **Section 3 — Injection Vulnerabilities (Command injection):** No `Runtime.getRuntime().exec()`, no `ProcessBuilder`, no external process invocation.
- **Section 3 — Injection Vulnerabilities (SSRF):** No `HttpURLConnection`, `HttpClient`, or URL construction present.
- **Section 4 — Session and CSRF:** No session access, no CSRF token logic, no form handling. Not applicable to bean classes.
- **Section 6 — Transport Security:** No network or connector configuration present in these files.
- **Section 7 — Dependency Vulnerabilities:** Single import (`java.util.ArrayList`) from the Java standard library. No third-party dependencies introduced by these files.
