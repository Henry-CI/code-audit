### J39-F01: SQL Injection — CustomerDAO.getCustomerName() — Unparameterised cust_cd
**Description:** `CustomerDAO.getCustomerName()` builds a SQL query by directly concatenating the `cust_cd` parameter into the query string using a `Statement` (not a `PreparedStatement`). The `cust_cd` value originates from the `params[]` array passed to the report constructors, which is ultimately sourced from the caller (scheduled email job or report trigger). An attacker who can influence the `cust_cd` parameter can inject arbitrary SQL.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL injection)
**Evidence:**
```java
// CustomerDAO.java, line 32
String sql = "select \"USER_NAME\" from \"FMS_CUST_MST\" where \"USER_CD\" = " + cust_cd;
rs=stmt.executeQuery(sql);
```
`cust_cd` is the raw value passed in from `Frm_excel.cust_cd` which is set from constructor parameters with only a string equality check against `"0"` and `""`.
**File:** `WEB-INF/src/com/torrent/surat/fms6/excel/reports/dao/CustomerDAO.java`
**Line(s):** 32–33

### J39-F06: Excel Formula Injection — Database Values Inserted Into Cell Formulas
**Description:** All three report classes construct Excel cell formulas by string-concatenating time values retrieved from the database directly into `VALUE("...")` formula strings. If any database record contains a value beginning with `=`, `+`, `-`, or `@`, or containing `"` characters, this could produce a malformed or maliciously crafted formula in the output `.xlsx` file. While the primary vector here is database content rather than direct user input, it still represents a formula injection risk if database values have ever been influenced by user-supplied strings (e.g. driver names or vehicle names stored in time fields due to data quality issues, or via SQL injection into upstream data).
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**
```java
// EmailDailyVehSummaryReport.java, line 193
```

### J39-F09: Stack Traces Exposed in Application Logs via e.printStackTrace()
**Description:** All five methods in `CustomerDAO` catch exceptions and call `e.printStackTrace()`. This prints full stack traces — including internal class names, method chains, SQL queries, and driver error messages — to the standard error stream, which in Tomcat is typically the `catalina.out` log file. If that log is accessible (e.g. via a misconfigured log shipping endpoint, or if an attacker gains read access to the server), this significantly aids reconnaissance. Additionally, the pattern `e.getMessage()` is called without assigning or using the return value (a no-op), indicating the error handling is incomplete and likely copied without understanding.

**Checklist item:** Section 5 — Data Exposure and Privacy (error responses)

### J39-F11: Raw (Unchecked) ArrayList Usage Throughout
**Description:** All three report classes use raw `ArrayList` (without type parameters) for the primary data collections returned from `Databean_report`. Raw collections suppress compile-time type checks and require unchecked casts, which can cause silent runtime `ClassCastException` errors. More relevant to security: unchecked collections from database result sets mean unexpected data types (e.g. `null` values, unexpected numeric types) will not be detected early and could cause NPEs or unexpected output that bypasses conditional formatting logic (such as the time-display guards that check `equalsIgnoreCase("0:0:00")`).
**Severity:** INFO
**Checklist item:** Section 3 — Injection Vulnerabilities (data handling)
**Evidence:**

- **Section 4 — Session and CSRF:** These are server-side Excel/email report generation classes with no HTTP request handling, no session management, and no form processing. CSRF is not applicable at this layer.

- **Section 6 — Transport Security:** These classes do not configure or use HTTP/TLS connectors. Transport security is not applicable at this layer.

- **Section 7 — Dependency Vulnerabilities:** No library version information is embedded in the reviewed source files. Apache POI is used for Excel generation; version cannot be determined from these source files alone. No other notable dependencies introduced by these specific files beyond those in the parent class.
