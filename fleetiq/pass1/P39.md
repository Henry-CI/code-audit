### P39-F02: Unescaped Database Values Output Directly to HTML (Pervasive XSS Risk)

**Description:** Every JSP file in this set outputs database-sourced values directly into HTML table cells using `<%= ... %>` expressions without any HTML escaping. Driver names, vehicle names, department names, licence types, serial numbers, and other data fields are all rendered raw. If any of these fields contain user-controlled content stored in the database (e.g., a driver name containing `<script>alert(1)</script>` entered via a management screen), the browser will execute it as JavaScript. This is a stored XSS vulnerability. The correct approach is `<c:out value="${...}" />` or `${fn:escapeXml(...)}`.

**Severity:** HIGH

**Checklist item:** Section 3 — XSS (Cross-Site Scripting)

**Evidence:**
```jsp
// daily_summary_report_by_driver.jsp line 168:
<td><b><%=dsum_driver_nm.get(i) %></b></td>

// detailed_report_by_driver.jsp line 244:
<td><strong><%=driv_nm %></strong></td>

// detailed_report_by_unit.jsp line 255:
<td><strong><%=veh_nm %></strong></td>

// driver_access_abuse_report.jsp line 137:
<td> <%=tmp %></td>   // where tmp = ""+a_driv_nm.get(i)

// driver_impact_report_au.jsp line 279:
<td><%=driv_name %></td>
```

### P39-F06: NullPointerException Risk from Unguarded `.toString()` on Request Parameters

**Description:** In `driver_impact_report_au.jsp`, `driver_league_report.jsp`, and `driver_league_table.jsp`, `request.getParameter("mnm").toString()` and `request.getParameter("sub").toString()` are called without null-checking the return value first. If either parameter is absent from the request, `request.getParameter()` returns `null`, and `.toString()` will throw a `NullPointerException`. This can cause server-side errors and potentially leak stack-trace information to the user if error pages are not properly configured. All other files in this set handle this with the pattern `request.getParameter("sub")==null?"":request.getParameter("sub")`.

**Severity:** MEDIUM

**Checklist item:** Section 5 — Data Exposure and Privacy (Error responses)

**Evidence:**
