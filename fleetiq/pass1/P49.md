## Findings

### P49-F01: Unescaped Request Parameters Written Directly to HTML (XSS)
**Description:** In both files, `request.getParameter("mnm")` and `request.getParameter("sub")` are emitted directly into HTML hidden input `value` attributes using raw `<%= %>` expressions, with no HTML escaping. An attacker can inject arbitrary HTML or JavaScript by crafting a malicious value for these parameters in the URL. Since the form uses `method="get"`, these parameters appear in the URL and can be delivered via a phishing link.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// report_head_with_filter.jsp, lines 65–66:
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">

// report_head_with_filter_keyword.jsp, lines 66–67:
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">
```

**Description:** In both files, customer names (`Vuser_nm`), location names (`Vloc_nm`), and department names (`vdept_nm`) sourced from the database are rendered into `<option>` element text content via raw `<%= %>` expressions. No HTML escaping is applied. If any of these database values contain HTML special characters (e.g., injected via a data-entry form or direct database manipulation), they will be rendered as HTML and executed as script. ### P49-F05: NullPointerException Risk from Unchecked `request.getParameter()` Calls
**Description:** In both files, `request.getParameter("mnm").toString()` and `request.getParameter("sub").toString()` are called without null-checking the return value of `getParameter()`. If either parameter is absent from the request, `getParameter()` returns `null`, and calling `.toString()` on `null` will throw a `NullPointerException`. This exception, if not caught by a generic error page, may expose a stack trace to the user, leaking internal class names and application structure (a data exposure finding). It also means the page will fail to render, constituting a denial-of-service vector for any request missing these parameters.
**Severity:** MEDIUM
**Checklist item:** 5. Data Exposure and Privacy — Error responses
**Evidence:**

### P49-F06: No CSRF Protection on Report Filter Form
**Description:** Both files render a `<form method="get">` that submits report filter parameters to `../pages/admin.jsp`. Neither form contains a CSRF token (no hidden field with a server-generated token). While HTTP GET forms are lower risk for state-changing operations than POST, this form drives report access, data export, and email sending functionality — the Export and Email/Subscribe buttons are rendered within the same form context and pass their parameters as part of the same request flow. The absence of CSRF protection means an attacker can craft a URL that, when visited by an authenticated user, triggers report export or email actions on their behalf. Notably, both files also construct and render direct links/buttons to `sendmail.jsp`, `subscribe.jsp`, and Excel export endpoints without any CSRF token.
**Severity:** MEDIUM
**Checklist item:** 4. Session and CSRF
**Evidence:**

### P49-F08: `search_crit` Embedded Unencoded in Constructed URLs Passed to External Endpoints

**Severity:** HIGH

## Checklist Items With No Issues

The following checklist items were reviewed against the content of these two files and no issues were found:

- **1. Secrets and Credentials:** No hardcoded passwords, API keys, JDBC connection strings, or credentials of any kind are present in either file.
- **3. Injection — SQL Injection:** Neither file contains any SQL statements, JDBC calls, or database query construction. SQL injection is not applicable to these view-layer fragments.
- **3. Injection — File Inclusion:** All includes in these files are static. There are no dynamic include paths derived from user input.
- **3. Injection — Command Injection:** Neither file contains `Runtime.getRuntime().exec()`, `ProcessBuilder`, or any OS command invocation.
- **3. Injection — SSRF:** Neither file makes outbound HTTP requests. The constructed URLs are rendered as client-side links/attributes; no server-side HTTP client calls are made in these fragments.
- **6. Transport Security:** No TLS configuration, connector definitions, or mixed-content resource loading is present in these JSP fragments. Transport security is configured at the container level and not applicable to these files individually.
- **7. Dependency Vulnerabilities:** Only two application-level imports are present (`LindeConfig`, `UtilBean`) plus the unused `RPattern`. No direct dependency declarations are present in these JSP fragments. Version analysis must be conducted at the `pom.xml`/JAR level by other agents.
