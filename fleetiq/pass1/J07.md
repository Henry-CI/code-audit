## Findings

### J07-F01: Package-private fields on Serializable beans expose internal state without access control
**Description:** All mutable fields in `ImpactLocBean` and `ImpactSummaryBean` are declared without an explicit access modifier, making them package-private. Any class in the same package (`com.torrent.surat.fms6.bean`) can read and write these fields directly, bypassing the public getter/setter methods entirely. For a Serializable bean carrying driver identity data (`driver_cd`, `driver_name`) and location data (`loc_cd`, `loc_name`), this means the data encapsulation that should protect against inadvertent mutation or unintended data exposure is absent. If any serialization pathway transmits these objects (e.g., to a session, to a distributed cache, or across a network), all fields are accessible to any code in the same package without restriction.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J07-F02: `ImpactSummaryBean` implements `Serializable` without declaring `serialVersionUID`
**Description:** `ImpactSummaryBean` implements `java.io.Serializable` but does not declare a `serialVersionUID` field. In contrast, `ImpactLocBean` correctly declares `serialVersionUID = 4253198288627637842L`. Without an explicit `serialVersionUID`, the JVM generates one at runtime based on the class structure. Any change to the class (adding or removing a field, changing a method signature) will silently change the generated UID, causing `InvalidClassException` during deserialization of previously-serialized instances — for example, those stored in HTTP sessions or a distributed cache. In a multi-tenant environment, a mid-session class change could cause session deserialization failures affecting all active users, potentially surfacing unhandled exceptions (and internal class names or stack traces) to end users.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (error responses leaking implementation details)
**Evidence:**

### J07-F03: Driver identity data (`driver_cd`, `driver_name`) carried in bean with no customer-scoping field
**Description:** `ImpactSummaryBean` holds `driver_cd` and `driver_name` fields alongside location data (`loc_cd`, `loc_name`). The bean itself contains no `customer_id` or tenant-scoping field. There is no mechanism within the bean to assert which customer's driver is represented. This is an observation at the bean layer: whether the code that populates and consumes this bean enforces customer scoping cannot be determined from the bean alone. However, the absence of any tenant identifier in a bean that carries driver PII (name and code) on a multi-tenant branch (`multi-customer-sync-to-master`) is a structural risk indicator. If this bean is ever populated from a query that does not filter by customer, or returned to a caller that does not verify tenant ownership, driver data from one customer could be exposed to another.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy; Section 5 — Data Exposure and Privacy (driver data)
**Evidence:**
```java

String loc_cd = "";
String loc_name = "";
ArrayList<ImpactBean> arrImpact = new ArrayList<ImpactBean>();
int index = 0;
int total = 0;
String driver_cd = "";    // driver identity — no customer_id field present
String driver_name = "";  // driver PII — no tenant scope enforced at bean level
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/ImpactSummaryBean.java`
