## Findings

### J47-F01: Excel Formula Injection via Unsanitised Bean Values
**Description:** All three report classes construct Excel cell formulas by directly concatenating values from the report beans (which originate from the database or caller-supplied parameters) into formula strings. The pattern `setCellFormula("VALUE(\"" + someValue + "\")")` is used throughout. If an attacker or malicious database record can place a value such as `\")&A1&VALUE(\"` into a field like start time, end time, duration, or stop-list entry, the resulting formula in the spreadsheet will be syntactically modified. When the generated `.xlsx` file is opened in Excel or LibreOffice, the injected formula executes. This is a CSV/spreadsheet formula injection vulnerability (also called "DDE injection" in older literature). It affects all three files.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**

ExcelDailyVehSummaryReport.java — duration value injected directly into formula (representative of the same pattern at lines 169, 182, 191, 200, 215, 225, 237):
```java
// line 169
contentCell.setCellFormula("VALUE(\""+tmp+"\")");
```
Where `tmp` is set at line 164:
```java
tmp = dsum_key_hr.get(i).toString();
```
And `dsum_key_hr` is sourced from `dsumBean.getDsum_key_hr()` which originates from database data.

ExcelDetailUnitReportBmw.java — start time and end time injected (lines 162, 172):
```java
// line 162
contentCell.setCellFormula("VALUE(\"" + itemBean.getStartTime() + "\")");
// line 172
contentCell.setCellFormula("VALUE(\"" + itemBean.getEndTime() + "\")");
```

ExcelDetailUnitReportBmw.java — stop list values injected (line 190):
```java
// line 190
contentCell.setCellFormula("VALUE(\"" + stop + "\")");
```

### J47-F02: No Customer/Tenant Scoping Enforced in Report Generation Layer
**Description:** All three Excel report classes accept `cust`, `loc`, and `dep` parameters as plain strings and write them into report subtitles. None of the three classes perform any check that the `cust` parameter belongs to the authenticated session's customer. The scoping decision is delegated entirely to the calling layer. If the calling servlet or JSP does not enforce customer ownership before constructing these beans and invoking these Excel classes, any authenticated user could supply a different customer name and receive that customer's fleet data in the exported report. This is the highest-risk area on the `multi-customer-sync-to-master` branch (per checklist section 2 — multi-tenancy). The risk is particularly pronounced for `ExcelDetailUnitReportBmw`, which is clearly a customer-specific report variant (named "Bmw"), indicating the application already acknowledges tenant-specific variants exist.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and Multi-tenancy; Section 5 — Data Exposure and Privacy (data export without authorisation check)
**Evidence:**

ExcelCurrUnitReport.java — cust parameter placed in subtitle with no validation:
```java
// lines 64-73
String[][] subtitles1 = {
    {"FleetFocus Reporting System"},
    {"Start Time", st},
    {"End Time", et},
    {"Customer", cust},      // <-- cust is caller-supplied, never validated
    {"Site", loc},
    {"Dept", dep},
    {"Report:", tabTitle}
};
```

ExcelDailyVehSummaryReport.java — same pattern (lines 94-103):
```java
String[][] subtitles1 = {
    {"FleetFocus Reporting System"},
    {"Start Time", st},
    {"End Time", et},
    {"Customer", cust},      // <-- cust is caller-supplied, never validated
    {"Site", loc},
    {"Dept", dep},
    {"Report:", tTitle}
};
```

The `cust` parameter is used only for display, but the data content of the report (vehicles, drivers, session hours) is drawn from bean data that was presumably fetched before calling these methods. Whether the bean-populating DAO queries are scoped to the authenticated customer's ID cannot be verified from these files alone, but the absence of any assertion or guard in the report-generation layer is a structural gap.

### J47-F05: Raw ArrayList Without Generics — Type-Safety Absence
**Description:** All three files make extensive use of raw `ArrayList` (without generic type parameters), sourced from the report beans. Items are cast to `String` via explicit casts such as `(String)vveh_nm.get(i)` and `.toString()`. A malicious or corrupt data value that is not a `String` would cause a `ClassCastException` at runtime, potentially crashing the report generation thread and leaking a stack trace to the user depending on error page configuration. This is a code quality issue that also has a minor data-exposure implication.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (stack trace leakage via error response)
**Evidence:**

## Checklist Items With No Issues

The following checklist items were reviewed against the three files in scope and no issues were found within these files:

**Section 1 — Secrets and Credentials:** No hardcoded passwords, database URLs, API keys, or credentials found in any of the three files. No `DriverManager.getConnection()` calls. No credential strings in comments.

**Section 2 — Authentication and Authorization (container-managed authentication):** These are report-generation helper classes, not servlets. They do not contain `HttpServletRequest` handling, session management, or login logic. Authentication is expected to be enforced by the calling servlet layer. The absence of authentication logic in these files is expected for this layer.

**Section 3 — SQL Injection:** No JDBC calls (`Statement`, `PreparedStatement`, `Connection`) are present in any of the three files. No SQL strings are constructed. The DAO objects that are instantiated (Finding J47-F03) are never actually called.

**Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage in any file.

**Section 3 — SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) in any file.

**Section 3 — File Inclusion:** Not applicable (Java classes, not JSP files).

**Section 3 — XSS:** Not applicable (output is to Excel binary format via Apache POI, not to HTML). Cell values set via `setCellValue()` are not rendered as HTML.

**Section 4 — Session and CSRF:** Not applicable. These classes are not servlets and do not handle HTTP requests or form submissions.

**Section 6 — Transport Security:** Not applicable to these classes. No HTTP connectors or TLS configuration is present in these files.

**Section 7 — Dependency Vulnerabilities:** The only third-party library directly used is Apache POI (`org.apache.poi.ss.usermodel.*`). The specific POI version cannot be determined from these source files alone; JAR version auditing is required separately. No Log4j, Struts, or Commons Collections imports are present in these files.
