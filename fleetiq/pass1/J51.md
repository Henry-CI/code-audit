## Findings

### J51-F01: Excel Formula Injection via Unsanitised Duration Value
**Description:** In all three report classes, a duration string retrieved from the bean (which originates from database query results) is directly embedded into an Excel formula string using `setCellFormula("VALUE(\"" + tmp + "\")")`. If the duration value contains Excel formula syntax (e.g., `"+HYPERLINK(...)"` or `"=CMD(...)`), it will be interpreted by Excel as an active formula when the file is opened. This is a CSV/Excel formula injection (also known as DDE injection). The same pattern exists in the parent `Frm_excel.setTotalDuration()`.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**
```java
// ExcelPreOpCheckDetailedReport.java line 186
contentCell.setCellFormula("VALUE(\""+tmp+"\")");

// ExcelPreOpCheckFailReport.java line 180
contentCell.setCellFormula("VALUE(\""+tot_dur+"\")");
contentCell.setCellFormula("VALUE(\""+tmp+"\")"); // line 319

// ExcelPreOpCheckReport.java line 158
contentCell.setCellFormula("VALUE(\""+tmp+"\")");

// Frm_excel.java line 855 (called by all three)
contentCell.setCellFormula("VALUE(\""+totalDuration+"\")");
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/excel/reports/ExcelPreOpCheckDetailedReport.java`, `ExcelPreOpCheckFailReport.java`, `ExcelPreOpCheckReport.java`, `Frm_excel.java`

### J51-F02: Customer Name Hard-coded in Business Logic — Tenant Isolation Leak
**Description:** `ExcelPreOpCheckDetailedReport.createPreOpReport()` branches on the literal string `"BMW"` to produce a different report format for that specific customer. This exposes the identity of a named customer in source code and indicates that customer-specific business logic is embedded in the shared codebase rather than being driven by configuration. This is an information disclosure issue and an architectural multi-tenancy concern: the fact that BMW receives a structurally different output means any future customer requiring similar treatment would require a code change, and the existing check provides no runtime access control — it only controls presentation, not data isolation.
**Severity:** LOW
**Checklist item:** Section 2 — Authorization and multi-tenancy; Section 5 — Data Exposure and Privacy

### J51-F03: No Authorization Check in Report Generation Classes — Customer Scoping Depends Entirely on Caller
**Description:** None of the three Excel report classes (`ExcelPreOpCheckDetailedReport`, `ExcelPreOpCheckFailReport`, `ExcelPreOpCheckReport`) perform any authentication or authorization check. The customer/site/department parameters `cust`, `loc`, `dep` are accepted as plain `String` arguments and passed directly into the report subtitle cells and (via the bean) into the report data. There is no check that the `cust` value matches the authenticated user's customer. The security boundary is entirely the responsibility of the caller. If a calling servlet passes a caller-supplied (request parameter) `cust` value without verifying it against the session's authenticated customer ID, any authenticated user can request report data for any customer — a cross-tenant data exposure. This was confirmed possible given the pattern observed in the related `EmailPreOpChkReport` and `EmailPreOpChkFailReport` classes, which take `cust_cd` as a constructor argument passed through from parent class `Frm_excel`.

### J51-F05: Driver ID Column Declared in Header But Data Never Written — Data Integrity Gap
**Description:** In `ExcelPreOpCheckReport.createPreOpReport()`, "Driver ID" is declared as a column header (line 85) and `driv_id` is fetched from the bean (line 53). However, `driv_id` is never written to any cell in the data rows. The same pattern occurs in `ExcelPreOpCheckFailReport.createPreOpFailReport()` (lines 70 and 209): `driv_id` is retrieved but never written. This means the exported report contains a header column with no corresponding data. While this is primarily a data quality defect, in a fleet management system context driver identity data appearing incomplete in audit exports could mask compliance or safety investigation requirements. The discrepancy may also indicate that a `po_veh_id` write (serial number) was substituted for the driver ID but the header was not corrected.

## Checklist Items With No Issues

The following checklist items were reviewed and no issues were found within the three assigned files:

- **Section 1 — Secrets and Credentials:** No hardcoded passwords, API keys, JDBC connection strings, or credentials were found in any of the three files.
- **Section 3 — SQL Injection:** These three files contain no SQL statements. They are pure Excel output renderers that receive pre-fetched data via bean objects. SQL injection risk is in the DAO/report layers (e.g., `UnitDAO`) that populate the beans, which are not in scope for this agent.
- **Section 3 — XSS:** These files produce `.xlsx` binary output, not HTML. XSS is not applicable to Excel cell values. (Note: Excel formula injection is covered in J51-F01.)
- **Section 3 — File inclusion:** No `jsp:include`, `Runtime.exec()`, or `ProcessBuilder` calls are present in these files.
- **Section 3 — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage.
- **Section 3 — SSRF:** No HTTP client calls in these three files. (Note: the parent class `Frm_excel` contains `new URL(image).openStream()` patterns in `addImage2`, `addImageUtilisationChart`, and `addDashboardChart` — these are in scope for the agent auditing `Frm_excel.java`.)
- **Section 4 — Session and CSRF:** These files are server-side Excel report generators with no HTTP request/response handling, form rendering, or session manipulation.
- **Section 6 — Transport Security:** Not applicable to server-side Excel generation classes.
- **Section 7 — Dependency Vulnerabilities:** No new dependency declarations are introduced. Apache POI is used via `org.apache.poi.*` imports; the version cannot be determined from these source files alone — it depends on the JAR version in the classpath (in scope for the inventory agent).
- **Section 8 — Cross-branch comparison:** Out of scope for a single-agent pass.
