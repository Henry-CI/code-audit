## Findings

### J59-F01: Static mutable fields in ImpactReportBean create cross-request data leakage risk
**Description:** Seven fields in `ImpactReportBean` are declared `private static`: `machineId`, `model`, `driverName`, `cardNo`, `dtImpact`, `severityLvl`, and `fileName`. These fields are mutable and their setters are also static. In a Tomcat servlet environment, a single class instance (and therefore its static fields) is shared across all threads and all HTTP requests. If one user's report request writes driver names, card numbers, or impact data into these static fields, a concurrent request from a different user — potentially belonging to a different customer/tenant — will read those same values. This is a data leakage and multi-tenancy isolation failure: one customer's driver PII (`driverName`, `cardNo`) and operational data (`machineId`, `dtImpact`, `severityLvl`) can bleed into another customer's report.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and multi-tenancy ("verify that every data access is scoped to the authenticated user's customer/organisation"); Section 5 — Data Exposure and Privacy ("Driver data: driver names, licence numbers, or contact details")
**Evidence:**

### J59-F02: Raw ArrayList fields carry PII with no type safety or defensive copying
**Description:** All three bean classes use raw `ArrayList` (without generic type parameters) throughout. All getters return direct references to internal mutable list instances — there is no defensive copy on get or set. In a multi-tenant report context, a caller that obtains the list via `getVrpt_veh_driv_nm()` or `getDriverName()` can mutate the list in place, causing unpredictable side effects in other code paths that hold a reference to the same object. The PII concern is heightened because `DynDriverReportBean` holds driver names (`Vrpt_veh_driv_nm`) and driver codes, and `ImpactReportBean` holds `driverName` and `cardNo` (likely a driver card or ID card number). The absence of generics also means unchecked casts at usage sites suppress compiler-level type errors that might catch injection of unexpected object types.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy ("Driver data: driver names, licence numbers, or contact details exposed beyond what is necessary")

### J59-F03: DynDriverReportBean and DynUnitReportBean fields have package-private (default) access
**Description:** Every data-holding ArrayList field in both `DynDriverReportBean` and `DynUnitReportBean` is declared without an access modifier, making them package-private. ### J59-F04: No customer/tenant scoping field in any of the three report beans
**Description:** None of `DynDriverReportBean`, `DynUnitReportBean`, or `ImpactReportBean` contains a field for customer ID, organisation ID, or tenant identifier. These beans carry driver identities, vehicle identifiers, and impact event data — all of which are sensitive multi-tenant assets. Without a customer scoping field in the bean itself, there is no structural guarantee that the data populated into the bean at the point of DB query population was filtered to the authenticated user's customer. The bean cannot be validated for correct tenant isolation before its data is rendered into a report. This is a structural gap: the beans should carry the tenant context so that any code consuming them can assert it.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy ("verify that every data access is scoped to the authenticated user's customer/organisation"); Section 5 — Data Exposure and Privacy ("Data export: report generation and data export features for missing authorisation checks")

### J59-F05: ImpactReportBean not serializable despite use as a session/report transfer object
**Description:** `ImpactReportBean` is used as a report data carrier. It does not implement `java.io.Serializable` and defines no `serialVersionUID`. In a Tomcat environment where session persistence is enabled (Tomcat's `PersistentManager` or clustering), any bean stored in the HTTP session will fail to serialize, causing `NotSerializableException` at runtime. While the same is true of `DynDriverReportBean` and `DynUnitReportBean`, the risk is elevated for `ImpactReportBean` because its fields include sensitive PII (`driverName`, `cardNo`) that, if session serialisation were silently attempted, would be written to disk in plaintext in Tomcat's session store. The absence of `Serializable` is the lesser issue; the PII-to-disk risk if it were added without encryption is the real concern.
**Severity:** INFO
**Checklist item:** Section 5 — Data Exposure and Privacy (driver PII in logs/storage); Section 6 — Transport Security (data at rest)
**Evidence:**

## Checklist Items With No Issues

The following checklist items were reviewed against these three files and no issues were found. These files are pure data-transfer beans with no SQL, no HTTP handling, no file I/O, and no external calls.

- **Section 1 — Secrets and Credentials:** No hardcoded credentials, passwords, API keys, connection strings, or comments containing sensitive values in any of the three files.
- **Section 3 — SQL Injection:** No JDBC calls, no SQL string construction, no `Statement` or `PreparedStatement` usage in any of the three files.
- **Section 3 — XSS:** No JSP expression output, no HTML rendering, no response writing in any of the three files.
- **Section 3 — File inclusion:** No file path construction, no `include` directives, no file I/O in any of the three files.
- **Section 3 — Command injection:** No `Runtime.exec()`, no `ProcessBuilder`, no external process invocation in any of the three files.
- **Section 3 — SSRF:** No HTTP client calls, no URL construction, no network I/O in any of the three files.
- **Section 4 — Session and CSRF:** These are pure bean classes with no request handling, no session access, and no form rendering. CSRF does not apply at the bean layer.
- **Section 6 — Transport Security:** No connector configuration, no HTTP/HTTPS handling in any of the three files.
- **Section 7 — Dependency Vulnerabilities:** The only import in all three files is `java.util.ArrayList` from the JDK standard library. No third-party dependencies are introduced by these files.
