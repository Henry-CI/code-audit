## Findings

### J88-F01: SQL Injection via String Concatenation — MessageDao.init()
**Description:** The `init()` method builds a SQL query by directly concatenating the instance field `fid` into the query string without using a `PreparedStatement`. The `fid` field is set by callers via `setFid()` and may originate from user-supplied input (e.g., a form field or URL parameter). An attacker controlling this value can inject arbitrary SQL.
**Severity:** CRITICAL
**Checklist item:** 3. Injection Vulnerabilities — SQL injection
**Evidence:**
```java
// MessageDao.java, lines 62–63
queryString = "select * from mnt_msg where \"FORM_CD\"="+fid;
rset = stmt.executeQuery(queryString);
```
**File:** `WEB-INF/src/com/torrent/surat/frms6/dao/MessageDao.java`
**Line(s):** 62–63

### J88-F11: Multi-Tenancy Isolation Relies Entirely on Delegated unitlst/driverlst Generation
**Description:** All customer-scoping in `LockOutDAO` and `PreCheckDAO` is delegated to `unitDAO.getUnitLstByModel()` and `driverDAO.getDriverFulllst()`. The DAOs in this review perform no independent verification that the returned vehicle or driver IDs actually belong to the authenticated customer. If `UnitDAO` or `DriverDAO` have any flaw in their customer scoping (e.g., if `cust_cd` can be manipulated by the caller), all downstream queries in these three files will silently cross tenant boundaries. The risk is amplified by the fact that this is the `multi-customer-sync-to-master` branch — the primary multi-tenancy branch. The `cust_cd` values passed into these methods are not verified against the authenticated session in any code visible in these files.
**Severity:** HIGH

## Checklist Items With No Issues

The following checklist items were reviewed against these three files and no issues were found in the scope of these files alone:

- **1. Secrets and Credentials:** No hardcoded database passwords, API keys, or credentials appear in any of the three files. The JNDI datasource lookup in `MessageDao` uses `RuntimeConf.penom_database` as the datasource name — no literal connection string or password present.
- **3. Injection — XSS:** These are DAO-layer Java files; no JSP output, HTML rendering, or `<%= %>` expressions are present. XSS is not applicable to this layer.
- **3. Injection — File inclusion:** No file path construction or inclusion mechanisms present.
- **3. Injection — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls present in any of the three files.
- **3. Injection — SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) present in these files.
- **4. Session and CSRF:** These are DAO-layer files; no session management or CSRF token handling is present or expected at this layer.
- **6. Transport Security:** These files contain no TLS or HTTP connector configuration.
- **7. Dependency Vulnerabilities:** Import analysis shows only standard JDK (`java.sql.*`, `javax.naming.*`, `javax.sql.*`) and application-internal classes. No third-party library imports are present in these three files.
