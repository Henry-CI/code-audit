### J68-F01: Pervasive SQL Injection via String Concatenation — Entire Reporting Layer
**Description:** Every SQL query in both `Databean_report.java` and `Databean_report1.java` is built by string concatenation of user-controlled parameters directly into SQL strings. Neither file uses `PreparedStatement` with `?` placeholders at any point. Parameters including `set_cust_cd`, `set_loc_cd`, `set_dept_cd`, `set_user_cd`, `set_gp_cd`, `st_dt`, `end_dt`, `set_sc` (search criteria), and `rptLockoutTypeFilter` are concatenated into query strings and executed with `Statement.executeQuery()`. This affects every report in the application — utilisation reports, impact reports, driver reports, GPS/location reports, pre-op check reports, and subscription management. The attack surface spans thousands of SQL statements across 22,208 lines. Any parameter the calling JSP page sets on the bean before calling `init()` is injectable.

**Severity:** CRITICAL

### J68-F05: Internal Error Information Leaked to System.out Including SQL Query Text
**Description:** Both `Databean_report.java` and `Databean_report1.java` catch exceptions in their `init()` methods and print the exception, the current SQL query string, and the method name to `System.out`. In a Tomcat deployment, `System.out` typically goes to the Tomcat `catalina.out` log. More critically, if any error path produces an HTTP response containing this output (e.g., via JSP scriptlets that read bean state), the full SQL query — including user-supplied parameter values and the internal table structure — could be disclosed to the user. Even if only logged internally, query text in logs is a significant finding: it reveals table names, column names, and parameter values.

### J68-F07: LIKE Pattern Injection — Search Criteria Parameter (set_sc) Injected Without Quoting
