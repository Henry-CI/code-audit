## Findings

### J40-F01: Excel Formula Injection via Database Values (EmailDynDriverReport)
**Description:** In `createDynDriverReport()`, database-sourced string values `sttm`, `endtm`, and `stop` are concatenated directly into Excel cell formula strings using `setCellFormula("VALUE(\"" + value + "\")")`. If any of these values contain formula characters (e.g., `=`, `+`, `-`, `@`), they will be interpreted as formula instructions when the generated Excel file is opened. An attacker with the ability to insert malicious data into the database (e.g., by controlling driver names, unit identifiers, or time values) could craft payloads that execute when the recipient opens the email attachment in a spreadsheet application. This is a well-documented attack vector known as CSV/Excel Formula Injection (also called Comma Injection or DDE Injection).
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**
```java
// Lines 250-251
contentCell.setCellFormula("VALUE(\""+sttm+"\")");
contentCell.setCellValue(sttm);

// Lines 255-256
contentCell.setCellFormula("VALUE(\""+endtm+"\")");
contentCell.setCellValue(endtm);

// Lines 269-270
contentCell.setCellFormula("VALUE(\""+stop+"\")");
contentCell.setCellStyle(styles.get("cell_48hour_time"));
```
**File:** WEB-INF/src/com/torrent/surat/fms6/excel/reports/EmailDynDriverReport.java
**Line(s):** 250–251, 255–256, 269–270

### J40-F03: Missing Customer Scoping Verification — cust_cd Emptied to Empty String
**Description:** In all three files, when `cust_cd` equals `"0"`, it is replaced with an empty string `""` (EmailDriverLicenceExpiry line 43-45, EmailDynDriverReport lines 51-53, EmailDynUnitReport lines 55-57). This empty string is then passed directly to the data bean as the customer code for the SQL query. Depending on how the downstream `Databean_report` and `Databean_dyn_reports` DAOs handle an empty-string customer code — specifically whether an empty string causes no WHERE clause filtering on customer — this could result in report data being returned across all customers rather than scoped to the authenticated user's customer. In a multi-tenant system this is a critical data isolation risk. The branch name `multi-customer-sync-to-master` makes this particularly relevant.

### J40-F04: No Authorization or Authentication Check in Report Generation Classes
**Description:** None of the three report classes perform any check to verify the caller is authenticated or that the `cust_cd` parameter matches the session user's customer. The classes are called with `cust`, `loc`, `dep`, `st`, `et`, `params`, and `dir` passed as constructor arguments. There is no session validation, no role check, and no verification that the supplied customer code is the one associated with the authenticated user. If the calling code (not visible in these files) does not enforce these checks, an authenticated user from one customer could generate reports for another customer's data by supplying a different `cust` value. This is a classic IDOR risk in multi-tenant report generation.

### J40-F05: Null Dereference Risk — search_crit Not Null-Guarded in EmailDynUnitReport
**Description:** In `EmailDynUnitReport.createDynUnitReport()`, `search_crit` is assigned from `params[1]` at line 45 with no null check. Compare this to the parallel code in `EmailDynDriverReport.createDynDriverReport()` at line 43, which uses a null guard: `params[1] == null ? "" : params[1]`. The inconsistency means that if `params[1]` is null in `EmailDynUnitReport`, a NullPointerException will not be caught at assignment, and the null value will be passed to `dbreport.setSet_sc(null)` at line 73. Depending on how the DAO handles a null search criteria, this could cause an unhandled exception that leaks stack trace information to the user, or produce a SQL error that aids SQL injection probing.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (error responses exposing stack traces)
**Evidence:**
```java
// EmailDynUnitReport.java line 45 — no null guard
if( params.length > 1){
    search_crit = params[1];   // params[1] could be null
}

// Compare to EmailDynDriverReport.java line 43 — has null guard
if( params.length > 1){
    search_crit = params[1] == null ? "" : params[1];
}
```
**File:** WEB-INF/src/com/torrent/surat/fms6/excel/reports/EmailDynUnitReport.java
**Line(s):** 44–46

### J40-F06: Driver PII (Names, Licence Data) Written to Filesystem Without Access Controls Visible in These Classes
**Description:** All three classes write their generated Excel reports to the filesystem via `FileOutputStream(result)` (EmailDriverLicenceExpiry line 27, EmailDynDriverReport line 32, EmailDynUnitReport line 32). The `result` path is provided by the parent class `Frm_excel` via the `dir` constructor argument. The Excel files contain driver names, licence expiry dates (EmailDriverLicenceExpiry), driver names paired with vehicle usage data (EmailDynDriverReport), and vehicle/driver operational records (EmailDynUnitReport). There is no evidence in these classes of the generated file being written to a secure, access-controlled directory, or of the file being deleted after email delivery. If the `dir` path is web-accessible, these files containing PII could be accessed directly by URL without authentication.

## Checklist Items With No Issues

The following checklist sections were reviewed against these three files and no issues were identified within the scope of these files:

**Section 1 — Secrets and Credentials:** No hardcoded database credentials, passwords, API keys, or connection strings found in any of the three files. No JDBC `DriverManager.getConnection()` calls. No credentials in comments.

**Section 3 — SQL Injection:** These three files do not contain any direct JDBC calls (`Statement.executeQuery`, `PreparedStatement`, etc.). All SQL execution is delegated to `Databean_report` and `Databean_dyn_reports`. SQL injection risk must be assessed in those DAO/Databean classes, which are outside the scope of this agent's assigned files. No direct SQL string construction was observed in these files.

**Section 3 — XSS:** These files produce Excel output (Apache POI), not HTML. There are no JSP expression outputs, no `<%= %>` expressions, and no HTML rendering. XSS does not apply to Excel generation code directly.

**Section 3 — File Inclusion:** No `<jsp:include>` or `<%@ include %>` directives — these are Java classes, not JSP files. No dynamic file path construction for inclusion purposes was observed.

**Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls found in any of the three files.

**Section 3 — SSRF:** No `HttpURLConnection`, `HttpClient`, or equivalent HTTP client calls found in any of the three files.

**Section 4 — Session and CSRF:** These are report-generation helper classes, not servlet or JSP controllers. They do not handle HTTP requests, session objects, or form submissions. CSRF controls must be evaluated in the calling servlets/JSPs, which are outside the scope of these files.

**Section 6 — Transport Security:** These files do not configure or interact with TLS, HTTP connectors, or cookie settings. Transport security must be evaluated in `server.xml`, `context.xml`, and `web.xml`.

**Section 7 — Dependency Vulnerabilities:** The only third-party library directly referenced in these files is Apache POI (`org.apache.poi.ss.usermodel.*`). No version can be determined from these source files alone — the JAR version must be checked separately in the repository's committed libraries. No Struts, Log4j, Commons Collections, or other high-risk libraries are directly imported here.

**Section 8 — Cross-Branch Security Comparison:** This comparison requires completion of both FleetIQ and FleetFocus audits and is performed at the orchestration level, not within this agent's scope.
