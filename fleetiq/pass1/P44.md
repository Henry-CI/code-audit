### P44-F01: Pervasive XSS — Unescaped DB and User-Supplied Values in JSP Expressions
**Description:** Every JSP file in this batch outputs data directly into HTML using raw `<%= ... %>` expressions without HTML escaping. Data includes database-sourced values (customer names, user names, vehicle names, driver names, model names, serial numbers, site names, department names, login times) as well as directly user-supplied request parameters. None of the output uses `<c:out>`, `${fn:escapeXml()}`, or any other escaping mechanism. If any of these stored values contain HTML special characters or a stored script payload, it will execute in the victim's browser. This constitutes stored XSS for DB-sourced values and reflected XSS for request-parameter values.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// user_login_report.jsp lines 201–218 — unescaped DB output
<td>  <%=user_fnm.get(i) %></td>
<td>  <%=user_lnm.get(i) %></td>
<td>  <%=user_login.get(i) %></td>
<td>  <%=user_login_time.get(i)%></td>
<td>  <%=vuser_nm.get(i) %></td>
<td > <%=loc_nm.get(i) %></td>
<td > <%=dep_nm.get(i) %></td>
<td > <%=vgroup_nm.get(i) %></td>
```

**Description:** In `vor_status_report.jsp`, request parameters `dealer`, `cust_cd`, `loc_cd`, `dept_cd`, `search_crit`, and `do_list` are injected directly into JavaScript variable assignment statements using single-quoted string literals. There is no escaping. An attacker can close the string literal, inject arbitrary JavaScript, and execute it in the victim's browser. The `dealer` value is injected unquoted as a numeric literal, meaning any non-numeric value becomes a JavaScript syntax injection point. ### P44-F05: Missing CSRF Protection on All Report Forms
**Description:** All report forms in this batch use `method="get"` and contain no CSRF token — no hidden field with a server-side token, no double-submit cookie, no synchroniser token pattern. Although GET forms are lower risk for state changes than POST forms, these forms construct email and export URLs that are submitted, and the broader application pattern of no CSRF tokens means any state-changing endpoint derived from these forms is also unprotected. Additionally the `form_cd` parameter controls which permissions are checked, and submitting forged requests with manipulated `form_cd` values could bypass permission gates.
**Severity:** MEDIUM
**Checklist item:** Section 4 — Session and CSRF
**Evidence:**
```jsp
// vor_report.jsp line 210 — GET form, no CSRF token
<form method="get" class="report_form" action="../pages/admin.jsp">
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">
<input type="hidden" value="<%=form_cd%>" name="form_cd">
// No CSRF token present anywhere in form

// user_login_report.jsp line 92
<form method="get" class="report_form">
// No CSRF token present
```

**Line(s):** user_login_report.jsp:92; veh_calib_report.jsp:124; vor_report.jsp:210; vor_report2.jsp:200; vor_report_simple.jsp:201; vor_unit_report.jsp:144

**Description:** All report pages accept `user_cd` (customer code), `loc_cd` (site code), and `dept_cd` (department code) as request GET parameters. The access-level enforcement relies on session attributes (`access_level`, `access_cust`, `access_site`, `access_dept`) being passed to the filter bean's setter methods before `filter.init()` is called. However, the `user_cd` (customer) that determines whose data is fetched is directly user-supplied from the request parameter. If the backend filter bean does not correctly enforce that the requested `user_cd` is within the tenant scope permitted by the session's access level, a user could enumerate data belonging to other customers by submitting a different `user_cd`. This pattern is present in all eight files. The risk is especially elevated because `user_cd` defaults to the first entry returned by the filter (which is itself scoped by the session), but if a user manually overrides `user_cd` with an arbitrary value, there is no visible JSP-level check that the provided value is within their allowed scope.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and multi-tenancy / IDOR
**Evidence:**
```jsp

filter.setAccess_site(access_site);
filter.setAccess_dept(access_dept);
...
filter.setSet_cust_cd(user_cd);  // user-controlled value, must be validated server-side in filter bean
filter.init();

// vor_status_report.jsp lines 5–8 — cust_cd directly from request
```

### P44-F11: Data Exposure — User PII (First Name, Last Name, Login Times, Access Level) Output to Authenticated Users Without Role Check at JSP Level
**Description:** `user_login_report.jsp` and `user_summary_report.jsp` output a full listing of user first names, last names, login timestamps, access levels, customer names, site names, and department names for all users visible to the session's access level. This is administrative-grade PII. The JSP relies solely on the `access_level` session attribute to scope the data returned by `Databean_report`. There is no additional JSP-level check that the requesting user is actually an admin-level user before rendering. If the backend bean's scoping has any flaw, or if the session `access_level` attribute can be manipulated, this entire user directory including last-login information would be exposed.
