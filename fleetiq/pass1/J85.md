### J85-F01: SQL injection via string concatenation ‚Äî userid in password_life.java
**Description:** The `loadDefaultValues()` method in `password_life` constructs four separate SQL queries by directly concatenating the `userid` field into SQL strings. `userid` is set by the caller via `setUser()` from outside the class with no sanitisation. If an attacker can supply a crafted value for `userid`, they can manipulate all four queries, including a login-status UPDATE. The fourth query also concatenates `ip`, `logindate`, and `logintime`, all of which are set by the caller.
**Severity:** CRITICAL
**Checklist item:** Section 3 ‚Äî Injection Vulnerabilities (SQL injection)
**Evidence:**
```java
// Line 76

// Line 97

queryString = "select nvl(emplpass_login_status,'&nbsp;') from empl_passwords where emplpass_empid='" + userid + "' ";
```

### J85-F07: All security-sensitive fields are public ‚Äî broken encapsulation on authentication objects
**Description:** In both `password_life` and `password_policy`, all instance fields including `userid`, `ip`, `logindate`, `logintime`, `login_status`, `emp_pass`, and the raw SQL query strings (`queryString`, `queryString1`, etc.) are declared `public`. This is particularly dangerous for `userid` and `emp_pass` ‚Äî if an attacker gains a reference to a shared or pooled instance, they can overwrite the user identity being queried or read another user's password-reset flag. The `queryString` fields are also public, meaning the SQL being executed is externally readable and writable.

### J85-F11: Database connection not closed on exception path in password_life and password_policy
**Description:** In `password_life.init()` and `password_policy.init()`, the database connection (`conn`) is opened and then `loadDefaultValues()` is called. If `loadDefaultValues()` throws (which it can, since it declares `throws SQLException`), the `catch` block in `init()` only logs the exception ‚Äî it does not close `conn`. This results in a connection leak on every exception, which will exhaust the connection pool under sustained error conditions. While primarily a reliability issue, connection pool exhaustion can be used as a denial-of-service vector.
**Severity:** LOW
**Checklist item:** Section 2 ‚Äî Authentication and Authorization (session/resource management)
**Evidence:**
