## Findings

### J37-F01: `cust_cd` normalised to empty string disables customer scoping
**Description:** In all three report classes, when `cust_cd` equals `"0"` it is replaced with an empty string (`""`). This empty string is then passed directly to the underlying data-access layer (`Databean_dyn_reports` / `Databean_report`) as the customer filter. If the downstream DAO or stored procedure interprets an empty string as "no filter" or "all customers", a caller who passes `"0"` as the customer code will receive data across all customers, breaking multi-tenant isolation. This is not verified in the files reviewed, but the pattern is a known multi-tenancy risk that must be confirmed in the DAO layer.
**Severity:** HIGH
**Checklist item:** Section 2 ‚Äî Authorization and multi-tenancy
**Evidence:**
```java
// EmailBatteryReport.java lines 52-54
if( cust_cd.equalsIgnoreCase("0")){
    cust_cd = "";
}
```
```java

    cust_cd = "";
}
```
```java
// EmailCimplicityShockReport.java lines 50-52

```

### J37-F02: `cust_cd` sourced from parent constructor ‚Äî origin and validation not visible in these files
**Description:** The `cust_cd` field (and `loc_cd`, `dept_cd`) are inherited from `Frm_excel` and populated by the parent constructor. These report classes receive `cust` as a constructor parameter and pass it to `super(...)`. It is not possible from these files alone to confirm whether `cust_cd` is validated against the authenticated session before the report class is instantiated. If a caller constructs these report objects directly with a caller-supplied `cust` parameter (e.g., from an HTTP request parameter) without verifying it belongs to the authenticated user, this is an IDOR vulnerability. The calling layer must be reviewed.
**Severity:** HIGH
**Checklist item:** Section 2 ‚Äî Authorization and multi-tenancy; IDOR

### J37-F03: File output path (`result`) inherited and not validated ‚Äî potential path traversal
**Description:** All three classes write report output to a path stored in the inherited field `result` using `new FileOutputStream(result)`. The `result` field is set by the parent class `Frm_excel` based on the `dir` constructor parameter. ## Checklist Items With No Issues

The following checklist items were reviewed against the three files and no issues were found within the scope of these files:

- **Section 1 ‚Äî Secrets and Credentials:** No hardcoded passwords, API keys, SMTP credentials, JDBC connection strings, or encryption keys found in any of the three files. No credentials in comments.
- **Section 3 ‚Äî SQL Injection:** These files do not contain any JDBC calls, `Statement.executeQuery()`, `PreparedStatement`, or SQL string construction. SQL access is entirely delegated to `Databean_dyn_reports` / `Databean_report` DAOs; SQL injection risk must be assessed in those classes.
- **Section 3 ‚Äî XSS:** These are server-side Excel report generation classes. They do not produce HTML output. There are no JSP expressions or JavaScript blocks. XSS is not applicable to these files.
- **Section 3 ‚Äî Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage found in any of the three files.
- **Section 3 ‚Äî SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) found in any of the three files.
- **Section 4 ‚Äî Session and CSRF:** These are report-generation utility classes, not servlet or JSP request handlers. Session and CSRF handling is not implemented here; it must be assessed in the calling servlet or JSP layer.
- **Section 6 ‚Äî Transport Security:** These classes generate local Excel files; they do not make network connections or configure TLS. Transport security is not applicable to these files directly.
- **Section 7 ‚Äî Dependency Vulnerabilities (Apache POI):** Apache POI is used (`org.apache.poi.ss.usermodel.*`). The specific version cannot be determined from these source files alone ‚Äî this must be verified against the JAR committed to the repository. Flagged for follow-up in repository inventory; no finding raised here as the version is unknown.

## Notes for Follow-Up (Not Findings)

1. **`Frm_excel` parent class must be reviewed.** The `result`, `dir`, `cust_cd`, `loc_cd`, `dept_cd`, `from`, `to`, `wb`, `fileName`, and `params` fields are all inherited. The security of `result` path construction, parameter sanitisation, and any authentication checks depends entirely on `Frm_excel`. This is out of scope for this agent but is a required follow-up.
2. **`Databean_dyn_reports` and `Databean_report` must be reviewed for SQL injection.** These DAO classes receive the op-codes and filter parameters and execute the actual database queries. The customer-scoping finding (J37-F01) can only be fully confirmed or dismissed by reviewing how `cust_cd = ""` is handled in those classes.
3. **Calling servlets/JSPs must be reviewed for IDOR.** The constructor parameters `cust`, `loc`, `dep`, `st`, `et`, `params`, `dir` are passed from a calling layer. Whether those values are validated against the authenticated session must be confirmed in the calling code (finding J37-F02 depends on this).
4. **Package name inconsistency:** `UnitDAO` is imported from `com.torrent.surat.frms6.dao.UnitDAO` (note `frms6` not `fms6`) in both `EmailBroadcastMsgReport` and `EmailCimplicityShockReport`. This inconsistency in package naming should be noted for the inventory.
