### J18-F01: Pervasive SQL Injection ‚Äî String Concatenation in All Fetch Methods
**Description:** Every SQL query in `BusinessInsightBean.java` is constructed by direct string concatenation of external-origin values (`customerCd`, `locationCd`, `st_dt`, `end_dt`, vehicle codes, driver codes, model IDs) into the query string. `Statement.executeQuery()` is used throughout ‚Äî no `PreparedStatement` with `?` placeholders is used anywhere in this file. The fields `customerCd` and `locationCd` originate from HTTP request parameters (`request.getParameter("customer")` and `request.getParameter("location")`) in `BusinessInsight.doPost()` with no sanitisation, and are passed directly to the bean. An attacker who controls these parameters can inject arbitrary SQL.
**Severity:** CRITICAL
**Checklist item:** 3. Injection Vulnerabilities ‚Äî SQL Injection
**Evidence:**

### J18-F06: Path Traversal Risk in getExportDir() ‚Äî Seven Levels of "../" from Code Location
**Description:** `getExportDir()` derives the export filesystem path by taking the JVM code-source location and appending `"/../../../../../../../excelrpt/"` (seven `../` segments). This is a fixed relative traversal that assumes a specific directory-depth relationship between the deployed WAR and the export directory. If deployment structure changes, or if the code-source path is influenced by a symbolic link or unusual container configuration, the resolved path may point outside the intended directory. While this is a fixed string (not user-controlled), the commented-out alternative (`"dir += "/../../excelrpt/";"` at line 113) suggests the path is fragile and has been changed during development.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities ‚Äî File inclusion
