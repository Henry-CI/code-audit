### J33-F01: SQL injection via unparameterised user-controlled date/time strings — Preop.java
**Description:** In all four data-fetch methods of `Preop.java` (`getVehicle`, `getDriver`, `getHour`, `getDate`), the request parameters `start_time` and `end_time` are taken directly from the HTTP request and concatenated into SQL timestamp literals without validation, sanitisation, or use of `PreparedStatement`. An attacker who can send arbitrary values for these parameters can inject SQL and read, modify, or delete data across the entire database.
**Severity:** CRITICAL
**Checklist item:** 3. Injection Vulnerabilities — SQL injection
**Evidence:**
```java
// Preop.java, getVehicle(), lines 127-128, 177:
String startTime = req.getParameter("start_time");
String endTime   = req.getParameter("end_time");
// ...
+ "where driver_cd !=2 and starttimestamp >= timestamp ' " + startTime + "' and starttimestamp <= timestamp '" + endTime + "' and vehicle_cd in " + vehicles + " group by r.id) as a "
```
Same pattern repeated in `getDriver` (lines 303–304, 353), `getHour` (lines 478–479, 528), `getDate` (lines 658–659, 710).
**File:** `WEB-INF/src/com/torrent/surat/fms6/dashboard/Preop.java`
**Line(s):** 127–128, 177; 303–304, 353; 478–479, 528; 658–659, 710

### J33-F04: Insufficient tenant isolation — cust_cd filter bypassable at access_level 1 — Preop.java and Summary.java
**Description:** At `access_level == 1` with `sessionCust != 0`, the cust/site/dept filter conditions are applied using the user-supplied request parameter rather than the session-stored customer value. The condition `level > 1 || !cust.equalsIgnoreCase("all")` allows a level-1 user to supply any `cust_cd` value and scope their query to that customer, enabling cross-tenant data access. A level-1 user can enumerate data belonging to other customers by varying the `cust_cd` request parameter. In `Preop.getVehicle()` and `Preop.getDriver()`, the `access_cust` session value is read but the enforced fallback (setting `cust` from session) that appears in some Summary.java methods is absent.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization — Authorization and multi-tenancy
**Evidence:**
```java
// Preop.java, getVehicle(), lines 121-125, 129, 143-144:

int sessionCust = Integer.parseInt(((String)(req.getSession().getAttribute("access_cust"))).substring(2));

if (level == 2 && sessionCust == 9999) level = 1;
// ...
String cust = req.getParameter("cust_cd");  // user-controlled
// ...
if (level > 1 || !cust.equalsIgnoreCase("all")) {
    vehiclesQuery += " and r.\"USER_CD\" = " + cust;  // no enforcement that cust == sessionCust
}
```
Contrast with Summary.java where some methods do enforce the session value:
```java
// Summary.java, getPreop(), lines 214-216:
if( cust.equals("all") && level > 1) {
    cust = String.valueOf(sessionCust);
}
```
But the Preop.java methods (`getVehicle`, `getDriver`, `getHour`, `getDate`) lack this enforcement entirely, and in Summary.java `getBattery` also adds this guard only for the "all" case, meaning a level-1 user supplying a non-"all" `cust_cd` value still bypasses it.
**File:** `WEB-INF/src/com/torrent/surat/fms6/dashboard/Preop.java`, `WEB-INF/src/com/torrent/surat/fms6/dashboard/Summary.java`

### J33-F05: Exception stack traces printed to stdout — information leakage — Preop.java and Summary.java
**Description:** Every catch block in both servlets calls `e.printStackTrace()`. On a typical Tomcat deployment, this writes the full Java stack trace — including class names, method names, line numbers, and sometimes SQL error messages — to `catalina.out`. While this does not expose the stack trace directly to the end user (the response is already committed with JSON content), it creates a permanent log of internal implementation detail that aids an attacker performing blind SQL injection. The SQL error messages returned by the database engine when an injection probe fails will be captured in the server log.

### J33-F07: No CSRF protection — state-reading endpoints exposed without token — Preop.java and Summary.java
**Description:** Both servlets respond to HTTP GET requests for data retrieval. ### J33-F09: Instance-level mutable field `cusList` in servlet — thread safety defect — Summary.java
**Description:** `Summary.java` declares `cusList` as an instance field (line 39): `LinkedList<HashMap<String, String>> cusList = new LinkedList<>()`. Servlet instances are singletons in Tomcat — a single instance handles all concurrent requests. This `cusList` field is never referenced in the code visible in this file, but its existence as a mutable, non-thread-safe instance field on a servlet is a defect that, if used in a future code path, would cause data leakage between concurrent requests (one user's data visible to another user's session). Additionally, `Gson gson` is also an instance field; while `Gson` is documented as thread-safe for serialisation, this combination indicates the developer may not have considered servlet instance concurrency.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization — multi-tenancy / 5. Data Exposure and Privacy
**Evidence:**

**3. Injection — File inclusion:** No `<jsp:include>` or `<%@ include>` with user-controlled paths. These are Java servlet files, not JSPs; no file inclusion vectors present.

**3. Injection — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls found in any of the three files.

**3. Injection — SSRF:** No `HttpURLConnection`, `HttpClient`, or equivalent HTTP client calls found in any of the three files.

**6. Transport Security:** No transport configuration present in these files. TLS configuration is a server/web.xml concern outside the scope of these three files.

**7. Dependency Vulnerabilities:** These files do not declare dependencies. Import of `com.google.gson.Gson` is noted; version assessment requires inspection of committed JARs, which is outside the scope of this agent's assigned files.

**SessionCleanupListener — design:** The session cleanup listener correctly implements `HttpSessionListener` and calls per-servlet `cleanupSession()` static methods when a session is destroyed, which is a sound approach to releasing session-keyed static map entries and preventing memory leaks. No security issues found in this file beyond the wildcard import noted in J33-F08.
