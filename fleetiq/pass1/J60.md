## Findings

### J60-F01: GPS coordinates and driver PII carried in report bean with no customer/tenant scoping field
**Description:** `ImpactReportBmwItemBean` stores GPS latitude, GPS longitude, and driver name as fields on every item in an impact report. There is no `customerId`, `tenantId`, or equivalent field on either the item bean or the parent `ImpactReportBmwBean`. On the multi-customer (multi-tenant) branch this means that if the code which populates these beans fails to filter by customer at the data-access layer, the resulting report bean will silently contain data belonging to any or all customers. The beans themselves provide no structural guarantee of tenant isolation. Any layer that renders the bean to Excel or returns it over an API cannot enforce isolation at the bean level — isolation is entirely dependent on the query or population logic upstream, which is not visible in these files.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and multi-tenancy; Section 5 — Data Exposure and Privacy (GPS/location data and driver data)
**Evidence:**

### J60-F03: KeyHourUtilBean fields declared with package-private visibility instead of private
**Description:** Every field in `KeyHourUtilBean` is declared without an explicit access modifier (lines 6-29). In Java this means package-private visibility: any class in the same package `com.torrent.surat.fms6.excel.reports.beans` can read or write these fields directly, bypassing the getter/setter methods. This is a violation of encapsulation but more relevantly for a security audit it means that report data (vehicle codes, names, utilisation figures) can be mutated or read by any code in the same package without going through a controlled interface. In a legacy codebase this increases the risk of accidental data leakage between bean instances or between report generations.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J60-F04: Raw (non-generic) ArrayList types used throughout KeyHourUtilBean
**Description:** All thirteen `ArrayList` fields in `KeyHourUtilBean` and all corresponding getter/setter method signatures use the raw `ArrayList` type without a type parameter (e.g., `ArrayList` rather than `ArrayList<String>`). This is not a direct security vulnerability but it is a code quality finding with indirect security relevance: the absence of generics means the compiler cannot enforce what type of objects are placed into these lists. If a caller inserts an unexpected type, a `ClassCastException` will occur at runtime, potentially bypassing normal report-generation logic or masking a data-handling error. In a multi-tenant report context, type confusion in list contents could contribute to data corruption or unintended data exposure.
**Severity:** INFO
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// KeyHourUtilBean.java lines 6-19 — representative examples
ArrayList veh_cd = new ArrayList();   // raw type, no generic parameter
ArrayList vutil1 = new ArrayList();   // raw type, no generic parameter
// ... same pattern for all 13 ArrayList fields
```
