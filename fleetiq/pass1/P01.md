## Findings

### P01-F01: Reflected XSS — Unescaped mail_id in Email Confirmation
**Description:** The `mail_id` parameter is read directly from the HTTP request and rendered unescaped into the HTML response at line 135 of xlsx_daily_veh_summary_report_mail.jsp. An attacker who can craft a POST request with a malicious `mail_id` value (e.g., `"><script>alert(1)</script>`) can inject arbitrary HTML/JavaScript into the page.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — XSS
**Evidence:**
```java
// Line 113
String mail_id=request.getParameter("mail_id");
// Line 135
<font color="white" face='verdana' size=2><b><%=mail_id%></b></font>
```
**File:** au_email/xlsx_daily_veh_summary_report_mail.jsp
**Line(s):** 113, 135

### P01-F02: Missing CSRF Protection on Email-Send Form
**Description:** The form in xlsx_daily_veh_summary_input_mail.jsp submits via POST to xlsx_daily_veh_summary_report_mail.jsp, which triggers sending an email and generating a report. No CSRF token is present in the form. An attacker can trick an authenticated user into submitting this form, potentially exfiltrating data to an attacker-controlled email address.
**Severity:** HIGH
**Checklist item:** 4. Session and CSRF
**Evidence:**
```html
<!-- Line 135 -->
<form name=xxxx method=post action="xlsx_daily_veh_summary_report_mail.jsp">
<!-- No hidden CSRF token field anywhere in the form -->
```
**File:** au_email/xlsx_daily_veh_summary_input_mail.jsp
**Line(s):** 135

### P01-F03: Unescaped DB Values in Dropdown (XSS)
**Description:** Mail group codes and names retrieved from the database are rendered directly into `<option>` HTML without escaping. If database values contain HTML special characters or JavaScript, XSS could result.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — XSS
**Evidence:**
```jsp
<!-- Line 149 -->
<option value=<%=grp_cd.get(i)%>><%=grp_nm.get(i)%></option>
```
**File:** au_email/xlsx_daily_veh_summary_input_mail.jsp
**Line(s):** 149

### P01-F04: Unescaped Request Parameters in Hidden Form Fields (XSS)
**Description:** Multiple request parameters (loc_nm, dept_nm, cust, gp_cd, user_cd, loc_cd, dept_cd, form_cd) are read and re-emitted directly into hidden HTML input `value=` attributes without HTML encoding. If the referrer page passed malicious values, these would be injected into the DOM.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — XSS
**Evidence:**
```jsp
<!-- Lines 167-177 -->
<input type="hidden" name="loc_nm" value="<%=loc_nm %>">
<input type="hidden" name="cust" value="<%=cust %>">
<input type="hidden" name="gp_cd" value="<%=gp_cd %>">
```
**File:** au_email/xlsx_daily_veh_summary_input_mail.jsp
**Line(s):** 167–177

### P01-F05: Path Traversal in Excel Report Directory Construction
**Description:** The temporary directory for Excel reports is constructed with `/../../../../../../../excelrpt/` relative to the class location. While this appears to be an intentional workaround, path construction using user-influenced values (the server's class path) through string manipulation without validation is a code smell that should be reviewed. No direct user control, so informational.
**Severity:** INFO
**Checklist item:** 3. Injection Vulnerabilities — File inclusion
**Evidence:**
```java
// Lines 98-103
String dir = this.getClass().getProtectionDomain().getCodeSource()
    .getLocation().toString().substring(6);
int charCount = dir.length() - dir.replaceAll("/", "").length();
int index = DataUtil.nthOccurrence(dir, '/', charCount - 1);
dir = dir.substring(0, index).replace("%20", " ");
dir += "/../../../../../../../excelrpt/";
```
**File:** au_email/xlsx_daily_veh_summary_report_mail.jsp
**Line(s):** 98–103
