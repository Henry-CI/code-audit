### J15-F01: Package-Private Field Access on Serializable Beans Bypasses Encapsulation
**Description:** In both `UnitUtilSummaryBean` (13 fields) and `UnitutilBean` (18 fields), every domain field is declared with package-private (default) access — no `private` modifier. Any class in the `com.torrent.surat.fms6.bean` package can read or write these fields directly, bypassing getters and setters entirely. In a legacy JSP application, JSP scriptlets frequently access bean fields directly rather than through property accessors. If a consuming scriptlet or servlet writes a field directly (e.g., `bean.hire_no = request.getParameter("hire_no")`), there is no interception point for validation or sanitisation before that value propagates to output or SQL.

### J15-F02: Mutable Internal State Exposed by Reference in UnitutilBean
**Description:** `UnitutilBean.getUtil()` returns a direct reference to the private `int[]` array. Any caller receiving this reference can modify the array contents without going through `setUtil()`. Similarly, `getUtilMap()`, `getImpactMap()`, and `getArrUtil()` return direct references to internal `HashMap` and `ArrayList` instances. In a multi-tenant context, if a bean instance is shared (e.g., cached at application or session scope and retrieved by multiple callers), this allows one caller to mutate another tenant's data structure. Even without sharing, it means there is no defensive copy and no validation on mutations made through the returned reference.

### J15-F03: chart_URL Field Has No Validation and Is Exposed for Direct Rendering
**Description:** `UnitutilBean` contains a `chart_URL` field (line 29) with a getter at line 182. The field name and context (utilisation charting) indicate this URL is likely rendered into a JSP page as an `<img src>`, `<iframe src>`, or JavaScript variable to display a chart. ### J15-F04: No Tenant/Customer Scoping in Any of the Three Beans
**Description:** None of the three beans (`UnitUtilSummaryBean`, `UnitVersionInfoBean`, `UnitutilBean`) contains a `customer_id`, `customerId`, `tenant_id`, or any equivalent tenant-discriminating field. These beans carry fleet utilisation data (hire numbers, serial numbers, vehicle codes, machine names, department codes, location codes, GMTP device IDs) that is inherently customer-specific in a multi-tenant system. The absence of a tenant field in the bean means that tenant scoping must be enforced entirely at the data-access layer (DAO/SQL) and correctly threaded through to every consumer. If any consuming servlet, JSP scriptlet, or DAO populates these beans without correctly applying a customer filter, the bean itself provides no backstop that would reveal the omission — the data simply flows through without a tenant identifier attached. This is a structural risk in a multi-tenant application, particularly given the branch name (`multi-customer-sync-to-master`) signals active multi-tenancy work.
**Severity:** MEDIUM

### J15-F05: UnitVersionInfoBean Does Not Implement Serializable
**Description:** `UnitVersionInfoBean` does not implement `java.io.Serializable`. The other two beans in this same package (`UnitUtilSummaryBean`, `UnitutilBean`) both implement `Serializable`, which is the standard pattern for beans stored in `HttpSession` in this application. ### J15-F06: Inconsistent Default Initialisation of hydl_hours in UnitUtilSummaryBean
**Description:** All hour-valued fields in `UnitUtilSummaryBean` are initialised to `"0"` except `hydl_hours`, which is initialised to `""` (empty string). If any consuming code performs numeric parsing on these fields (e.g., `Double.parseDouble(bean.getHydl_hours())`) without a null/empty check, the empty default will cause a `NumberFormatException`. This is a data integrity issue rather than a direct security vulnerability, but in a reporting context it can produce incorrect utilisation calculations and potentially mask data from one customer appearing in another customer's report (wrong totals that obscure IDOR).

## Checklist Items With No Issues

The following checklist items were reviewed against the three files in scope and no issues were found:

1. **Section 1 — Secrets and Credentials:** No hardcoded credentials, passwords, API keys, JDBC connection strings, or secret values of any kind are present in any of the three bean files. No comments containing credentials.

2. **Section 1 — Wildcard imports:** No wildcard imports (`import java.sql.*` or similar) are present in any of the three files.

3. **Section 3 — SQL Injection:** These are pure data-transfer bean (POJO) classes. They contain no JDBC calls, no `Statement`, no `PreparedStatement`, no SQL strings of any kind. SQL injection cannot be introduced at this layer.

4. **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()`, no `ProcessBuilder`, no OS-level invocation of any kind in any of the three files.

5. **Section 3 — SSRF:** No `HttpURLConnection`, `HttpClient`, or any outbound HTTP call in any of the three files. The `chart_URL` field stores a URL as a string value only — no HTTP request is made from within the bean itself (see J15-F03 for the rendering risk in consuming JSPs).

6. **Section 3 — File Inclusion:** No file I/O, no `InputStream`, no file path construction in any of the three files.

7. **Section 4 — Session and CSRF:** These beans are not servlets or JSP files and do not handle requests, generate responses, or manage session state directly. CSRF is not applicable at this layer.

8. **Section 6 — Transport Security:** No TLS configuration, connector configuration, or HTTP connection handling in these bean files.

7. **Section 7 — Dependency Vulnerabilities:** No third-party library dependencies introduced by these files beyond the Java standard library (`java.io`, `java.util`).

---

**Note on scope limitation:** These three files are pure data-transfer objects (DTOs/beans). The most significant security risks associated with this data (IDOR, XSS via unescaped rendering, SQL injection via unparameterised queries) will manifest in the servlet and JSP files that populate and consume these beans, not in the beans themselves. Findings J15-F03 and J15-F04 are flagged here as structural risks that must be verified in the consuming layer. The consuming JSPs and servlets for unit utilisation (likely named `unitutil*`, `vehicleUtil*`, or similar) should be reviewed as a priority in subsequent passes.
