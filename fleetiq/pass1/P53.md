### P53-F01: Reflected XSS via Unescaped JSP Expressions Interpolated into JavaScript
**Description:** In all six JSP files, request parameters `cust_cd`, `loc_cd`, and `dept_cd` are read directly from `request.getParameter(...)` and then interpolated unescaped into JavaScript string variables using `<%= %>` expressions. An attacker can inject arbitrary JavaScript by crafting a URL with a payload such as `cust_cd='; alert(1); var x='`. Because the values are written into a `<script>` block without any escaping or encoding, this constitutes a reflected Cross-Site Scripting (XSS) vulnerability. All six files are affected.

Note: `driver_licence_expiry.jsp` additionally interpolates the `category` parameter unescaped on line 60.

**Severity:** HIGH

**Checklist item:** Section 3 — XSS (Cross-Site Scripting)

**Evidence:**
```javascript
// critical_battery.jsp lines 110–112
```

### P53-F02: cust_cd Accepted From Request Parameter — Potential IDOR / Tenant Isolation Bypass
**Description:** In all six files, `cust_cd` (customer code) is accepted directly from the request parameter if supplied. Although the pages rely on `Expire.jsp` for session authentication, the customer scoping logic allows the caller to override the customer by passing `cust_cd` as a GET/POST parameter. The pattern is identical across all files:

```java
String cust_cd = request.getParameter("cust_cd") == null
    ? ((String)session.getAttribute("access_cust")).substring(2)
    : request.getParameter("cust_cd");
```

If the downstream servlets (e.g., `CriticalBatteryServlet`, `ImpactsServlet`, `LicenceServlet`, `PreopServlet`, `UtilisationServlet`, `pct_utilisationServlet`) do not independently verify that the supplied `cust_cd` belongs to the authenticated user's account, a user from Customer A can supply Customer B's `cust_cd` as a URL parameter and retrieve that customer's fleet data (vehicle battery, impact, licence, utilization, or preop data). This is a classic Insecure Direct Object Reference (IDOR) / multi-tenant isolation failure. The JSP pages themselves do not perform any validation that the supplied `cust_cd` matches the session user's permitted customer. The same applies to `loc_cd` and `dept_cd`.

**Severity:** HIGH

**Checklist item:** Section 2 — Authorization and multi-tenancy; IDOR

**Evidence:**
```java
// Pattern repeated in all six files (shown from critical_battery.jsp lines 4–6)
String cust_cd = request.getParameter("cust_cd") == null
    ? ((String)session.getAttribute("access_cust")).substring(2)
    : request.getParameter("cust_cd");
String loc_cd = request.getParameter("loc_cd") == null
    ? ((String)session.getAttribute("access_site")).substring(2)
    : request.getParameter("loc_cd");
String dept_cd = request.getParameter("dept_cd") == null
    ? ((String)session.getAttribute("access_dept")).substring(2)
    : request.getParameter("dept_cd");
```
**File:** `pages/admin/statistics/critical_battery.jsp`, `pages/admin/statistics/driver_licence_expiry.jsp`, `pages/admin/statistics/impacts.jsp`, `pages/admin/statistics/pct_utilization.jsp`, `pages/admin/statistics/preop.jsp`, `pages/admin/statistics/vehicles_utilization.jsp`

**Line(s):** Lines 4–6 in each file.

### P53-F06: No CSRF Protection on Filter/Date-Range Forms
**Description:** All six JSP files contain HTML forms that submit to backend servlets using GET method (e.g., `<form action="../Servlet/CriticalBatteryServlet?part=dates&mode=" method="get" id="filter">`). None of the forms include a CSRF token hidden field or any other CSRF protection mechanism visible in the JSP layer. While GET forms are less critical for CSRF than POST forms for state-changing operations, the broader application pattern of no CSRF tokens is confirmed here. These forms submit `cust_cd`, `loc_cd`, `dept_cd`, `start_time`, and `end_time` to reporting servlets — if the underlying servlets accept cross-origin requests, CSRF could be used to force a user to exfiltrate data. This is consistent with the pattern expected to be flagged as a wider application-level finding.

**Severity:** MEDIUM

### P53-F07: Driver PII (Name, Card Number, Licence Expiry) Exposed in Table Output Without Visible Access Control Scoping in JSP Layer

## Checklist Items With No Issues

The following checklist items were reviewed against the six JSP files in this assignment. No findings were identified for:

- **Section 1 — Secrets and Credentials:** No database credentials, API keys, SMTP passwords, or encryption keys found in these JSP files. The `LindeConfig` references are calls to a utility class, not inline credential strings. (Hardcoded customer ID `20` and demo user IDs are reported under F04/F05 as separate concerns.)
- **Section 1 — Bitbucket Pipelines:** Not applicable to these JSP files.
- **Section 1 — Binary artifacts:** Not applicable to these JSP files.
- **Section 3 — SQL Injection:** No direct SQL execution (`Statement`, `PreparedStatement`, `executeQuery`, etc.) appears in any of the six JSP scriptlet blocks. All data retrieval is delegated to backend servlets via AJAX. SQL injection risk exists at the servlet layer (outside scope of this agent's assigned files).
- **Section 3 — File inclusion:** All `<%@ include %>` and `<jsp:include>` directives use static, hardcoded paths (`../../../sess/Expire.jsp`). No user-controlled file inclusion found.
- **Section 3 — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls found in any of the six files.
- **Section 3 — SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) found in these JSP files. All external requests are client-side JavaScript `$.get()` calls to relative servlet paths, not server-side HTTP calls.
- **Section 6 — Transport Security:** These JSP files contain no Tomcat connector configuration, TLS settings, or mixed content resource loading. Transport security must be assessed at the `server.xml` / `context.xml` level (outside scope of these files). No HTTP resource links (`http://`) found in any of the six files.
- **Section 7 — Dependency Vulnerabilities:** These JSP files do not declare or reference library versions directly. Dependency assessment requires `pom.xml` / `build.xml` / JAR inventory review outside this file set.
