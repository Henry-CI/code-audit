### J45-F01: Excel Formula Injection via Unvalidated Database Values
**Description:** In `EmailUtilisationReport.java`, data values retrieved from `UnitUtilSummaryBean` (which originates from `Databean_report`, itself populated from the database) are embedded directly into Excel cell formula strings using string concatenation. If an hour value such as `total_hours`, `key_hours`, `seat_hours`, `track_hours`, or `hydl_hours` contains a formula-like string (e.g., beginning with `=`, `+`, `-`, or `@`), it will be interpreted as a formula by spreadsheet applications. In Apache POI's `setCellFormula()`, the constructed string is written verbatim into the XLSX formula field. A malicious or corrupted value such as `+cmd|' /C calc'!A0` or `=HYPERLINK("http://attacker.example/",1)` in a duration field would execute when the recipient opens the file. This is a classic CSV/Excel injection (also called formula injection) vulnerability.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**

### J45-F02: Broadcast Message Text Written to Excel Without Sanitisation
**Description:** In `ExcelBroadcastMsgReport.java`, the field `broadcastmsgBean.getText()` — which contains the text of a broadcast message sent from the system to vehicle-mounted devices or received in response — is written directly to an Excel cell with no sanitisation or formula-prefix stripping. Broadcast messages may be operator-authored or vehicle-generated, and an attacker with access to the messaging system (or a rogue vehicle unit) could craft a message that executes on the report recipient's machine. ### J45-F03: Missing Customer Scoping Check in EmailUtilisationReport — Potential IDOR
**Description:** `EmailUtilisationReport.createUtilReport()` accepts a `cust_cd` (customer code) parameter passed through the constructor from the caller. When `cust_cd` equals `"0"` it is blanked out to `""` (line 46-48), which may result in a query that returns data for all customers rather than being restricted to the authenticated user's customer. There is no check within this class that `cust_cd` matches the authenticated session user's customer. The class itself does not perform any authentication or authorisation check — it trusts that the caller has validated the customer code. Given the multi-tenant context (branch `multi-customer-sync-to-master`) this is a significant concern: if any calling servlet or JSP passes a user-supplied `cust_cd` to this constructor without validating it against the session, data from another customer will be returned in the report. This class's design makes IDOR possible at the caller level.
**Severity:** HIGH
