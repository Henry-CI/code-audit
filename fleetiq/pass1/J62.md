## Findings

### J62-F01: Raw ArrayList fields carry no customer/tenant scoping — multi-tenancy isolation absent at bean level
**Description:** All three report bean classes hold report data in raw, untyped `ArrayList` fields with no customer ID, tenant ID, or owner reference anywhere in the bean structure. `ServMaintenanceReportBean` contains vehicle codes (`vmachine_cd`), vehicle names (`vmachine_nm`), service IDs (`service_id`), and service dates. `UnitUnlockReportBean` contains unit names (`vunit_name`), lock/unlock times, lock reasons, and driver identifiers. Neither bean carries any field binding the data to a specific customer account. In a multi-tenant system (this branch is `multi-customer-sync-to-master`), if the code that populates these beans does not rigorously filter by customer ID at the SQL layer, data from other tenants will silently flow into the bean and into the generated report. The bean design provides no safety net — it cannot detect or reject cross-tenant data. This is a structural data isolation gap.
**Severity:** HIGH
**Checklist item:** Section 2 — Authorization and multi-tenancy; Section 5 — Data Exposure and Privacy
**Evidence:**

### J62-F02: SuperMasterAuthReportBean exposes superMasterCode in a report bean with no tenant isolation
**Description:** `SuperMasterAuthReportBean` holds a list of `SuperMasterAuthBean` objects. Each `SuperMasterAuthBean` carries a field named `superMasterCode` — a code associated with "super master" authorisation. This is exposed via a public getter `getSuperMasterCode()` with no restrictions. The report bean itself has no customer-scoping field. If any report generation layer (servlet, JSP, or export function) that populates this bean has a flaw in its SQL query — missing or bypassable `customer_id` filter — then a user could receive `superMasterCode` values belonging to other tenants in their exported report. Super-master codes represent elevated authorisation credentials; their cross-tenant exposure would be a privilege escalation risk. Additionally, the `custName` field in `SuperMasterAuthBean` suggests this bean is already aware of a customer dimension, but the parent report bean (`SuperMasterAuthReportBean`) provides no guard or constraint on which customers' records may be present.

### J62-F04: Raw (untyped) ArrayList use throughout all three beans
**Description:** Every `ArrayList` field in all three bean classes is raw — no generic type parameter is used (e.g., `ArrayList` instead of `ArrayList<String>`). This means the compiler cannot enforce what type of object is placed in any collection. In a security context, this increases the risk that mixed-customer data could be inserted without a type mismatch being detected at compile time, and it eliminates the ability to add type-checked validation. `UnitUnlockReportBean` and `ServMaintenanceReportBean` use exclusively raw `ArrayList` for all fields. `SuperMasterAuthReportBean` declares its field with a generic type (`ArrayList<SuperMasterAuthBean>`) but initialises it with a raw `new ArrayList()` (line 9), producing an unchecked warning and the same underlying risk.
**Severity:** LOW
**Checklist item:** Section 2 — Authorization and multi-tenancy (type safety as a defence control); Section 5 — Data Exposure and Privacy

### J62-F06: UnitUnlockReportBean carries driver identity and vehicle lock/unlock event data with no tenant scope
**Description:** `UnitUnlockReportBean` stores driver names associated with lock (`vunit_lodriver`) and unlock (`vunit_uldriver`) events, lock timestamps (`vunit_lotm`), unlock timestamps (`vunit_ultm`), lock reasons (`vunit_loreason`), and unit names (`vunit_name`). This constitutes driver PII (identity tied to security events) and vehicle operational data. The bean has no customer ID field, no tenant scope, and no indication of which account the data belongs to. In a multi-tenant deployment, a report generation bug at the data access layer (missing or bypassed `customer_id` WHERE clause) would silently expose another tenant's driver identities and vehicle security event history through this bean.
**Severity:** HIGH
**Checklist item:** Section 5 — Data Exposure and Privacy; Section 2 — Authorization and multi-tenancy

## Checklist Items With No Issues

The following checklist items were reviewed against the three bean files and found to have no issues within the scope of these files. Note that these are data transfer/report bean classes only — they contain no SQL, no HTTP handling, no authentication logic, and no output rendering. The checklist items below are assessed as not applicable or clean at this layer; they must be assessed in the servlet and JSP layers that use these beans.

1. **Section 1 — Secrets and Credentials:** No hardcoded credentials, passwords, API keys, JDBC connection strings, or encryption keys are present in any of the three files. No `.properties` file references with embedded secrets.

2. **Section 3 — SQL Injection:** No SQL statements, `Statement`, `PreparedStatement`, or `DriverManager` calls are present in any of the three files. These are pure data-holder beans.

3. **Section 3 — XSS:** No JSP expressions, `out.print()`, or HTML rendering code is present. These are Java beans, not JSP files. XSS risk assessment must be performed on the JSP/servlet layer that reads from these beans and renders output.

4. **Section 3 — File Inclusion:** No file I/O, `ClassLoader.getResource()`, or include directives present.

5. **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()`, `ProcessBuilder`, or process execution present.

6. **Section 3 — SSRF:** No HTTP client calls, URL construction, or external service invocation present.

7. **Section 4 — Session and CSRF:** These are data beans with no HTTP request handling. No session manipulation, no form rendering, no CSRF token handling is present or expected.

8. **Section 6 — Transport Security:** No network I/O, connector configuration, or TLS-relevant code present.

9. **Section 7 — Dependency Vulnerabilities:** Imports are limited to `java.util.ArrayList` and one internal project class (`SuperMasterAuthBean`). No third-party library dependencies are introduced by these files.
