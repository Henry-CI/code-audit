## Findings

### P68-F01: Cross-Site Scripting (XSS) - Unescaped Output in All JSP Files
**Description:** All 10 JSP files output database values and user-controlled data without proper HTML escaping using the `<%= ... %>` expression syntax. While these are Excel export files (application/vnd.ms-excel content type), Excel will still render HTML content and browsers may display these files inline. User-controlled data such as vehicle names, driver names, broadcast message text, and response text are output directly into the HTML without escaping, creating a stored XSS vulnerability. An attacker who can inject malicious content into vehicle names, driver names, or broadcast messages could execute JavaScript when these reports are viewed.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:** Multiple instances across all files. Examples:
```jsp
Line 93 (excel_battery_charge.jsp): <%=batteryBean.getBef_veh_nm() %>
Line 95 (excel_battery_charge.jsp): <%=batteryBean.getBef_dri_nm() %>
Line 92 (excel_broadcastmsg.jsp): <%=broadcastmsgBean.getText() %>
Line 93 (excel_broadcastmsg.jsp): <%=broadcastmsgBean.getResponse() %>
Line 115 (excel_blacklist_driv.jsp): <%=Vuser_fnm.get(i) %> <%=Vuser_lnm.get(i) %>
Line 164 (excel_curr_driv_report.jsp): <%=vdriv_nm.get(i) %>
Line 167 (excel_curr_driv_report.jsp): <%=vveh_nm.get(i) %>
```

**File:** All 10 files
**Line(s):** Multiple lines in each file as documented in reading evidence

### P68-F02: Potential Insecure Direct Object Reference (IDOR) - Missing Parameter Validation
**Description:** All JSP files accept customer, location, and department parameters directly from the request (`cust_cd`, `loc_cd`, `dept_cd`) without visible validation against the authenticated user's session-based access rights before passing them to the filter beans. While session attributes for access control are retrieved (access_cust, access_site, access_dept), the request parameters are set on the filter bean alongside these session attributes. Without seeing the backend filter implementation, we cannot verify that the filter properly enforces that request parameters cannot override or bypass session-based access controls. This creates a potential IDOR vulnerability where a user might manipulate URL parameters to access reports for other customers, locations, or departments.

**Severity:** HIGH

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:** Pattern repeated in all files:
```jsp
// excel_battery_charge.jsp lines 16-22
String gp_cd = request.getParameter("gp_cd")==null?"1":request.getParameter("gp_cd");
String user_cd = request.getParameter("user_cd")==null?"":request.getParameter("user_cd");
String loc_cd = request.getParameter("loc_cd")==null?"":request.getParameter("loc_cd");
String dept_cd = request.getParameter("dept_cd")==null?"":request.getParameter("dept_cd");

// Lines 31-35
String access_level = (String)session.getAttribute("access_level");

String access_site=(String)session.getAttribute("access_site");
```

### P68-F03: Insecure Default Access Level
**Description:** All JSP files implement a fallback where if the `access_level` session attribute is empty, it defaults to "5". Without understanding the access level scheme (whether higher numbers mean more or less privilege), this default assignment is concerning. In multi-tenant applications, defaulting to any access level when the session value is missing could bypass authentication or grant unintended privileges. The proper behavior should be to deny access or redirect to login if the access_level is not properly set in the session.

**Severity:** MEDIUM

**Checklist item:** 2. Authentication and Authorization - Session management

**Evidence:** Pattern in all files with access control:
```jsp
// excel_battery_charge.jsp lines 36-40
if(access_level.equalsIgnoreCase(""))
{
    access_level = "5";
}
int al = Integer.parseInt(access_level);
```

**File:** All files except excel_driver_league.jsp which doesn't include this check
**Line(s):** Lines 36-40 (battery_charge), 37-40 (blacklist_driv), 39-42 (broadcastmsg), 45-48 (curr_driv_report), 45-48 (curr_unit_report), 31-34 (daily_driver_summary), 31-34 (daily_veh_summary), 31-34 (driver_access_abuse), not present in driver_league, not present in driver_licence_expiry_detail

### P68-F04: No CSRF Protection on Report Generation
**Description:** None of the JSP files implement CSRF protection mechanisms. While these are report export endpoints that primarily perform read operations, they are triggered via HTTP requests (likely GET based on the parameter retrieval pattern) without CSRF tokens. In a multi-tenant fleet management system, an attacker could craft malicious links that, when clicked by an authenticated user, generate reports containing sensitive data (GPS locations, driver information, vehicle data) and potentially exfiltrate this data. The broadcast message report (excel_broadcastmsg.jsp) is particularly concerning as it could be used to view messages sent to/from specific vehicles.

**Severity:** MEDIUM

**Checklist item:** 4. Session and CSRF

**Evidence:** No CSRF token validation found in any file. Example structure:
```jsp
// excel_battery_charge.jsp - no token validation
String gp_cd = request.getParameter("gp_cd")==null?"1":request.getParameter("gp_cd");
// Direct parameter usage with no CSRF check
```

**File:** All 10 files
**Line(s):** N/A - absence of CSRF protection throughout

### P68-F05: Sensitive Data Exposure in Driver Reports
**Description:** Multiple reports expose personally identifiable information (PII) and business-sensitive data including driver names (first and last), vehicle identifiers, GPS-derived data (locations, times), license information, and performance metrics. The excel_driver_licence_expiry_detail.jsp specifically exposes driver license types and expiry dates. The excel_driver_access_abuse.jsp exposes which drivers accessed which vehicles at specific times, creating a detailed audit trail that could reveal sensitive operational patterns. Without proper access controls verified at the JSP level (only session attributes are retrieved), this data could be exposed to unauthorized users within the same tenant or across tenants if IDOR vulnerabilities (P68-F02) are exploited.

### P68-F06: Hardcoded External HTTP URLs in Stylesheets
**Description:** Two files (excel_blacklist_driv.jsp and excel_curr_driv_report.jsp, excel_curr_unit_report.jsp, excel_driver_licence_expiry_detail.jsp) contain hardcoded HTTP (non-HTTPS) links to external stylesheets at "http://fms.fleetiq360.com/css/". One file (excel_blacklist_driv.jsp line 8) contains a malformed URL starting with "ttp://" instead of "http://". Using HTTP instead of HTTPS for external resources creates a mixed content vulnerability if the application is served over HTTPS, and exposes users to man-in-the-middle attacks where the stylesheet could be intercepted and modified to inject malicious content.

### P68-F07: Potential Null Pointer Exception on Access Level Parsing
**Description:** All files that implement access control retrieve the `access_level` from session, check if it's an empty string, default to "5" if empty, and then parse it to an integer. However, the session.getAttribute() can return null (not just an empty string). The code uses .equalsIgnoreCase("") which will throw a NullPointerException if access_level is null. While variable `al` is assigned the parsed value, it's never used in the JSP (suggesting the validation happens in the filter bean), but the NullPointerException could cause the request to fail before reaching the filter.

**Severity:** LOW

**Checklist item:** 2. Authentication and Authorization - Session management

**Evidence:**
```jsp
// excel_battery_charge.jsp lines 31-40
String access_level = (String)session.getAttribute("access_level");
// ... other session attributes ...

if(access_level.equalsIgnoreCase(""))  // NPE if access_level is null
{
    access_level = "5";
}
int al = Integer.parseInt(access_level);
```

**File:** All files with access control except excel_driver_league.jsp
**Line(s):** Lines 36-40 (battery_charge), 37-40 (blacklist_driv), and similar in other files
