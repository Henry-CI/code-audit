### J72-F01: SQL Injection — `send_data` Request Parameter Concatenated Directly into UPDATE Statement
**Description:** In `GetMessages.doPost()`, the request parameter `send_data` is read from the HTTP POST body and inserted directly into a SQL UPDATE statement with no sanitisation, no validation, and no use of `PreparedStatement`. An attacker who can reach this servlet can inject arbitrary SQL, including subqueries, UNION selects, or destructive statements, against the database.
**Severity:** CRITICAL
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL Injection)
**Evidence:**

### J72-F02: SQL Injection — Session-Derived `uid` Concatenated into Multiple SELECT Queries
**Description:** In `GetMessages.doPost()`, the value of `user_cd` is retrieved from the session and concatenated directly into two SQL SELECT queries using `Statement` (not `PreparedStatement`). While `uid` originates from the session rather than a request parameter, it represents a trust boundary: if session data is ever tampered with (e.g., through session fixation, or if `user_cd` is stored without sanitisation at login), SQL injection is possible. The use of `Statement` with concatenation is also a defence-in-depth failure — the correct pattern for any value, regardless of source, is `PreparedStatement` with `?` placeholders.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL Injection)
**Evidence:**

### J72-F05: Missing Null Check on Session Attribute — Potential NullPointerException / Denial of Service
**Description:** In `GetMessages.doPost()`, `session.getAttribute("user_cd")` is immediately called with `.toString()` without a null check. If `user_cd` is absent from the session (e.g. for an unauthenticated or timed-out session that still has an active `HttpSession` object), this throws a `NullPointerException`. The exception is caught by the broad `catch (Exception e)` block and the error message (including the exception detail) is returned to the caller. Beyond the error disclosure issue (covered in J72-F03), the absence of an explicit authentication gate means the servlet does not actively reject unauthenticated requests — it simply crashes on the missing attribute.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization
**Evidence:**
```java
// Line 44
String uid = session.getAttribute("user_cd").toString();
```
No preceding null-check or authentication verification exists before this line.
**File:** `WEB-INF/src/com/torrent/surat/fms6/security/GetMessages.java`
**Line(s):** 43–44

### J72-F06: Long-Polling Servlet Holds Thread for Up to 30 Seconds
**Description:** `GetMessages.doPost()` implements a long-polling mechanism using `Thread.sleep(1000)` inside a `while(true)` loop that runs for up to `maxInvalidCount` (30) iterations. This means each request to this servlet can hold a Tomcat worker thread for up to 30 seconds. Under moderate load, this will exhaust the Tomcat thread pool and cause a denial-of-service condition for all other users. The correct pattern for long-polling in a Java servlet container is asynchronous servlet support (`AsyncContext`), not `Thread.sleep`.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (availability / session management context)
**Evidence:**
```java
// Lines 78–122
while(true)
{
    // ...
    if(i==maxInvalidCount)
    {
        // break after 30 iterations
        break;
    }
    Thread.sleep(1000); //sleep one second
}
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/security/GetMessages.java`
**Line(s):** 78–122

### J72-F10: No Multi-Tenant Scoping Validation in `GetMessages` — IDOR Risk
**Description:** In `GetMessages.doPost()`, the servlet reads `user_cd` from the session and uses it to filter queries. However, the `send_data` parameter (a list of message IDs) used in the `comfrim_read` UPDATE is not validated against the authenticated user's ownership of those IDs. Any authenticated user who knows or can guess message IDs belonging to another user can mark those messages as read by submitting them in `send_data`. This is an Insecure Direct Object Reference (IDOR) vulnerability in a multi-tenant context.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / IDOR)
**Evidence:**
```java
// Line 46 — send_data taken from request with no ownership check
String send_data = req.getParameter("send_data");
// Lines 53–54 — used directly in UPDATE with no WHERE user_id check
queryString = "update driver_access_abuse_subscription set status = 2 where id in ("+send_data+")";
stmt.executeUpdate(queryString);
```
The UPDATE does not include `AND user_id = uid`, meaning any valid `id` value can be targeted regardless of which user it belongs to.
**File:** `WEB-INF/src/com/torrent/surat/fms6/security/GetMessages.java`
**Line(s):** 46, 53–54

## Checklist Items With No Issues

The following checklist sections were reviewed against the three assigned files and no issues were identified:

- **Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, passwords, or connection strings found in any of the three files. Database connections are obtained via `DBUtil.getConnection()`, an abstraction layer — the credentials themselves are not visible in these files.
- **Section 3 — File Inclusion:** No JSP file includes are present in any of these files (all three are pure Java). No `Runtime.exec()` or `ProcessBuilder` usage found — no command injection risk in these files.
- **Section 3 — SSRF:** No `HttpURLConnection`, `HttpClient`, or HTTP client calls of any kind found in these three files.
- **Section 4 — CSRF:** Not directly applicable to `BeanComparator` (utility class) or `CftsAlert` (background scheduler utility). `GetMessages` is a servlet receiving POST requests; CSRF token validation is not present, but CSRF protection for non-browser API endpoints (long-poll style) is a lower concern — the broader application CSRF posture requires separate review of `web.xml` and JSP forms. This is noted but not raised as a finding against these files in isolation.
- **Section 6 — Transport Security:** No TLS configuration is present in these files; transport security must be assessed in `server.xml` and `context.xml` (not in scope for this agent).
- **Section 7 — Dependency Vulnerabilities:** No dependency configuration is contained in these files; library versions must be assessed via `pom.xml`/`build.xml` and committed JARs (not in scope for this agent).
- **BeanComparator.java — Security:** Beyond the wildcard import note (J72-F11), this class is a generic sorting utility using reflection. The reflection usage is constrained to method lookup by name on a caller-supplied class — the `methodName` and `beanClass` parameters are not user-controlled at the HTTP layer (they are supplied by internal Java callers), so there is no meaningful reflection-based attack surface in this file.
