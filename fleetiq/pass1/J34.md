### J34-F05: Instance-level field shared across concurrent requests — TableServlet.java (data leakage between users)
**Description:** `TableServlet` stores mutable request-scoped state in instance fields: `cusList`, `rsList`, `vehicles`, and `drivers` (lines 36–41). Servlets are singletons — there is one instance shared across all concurrent requests. When two requests execute simultaneously, the `vehicles` or `drivers` string built for user A can be overwritten by user B's request before user A's query executes, causing user A's query to run with user B's vehicle/driver list or vice versa. This causes both incorrect results and potential data leakage between tenants. The `Utilisation.java` file explicitly refactored this pattern into session-keyed `ConcurrentHashMap`s to fix exactly this problem, but `TableServlet.java` was never updated.

### J34-F07: `search_crit` parameter not validated before use as SQL LIKE value — multiple methods
**Description:** The `search` parameter retrieved from `search_crit` in `getDetailUnit()`, `getDetailDriver()`, `getDriverImpact()`, `getVORStatus()`, and `getPreop()` is concatenated into SQL `ILIKE` or `LIKE` predicates without sanitisation. While SQL metacharacters in a `LIKE` clause can be used for pattern injection (e.g., `%` wildcard amplification, leading to excessive table scans), more critically, because the surrounding query uses `Statement` rather than `PreparedStatement`, the single-quote in the `ILIKE` value is unsanitised, enabling full SQL injection through the search field. Additionally, in `getVORStatus()` and `getPreop()`, the search string is inserted into `nameCon` which is appended after other query clauses without an `AND` keyword guard in `getPreop()` (line 325 uses `WHERE` instead of `AND`), potentially producing malformed or injected SQL differently from the other methods.

**Checklist item:** 3. Injection Vulnerabilities — SQL injection

### J34-F10: Exception detail written to `System.out` / `e.printStackTrace()` — information disclosure
**Description:** Every catch block in both `TableServlet.java` and `Utilisation.java` calls `System.out.println(servlet + ": " + e)` and `e.printStackTrace()`.
