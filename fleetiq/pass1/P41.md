## Findings

### P41-F01: Reflected XSS via unescaped request parameters in HTML output
**Description:** Multiple report pages output `request.getParameter(...)` values directly into HTML attributes and element content using raw `<%= %>` expressions with no escaping. An attacker who can cause a victim to load a crafted URL can inject arbitrary HTML/JavaScript. Affected parameters include `mnm`, `sub`, `search_crit`, `message`, `user_cd`, `loc_cd`, `dept_cd`, `st_dt`, `to_dt`, `sev`, `form_cd`, `rpt_filter`, `lockouttype`, `inc`. These appear in hidden input values, breadcrumb href attributes, filter form fields, and action button URLs. No JSTL `<c:out>` or `fn:escapeXml` is used anywhere in these files.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// impact_meter_report.jsp lines 224–225
<input type="hidden" value="<%=request.getParameter("mnm").toString()%>" name="mnm">
<input type="hidden" value="<%=request.getParameter("sub").toString()%>" name="sub">

// impact_meter_report.jsp line 316
<input type="text" id="search_crit" name="search_crit" value="<%=search_crit %>" class="form-control input-lg" />

// impact_unlock_report.jsp line 330
<td align=center style="color: red" colspan="7" >  &nbsp; <%=message %></td>

// machine_unlock_report.jsp line 377
<td align=center style="color: red" colspan="7" >  &nbsp; <%=message %></td>
```
**File:** pages/admin/reports/impact_meter_report.jsp, impact_report.jsp, impact_unlock_report.jsp, inspection_report.jsp, machine_unlock_report.jsp, message_status_report.jsp, monthly_report.jsp, pre-op_check_list_report_original_with_no_filters.jsp, pre-op_check_report.jsp
**Line(s):** impact_meter_report.jsp:224–225, 316; impact_unlock_report.jsp:330; machine_unlock_report.jsp:377

### P41-F02: Stored XSS via unescaped database values in HTML output
**Description:** Every report page renders database-sourced values directly via `<%= %>` with no HTML escaping. Values include driver names, machine names, model numbers, vehicle serial numbers, site names, lockout reasons, unlock reasons (free-text), failed pre-op checklist answers (free-text), notes/comments fields, messages, and "sent by" user strings. Any of these fields that accept user-supplied input at the data-entry layer become stored XSS vectors when rendered in reports without escaping.

### P41-F03: DOM-based / reflected XSS via unescaped values injected into JavaScript event handlers and inline script
**Description:** Several files inject server-side variables directly into JavaScript `onClick` event handler attributes and Highcharts JavaScript configuration without HTML or JavaScript encoding. The `tmpTime` (UTC timestamp from DB), `Vveh_cd` (vehicle code), `dcd` (driver code), `msg_cd` (message code), `msg.get(i)` (message text), and `veh_hn` (hire number) values are all placed between JavaScript string delimiters in inline event attributes. A value containing a single quote terminates the JS string; a value containing `</script>` can break out of the script context. The `impact_meter.jsp` file also injects a user-controlled `imp` parameter after arithmetic into a Highcharts `data:` array.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — XSS (Cross-Site Scripting)
**Evidence:**

### P41-F04: XSS via unescaped values injected into HTML attribute (inline style and data-url)
**Description:** Multiple files inject database-derived or request-parameter values directly into HTML tag attributes using `<%= %>` — including `style` attributes (for background colours derived from DB impact levels), `data-url` attributes, and `bgcolor` attributes. While some colour values come from internally-derived logic, the `bgc` variable in impact reports and the `msg_col` in message_status_report.jsp are constructed from DB string comparisons and injected into `style='...'` attribute contexts without quoting or escaping. ## Checklist Items With No Issues

The following checklist items were reviewed and found to have no issues within the scope of these 10 JSP files:

- **1. Secrets and Credentials:** No hardcoded credentials, API keys, passwords, JDBC connection strings, or `.env` references were found in any of the 10 reviewed files. All database access is abstracted through the `Databean_report` bean.

- **2. Authentication — session fixation:** Session fixation cannot be assessed from these view-layer JSP files; these files read from the session but do not perform login or session creation. The `Expire2.jsp` include (session expiry check) is consistently present across all 10 files, indicating session expiry enforcement is in place at the include level.

- **3. Injection — File inclusion:** All `<%@ include file="..." %>` and `<jsp:include>` directives in these files use static, hardcoded paths. No user-controlled values are used in include paths.

- **3. Injection — Command injection:** No use of `Runtime.getRuntime().exec()` or `ProcessBuilder` was found in any of the 10 reviewed files.

- **3. Injection — SSRF:** No `HttpURLConnection`, `HttpClient`, or HTTP client calls were found in the JSP scriptlet code of the 10 reviewed files. The chart image URLs (`impactChart`, `utilChart`, `fleetcheckChart`) in `monthly_report.jsp` are rendered as `<img src="...">` in HTML — any SSRF risk depends on what the `filter` bean generates for those URLs, not the JSP itself.

- **3. Injection — SQL injection:** No direct JDBC or SQL calls (e.g., `Statement.executeQuery()`, `PreparedStatement`, `DriverManager.getConnection()`) are present in the scriptlet code of any of the 10 reviewed JSP files. All data access is delegated to the `Databean_report` bean class. SQL injection risk must be assessed in the Java source for that bean (out of scope for this agent batch).

- **6. Transport Security:** No TLS configuration (`server.xml`, `context.xml`) was reviewed in this batch; those files are out of scope. No mixed-content HTTP resource references (e.g., `http://` URLs in `<script src>` or `<link href>`) were observed in the 10 reviewed files; all resource references visible are relative paths.

- **7. Dependency Vulnerabilities:** No dependency files (`pom.xml`, `build.xml`, JAR manifests) were reviewed in this batch. The `org.apache.commons.codec` import in `impact_meter_report.jsp` and `impact_report.jsp` (line 2 in each: `org.apache.commons.codec.language.bm.Rule.RPattern`) indicates Apache Commons Codec is on the classpath; version and CVE status must be assessed via the inventory agent.
