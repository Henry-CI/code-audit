## Findings

### P72-F01: Critical Authentication Bypass - Missing Session Check
**Description:** The file `gpsImpactEmail.jsp` does not include the session expiry check file (`../sess/Expire.jsp`) that is present in other GPS-related JSP files (e.g., `gpsImpact.jsp`, `hire_vehicle_lst.jsp`). This means the page can be accessed without authentication, potentially exposing sensitive GPS location data, vehicle tracking information, and customer data to unauthenticated users. The file processes parameters such as customer codes, vehicle codes, and GPS timestamps without verifying that a valid user session exists.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization - Session management and multi-tenancy

**Evidence:**
```jsp
<%@ page import="java.util.*,com.torrent.surat.fms6.util.*"%>
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
```

Compare with gpsImpact.jsp which correctly includes:
```jsp
<%@ page import="java.util.*,com.torrent.surat.fms6.util.*"%>
<%@ include file="../sess/Expire.jsp" %>
```

**File:** reports/gpsImpactEmail.jsp

**Line(s):** 1-2

### P72-F02: Critical Authorization Bypass - Hardcoded Privileged Access Level
**Description:** The file `gpsImpactEmail.jsp` hardcodes the access level to "1" (the most privileged level in the system) instead of reading it from the user's session. This allows any user who can access the page to bypass authorization controls and view data across all customers, sites, and departments regardless of their actual permission level. This is a multi-tenant security violation that breaks customer data isolation.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
```jsp
// Session variables for the access level control
String access_level = "1";
String access_cust="";
String access_site="";
String access_dept="";
```

Compare with correct implementation in gpsImpact.jsp:
```jsp
// Session variables for the access level control
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");
```

**File:** reports/gpsImpactEmail.jsp

**Line(s):** 55-58

### P72-F04: High - Cross-Site Scripting (XSS) in XML/JSON Output
**Description:** All AJAX endpoint JSP files output data retrieved from the database directly into XML or JSON responses without any escaping or sanitization. If database values contain special characters like `<`, `>`, `"`, `'`, or if they have been previously compromised with malicious content, these will be rendered as-is in the response. When client-side JavaScript parses this XML/JSON and renders it in the DOM, XSS attacks can occur. This is a stored XSS vulnerability vector if attackers can influence database content (e.g., through customer names, department names, driver names).

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**

get_customer.jsp lines 23-25:

### P72-F06: Medium - Sensitive GPS and Fleet Data in URL Parameters
**Description:** The GPS tracking pages (`gpsImpact.jsp` and `gpsImpactEmail.jsp`) accept sensitive parameters including GPS timestamps, vehicle codes, and driver codes via URL query parameters. These parameters are then embedded in hidden form fields and passed to AJAX endpoints. URL parameters are logged in web server access logs, browser history, and referrer headers, which can lead to information disclosure. GPS coordinates and vehicle tracking data should be considered sensitive in a fleet management system, and exposure in logs could violate privacy requirements or customer data protection agreements.

**Severity:** MEDIUM

**Checklist item:** 5. Data Exposure and Privacy - GPS/location data

**Evidence:**

String map_dt = request.getParameter("map_dt")==null?"":request.getParameter("map_dt");

### P72-F09: Info - Wildcard Import Statements
**Description:** All JSP files use wildcard imports (`import="java.util.*,com.torrent.surat.fms6.util.*"`), which import all classes from the specified packages. While not directly a vulnerability, wildcard imports can lead to namespace pollution, make code harder to audit (unclear which specific classes are used), and in rare cases can introduce security issues if a malicious class with the same name as a commonly used class is added to the imported package. Best practice is to import only the specific classes needed.

## Summary

This audit reviewed 10 JSP files in the `reports/` directory. The most critical findings are:

1. **P72-F01 (CRITICAL)**: `gpsImpactEmail.jsp` has no authentication check, allowing unauthenticated access to GPS tracking data.
2. **P72-F02 (CRITICAL)**: `gpsImpactEmail.jsp` hardcodes the access level to "1" (admin), allowing any user to bypass authorization and view all customer data.
3. **P72-F03 (HIGH)**: Cross-site scripting vulnerabilities in JavaScript contexts across three files.
4. **P72-F04 (HIGH)**: Cross-site scripting vulnerabilities in XML/JSON output across seven AJAX endpoint files.
5. **P72-F05 (MEDIUM)**: Potential IDOR vulnerabilities where user-supplied resource IDs are not verified against the authenticated user's authorized scope at the JSP layer.

These findings indicate significant security gaps in authentication, authorization, and input/output handling. The `gpsImpactEmail.jsp` file appears to be a debug or test version of `gpsImpact.jsp` that was never properly secured and should be investigated immediately.
