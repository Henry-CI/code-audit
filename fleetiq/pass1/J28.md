### J28-F01: SQL Injection â€” `custCd` and `locCd` concatenated without sanitisation in `getEmailList`
**Description:** In `ChartsExcelDao.getEmailList()`, the `custCd` parameter is concatenated directly into a SQL query without quotes and without a prepared statement. An attacker able to influence the `custCd` value (e.g., via a request parameter that flows through `ChartDashboardUtil.cust_cd`) can inject arbitrary SQL. `locCd` is also concatenated inside single quotes, which is insufficient protection. The entire query uses `Statement.executeQuery()` not `PreparedStatement`.
**Severity:** HIGH
**Checklist item:** Section 3 â€” Injection Vulnerabilities (SQL injection)
**Evidence:**

### J28-F07: Google Charts API URL Sent to External Server â€” Potential Data Leakage
**Description:** Both `ImpactChart.createChart()` (line 112) and `MachineUnlockChart.createChart()` (line 112) call `chart.toURLString()`, which produces a URL for the Google Charts API (charts.googleapis.com). This URL encodes the chart data as query parameters. In the context of this application, chart data includes aggregated counts of vehicle impacts and machine unlock events per department. When the URL is used to embed or fetch a chart image, fleet operational data is transmitted to Google's servers. Depending on contractual obligations and applicable privacy regulations, this may constitute unauthorised disclosure of customer fleet data to a third party. Additionally, if the chart URL is logged, fleet metrics appear in logs in plain text.
**Severity:** MEDIUM
**Checklist item:** Section 5 â€” Data Exposure and Privacy
**Evidence:**

## Checklist Items With No Issues

The following checklist items were reviewed against the three assigned files and their direct context. No issues were found in these areas:

- **Section 1 â€” Secrets and Credentials:** No hardcoded credentials, API keys, JDBC connection strings with passwords, or commented-out credentials found in `ImpactChart.java`, `MachineUnlockChart.java`, or `PreopFailBean.java`. Database connections are obtained via `DBUtil.getConnection()` (a pooled utility); credentials are not visible in the files reviewed.

- **Section 3 â€” XSS:** `ImpactChart.createChart()` and `MachineUnlockChart.createChart()` return a URL string. The axis labels embedded in the URL are hardcoded string literals ("Amber", "Red", "Question", "Impact", "Unlocks") â€” no user-controlled or database-sourced text is interpolated into these labels in the chart classes themselves. `PreopFailBean` contains no output logic.

- **Section 3 â€” File Inclusion:** No `JSP include`, `Runtime.exec()`, or `ProcessBuilder` usage in any of the three assigned files.

- **Section 3 â€” Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls found in the three assigned files or in `ChartDashboardUtil.java` or `ChartsExcelDao.java`.

- **Section 3 â€” SSRF:** The Google Charts API call is client-side URL construction only (the URL string is returned to be used as an `<img src>` in a report or embedded in an Excel sheet). The server does not itself make an outbound HTTP request to the Charts API in the code reviewed. The SSRF concern is therefore not applicable here, though the data-leakage concern (F07) stands.

- **Section 4 â€” Session and CSRF:** These classes are pure chart/data-transfer logic with no HTTP request or session handling.

- **Section 6 â€” Transport Security:** Not applicable to these classes; they do not configure or use network connections.

- **Section 7 â€” Dependency Vulnerabilities:** The `charts4j` library is used in `ImpactChart` and `MachineUnlockChart`. The library version cannot be determined from these source files alone; this is noted for the inventory agent. No other third-party dependencies are imported in the three assigned files beyond standard Java collections.
