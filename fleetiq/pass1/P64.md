### P64-F01: Cross-Site Scripting (XSS) - Unescaped User-Controlled Data in All JSP Files
**Description:** All 10 JSP files output potentially user-controlled data from the database and request parameters directly into HTML without any escaping or sanitization. Data such as customer names, driver names, vehicle names, department names, dates, and other values retrieved from the database are rendered using raw JSP expressions (`<%= ... %>`). If an attacker can inject malicious JavaScript into any of these fields (e.g., by creating a vehicle with name `<script>alert(1)</script>`, or a driver with a malicious name), the script will execute when the report is viewed. This is a classic Stored XSS vulnerability affecting all report pages.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

### P64-F02: Cross-Site Scripting (XSS) - Request Parameter Directly Used in Locale Setting
**Description:** In email_curr_unit_report.jsp, the `sess_lang` request parameter is directly used in the `<fmt:setLocale>` tag without validation or sanitization. While JSTL's fmt:setLocale may provide some internal validation, accepting arbitrary locale values from user input can potentially lead to unexpected behavior or XSS if the locale value is later echoed back to the page. Additionally, this parameter is also stored in the session without validation (line 21).

**Severity:** MEDIUM

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**
```jsp
14: <fmt:setLocale value = '<%=request.getParameter("sess_lang") %>'/>
21: request.getSession().setAttribute("lang",request.getParameter("sess_lang"));
```

**File:** reports/email_curr_unit_report.jsp
**Line(s):** 14, 21

### P64-F04: Potential Server-Side Request Forgery (SSRF) via Chart URL
**Description:** In email_driver_util.jsp line 145, a chart URL is retrieved from the filter bean and directly inserted into an `<img src>` tag without validation. If the `chartURL` value can be influenced by user input (either directly or through database manipulation), an attacker could potentially cause the browser to make requests to arbitrary URLs, including internal network resources. This could be used for SSRF attacks or to exfiltrate data.

### P64-F05: Hardcoded Access Level Bypassing Session-Based Authentication
**Description:** All files except email_driver_impact_report.jsp and email_driver_league.jsp contain hardcoded access level controls that set `access_level = "1"` (highest privilege) without retrieving it from the session. This completely bypasses any session-based access control mechanism. While the access level is set on the filter bean, there's no actual enforcement visible in these JSP files. ### P64-F06: Missing Multi-Tenant Data Isolation Verification
