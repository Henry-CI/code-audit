## Findings

### S01-F01: Stored function accepts raw GMTP message and parses by string splitting — no input validation
**Description:** Both stored procedures `sp_store_preop_comments` (FLEETIQ_JAN_2022.sql) and `sp_store_prompt_type` (FLEETIQ_NOV_2022.sql) accept a raw device message string as `$3` and parse it entirely using string operations (`split_part`, `string_to_array`, `position`, `substring`). There is no validation that the message conforms to the expected format, no length check, and no sanitisation. A malformed or adversarially crafted GMTP message could cause unexpected behaviour: null values silently inserted, wrong driver or vehicle associations, or data corruption. While stored procedures execute server-side and are not directly SQL-injectable through the function interface, the raw-message parsing pattern is fragile and the absence of any input validation is a design defect with security implications if the GMTP channel is accessible to untrusted devices.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — SQL injection / input validation
**Evidence:**
```sql
-- FLEETIQ_JAN_2022.sql, lines 55–61
SELECT INTO opcmntLeft split_part(mess, 'OPCMNT=', 1);
utcTime = trim(split_part(opcmntLeft, ',', 2));
driverId = split_part(split_part (opcmntLeft, ',', 1), '=', 2);
SELECT INTO opcmntRight split_part(mess, 'OPCMNT=', 2);
```
```sql
-- FLEETIQ_NOV_2022.sql, lines 94–115
select string_to_array($3, ' ') into params;
authmsg := replace(params[1] , 'AUTH=', '');
SELECT INTO cardID substring(authmsg, 1, position(',' in authmsg)-1);
SELECT INTO utcTime substring(authmsg, position(',' in authmsg)+1, length(authmsg));
...
SELECT INTO mess substring($3, position('PROMPT=' in $3), length($3));
data_lst := string_to_array(mess, ',');
dc2 := data_lst[2];
if(dc2 IS NULL) then
    q_type := 0;
else
    q_type := dc2::integer;   -- cast of unsanitised string to integer; will raise exception on non-numeric input
end if;
```

### S01-F04: ctgry_type_mst stores a file_path column — potential path traversal risk
**Description:** The `ctgry_type_mst` table (FLEETIQ_MAY_2022.sql) has a `file_path character varying(255)` column. Storing file system paths in a database table is a pattern that frequently enables path traversal vulnerabilities if application code later uses the stored path to read, serve, or include a file without validating that the resolved path is within an allowed directory. The SQL alone does not confirm exploitation, but the schema design is a risk indicator that must be reviewed in the application code that reads this column.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — File inclusion
**Evidence:**

### S01-F06: Driver lookup uses ILIKE (case-insensitive) on CARD_ID — potential authentication bypass via case manipulation
**Description:** In `sp_store_prompt_type` (FLEETIQ_NOV_2022.sql line 100), the driver lookup uses `CARD_ID ilike cardID`. ILIKE is case-insensitive matching. If CARD_ID values are intended to be exact identifiers (wiegand card IDs, numeric or hex strings), using ILIKE instead of `=` or `LIKE` means a manipulated card ID string that differs only in case would match a legitimate record. In normal operation wiegand IDs are numeric and case is irrelevant, but the use of ILIKE on an identity credential field is a non-standard pattern worth flagging, particularly because it also does not anchor the match (no `^` equivalent) when used on a varchar field.
**Severity:** LOW
**Checklist item:** 2. Authentication and Authorization
**Evidence:**
```sql
-- FLEETIQ_NOV_2022.sql, line 100
SELECT INTO drivercd "USER_CD" from "FMS_USR_MST" where "CARD_ID" ilike cardID and "USER_CD" in (select "USER_CD" from "FMS_USER_DEPT_REL" where "CUST_CD" = c_cd and "LOC_CD" = s_cd);
```

### S01-F09: No GRANT/REVOKE statements in any migration script — database permission baseline is not managed through migrations
**Description:** None of the eight SQL migration files contains any GRANT or REVOKE statement controlling which database roles have access to the new tables, sequences, or functions introduced. All new objects are owned by the `gmtp` role (via `OWNER TO gmtp`), and access rights to application roles or read-only roles are never defined. This means that any role with broad schema-level privileges can read operational check comments, driver card IDs, vehicle associations, PII contact fields, and salesperson data. The absence of least-privilege grants in migrations is a systemic deficiency.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization
