## Findings

### J29-F01: Mislabelled Chart Plot — "Unlocks" Label in Pre-op Fail Chart
**Description:** In `PreopFailChart.createChart()`, the bar chart plot is hardcoded with the label `"Unlocks"` (line 78), but this class is named `PreopFailChart` and its report title is set to `"Incorrect Pre-op Checks"` (line 43 of constructor). This is a copy-paste error from what is likely `UnlockBean`-related code. While this is not a security vulnerability, it indicates this code was copy-pasted from another chart class without thorough review, which is a code-quality signal that copy-paste patterns may have introduced similar errors elsewhere in the codebase — including potentially in security-sensitive logic such as customer ID scoping.
**Severity:** INFO
**Checklist item:** Section 2 — Authorization and multi-tenancy (code quality indicator relevant to audit confidence)
**Evidence:**
```java
// Line 76-79
BarChartPlot plot = Plots.newBarChartPlot(
        DataUtil.scaleWithinRange(0, this.yAxis, datalist),
        this.colors.get(i), "Unlocks");   // label should not say "Unlocks" in a PreopFail chart
plots.add(plot);
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/chart/excel/PreopFailChart.java`
**Line(s):** 78

### J29-F02: `PreopFailChart.createChart()` Returns an External Google Chart API URL — Potential Data Exfiltration via Third-Party Service
**Description:** `createChart()` constructs a URL via `chart.toURLString()` at line 112 using the `charts4j` library, which generates Google Chart API (`chart.apis.google.com`) URLs. The chart data — including labels derived from fleet data such as `yAxisLabel` entries — is encoded into the URL and sent to Google's servers when rendered in a browser (the URL is the `src` attribute of an `<img>` tag in the JSP layer). This means pre-op failure data (vehicle/unit inspection results) is transmitted to Google as part of every chart render. In a multi-tenant fleet management context, this constitutes a data exposure risk: customer operational data leaves the application boundary and is logged by a third-party service. This is an architectural-level finding applicable wherever `charts4j`/Google Chart API URLs are rendered.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// Line 112
String url = chart.toURLString();
// ...
return url;
```
The returned URL is of the form `https://chart.apis.google.com/chart?cht=bvg&chs=650x350&chd=...&chl=...` with chart data encoded in query parameters.
**File:** `WEB-INF/src/com/torrent/surat/fms6/chart/excel/PreopFailChart.java`
**Line(s):** 112–114

### J29-F03: No Customer/Tenant Scoping in Data Transfer Objects
**Description:** `UnitUtilBean` (fields: `deptName`, `utilNo`) and `UnlockBean` (fields: `deptName`, `reason`) are DTOs used to carry chart data. Neither bean contains a `customerId`, `tenantId`, or any multi-tenancy identifier. The `deptName` field in both beans is the only organisational discriminator. If the code that populates these beans (not in scope for this agent) does not enforce customer-scoped SQL queries, data from one tenant's departments could be returned to another tenant. This finding flags an architectural absence: the DTO design itself provides no enforcement layer for multi-tenancy. The risk must be confirmed by reviewing the data-population layer (the relevant servlet or DAO code).
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy (IDOR / multi-tenant data isolation)
**Evidence:**
```java
// UnitUtilBean.java — fields at lines 5-6
private String deptName;
private double[] utilNo;
// No customerId, tenantId, or organisationId field present.

// UnlockBean.java — fields at lines 5-6
private String deptName;
private String reason;
// No customerId, tenantId, or organisationId field present.
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/chart/excel/UnitUtilBean.java` (lines 5–6); `WEB-INF/src/com/torrent/surat/fms6/chart/excel/UnlockBean.java` (lines 5–6)

## Checklist Items With No Issues

The following checklist sections were reviewed against these three files and no issues were found in the code contained within these specific files:

1. **Section 1 — Secrets and Credentials:** No hardcoded credentials, passwords, API keys, JDBC connection strings, or secrets in any of the three files.
2. **Section 2 — Authentication and Authorization (authentication):** No session access, no `HttpServletRequest`, no login/logout logic in any file. Authentication is not bypassed here.
3. **Section 3 — SQL Injection:** No JDBC calls, no `Statement`, no `PreparedStatement`, no SQL strings in any of the three files.
4. **Section 3 — XSS (in these files):** No JSP expressions, no direct HTML output, no `PrintWriter` output. XSS risk from `UnlockBean.reason` or `UnitUtilBean.deptName` exists only at the rendering layer (not in scope for these files).
5. **Section 3 — File Inclusion:** No `jsp:include`, no file I/O, no `Runtime.exec()` or `ProcessBuilder` in any file.
6. **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage in any file.
7. **Section 3 — SSRF:** `PreopFailChart` uses `charts4j` which generates external URLs, but the URL construction itself is not driven by user-supplied URL input. The data exfiltration concern is captured under J29-F02. No user-controlled SSRF vector exists within these files.
8. **Section 4 — Session and CSRF:** No session manipulation, no form rendering, no CSRF token logic in any file.
9. **Section 6 — Transport Security:** No Tomcat connector configuration, no HTTP client URL construction from user input in these files.
10. **Section 7 — Dependency Vulnerabilities:** `charts4j` is referenced via import in `PreopFailChart.java`. Version cannot be determined from source alone — JAR version must be checked in the repository inventory. No other third-party library calls beyond `charts4j` and the internal `com.torrent.surat.fms6.chart.Chart` parent class.
