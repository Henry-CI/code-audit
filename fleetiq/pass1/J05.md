## Findings

### J05-F01: Package-private fields on serializable bean expose internal state
**Description:** `EntityBean` implements `java.io.Serializable` but all six of its data fields (`id`, `name`, `totalno`, `attribute`, `locs`, `depts`) are declared with package-private (default) access rather than `private`. This means any class within the same package (`com.torrent.surat.fms6.bean`) can read and write these fields directly, bypassing the setter methods entirely. In a legacy JSP application, scriptlet code in JSP pages can access these fields without going through the public API, making it impossible to apply any future validation logic in the setters. The risk is compounded by `Serializable`: if this object is deserialised from an untrusted source (e.g., session replication, HTTP session stored in a database, or an object cache), the fields can be set without any setter validation.
**Severity:** LOW
**Checklist item:** 2. Authentication and Authorization — session management; 5. Data Exposure and Privacy
**Evidence:**
```java

String id = "";
String name = "";
double totalno = 0;

String depts = "";
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/EntityBean.java`

### J05-F02: Package-private fields on EquipmentBypassAlert expose vehicle alert state
**Description:** All five fields of `EquipmentBypassAlert` (`id`, `vehicle`, `triggerTime`, `duration`, `impactCount`) are declared with package-private (default) access rather than `private`. This bean represents equipment bypass alert records, which are security-relevant: `impactCount` tracks safety impact events and `vehicle` ties the record to a fleet asset. Any class within the `com.torrent.surat.fms6.bean` package can mutate these fields directly without using the setter methods. If a JSP scriptlet or another bean in the same package assigns values to these fields outside the setter, any future validation added to the setters will be silently bypassed.
**Severity:** LOW

### J05-F03: DriverLeagueBean carries driver PII with no customer/tenant scoping field
**Description:** `DriverLeagueBean` holds `driverName` and associated performance metrics (`redImpact`, `preOp`, `keyHour`, `tracHours`, `percActive`, `department`). It contains no `customerId` or `tenantId` field. On the `multi-customer-sync-to-master` branch the application is multi-tenant. A bean that carries driver data without a customer identifier cannot enforce tenant isolation at the object level. If a query populates a list of these beans, there is no in-bean mechanism to verify at the consumer (JSP, servlet) that the driver records being displayed belong to the authenticated user's customer. The absence of a tenant field shifts the entire isolation burden to the SQL query, with no defensive layer at the bean or presentation tier.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization — multi-tenancy; 5. Data Exposure and Privacy — driver data
**Evidence:**

### J05-F04: EntityBean carries entity data with no customer/tenant scoping field
**Description:** `EntityBean` contains `id`, `name`, `totalno`, `attribute`, `locs` (locations), and `depts` (departments). The fields `locs` and `depts` suggest this bean represents organisational entities (locations and departments within a customer account). On a multi-tenant branch this bean contains no `customerId` or `tenantId` field. The `id` field is a `String` with no typing or format constraint, meaning a consumer could potentially pass an arbitrary ID originating from user input and the bean provides no mechanism to verify it belongs to the authenticated tenant. This finding is structural: like J05-F03, it shifts the entire multi-tenancy enforcement responsibility to SQL queries or servlets, with no safety net at the bean level.

### J05-F05: EquipmentBypassAlert carries vehicle safety data with no customer/tenant scoping field
**Description:** `EquipmentBypassAlert` records equipment bypass events including `vehicle` identifier, `triggerTime`, `duration`, and `impactCount`. This is safety-critical and customer-sensitive data. The bean contains no `customerId` or `tenantId` field. In a multi-tenant deployment, a list of these beans populated from a query that is missing a `WHERE customer_id = ?` clause would silently include bypass alerts from all customers, with no way for consuming code to detect the leakage from the bean's own structure. The `id` field (int) is an auto-increment database surrogate, meaning IDOR is possible at the query layer if the caller does not scope by customer.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization — multi-tenancy / IDOR; 5. Data Exposure and Privacy
