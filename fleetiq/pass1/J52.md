### J52-F06: Missing Authentication Guard on Excel Export JSPs
**Description:** The JSPs `au_excel/xlsx_restricted_access_usage_report.jsp`, `au_excel/xlsx_seat_hour_util_report.jsp`, and `au_excel/xlsx_serv_maintenance_report.jsp` read session attributes (`access_level`, `access_cust`, etc.) but perform no null-check or redirect guard to verify that a valid authenticated session exists before executing. If a session has no `access_level` attribute (unauthenticated or expired session), the code sets it to the string `"5"` and continues. Additionally, the `/au_excel/` directory is not covered by any `<security-constraint>` in `web.xml` — only `/pages/*` has a transport guarantee constraint, and there is no `<auth-constraint>` with a `<role-name>` on any URL pattern in the reviewed web.xml excerpt. ### J52-F10: Formula Injection in Excel Cell Formulas — Data from Bean Written into setCellFormula()
**Description:** `ExcelSeatHourUtilReport.createUtilReport()` writes bean values directly into Excel cell formula strings using `setCellFormula("VALUE(\"" + vutil1.get(i).toString() + "\")")`. If any `vutil` value in the bean contains characters that alter the Excel formula (e.g. `"`, `)`, `+`, `-`, `@`, `=`), this can result in an Excel formula injection vulnerability. When a victim opens the exported file, a crafted formula could execute in Excel (e.g. shell commands via DDE on older Excel versions, or HYPERLINK/WEBSERVICE calls in modern ones). The values in the bean originate from database queries that are themselves injectable (see F01–F05), amplifying the risk.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities (command injection / data injection via export)

### J52-F11: Exception Stack Traces Printed to Server Log / Potentially Exposed

## Checklist Items With No Issues

- **Section 1 — Secrets and Credentials:** No hardcoded database credentials, API keys, passwords, or connection strings found in the three primary files or in CustomerDAO. Connection is obtained via `DBUtil.getConnection()` (connection pool abstraction). No `.env` or `secrets.properties` files referenced in these files.

- **Section 3 — File Inclusion:** No `<jsp:include>` or `<%@ include %>` with user-controlled paths in the reviewed JSPs. All includes (if any) use static paths. The report classes themselves do not perform file inclusion.

- **Section 3 — Command Injection:** No calls to `Runtime.getRuntime().exec()` or `ProcessBuilder` found in any of the three primary files or CustomerDAO.

- **Section 3 — SSRF:** The primary Java report files do not make HTTP client calls. `Frm_excel` (parent) has `addImage2` and `addImageUtilisationChart` methods that open a URL via `new URL(image).openStream()`, but these are not called from the three files under review and the `image` parameter in context comes from server-side configuration, not user input visible in these files.

- **Section 3 — XSS:** These are server-side Excel generation classes (not JSP output). No HTML output is generated; output is written to `.xlsx` files. XSS via JSP expression is not applicable to these specific classes.

- **Section 6 — Transport Security:** Not determinable from these files alone; assessed at the web.xml/server.xml level. The `/au_excel/` directory's absence from the `CONFIDENTIAL` transport constraint is noted in F06 above.

- **Section 7 — Dependency Vulnerabilities:** No dependency configuration is present in these files. Apache POI is used for Excel generation; specific version not determinable from source alone — requires JAR manifest inspection (out of scope for these files).

- **Section 4 — CSRF:** These JSPs generate file downloads (GET-initiated report exports), not state-changing operations. CSRF for read-only data export is lower risk, though the authentication gap in F06 is the more significant concern for these endpoints.
