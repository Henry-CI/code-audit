## Findings

### P75-F01: Cross-Site Scripting (XSS) - Unescaped Output in All Report Files
**Description:** All 10 JSP files output database values, user-supplied parameters, and session variables using unescaped `<%= ... %>` expressions. These values are rendered directly into HTML without proper encoding, allowing attackers to inject malicious JavaScript. In a multi-tenant fleet management system, one customer could inject scripts that execute in another customer's browser session, potentially stealing session tokens, performing actions on behalf of victims, or exfiltrating sensitive fleet data.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**
Examples across all files:
```jsp
// print_driver_licence_expiry_detail.jsp, line 126
<td class="textgreen"><%=driver_nm.get(i)%></td>

// print_driver_util.jsp, line 133
<td><b><%=get_user %></b> </td>

// print_equipment_bypass_alert_report.jsp, line 78
<td class="textgreen"><%=equipmentBypassAlert.getVehicle() %></td>

// print_hour_counter.jsp, line 107
<td class="textgreen"><b><%=a_veh_nm.get(i)%></b></td>

// print_impact_meter_report.jsp, lines 172, 236-237
<td style="<%=subborderstyle%>"><%=tmp %></td>
<fmt:message key="global.latitude"/> <%=lat %> <br/>
<fmt:message key="global.longitude"/> <%=lon %>

// print_inspection_report.jsp, line 299
<td><%=notesCommentss.get(i) %>}</td>

// print_key_hour_util.jsp, line 127
<td class="textgreen"><b><%=veh_nm.get(i) %></b></td>
```

All database-derived values (driver names, vehicle names, customer names, location names, department names, notes/comments, GPS coordinates, etc.) and request parameters are output without HTML entity encoding. The correct approach is to use JSTL `<c:out>` or `${fn:escapeXml()}`.

**File:** All 10 files
**Line(s):** Multiple instances in every file

### P75-F02: Cross-Site Scripting (XSS) - Unescaped Request Parameters in Hidden Inputs and URLs
**Description:** Request parameters are echoed directly into HTML attributes without escaping, creating XSS vulnerabilities through attribute injection.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**
```jsp
// print_driver_licence_expiry_detail.jsp, line 133
<input type="hidden" value="<%=type%>" name="type"/>

// print_hourmeter_exception.jsp, line 125
<input type="hidden" name="form_cd" value="<%=form_cd %>"/>

// print_inspection_report.jsp, line 121
<li><a href="../pages/admin.jsp?mnm=Reports&sub=<%=sub%>">
```

The `type`, `form_cd`, and `sub` parameters are request parameters that are output without escaping. An attacker could inject JavaScript through these parameters: `type="><script>alert(document.cookie)</script>`

**File:**
- reports/print_driver_licence_expiry_detail.jsp
- reports/print_hourmeter_exception.jsp
- reports/print_inspection_report.jsp

**Line(s):** 133, 125, 121 respectively

### P75-F03: Path Traversal in Image File Inclusion
**Description:** Image file paths are constructed by concatenating a configuration directory with database values without validation, allowing potential path traversal attacks. An attacker who can control the `img_link` database value could access files outside the intended impact image directory.

**Severity:** MEDIUM

**Checklist item:** 3. Injection Vulnerabilities - File inclusion

**Evidence:**
```jsp
// print_impact_meter_report.jsp, line 222
tmp = "../"+ RuntimeConf.impactDir+"/"+img_link.get(j);
// ...
<img src="<%=tmp%>" width=100% height=90% />

// print_impact_report.jsp, line 215
tmp = "../"+ RuntimeConf.impactDir+"/"+img_link.get(j);
// ...
<img src="<%=tmp%>" width=100% height=90% />
```

If an attacker can modify the database to set `img_link` to a value like `../../../../etc/passwd` or `..\..\..\web.xml`, they could attempt to access sensitive files. The path is constructed without validation and output without escaping.

**File:**
- reports/print_impact_meter_report.jsp
- reports/print_impact_report.jsp

**Line(s):** 222-228, 215-220 respectively

### P75-F04: Insecure Transport - HTTP URLs for External Resources
**Description:** Multiple JSP files reference external resources (images, CSS files) using HTTP instead of HTTPS. In a fleet management system that handles GPS data, driver information, and business-critical vehicle data, all resources must be loaded over encrypted connections to prevent man-in-the-middle attacks and mixed content warnings.

**Severity:** MEDIUM

**Checklist item:** 6. Transport Security

**Evidence:**
```jsp
// print_driver_licence_expiry_detail.jsp, lines 49, 51, 67, 99
<img src="//fms.fleetiq360.com//images/linde/logo_rpt.png" width="200" height="90" />
<img src="//fms.fleetiq360.com//images/linde/header.png" width="947" height="90" />
<img src="http://fms.fleetiq360.com//images/linde/left-round.jpg" width="30" height="262" />
<img src="http://fms.fleetiq360.com//images/linde/right-round.jpg" width="30" height="262" />

// print_driver_util.jsp, lines 8-9
<link href="http://fms.fleetiq360.com/css/alternate_rows.css" rel="stylesheet" type="text/css">
<link href="http://fms.fleetiq360.com/css/reportTables1.css" rel="stylesheet" type="text/css">

// print_inspection_report.jsp, lines 209, 223, 250
<img src="http://fms.fleetiq360.com//images/linde/header.png" width="947" height="90" />

```

Protocol-relative URLs (`//fms.fleetiq360.com`) will use HTTP if the page is served over HTTP. Explicit `http://` URLs will never upgrade to HTTPS. Both should be changed to `https://`.

**File:**
- reports/print_driver_licence_expiry_detail.jsp
- reports/print_driver_util.jsp
- reports/print_inspection_report.jsp

**Line(s):** Multiple instances as listed above

### P75-F05: Potential Multi-Tenancy Data Isolation Issue - Access Control Implementation
**Description:** All report files implement access level control using session attributes (access_level, access_cust, access_site, access_dept) passed to a filter bean. However, the JSP files do not verify that returned data belongs to the authenticated customer. The branch name `multi-customer-sync-to-master` indicates this is a multi-tenant system. If the `Databean_report` filter bean does not properly enforce customer scoping in its database queries, this could lead to cross-customer data leakage (IDOR vulnerability).

**Severity:** HIGH (if filter bean is not properly scoped) / INFO (if verification needed)

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
All files follow this pattern:
```jsp
// Example from print_equipment_bypass_alert_report.jsp, lines 24-38
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");

if(access_level.equalsIgnoreCase(""))
{
    access_level = "5";
}
int al = Integer.parseInt(access_level);

filter.setAccess_level(access_level);
filter.setAccess_cust(access_cust);
filter.setAccess_site(access_site);
filter.setAccess_dept(access_dept);
```

The JSP files retrieve session attributes and pass them to the filter bean, then trust the filter bean to return correctly scoped data. There is no verification in the JSP that returned data matches the authenticated customer's scope. Additionally, if `access_level` is empty, it defaults to "5" (the least privileged level), which is safe, but the lack of null checking on other access attributes could cause issues.

**Recommendation:** Verify that `com.torrent.surat.fms6.reports.Databean_report` implements proper customer scoping in all SQL queries. All queries should include `WHERE customer_id = ?` predicates derived from `access_cust` session attribute, not from request parameters.

**File:** All 10 files
**Line(s):** Access control blocks in each file (lines vary)

### P75-F08: Potentially Dangerous Session Attribute Substring Operation
**Description:** In the inspection report, session attributes are processed using substring operations without length validation, which could cause IndexOutOfBoundsException if the session attributes are malformed or have unexpected values.

**Severity:** LOW

**Checklist item:** 5. Data Exposure and Privacy - Error responses

**Evidence:**
```jsp
// print_inspection_report.jsp, lines 63-65
String acc_cus = access_cust.substring(2, access_cust.length());
String acc_sit = access_site.substring(2, access_site.length());
String acc_dep = access_dept.substring(2, access_dept.length());
```

These substring operations assume session attributes are at least 2 characters long. If `access_cust` is null, empty, or a single character, this will throw an exception. The extracted values (`acc_cus`, `acc_sit`, `acc_dep`) are assigned but never used in the subsequent code, suggesting this may be dead code.

**File:** reports/print_inspection_report.jsp
**Line(s):** 63-65
