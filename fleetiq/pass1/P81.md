### P81-F01: Cross-Site Scripting (XSS) - Unescaped Output in JavaScript Context
**Description:** Multiple JSP files output user-controlled or database-derived values directly into JavaScript contexts without escaping. This allows attackers to inject malicious JavaScript code that will execute in victims' browsers. In a multi-tenant system, this enables cross-customer attacks where one customer can inject code that steals data from other customers or performs unauthorized actions.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**

### P81-F05: Insecure Direct Object Reference (IDOR) - Missing Multi-Tenant Authorization
**Description:** All report JSP files retrieve session-based access control attributes (access_level, access_cust, access_site, access_dept) and pass them to backend beans, but there is no visible validation that the user_cd, loc_cd, dept_cd, or other resource identifiers in the request parameters actually belong to the authenticated user's customer/tenant. The access control appears to rely on session attributes being set correctly, but request parameters are user-controllable. An attacker could manipulate parameters like user_cd, loc_cd, dept_cd to access other customers' data if the backend doesn't properly validate ownership.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
```jsp
// Example from rpt_hour_counter.jsp:9-16, 22-36
String user_cd = request.getParameter("user_cd")==null?"":request.getParameter("user_cd");
String loc_cd = request.getParameter("loc_cd")==null?"":request.getParameter("loc_cd");
String dept_cd = request.getParameter("dept_cd")==null?"":request.getParameter("dept_cd");

String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");

filter.setAccess_level(access_level);
filter.setAccess_cust(access_cust);
filter.setAccess_site(access_site);
filter.setAccess_dept(access_dept);

filter.setSet_cust_cd(user_cd);  // User-controlled parameter
filter.setSet_loc_cd(loc_cd);    // User-controlled parameter
filter.setSet_dept_cd(dept_cd);  // User-controlled parameter
```

This pattern repeats in all files. While session attributes are set, the user_cd, loc_cd, and dept_cd from request parameters are also passed to the bean. Without seeing the backend bean implementation, it's unclear if these user-controlled parameters could override or bypass the session-based access control, potentially enabling cross-tenant data access.

**File:** All 10 JSP files reviewed
**Line(s):** Multiple instances in initialization scriptlets

### P81-F06: Session Fixation Risk - No Session Regeneration Evidence
**Description:** All JSP files rely on session attributes for access control (access_level, access_cust, access_site, access_dept) but there is no evidence in these files of session regeneration after authentication. While the actual login logic is not visible in these report files, this indicates a potential session fixation vulnerability where an attacker could set a known session ID and trick a user into authenticating with it.

**Severity:** MEDIUM

**Checklist item:** 2. Authentication and Authorization - Session management

**Evidence:**
```jsp
// Pattern repeated in all files:
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");
```

No session.invalidate() or session creation with new ID is observed in any report file. This is expected in report pages, but indicates the login flow must properly regenerate sessions (not visible in these files).

**File:** All 10 JSP files reviewed (indicates authentication flow concern)
