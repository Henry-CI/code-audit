### P88-F01: Static (Class-Level) Fields Used for Per-Request Session State — Thread Safety / Tenant Data Leakage
**Description:** In `Expire.jsp` (lines 15–18) and `Expire2.jsp` (lines 15–18), the session-specific values `dealerId`, `isSuperAdmin`, `adminAccessLvl`, and `date_only_format` are written to **static fields** of `SessionVar`. JSP scriptlets execute on shared servlet instances in a multi-threaded Tomcat environment. Writing per-user session data into static (shared) fields means one user's request can overwrite the values set by another user's concurrent request. In a multi-tenant fleet management application this is a data isolation vulnerability: the `dealerId` and access-level values used for subsequent tenant scoping checks could be those of a different user's session. `_expire.jsp` avoids this by using local variables, confirming the developers are aware of the pattern but have not corrected the other two files.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / customer ID scoping)
**Evidence:**
```jsp
// Expire.jsp lines 15–18
SessionVar.dealerId = session.getAttribute("dealer_id") == null ? 0:Integer.parseInt(session.getAttribute("dealer_id").toString());
SessionVar.isSuperAdmin = session.getAttribute("admin_access_level") == null ? false:session.getAttribute("admin_access_level").toString().trim().equals("0") ? true:false;
SessionVar.adminAccessLvl = session.getAttribute("admin_access_level") == null ? 99 : Integer.parseInt(session.getAttribute("admin_access_level").toString());
SessionVar.date_only_format =  session.getAttribute("admin_access_level") == null ? "dd/MM/yyyy" :session.getAttribute("date_format").toString();

// Expire2.jsp lines 15–18 (identical pattern)
SessionVar.dealerId = session.getAttribute("dealer_id") == null ? 0:Integer.parseInt(session.getAttribute("dealer_id").toString());
SessionVar.isSuperAdmin = session.getAttribute("admin_access_level") == null ? false:session.getAttribute("admin_access_level").toString().trim().equals("0") ? true:false;
SessionVar.adminAccessLvl = session.getAttribute("admin_access_level") == null ? 99 : Integer.parseInt(session.getAttribute("admin_access_level").toString());
SessionVar.date_only_format =  session.getAttribute("admin_access_level") == null ? "dd/MM/yyyy" :session.getAttribute("date_format").toString();

// _expire.jsp lines 15–17 — correctly uses local variables
int current_dealer = DataUtil.toInt(request.getSession(), "dealer_id");
int adminAccessLevel = DataUtil.toInt(request.getSession(), "admin_access_level");
boolean isSuperAdmin = adminAccessLevel == 0 ? true : false;
```
**File:** sess/Expire.jsp, sess/Expire2.jsp

### P88-F02: NullPointerException Risk on `access_level` Session Attribute — Authentication Bypass Potential
**Description:** In `Expire2.jsp` line 95, `session.getAttribute("access_level")` is cast directly to `String` and passed to `Integer.parseInt()` with no null guard. If the `access_level` session attribute is absent (e.g., corrupted session, partially initialised session after a failover, or during certain error states), this line throws a `NullPointerException`. Depending on Tomcat's error handling configuration, an unhandled NPE in a JSP scriptlet may result in a 500 error page rather than a redirect to login, potentially allowing continued request processing past the access check, or exposing stack trace information. `Expire.jsp` line 92 and `_expire.jsp` line 93 both guard this with a null check (`== null ? 5 : ...`), making Expire2.jsp inconsistent and unsafe by comparison.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (session management); Section 5 — Data Exposure and Privacy (error responses / stack traces)
**Evidence:**
```jsp
// Expire2.jsp line 95 — NO null guard
int access_level = Integer.parseInt((String)session.getAttribute("access_level"));

// Expire.jsp line 92 — null guard present (correct)
int access_level =(String)session.getAttribute("access_level") == null ?  5 : Integer.parseInt((String)session.getAttribute("access_level"));

// _expire.jsp line 93 — null guard present (correct)
int access_level =(String)session.getAttribute("access_level") == null ?  5 : Integer.parseInt((String)session.getAttribute("access_level"));
```
**File:** sess/Expire2.jsp
**Line(s):** 95

### P88-F04: Tenant Scoping Bypass via Empty or Special-Value Customer Codes
**Description:** The access-control logic in all three files grants full access (`custAcc = true`) when the `_cust_cd_` parameter is empty string, `"all"`, or `"none"`. An attacker who is an authenticated user (adminAccessLvl > 1) can omit the customer parameter entirely, or pass `customer=all` or `customer=none`, and the customer access check is bypassed — `custAcc` is set to `true` without verifying the user has access to all customers. Similarly, `deptAcc` is set using `!_dept_cd_.trim().equalsIgnoreCase("all")` in the else branch (lines 163/166/164), which evaluates to `false` only when `_dept_cd_` is literally `"all"` — making the logic hard to reason about and potentially inconsistent. This allows cross-tenant data access when pages relying on these guard files do not enforce their own customer scoping.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (IDOR; multi-tenancy)
**Evidence:**
```jsp
// Expire.jsp lines 129–139 (identical in Expire2.jsp lines 132–142, _expire.jsp lines 130–140)

}else{
    custAcc = true;  // empty, "all", or "none" — no check performed
}

// deptAcc logic (Expire.jsp line 163):
deptAcc = !_dept_cd_.trim().equalsIgnoreCase("all") ;
// When _dept_cd_ is empty (not supplied), equalsIgnoreCase("all") is false, so deptAcc = true — bypass
```

An attacker operating a dealer-level admin account can pass `customer=9999` (or equivalent parameter) to any page guarded by these files to skip the `Databean_cdisp` check and access data without verification. ### P88-F06: Duplicate and Divergent Session Guard Files — Inconsistent Security Posture
**Description:** There are three near-identical session guard files (`Expire.jsp`, `Expire2.jsp`, `_expire.jsp`) performing the same role. They differ in subtle but security-relevant ways: (1) Expire.jsp and Expire2.jsp use static `SessionVar` fields (thread-unsafe) while `_expire.jsp` uses local variables; (2) Expire2.jsp lacks the null guard on `access_level`; (3) the parameter priority order for `_cust_cd_` differs between Expire2.jsp (`user_cd` first) and the other two (`customer` first). Maintaining three variants of security-critical logic guarantees divergence over time and means a security fix applied to one file may not be applied to the others. The existence of three files also raises the question of which pages include which guard, and whether any pages include none of them.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (session management; multi-tenancy)
**Evidence:**
```
Expire.jsp   line 59:  _cust_cd_ = request.getParameter("customer")  [first priority]
_expire.jsp  line 60:  _cust_cd_ = request.getParameter("customer")  [first priority]
Expire2.jsp  line 61:  _cust_cd_ = request.getParameter("user_cd")   [first priority — different]

Expire.jsp   line 92:  null guard present on access_level
Expire2.jsp  line 95:  NO null guard on access_level
_expire.jsp  line 93:  null guard present on access_level
```
**File:** sess/Expire.jsp, sess/Expire2.jsp, sess/_expire.jsp

### P88-F07: Unescaped Session Attribute in `<fmt:setLocale>` Tag
**Description:** All three files output `<%=session.getAttribute("lang") %>` directly as the value of `<fmt:setLocale>`. While `<fmt:setLocale>` is a JSTL tag attribute rather than raw HTML output, the value is taken from the session attribute `lang` with no sanitisation. If the `lang` session attribute can be set or influenced by user-supplied input (e.g., from a login form's language selector), a maliciously crafted value could produce unexpected tag behaviour. More broadly this is a pattern of outputting session-derived data without escaping in a JSP context, consistent with missing XSS discipline across these files.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (XSS)
**Evidence:**
```jsp
// Expire.jsp line 8, Expire2.jsp line 9, _expire.jsp line 8
<fmt:setLocale value = '<%=session.getAttribute("lang") %>'/>
```
**File:** sess/Expire.jsp, sess/Expire2.jsp, sess/_expire.jsp
**Line(s):** Expire.jsp: 8; Expire2.jsp: 9; _expire.jsp: 8
