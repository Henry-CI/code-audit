### P35-F01: Static isSuperAdmin Field Used as Per-User Authorization Gate
**Description:** `SessionVar.isSuperAdmin` is declared as a `public static boolean` in `com.torrent.surat.fms6.util.SessionVar`. Static fields in Java are class-level, shared across all threads in the JVM. In `firmware.jsp` at line 151, the firmware version dropdown is only populated when `SessionVar.isSuperAdmin` is true. Because this is a static field and not a per-session/per-request value, any user login that sets `isSuperAdmin = true` (e.g., a super-admin logging in) will cause the field to remain true for all subsequent requests from all users — including non-admin users — until the JVM is restarted or another request explicitly resets it. This is a broken access control vulnerability: non-admin users may see firmware versions intended only for super-admins.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization
**Evidence:**
```java
// WEB-INF/src/com/torrent/surat/fms6/util/SessionVar.java, line 13:
public static boolean isSuperAdmin = false;

// firmware.jsp, lines 149-155:
for (int x = 0; x < firm_vers.size(); x++)
{
    if( SessionVar.isSuperAdmin ){
    %>
        <option value="<%=firm_vers.get(x) %>"><%=firm_vers.get(x) %></option>
    <%
    }
}
```
**File:** pages/admin/customer/firmware.jsp
**Line(s):** 149–157; SessionVar defined in WEB-INF/src/com/torrent/surat/fms6/util/SessionVar.java line 13

### P35-F04: Unescaped Request Parameter Injected into JavaScript Variable (XSS)

### P35-F05: Unescaped Database Values in JavaScript Template Strings (XSS)
**Description:** In `edit-sites.jsp` (lines 125, 131) and `edit-sites_b.jsp` (line 122), database-sourced values (`st_cd`, `st_nm`, salesperson names and IDs) are injected directly into multi-line JavaScript template strings using `<%= %>`. These template strings are assigned to JavaScript variables (`trTemplate`) and later inserted into the DOM. If any of these DB values contain single quotes, backslashes, or HTML special characters, an attacker who controls data in the database (e.g., a malicious salesperson name) can inject JavaScript into the page for all users who view the affected customer's site list.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```javascript
// edit-sites.jsp, lines 110-137 (representative excerpt):

...

  <option value="0" ></option>
  <% for (SalespersonBean salesperson : filter.getSalespersonList()) { %>
    <option value="<%=salesperson.getId()%>"><%=salesperson.getCompleteName()%></option>
  <% } %>
</select></td>
```
