## Findings

### J69-F01: SQL Statement fields exposed at package-private visibility — potential misuse from same-package classes
**Description:** In `Databean_reports`, the JDBC fields `conn`, `stmt`, `stmt1`, `stmt3`, `rset`, `rset1`, `rset3` are all declared at package-private (default) visibility, not `private`. Any class in the same package (`com.torrent.surat.fms6.reports`) can directly access, modify, or close these live database handles without going through any controlled access path. If any of the sibling classes in this package are reachable from a web request, an attacker who influences how those classes interact could disrupt query execution, read result sets intended for another tenant's query, or interfere with connection state.
**Severity:** MEDIUM
**Checklist item:** 2 — Authorization and multi-tenancy (multi-tenant data isolation); 3 — Injection Vulnerabilities (SQL)
**Evidence:**

### J69-F02: Access control fields exposed at package-private visibility — multi-tenancy bypass risk
**Description:** The access control fields `access_level`, `access_cust`, `access_site`, and `access_dept` that gate tenant-level data scoping are declared package-private. In a multi-tenant application on the `multi-customer-sync-to-master` branch, these are the critical identifiers that should restrict data to the authenticated user's customer. Any class in `com.torrent.surat.fms6.reports` can read or overwrite these values without calling any setter. If a request path through the application allows two threads to share a bean instance, or if another class in the package can replace `access_cust`, tenant isolation is broken.
**Severity:** HIGH
**Checklist item:** 2 — Authorization and multi-tenancy (multi-tenant data isolation; IDOR)

### J69-F05: Raw (unparameterized) ArrayList use — type safety absent, unchecked cast risk
**Description:** `Databean_reports` (lines 44–46), `UtilModelBean` (line 20), and `UtilBean` (implicitly via `UtilModelBean.getUb()`) all use raw `ArrayList` without generic type parameters. ### J69-F07: `Statement` used instead of `PreparedStatement` — SQL injection risk in active field declarations
**Description:** The class declares three `Statement` fields (`stmt`, `stmt1`, `stmt3`) rather than `PreparedStatement`. `Statement` does not support parameterized queries; SQL must be built by string concatenation. The presence of `query` and `queryString` private fields (lines 23–24) further suggests that the pattern in use is string-building followed by `Statement.executeQuery(query)`. Although the active query execution code has been commented out, the field structure of the class is designed exclusively for non-parameterized SQL execution. Any future (or existing, in caller JSPs) use of this class's `stmt` fields will be via string-concatenated SQL.
**Severity:** HIGH
**Checklist item:** 3 — Injection Vulnerabilities — SQL injection (Statement vs PreparedStatement; string concatenation in SQL)
**Evidence:**
```java

Statement  stmt3;   // line 17 — not PreparedStatement
```
