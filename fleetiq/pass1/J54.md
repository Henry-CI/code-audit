### J54-F01: Unescaped user-/database-controlled values injected into HTML email body (Stored XSS / HTML Injection)
**Description:** Both overloads of `MailBase.setMailHeader()` build an HTML email body by directly concatenating string parameters ‚Äî `cust`, `loc`, `dep`, `rptName`, `model`, `st`, `et` ‚Äî into the HTML without any HTML escaping. These parameters originate from database values (customer names, site names, department names, model identifiers) and/or report filter parameters that users may influence. If any of these values contain HTML metacharacters (`<`, `>`, `"`, `&`) ‚Äî for example, a customer name stored as `Acme <Pty> Ltd` ‚Äî the resulting email will contain malformed or injected HTML. A malicious or malformed value could inject arbitrary HTML or script tags into the email body sent to recipients. Since these are outbound emails, the attack surface is the email client of the recipient; if the email client renders HTML, injected `<script>` or `<a href="...">` content would execute.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî XSS (Cross-Site Scripting)

### J54-F03: Outbound email assets loaded over plain HTTP (mixed content / transport security)
**Description:** All three asset URLs embedded in the HTML email (`alternate_rows.css`, `linde_long_logo.gif`, `Logo1.gif`) use `http://` not `https://`. If the application is accessed over HTTPS and the email is opened in a web-based email client served over HTTPS, these HTTP asset loads constitute mixed content. More broadly, loading assets over plain HTTP from a fleet management system email leaks the recipient's IP address and email-open event to a passive network observer. ### J54-F04: Excel formula injection via unvalidated database values in setCellFormula()
**Description:** `ExcelUtilisationReport.createUtilReport()` constructs Excel cell formulas by direct string concatenation with values retrieved from database-backed bean objects (`utilBean.getTotal_hours()`, `getKey_hours()`, `getSeat_hours()`, `getTrack_hours()`, `getHydl_hours()`). The resulting formula string is passed directly to `setCellFormula()`. If any of these hour values contain characters meaningful to Excel formula syntax (e.g., `+`, `-`, `=`, `@`, pipe, or embedded formula fragments), the generated `.xlsx` file will contain injected formulas. When a user opens the exported Excel file, Excel will evaluate those formulas. This is a classic CSV/Excel injection (also called formula injection) vulnerability. The attack vector is indirect: a malicious value stored in the database propagates into the spreadsheet formula at export time.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî SQL injection / formula injection (data export feature)

### J54-F05: No authorisation or customer-scoping check within report generation classes
**Description:** Neither `ExcelUtilWOWReportEmail` nor `ExcelUtilisationReport` perform any check that the data being rendered belongs to the authenticated user's customer/tenant. Both classes accept pre-populated bean objects (`UtilWowReportBean`, `UtilisationReportBean`) and render whatever data those beans contain into the Excel file without verifying customer ownership. This means the security of the report is entirely dependent on whatever servlet or controller populates the bean before passing it to these classes. If the calling layer fails to scope the data query to the authenticated customer ‚Äî or if a parameter is manipulated to request another customer's data ‚Äî these classes will faithfully render the wrong customer's fleet data into the exported file with no safeguard. This is an absence of defence-in-depth for the multi-tenant data isolation requirement specific to this branch.

### J54-F06: `chartUrl` from bean embedded in spreadsheet without validation (potential SSRF / malicious URL injection)
**Description:** In `ExcelUtilWOWReportEmail.createUtilWowReport()`, the value `chartUrl = utilBean.getChartUrl()` is retrieved from the bean at line 54 and passed directly to `addImageUtilisationChart(sheet, chartUrl, 18, true, 0)` at line 97. No validation of `chartUrl` is performed within this class. If `chartUrl` is derived from a user-controlled parameter or a database value that can be influenced by a tenant, an attacker could supply an arbitrary URL. Depending on how `addImageUtilisationChart` (defined in superclass `Frm_excel`) fetches the image ‚Äî if it makes an outbound HTTP request to `chartUrl` ‚Äî this is a Server-Side Request Forgery (SSRF) vulnerability. Even if the image is fetched server-side from a file path rather than a URL, a path traversal attack is possible if `chartUrl` can reference `../../` paths. The risk level depends on the `Frm_excel` implementation, which is out of scope for this file set, but the unvalidated pass-through is a finding.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî SSRF (Server-Side Request Forgery)

### J54-F08: Commented-out sheet protection with null password in ExcelUtilisationReport
**Description:** Line 77 of `ExcelUtilisationReport.java` contains `//sheet.protectSheet(null);`. This is dead commented-out code. Had it been uncommented, passing `null` to `protectSheet()` would enable worksheet protection with no password, which in Apache POI results in a protection that can be trivially bypassed by any user (no password is required to unprotect). The comment indicates an intent to add protection that was abandoned. Exported financial/utilisation reports with no sheet protection allow recipients to silently modify the data and potentially re-share altered reports.
