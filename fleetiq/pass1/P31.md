### P31-F01: Static test pages committed to repository expose full customer enumeration and real vehicle data with no authentication
**Description:** Vtest.jsp and Vttest.jsp are static HTML snapshots of the Manage Vehicles screen committed to the repository. Both files are full HTML pages (not JSP fragments) with no `_expire.jsp` include, no session check, and no scriptlet logic. They render a complete dropdown list of all customers in the production database (approximately 130 real customers with their internal numeric IDs), real vehicle serial numbers, GMTP IDs, SIM CCID numbers, a real username ("Hui Ke"), and vehicle action links with hardcoded vehicle IDs. Because these files are standard `.jsp` files located in the `pages/admin/` directory, the Tomcat container will serve them directly when accessed. If the Tomcat container routes `pages/admin/Vtest.jsp` and `pages/admin/Vttest.jsp` without requiring session authentication (which is the common pattern in this application where auth is enforced by the included `_expire.jsp` fragment rather than container-managed security constraints), these pages will be accessible to unauthenticated users and expose the full customer registry and vehicle inventory data.

### P31-F02: Unescaped output of database-sourced values — XSS in multiple admin pages
**Description:** All eight active JSP files in this batch (Reports, Send_Driver_Limiting_by_Truck, Send_Driver_Settings_to_Unit, Settings, SpecialAdminRights, Users, Vehicles) render database-sourced values directly into HTML using `<%= ... %>` expressions without any escaping. There is no use of `<c:out value="..." />` or `${fn:escapeXml(...)}`. Values rendered include customer names, user first/last names, form names, form descriptions, reskin path strings, department names, vehicle serial numbers, and vehicle type names. If any of these database values contain HTML metacharacters (e.g. from a stored injection attack or corrupt data), the content will be interpreted as HTML by the browser, enabling stored XSS. Additionally, two search box values (Users.jsp line 307 and Vehicles.jsp line 309) reflect the `search` request parameter directly into an `<input value="...">` attribute, enabling reflected XSS.

### P31-F03: Unescaped URL parameters reflected in href attributes — reflected XSS via URL construction
**Description:** Multiple pages reflect raw, unescaped request parameters (`module_cd`, `form_cd`, `customer`, `location`, `department`, `status`, `search`) directly into HTML href attribute values and hidden form field values. An attacker who can craft a link containing HTML metacharacters or a javascript: scheme in these parameters may be able to inject markup into the rendered page. While some values are validated indirectly (e.g. numeric IDs from the dropdown), the `search` parameter and string-typed parameters receive no validation or escaping before being written into the HTML.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (XSS)
