## Findings

### P20-F01: Pervasive unescaped XSS across all 10 files — database and user-input values output raw

**Description:** Every JSP file in this batch outputs database-sourced values (customer names, driver names, vehicle identifiers, question text, alert types, department names) and user-supplied request parameters (`search_crit`, `message`, `cust_cd`, `loc_cd`, etc.) using raw `<%= ... %>` expressions with no HTML escaping. No use of `<c:out>`, `${fn:escapeXml()}`, or any other escaping mechanism is present anywhere in any of these files. An attacker who can control any stored value (via a customer name, driver first/last name, vehicle description, question text, etc.) can inject arbitrary HTML and JavaScript that will execute in all users' browsers when they view the affected pages.

The `message` request parameter is of particular concern: it is echoed back verbatim in every file and is typically set by servlet redirects after form submissions. If an attacker can influence the servlet to return a crafted message value — or can directly craft a URL — this is a reflected XSS vector.

**Severity:** HIGH

**Checklist item:** Section 3 — XSS (Cross-Site Scripting)

**Evidence (representative samples across files):**

### P20-F02: Unescaped database/user values embedded directly in JavaScript event handlers (Stored XSS / JS injection)

**Description:** Throughout these files, database values and URL parameters are embedded directly inside JavaScript `onclick` attributes and `onload` event handler calls using `<%= ... %>` with no escaping. If any embedded value contains a single quote, closing parenthesis, or script-relevant characters, it will break out of the string context and execute arbitrary JavaScript. This is a separate, higher-exploitability XSS vector from the HTML context findings in F01.

**Severity:** HIGH

**Checklist item:** Section 3 — XSS (Cross-Site Scripting) / JavaScript context injection
