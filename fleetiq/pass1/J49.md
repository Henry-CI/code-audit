## Findings

### J49-F01: Hardcoded Production HTTP URL in ExcelImpactMeterReport
**Description:** Line 170 of `ExcelImpactMeterReport.java` contains a hardcoded production hostname (`fleetfocus.lindemh.com.au`) and uses the plaintext `http://` scheme. This bakes a production URL into the compiled binary, meaning any environment (test, staging, UAT) that executes this code will reach out to the production server for impact photo images. Additionally, using HTTP (not HTTPS) means the image URL and any session-related parameters transmitted with the request are exposed in plaintext in transit.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials (hardcoded infrastructure); Section 6 — Transport Security (HTTP instead of HTTPS)
**Evidence:**
```java
// Line 170
String tmp = "http://fleetfocus.lindemh.com.au/fms/"+ RuntimeConf.impactDir+"/"+img_link.get(j);
```
**File:** WEB-INF/src/com/torrent/surat/fms6/excel/reports/ExcelImpactMeterReport.java
**Line(s):** 170

### J49-F02: SSRF via Database-Controlled Image URL in ExcelImpactMeterReport
**Description:** The URL passed to `addImage2()` at line 171 is constructed by concatenating a base URL, `RuntimeConf.impactDir` (a runtime configuration value), and `img_link.get(j)` (a value originating from the `ImpactReportBean`, which is populated from database query results). If `img_link` values are attacker-influenced — for example, by an authenticated user who can insert or manipulate impact records, or through SQL injection in the layer that populates the bean — the server-side fetch triggered by `addImage2()` can be redirected to an arbitrary internal or external host. This is a Server-Side Request Forgery (SSRF) risk. There is no URL validation, whitelisting, or scheme enforcement at this layer.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (SSRF)
**Evidence:**

### J49-F03: Excel Formula Injection via Unsanitised Data Values (All Three Files)
**Description:** All three Excel report classes construct Excel cell formulas by directly concatenating data values retrieved from the report beans into formula strings using `setCellFormula("VALUE(\"" + value + "\")")`. The values `sttm`, `endtm`, and `stop` are written into formulas without any sanitisation or validation that they contain only expected time/duration data. If any of these values contain characters meaningful to the Excel formula parser (e.g., `"`, `)`, or formula prefixes), a malicious or malformed value in the underlying data could result in formula injection. When a recipient opens the exported Excel file, the injected formula executes in their Excel environment. This is a well-documented attack class (CSV/Formula Injection, CWE-1236). Fleet data (vehicle IDs, driver names used nearby in the same rows) originates from operator input and database records — both potentially attacker-influenced in a multi-tenant environment.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities (general injection); Section 5 — Data Exposure and Privacy (data export)

### J49-F04: No Authorization or Customer Scoping in Report Generation Classes
**Description:** All three Excel report classes accept `cust`, `loc`, and `dep` as plain `String` parameters passed in by the caller. None of these classes perform any check that the caller is authorized to generate a report for the supplied customer/location/department. The classes blindly render whatever data is in the bean that was passed to them. Authorization and tenant scoping are entirely the caller's responsibility, with no enforcement at this layer. This is not a defect in isolation — it is a design risk: if any upstream caller fails to validate the customer parameter against the authenticated session, cross-tenant data will be exported. Given this branch is named `multi-customer-sync-to-master`, the multi-tenant risk is heightened. The `cust` value is also embedded verbatim in the Excel report subtitle (e.g., `ExcelDynDriverReport.java` line 114: `{"Customer", cust}`), so whatever customer name is supplied appears in the exported file.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy, IDOR); Section 5 — Data Exposure and Privacy (data export)

### J49-F05: Instantiation of UnitDAO with No Apparent Use (Dead Code / Attack Surface Ambiguity)
**Description:** All three report classes instantiate `UnitDAO unitDAO = new UnitDAO()` but never call any method on the object. In `ExcelDynDriverReport.java` at line 107, `ExcelDynUnitReport.java` at line 107, and `ExcelImpactMeterReport.java` at line 76. This is dead code. The security concern is ambiguity: if `UnitDAO`'s constructor performs a database connection or other side effect (which is common in legacy DAO patterns), that side effect occurs unconditionally on every report generation. Without reading `UnitDAO`, the impact is uncertain, but the pattern is a red flag. ## Checklist Items With No Issues

The following checklist items were reviewed against the three files and no issues were found within the scope of these files:

- **Section 1 — Secrets and Credentials:** No hardcoded database passwords, API keys, SMTP credentials, or encryption keys found in these three files. (The hardcoded hostname in F01 is infrastructure exposure, not a credential.)
- **Section 3 — SQL Injection:** These three files contain no direct JDBC calls, `Statement.executeQuery()`, `PreparedStatement`, or SQL string construction. SQL access is delegated entirely to bean/DAO layers not present in these files.
- **Section 3 — XSS:** These are server-side Excel generation classes with no JSP, HTML output, or `<%= %>` expressions. XSS is not applicable.
- **Section 3 — File Inclusion:** No JSP includes, no `Runtime.getRuntime().exec()`, no `ProcessBuilder` found.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls found in any of the three files.
- **Section 4 — Session and CSRF:** These are pure report-generation utility classes. They do not handle HTTP sessions, form submissions, or CSRF tokens. Session and CSRF controls are the responsibility of the calling servlet layer.
- **Section 6 — Transport Security:** No TLS configuration, connector configuration, or session cookie handling present in these files (the HTTP issue in F01 is a request origination concern, already reported).
- **Section 7 — Dependency Vulnerabilities:** Apache POI is used for Excel generation (imports from `org.apache.poi.ss.usermodel`). The specific POI version cannot be determined from these source files alone — version identification requires inspection of the JAR in `WEB-INF/lib` or the build descriptor. No findings raised at this layer; flagged for inventory review.
