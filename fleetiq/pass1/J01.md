## Findings

### J01-F01: `dealerName` field is public in `Dealer.java`
**Description:** The field `dealerName` in `Dealer` is declared `public` rather than `private`. In a Java bean/entity, fields should be `private` to enforce encapsulation. A public field bypasses any validation or logic that might otherwise be placed in the setter. Any code in the application can write directly to `dealerName` without going through `setDealerName()`, which means input handling or tenant-scoping logic attached to the setter can be bypassed.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (data integrity and access control at the model layer); Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// Dealer.java, line 13
public String dealerName;
```
**File:** `WEB-INF/src/com/ci/entity/Dealer.java`
**Line(s):** 13

### J01-F02: `BatteryBean` fields are package-private (no access modifier)
**Description:** All 21 data-carrying fields in `BatteryBean` are declared without an access modifier, making them package-private. Any class in the `com.torrent.surat.fms6.bean` package can read and write these fields directly, bypassing all getter/setter methods. This includes fields that hold driver names (`bef_dri_nm`, `aft_dri_nm`), vehicle names (`bef_veh_nm`, `aft_veh_nm`), unit codes, and battery identifiers — all of which are PII-adjacent fleet management data. The lack of `private` on these fields is a structural data-exposure weakness; it makes it impossible for auditors or the runtime to assert that access to driver and vehicle data goes through any controlled access path.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J01-F03: `DealerConfigBean` does not implement `Serializable`
**Description:** `DealerConfigBean` is used as a configuration bean (branding, subdomain, timezone, dateFormat, website, phone). It does not implement `java.io.Serializable`. If this bean is placed in the HTTP session (which is common for per-tenant config objects in legacy JSP applications) and Tomcat attempts to persist or replicate the session, serialization will fail with a `NotSerializableException`. This could expose session state handling issues or cause application instability, depending on Tomcat configuration. The sister class `Dealer` does implement `Serializable`. The inconsistency suggests this was overlooked.

This is not directly a security vulnerability in isolation, but session persistence failures in multi-tenant contexts can result in configuration bleed or unexpected fallback to default state — for example, falling back to a default dealer's subdomain, branding, or timezone config.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (session management); Section 4 — Session and CSRF
**Evidence:**
```java
// DealerConfigBean.java, lines 3-4
// No "implements Serializable" declaration
public class DealerConfigBean {
```
Compare to `Dealer.java`, line 5:
```java
public class Dealer implements Serializable{
```
**File:** `WEB-INF/src/com/ci/entity/DealerConfigBean.java`
**Line(s):** 3–4

### J01-F04: `BatteryBean` exposes driver names and vehicle names as plain `String` fields with no tenant-scoping identifier
**Description:** `BatteryBean` carries driver names (`bef_dri_nm`, `aft_dri_nm`), vehicle names (`bef_veh_nm`, `aft_veh_nm`), unit codes, and battery identifiers, but contains no customer ID or tenant ID field. In a multi-tenant application (this branch is named `multi-customer-sync-to-master`), a data bean that represents battery swap events for vehicles and drivers must carry or be validatably linked to a tenant/customer identifier so that the layer that populates the bean can be verified to be scoping queries to the correct customer. The absence of a tenant identifier at the bean level means tenant isolation cannot be enforced or audited at the model layer — it must be trusted entirely to the DAO or query layer, with no defensive check possible at the bean level.

This is a structural IDOR/multi-tenancy risk indicator. It does not confirm a vulnerability in this file alone, but it flags that battery swap data — including driver and vehicle identity — may be fetchable across tenant boundaries if the query layer has any deficiency. The orchestrating session should prioritise auditing the DAO/service layer that populates `BatteryBean` instances to verify customer-ID scoping in the underlying SQL.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / IDOR); Section 5 — Data Exposure and Privacy
**Evidence:**
```java

//   duration, isTruck1120, bef_bat_fleet_num, aft_bat_fleet_num, bat_fleet_num
```

**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/BatteryBean.java`

### J01-F05: `Dealer` entity has no customer/tenant isolation field — `dealerId` relationship is opaque
**Description:** The `Dealer` entity carries `id` and `dealerId` as separate `int` fields with no documentation or naming convention that makes clear which is the primary key and which is the tenant/customer association. In a multi-tenant system, it is critical that the entity model makes tenant ownership unambiguous. The field `dealerId` could be a self-referential FK, a parent dealer ID, or a customer org ID — it is impossible to determine from this file alone. Additionally, there is no `customerId` field, meaning the relationship between a `Dealer` record and a customer tenant is not represented in the entity. If application code resolves dealer-to-customer relationships by joining on `dealerId`, any query bug could allow cross-tenant dealer data access. This should be reviewed alongside the DAO layer.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / IDOR)
**Evidence:**
```java
// Dealer.java, lines 11-12
private int id;
private int dealerId;
```
No Javadoc, no annotation, and no naming convention distinguishes the purpose of `id` vs `dealerId` or documents the tenant ownership relationship.
**File:** `WEB-INF/src/com/ci/entity/Dealer.java`
**Line(s):** 11–12
