## Findings

### J13-F01: All fields in SiteConfigurationBean use package-private access instead of private
**Description:** Every non-static field in `SiteConfigurationBean` is declared without an explicit access modifier, making them package-private. In a JavaBean that exposes public getters and setters, fields should be `private` to enforce encapsulation. Package-private fields can be read and written directly by any class in the same package (`com.torrent.surat.fms6.bean`) without going through the accessor methods, bypassing any future validation or audit logic that might be added to those accessors. This is particularly notable for fields such as `super_card`, `facility_code`, `contact_email`, `contact_no`, and `contact_nm`, which carry contact and hardware-credential data.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (access control on data carriers)
**Evidence:**

### J13-F02: All fields in SpecialAccessBean use package-private access instead of private
**Description:** Every field in `SpecialAccessBean` is package-private. This bean is the most security-sensitive of the three files reviewed: it models cross-customer special access grants, carrying `cust_cd` (customer code), `user_cd` (user code), `loc_cd` (location code), `dept_cd` (department code), and `module_cd` (module/feature code), along with an `enabled` flag. Because these fields are package-private, any class in `com.torrent.surat.fms6.bean` can directly set `cust_cd` or `enabled` without using the setter, circumventing any validation that may be added to the accessor. The multi-tenant nature of the application (branch `multi-customer-sync-to-master`) means a cross-customer access assignment is a high-impact error; package-private fields increase the chance of accidental or deliberately crafted misassignment.
**Severity:** LOW

### J13-F03: SpecialAccessBean does not implement Serializable despite carrying cross-tenant access grant data
**Description:** `SpecialAccessBean` models special access grants that cross customer boundaries (it holds both `cust_cd` and `user_cd`). Unlike `SiteConfigurationBean`, this class does not implement `java.io.Serializable`. If instances of this bean are ever stored in an `HttpSession` (common in JSP/Servlet applications for caching permission lists), Tomcat will fail to serialize the session to disk during shutdown/restart cycles, causing silent session loss. More broadly, the absence of `Serializable` indicates this class was not designed with session persistence in mind, which may mean its presence in sessions was not reviewed for security implications. This is an architectural observation that warrants investigation of how `SpecialAccessBean` instances are used at runtime — specifically whether they are stored in session scope and whether session attributes are ever audited for cross-customer leakage.
**Severity:** INFO
**Checklist item:** Section 2 — Session management; Section 2 — Authorization and multi-tenancy
**Evidence:**
```java
// Line 3: no 'implements Serializable'

```
**File:** `WEB-INF/src/com/torrent/surat/fms6/bean/SpecialAccessBean.java`

### J13-F05: SiteConfigurationBean carries hardware/SIM credential fields with no tenant-scoping visible in the bean
**Description:** `SiteConfigurationBean` contains fields `sim_supplier`, `sim_type`, `facility_code`, and `super_card`. The field `super_card` is particularly notable: the name implies an elevated-privilege physical access card or hardware token value. There is no `customer_id` or `cust_cd` field in this bean to scope it to a specific tenant. In a multi-tenant fleet management system, site configuration objects must be unambiguously tied to a single customer. If the bean is populated from a query that does not enforce customer scoping, a user could retrieve another tenant's SIM configuration, facility code, or super-card value. This cannot be confirmed from the bean class alone — the risk must be traced to the DAO or servlet that populates this bean — but the absence of any customer identifier in the bean's structure is a red flag requiring follow-up.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy; Section 5 — Data Exposure and Privacy
