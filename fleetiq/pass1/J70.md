## Findings

### J70-F01: SQL Injection in Frm_login — Login Query Built by String Concatenation
**Description:** The `doPost` method in `Frm_login` constructs two SQL queries by directly concatenating the user-supplied `login` parameter without any sanitisation or use of `PreparedStatement`. An attacker can inject arbitrary SQL by providing a crafted username such as `' OR '1'='1`. The first query leaks whether the username exists (a count query), and the second retrieves the cleartext password and user code. Because password comparison is done in Java (`password.equals(pass_word)`) rather than in SQL, this attack surface affects authentication bypass, credential extraction, and enumeration.
**Severity:** CRITICAL
**Checklist item:** Section 3 — SQL Injection
**Evidence:**
```java
// Line 58-59
queryString = "select count(*) from \"FMS_USR_MST\" where \"USER_NAME\" = '"
        + login + "'";

// Line 72-73
queryString = "select \"PASSWORD\",\"USER_CD\" from \"FMS_USR_MST\" where \"USER_NAME\" = '"
        + login + "'";
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/security/Frm_login.java`
**Line(s):** 58–59, 72–73

### J70-F03: No Session Fixation Protection on Login
**Description:** On successful authentication, `Frm_login.doPost` sets a session attribute on the **existing** session (`request.getSession().setAttribute(...)`) without first invalidating it and issuing a new session ID. An attacker who can pre-set a session cookie value (e.g., on a shared network or via a pre-login XSS) can fixate a session ID and hijack the authenticated session immediately after the victim logs in.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization (Session Management — session fixation)
**Evidence:**
