## Findings

### J46-F01: No customer/tenant scoping enforced in report generation layer
**Description:** All three Excel report classes accept `cust`, `loc`, and `dep` parameters as plain strings passed directly to the report builder. There is no code in any of these classes that validates or verifies that the `cust` parameter matches the authenticated session's customer. The values are used only for display in the report subtitle header (e.g., `{"Customer", cust}`). If the caller â€” a servlet or JSP upstream â€” passes an attacker-supplied customer name without server-side verification that the customer belongs to the authenticated user, reports can be generated for arbitrary customers. The data itself (machines, users, vehicles, driver names) is pre-loaded into the bean objects (`CimplicityShockReportBean`, `CimplicityUtilReportBean`, `CurrDrivReportBean`) before these classes are invoked, so the isolation risk is at the bean-population layer; however, no guard exists here either, meaning this layer provides no defence in depth for multi-tenant isolation.
**Severity:** HIGH
**Checklist item:** Section 2 â€” Authorization and multi-tenancy (IDOR / multi-tenant data isolation)
**Evidence:**

### J46-F02: Excel formula injection via user/database-controlled data in setCellFormula
**Description:** In `ExcelCimplicityUtilReport.java`, line 112, a value drawn from the `dsum_key_hr` data list is directly concatenated into a `setCellFormula()` call: `contentCell.setCellFormula("VALUE(\""+temp+"\")")`. The variable `temp` is populated from `dsum_key_hr`, which originates from the database via `CimplicityUtilReportBean`. If any database record contains formula-injection characters (e.g., a value starting with `=`, `+`, `-`, `@`, or containing `")`), the resulting `.xlsx` file will contain a crafted formula. When a recipient opens the Excel file, the formula executes in their spreadsheet client. This is a well-documented Excel/CSV injection vulnerability (also called formula injection or DDE injection). The impact is arbitrary formula execution in the recipient's Excel environment, which can include data exfiltration via `HYPERLINK()` or remote DDE calls.
**Severity:** HIGH
**Checklist item:** Section 3 â€” Injection Vulnerabilities
**Evidence:**

### J46-F03: Unclosed FileOutputStream on exception path â€” potential resource leak with predictable output file paths
**Description:** In all three classes, the `createExcel` method opens a `FileOutputStream` and writes the workbook, but does not use a try-with-resources block or a finally clause. If `wb.write(fileOut)` throws an exception, `fileOut.close()` at line 41 (same pattern in all three files) is never called, leaking the file handle. Additionally, the output file path is derived from the `result` field (inherited from `Frm_excel`) which is constructed from `docRoot` and `fileName` parameters supplied by the caller. If `fileName` is user-influenced and not sanitized, a path traversal attack could cause the Excel file to be written to an arbitrary location on the server filesystem (e.g., `fileName = "../../webapps/ROOT/shell.xlsx"`).
**Severity:** MEDIUM
**Checklist item:** Section 3 â€” Injection Vulnerabilities (file inclusion / path traversal); Section 5 â€” Data Exposure

### J46-F05: Driver names and vehicle data written to Excel without output sanitization
**Description:** `ExcelCurrDrivReport.java` writes driver names (`vdriv_nm`), vehicle names (`vveh_nm`), fleet numbers (`veh_fno`), and vehicle type names (`vveh_typ`) directly from ArrayLists populated from the database into Excel cells without any sanitization. Similarly, `ExcelCimplicityShockReport.java` writes user names (`vuser_nm`) and machine names (`vmachine_nm`) directly, and `ExcelCimplicityUtilReport.java` writes driver/vehicle names and types. While Excel cell values (set via `setCellValue`) are generally safe from HTML XSS, they are not safe from Excel formula injection (see J46-F02), and they expose driver PII (names, last reported times) in a downloadable file. There is no redaction, masking, or role-based filtering of PII fields before writing to the export.
**Severity:** LOW
**Checklist item:** Section 5 â€” Data Exposure and Privacy
