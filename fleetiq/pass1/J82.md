## Findings

### J82-F01: Static shared fields used for per-session authentication state — critical multi-tenancy isolation failure
**Description:** `SessionVar` stores the authenticated tenant identifier (`dealerId`), the superadmin flag (`isSuperAdmin`), and the access level (`adminAccessLvl`) as `public static` fields on a plain Java class. `public static` fields in a Tomcat application are JVM-wide singletons shared across all concurrent requests from all users and all tenants. When one user logs in, their `dealerId` overwrites the value for every other concurrent request. Any subsequent data access by a different user on a concurrent thread will use the last-written `dealerId`, `isSuperAdmin`, and `adminAccessLvl` values. This is a complete multi-tenancy isolation failure and a complete authentication bypass: a superadmin login by any user simultaneously grants superadmin rights to all concurrent sessions; a login by tenant A simultaneously changes the active tenant for all concurrent sessions of tenant B. The `multi-customer-sync-to-master` branch name directly indicates this is a multi-tenant deployment, making this the highest severity finding in scope.
**Severity:** CRITICAL
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / per-request isolation); Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// SessionVar.java lines 11–16
public static int dealerId = 0;
public static int adminAccessLvl = 99;
public static boolean isSuperAdmin = false;

public static List<Dealer> dealerList = new ArrayList();
public static DealerConfigBean dealerConf = new DealerConfigBean();
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/SessionVar.java`
**Line(s):** 11–16

### J82-F03: SQL string concatenation with database-sourced value — SQL injection via sms_outgoing.id
**Description:** In `SendMessage.send_sms_message()`, the `id` parameter (sourced directly from `rset.getString(1)` on the `sms_outgoing` table) is concatenated into a SQL DELETE statement without using a `PreparedStatement`. While this specific value originates from the database rather than directly from HTTP user input, it represents a SQL injection risk: if an attacker can insert a crafted value into the `sms_outgoing.id` column (through any other injection point or database write path), the DELETE statement will execute arbitrary SQL. Additionally, `Statement` is used throughout instead of `PreparedStatement`, which is contrary to best practice for all parameterised queries.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities (SQL injection)
**Evidence:**
```java
// SendMessage.java line 132
query = "DELETE FROM sms_outgoing WHERE id = "+id;
stmt.execute(query);

// And at line 203–204 (same pattern, second occurrence):
query = "DELETE FROM sms_outgoing WHERE id = "+id;
stmt.execute(query);
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/SendMessage.java`
**Line(s):** 132–133, 203–204

### J82-F04: SMS gateway credentials passed in HTTP URL query string — credential exposure in logs and process memory
**Description:** In `SendMessage.send_sms_message()`, the SMS gateway authentication call constructs a URL containing `USERNAME`, `PASSWORD`, and `API_ID` as plain query string parameters. HTTP GET requests with credentials in the URL are logged by all intermediary proxies, web server access logs, and browser/HTTP client history. Even over HTTPS the URL path including query string appears in server-side access logs. This means SMS gateway credentials (username, password, and API ID) are written to every Tomcat access log entry for this request. The credential values themselves are sourced from `RuntimeConf`, which warrants separate review, but the transmission pattern is independently a finding.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials; Section 6 — Transport Security
**Evidence:**
```java
// SendMessage.java line 180
url = RuntimeConf.LOCATION + "/http/auth?user=" + RuntimeConf.USERNAME + "&password=" + RuntimeConf.PASSWORD + "&api_id=" + RuntimeConf.API_ID;
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/SendMessage.java`
**Line(s):** 180

### J82-F05: GPS location data written to stdout without sanitisation
**Description:** In `SkyhookService.getWifiPos()`, the full JSON response from the Skyhook location API is printed to `System.out` (Tomcat's catalina.out log) unconditionally. The Skyhook API response contains GPS coordinates (latitude, longitude, accuracy) for the located device. Writing raw GPS coordinates to application stdout logs means location data for every WiFi positioning lookup is stored in Tomcat logs in plain text. This is a data exposure issue for location/PII data and is contrary to Section 5 of the checklist, which specifically flags GPS coordinate logging.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy (logging of GPS/location data)
**Evidence:**

### J82-F08: `SELECT *` query with no customer_id scoping on sms_outgoing table
**Description:** The query `SELECT * FROM sms_outgoing` at line 84 of `SendMessage.java` retrieves all SMS outgoing records from the database with no `WHERE` clause filtering by customer, tenant, or any scoping predicate. In a multi-tenant deployment this means the `SendMessage` job processes SMS alerts for every customer indiscriminately from a single shared table. If the table contains rows for multiple customers and the SMS gateway credentials belong to one customer's account, alerts from other customers will be sent via that account. More broadly, the absence of tenant scoping on this query is consistent with the static `SessionVar.dealerId` pattern — neither construct enforces tenant isolation.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy / customer_id scoping); Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// SendMessage.java line 84
query =  "SELECT * FROM sms_outgoing";
rset = stmt1.executeQuery(query);
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/SendMessage.java`
**Line(s):** 84–85

### J82-F09: Unvalidated external JSON input forwarded to third-party API (SSRF-adjacent)
**Description:** `SkyhookService.getWifiPos()` accepts a `String jsonInputString` parameter and writes its raw bytes directly to the POST body of the Skyhook API request without any validation, sanitisation, or size limit. The Skyhook URL itself is fixed (not user-controlled), so this is not a classic SSRF; however, if `jsonInputString` is populated from user-controlled or unvalidated input upstream (WiFi scan data received from a device or a client), a malicious caller could send crafted payloads to the Skyhook API using the application's hardcoded API key. This should be confirmed by tracing callers of `getWifiPos()` in other files; flagged here as a finding based on the lack of input validation visible in this file.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (SSRF); Section 1 — Secrets and Credentials (API key abuse)
**Evidence:**
```java
// SkyhookService.java lines 30–33
try(OutputStream os = con.getOutputStream()) {
    byte[] input = jsonInputString.getBytes("utf-8");
    os.write(input, 0, input.length);
}
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/SkyhookService.java`
**Line(s):** 30–33
