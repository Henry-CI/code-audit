### P61-F01: Cross-Site Scripting (XSS) - Widespread Unescaped Output
**Description:** All JSP files use unescaped JSP expression syntax (`<%= ... %>`) to output user-controlled values, database values, and request parameters directly into HTML without any escaping or sanitization. This allows attackers to inject malicious JavaScript that will execute in other users' browsers.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// new-general.jsp line 126 - customer name from database
<option value="<%=Fms_Usr_Mst_Cd.get(i)%>"><%=Fms_Usr_Mst_Name.get(i)%></option>

// search.jsp lines 347-348 - vehicle data from database
<td><%=Fms_Vehicle_List_Serial_No.get(i)%></td>
<td><%=Fms_Vehicle_List_Id.get(i)%></td>

// other-settings.jsp line 127 - note field from database
<textarea class="form-control"  name="note" rows="2" style="width:100%;"><%=note%></textarea>

// service.jsp line 557 - filename in URL
<a href="../servlet/GetFile?method=inspection&vehicle_cd=<%=vehicle_cd %>&filename=<%=fileName %>">

// new-general-b.jsp line 184 - CAN bus model name
<option value="<%=Canbus_Models.get(x).toString()%>" > <%=Canbus_Models.get(x).toString()%> </option>
```
**File:** Multiple files (all 10 JSP files affected)
**Line(s):** Throughout all files - examples: new-general.jsp:126, search.jsp:347-348, other-settings.jsp:127, service.jsp:557, new-general-b.jsp:184

## Summary

This security audit reviewed 10 JSP files in the FleetIQ vehicle administration module and identified 15 security findings ranging from INFO to HIGH severity. The most critical issues are:

1. **Widespread XSS vulnerabilities** (HIGH) - All files output user-controlled and database values without HTML escaping
2. **Missing CSRF protection** (HIGH) - All forms lack CSRF tokens, enabling request forgery attacks
3. **IDOR vulnerabilities** (HIGH) - Direct object references without authorization checks for vehicles and customers
4. **SQL injection risk** (MEDIUM) - Parameters passed to filter methods without visible validation

The application appears to be a legacy JSP/Servlet architecture with multi-tenant functionality (indicated by the branch name). The lack of modern security controls (CSRF protection, output encoding, parameterized queries) combined with multi-tenancy creates significant security risk, particularly for:
- Cross-customer data access (IDOR)
- Stored XSS attacks affecting multiple tenants
- Session hijacking via XSS
- Data manipulation via CSRF

Immediate remediation recommended for all HIGH severity findings.
