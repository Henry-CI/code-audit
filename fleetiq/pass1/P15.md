### P15-F01: Unescaped JSP Expressions Injected Into JavaScript Event Handlers (XSS)
**Description:** In `report_nevigation.jsp` line 56, three server-side variables (`user_cd`, `loc_cd`, `dept_cd`) are interpolated directly into the `onload` inline JavaScript event handler attribute without any HTML or JavaScript encoding. These values are drawn from the `filter` object which is populated from request parameters or DB data. An attacker who can influence these values (e.g., through a specially crafted customer/site/department name stored in the DB, or by manipulating request parameters before they are stored) could inject arbitrary JavaScript that executes when the page loads. Similarly, lines 149–158 inject URL variables directly into `onclick` handlers without escaping.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// Line 56 — report_nevigation.jsp
<body bgcolor="white" onload="set('<%=user_cd %>','<%=loc_cd %>','<%=dept_cd %>')">

// Lines 149, 155–158 — report_nevigation.jsp
```

### P15-F03: Dynamic Key Injected Into `<fmt:message>` Tag Attribute (Potential Resource Injection / Information Disclosure)
**Description:** In `report_export_keyword.jsp` line 109, the `localeKey` variable — derived from the `filter` object — is used as a dynamic attribute value in `<fmt:message key="<%=localeKey %>" />`. While JSTL message bundles are server-side resource files (not SQL or OS calls), injecting an attacker-controlled key allows enumeration of any key in the `Messages` resource bundle. If `localeKey` can be manipulated through a request parameter or stored DB value, an attacker may be able to probe the contents of the message bundle by trying arbitrary key names. Additionally, if an unrecognised key is returned as-is by the JSTL implementation, it may result in information leakage about internal message key naming conventions.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities; Section 5 — Data Exposure and Privacy
**Evidence:**
```jsp
// report_export_keyword.jsp line 109
<fmt:message key="<%=localeKey %>" />
```
Where `localeKey` is set at line 33:
```jsp
String localeKey = filter.getLocaleKey();
```
**File:** includes/report_export_keyword.jsp
**Line(s):** 33, 109

### P15-F08: Session Guard Bypass Risk — Site-Name Bypass in Expire.jsp
**Description:** The session guard included by all report header files (`sess/Expire.jsp`) contains a conditional at line 20 that skips the `dealerId == 0` check for sites named "AU" or "UK". This means that on AU and UK deployments, a user with `dealerId == 0` (which normally indicates an invalid/unauthenticated state) is not redirected to the login page. The multi-tenancy access control check block at lines 57–187 is similarly gated behind `!LindeConfig.siteName.equalsIgnoreCase("AU") && !LindeConfig.siteName.equalsIgnoreCase("UK")` — meaning the entire tenant-scoping check is **skipped entirely** for AU and UK site deployments. Any report page on an AU or UK deployment is therefore served without verifying that the requesting user's customer/site/department parameters are within their permitted access scope.
**Severity:** CRITICAL
**Checklist item:** Section 2 — Authentication and Authorization; Multi-tenancy (highest-risk area per checklist)
**Evidence:**
```java
// sess/Expire.jsp lines 20–27
if( !SessionVar.isSuperAdmin && SessionVar.dealerId == 0 && !LindeConfig.siteName.equalsIgnoreCase("AU")  && !LindeConfig.siteName.equalsIgnoreCase("UK")){
    request.getSession().invalidate();
    // ... forward to login ...
}

// sess/Expire.jsp lines 57–187
if( !LindeConfig.siteName.equalsIgnoreCase("AU")  && !LindeConfig.siteName.equalsIgnoreCase("UK") ){
    // ... entire customer/site/dept access check block ...
    // This block is NOT executed for AU or UK deployments.
}
```
**File:** sess/Expire.jsp (included by all report_header*.jsp files)
**Line(s):** 20, 57
