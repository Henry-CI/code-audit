### P30-F01: Unescaped Database Values Written to HTML ‚Äî Stored XSS Risk (Multiple Files)
**Description:** Across Account.jsp, Battery.jsp, BusinessInsight.jsp, Customers.jsp, Dealers.jsp, Home.jsp, and Limiting.jsp, values retrieved from the database (customer names, dealer names, subdomains, module names, module descriptions, form descriptions, battery menu names/descriptions, user first/last name, email, group name, access level) are output directly with `<%= ... %>` into HTML attribute values and HTML element bodies without any escaping. There is no use of `<c:out>`, `${fn:escapeXml(...)}`, or any equivalent escape mechanism anywhere in these files. If any of these database-stored values contain HTML metacharacters (e.g., `<`, `>`, `"`, `&`) ‚Äî whether introduced through a compromised admin session, a SQL injection, or an XSS in a data-entry form ‚Äî they will be rendered as raw HTML in every user's browser (stored XSS). Fields of particular concern are customer names, dealer names, module descriptions, and subdomain values, all of which are database-driven and rendered unescaped.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî XSS (Cross-Site Scripting)
**Evidence:**
```
// Customers.jsp line 186 ‚Äî customer name from DB, no escaping:
<td><%=Customers_Name.get(i).toString()%></td>

// Dealers.jsp line 181 ‚Äî dealer name from DB, no escaping:
<td><%=Dealers_Name.get(i).toString()%></td>

// Dealers.jsp line 182 ‚Äî subdomain from DB, no escaping:
<td><%=Dealer_subd.get(i).toString()%></td>

// Home.jsp line 103 ‚Äî module description from DB as raw HTML content, no escaping:
<%=Module_Desc.get(x).toString()%>

// Account.jsp line 35 ‚Äî user first name from DB in input value, no escaping:
<input type="text" ... value="<%=Fms_Usr_First_Name%>">

// Limiting.jsp line 46/47 ‚Äî form name and description from DB, no escaping:

<%=Form_Desc.get(x).toString()%>
```

### P30-F02: Unescaped User-Controlled Request Parameters Reflected in HTML ‚Äî Reflected XSS
**Description:** In Customers.jsp and Dealers.jsp, the request parameter `sc` (search_crit) is read directly from `request.getParameter("sc")` and reflected unescaped into an HTML input field's `value` attribute. Additional request parameters (`st_dt`, `end_dt`, `mnm`, `module_cd`, `form_cd`) are also reflected unescaped into hidden fields. These are directly user-controlled (URL parameters) and are reflected back into HTML without any encoding, constituting reflected XSS vulnerabilities. An attacker can craft a URL with a malicious value for `sc` and trick an authenticated user into executing arbitrary JavaScript.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities ‚Äî XSS (Cross-Site Scripting)
**Evidence:**
```
// Customers.jsp line 13/20 ‚Äî read from request:
String search_crit = request.getParameter("sc") == null ? "" : request.getParameter("sc");

// Customers.jsp line 95 ‚Äî reflected unescaped into input value:
<input type="text" name="sc" class="form-control input-lg"
    placeholder="..." value="<%=search_crit%>" />

// Customers.jsp line 98 ‚Äî reflected unescaped into hidden field:
<input type="hidden" name="frm_dt" value="<%=st_dt%>" ... />

// Dealers.jsp line 95 ‚Äî identical pattern:
<input type="text" name="sc" class="form-control input-lg"
```

### P30-F06: `search_crit` (User Input) Passed to Data Query Without Visible Sanitisation
