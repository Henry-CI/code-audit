**Description:** The `writelog` method builds a SELECT query by directly concatenating `tuid` (a substring parsed from the caller-supplied `msg` parameter) into the SQL string. No `PreparedStatement` is used. Any caller that controls the content of the `msg` string can inject arbitrary SQL into the SELECT, including UNION attacks, subqueries, or destructive statements.
**Severity:** CRITICAL
**Checklist item:** 3. Injection Vulnerabilities â€” SQL injection
**Evidence:**

### J79-F05: XXE (XML External Entity) injection â€” DocumentBuilderFactory with no external entity disabling
**Description:** `LindeConfig.readXMLFile()` creates a `DocumentBuilderFactory` and immediately calls `newDocumentBuilder()` without disabling external entity processing. The default Java XML parser supports external entity references (`DOCTYPE` with `SYSTEM` or `PUBLIC` identifiers). If the XML file at `/home/gmtp/linde_config/settings_multi.xml` can be written by any lower-privileged process or external party, an attacker can inject an XXE payload that reads arbitrary files from the server filesystem (e.g. `/etc/passwd`, private keys) or triggers server-side request forgery.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities â€” SSRF; general XML injection
**Evidence:**
```java
// LindeConfig.java, lines 74â€“76
DocumentBuilderFactory dbFactory = DocumentBuilderFactory.newInstance();
DocumentBuilder dBuilder = dbFactory.newDocumentBuilder();
Document doc = dBuilder.parse(fXmlFile);
```
Correct mitigation would be:
```java
dbFactory.setFeature("http://xml.org/sax/features/external-general-entities", false);
dbFactory.setFeature("http://xml.org/sax/features/external-parameter-entities", false);
dbFactory.setFeature("http://apache.org/xml/features/nonvalidating/load-external-dtd", false);
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/LindeConfig.java`
**Line(s):** 74â€“76

### J79-F08: Unsanitised user-controlled value used in ResourceBundle name lookup
**Description:** `LoadBundle.getBundle()` reads the `lang` attribute from the HTTP session (originally set from `req.getLocale().getLanguage()`, which is derived from the browser-supplied `Accept-Language` header). This value is appended directly to the string `"Messages_"` and passed to `ResourceBundle.getBundle()` with no validation, whitelist check, or sanitisation. An attacker who can manipulate the session `lang` attribute (e.g. via a prior session attribute injection) or who controls the `Accept-Language` header on first request could supply a crafted language code that traverses the resource directory or causes unexpected class loading via the `URLClassLoader`. While `ResourceBundle` itself is not a direct path traversal vector, the unsanitised concatenation violates the principle of input validation and should be whitelisted to known locale codes.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities â€” file inclusion
**Evidence:**

### J79-F10: Exception stack traces written to stderr / `System.out` â€” information leakage
**Description:** All three files handle exceptions by writing to `System.out.println(...)` or `e.printStackTrace()` rather than using a structured logging framework. Stack traces written to standard output are frequently captured in Tomcat's `catalina.out` log. Specifically: `InfoLogger` prints the full exception including the SQL query string on line 72; `LindeConfig` calls `e.printStackTrace()` on line 129 which may include the XML file path; `LoadBundle` calls `ex.printStackTrace()` on line 39.

### J79-F12: `LindeConfig` mutable public static fields â€” race condition and tenant bleed risk
**Description:** Every configuration value in `LindeConfig` is declared `public static` and non-final. The constructor calls `readXMLFile()` which overwrites all static fields. In a multi-threaded Tomcat environment, if multiple threads instantiate `LindeConfig` concurrently, or if a re-read is triggered at runtime, these unsynchronised writes to static fields create a race condition. In a multi-tenant context this is particularly dangerous: a field such as `customerLinde` could briefly hold a different tenant's value between the write and any subsequent read in a concurrent request, causing cross-tenant data leakage.
**Severity:** HIGH
**Checklist item:** 2. Authentication and Authorization â€” multi-tenancy; 4. Session and CSRF (shared mutable state)
**Evidence:**
