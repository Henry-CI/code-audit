### J73-F03: SQL Injection via String Concatenation in firmware Action (CustomUpload.java)
**Description:** In the `firmware` action branch, two SQL INSERT statements are constructed by direct string concatenation of variables derived from database query results. While `gmtpId` originates from a PreparedStatement query result (and thus from the database), the variable `message` at line 220 is a hardcoded string in this specific path, and `sno` and `tm` come from a prior SELECT result. However, the pattern of building SQL with string concatenation is inherently fragile: if the values written into `outgoing` are later replayed or the code path changes, injection becomes possible. More critically, the INSERT at line 222 concatenates `gmtpId` and `message` directly; if a device HIRE_NO or vehicle name in the database contained a SQL metacharacter, it would propagate through this query. The INSERT at line 235 concatenates four variables. PreparedStatement is available and used elsewhere in the same method ‚Äî its non-use here is inconsistent and a defect.
**Severity:** HIGH
**Checklist item:** Section 3 ‚Äî Injection Vulnerabilities (SQL Injection) ‚Äî "Flag every instance of string concatenation in a SQL query"
**Evidence:**
```java
// CustomUpload.java lines 222‚Äì223
queryString="insert into \"outgoing\" (destination,message) values('"+gmtpId+"','"+message+"')";
ps = dbcon.prepareStatement(queryString);
ps.executeUpdate();

// CustomUpload.java lines 235‚Äì237
queryString="insert into \"outgoing_stat\" (outgoing_id_s,destination_s,message_s,timestamp_s,ack) values('"+sno+"','"+gmtpId+"','"+message+"','"+tm+"','f')";

ps.executeUpdate();
```

### J73-F05: Unrestricted File Upload ‚Äî No File Type Validation
**Description:** The servlet accepts a multipart file upload and writes it directly to the server filesystem using the client-supplied filename, without any validation of the file extension, MIME type, or content. The `fileName` is extracted from the HTTP `content-disposition` header (controlled by the attacker), and is used without sanitisation to construct the output path `path + File.separator + fileName`. This permits an attacker to upload any file type (e.g. a JSP web shell) to the upload directory. If that directory is within the web application root, the uploaded file could be executed as a servlet. Path traversal sequences in the filename (e.g. `../../`) could also place files outside the intended directory.
**Severity:** CRITICAL
**Checklist item:** Section 3 ‚Äî Injection Vulnerabilities (File Inclusion) / Command Injection / general injection
**Evidence:**
