## Findings

### P02-F01: Access Level Control Entirely Commented Out — xlsx_curr_unit_report.jsp
**Description:** The entire session-based access level control block in `xlsx_curr_unit_report.jsp` is commented out (lines 31–57). This means any authenticated user can download the Current Unit Report Excel file for any customer, bypassing tenant isolation. The `access_level`, `access_cust`, `access_site`, and `access_dept` session attributes are never applied to the data query.
**Severity:** CRITICAL
**Checklist item:** 2. Authentication and Authorization — multi-tenancy / IDOR
**Evidence:**
```java
// Lines 31-57 — entire access control block commented out:
// 	String access_level = (String)session.getAttribute("access_level");

// 	String access_dept=(String)session.getAttribute("access_dept");
// 	if(access_level.equalsIgnoreCase(""))
// 	{
// 		access_level = "5";
// 	}
// 	int al = Integer.parseInt(access_level);

// 	filter.setAccess_cust(access_cust);

// 	filter.setAccess_dept(access_dept);
```
**File:** au_excel/xlsx_curr_unit_report.jsp
**Line(s):** 31–57

### P02-F02: Unescaped Database-Sourced Values in HTML Output — XSS
**Description:** In `excel_hourmeter_exception.jsp`, multiple values retrieved from the database are rendered unescaped into HTML output: `vmachine_nm.get(i)`, `sm_hour_meter_ts.get(i)`, `sm_hour_meter.get(i)`, `get_user`, `get_loc`, `get_dep`, `form_nm`. If any database field contains HTML special characters, XSS is possible.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — XSS
**Evidence:**
```jsp
<!-- Lines 158-165 -->
<td><%=vmachine_nm.get(i) %></td>
<td><%=sm_hour_meter_ts.get(i) %></td>
<td><%=sm_hour_meter.get(i) %></td>
<!-- Line 127-128 -->
<td><b><%=get_user %></b></td>
```
**File:** au_excel/excel_hourmeter_exception.jsp
**Line(s):** 127–128, 158–165

### P02-F03: Hardcoded External HTTP URL (Mixed Content / Information Disclosure)
**Description:** `excel_hourmeter_exception.jsp` hardcodes `http://fleetfocus.lindemh.com.au/fms/images/...` for logo images. This is HTTP (not HTTPS), leaks the internal server hostname, and constitutes mixed content if the application is served over HTTPS.
**Severity:** LOW
**Checklist item:** 6. Transport Security
**Evidence:**

**File:** au_excel/excel_hourmeter_exception.jsp
**Line(s):** 110, 120

### P02-F04: Stack Trace Exposure via e.printStackTrace()
**Description:** All xlsx_*.jsp Excel download files catch `Exception` and call `e.printStackTrace()`. While this outputs to the server log (not the browser), it may expose internal class paths, library versions, and logic structure in log files accessible to attackers with log access. This is a consistent pattern across xlsx_battery_report.jsp, xlsx_broadcastmsg_report.jsp, xlsx_cimplicity_shock_report.jsp, xlsx_cimplicity_util_report.jsp, xlsx_curr_driv_report.jsp, xlsx_curr_unit_report.jsp, xlsx_daily_driver_summary_report.jsp, xlsx_daily_veh_summary_report.jsp, xlsx_driver_access_abuse_report.jsp.
**Severity:** LOW
**Checklist item:** 5. Data Exposure and Privacy — Error responses
**Evidence:**
```java
} catch (Exception e) // file IO errors
{
    e.printStackTrace();
}
```
**File:** au_excel/xlsx_battery_report.jsp (and others in this partition)
**Line(s):** 91–94 (pattern repeated across all xlsx_*.jsp files)

### P02-F05: Missing CSRF Protection on Report Download Forms
**Description:** These JSP files are invoked via form submission from parent report pages. No CSRF token mechanism is observed in these pages or their parents. State-changing actions (triggering report generation and email sending) are not CSRF-protected.
**Severity:** HIGH
**Checklist item:** 4. Session and CSRF
**Evidence:** No hidden CSRF token fields in any of the reviewed au_excel/*.jsp files. All are invoked via POST from parent pages.
**File:** au_excel/xlsx_battery_report.jsp (pattern across all in partition)
**Line(s):** N/A — absence of CSRF token throughout
