### S02-F03: equipment_bypass and equipment_bypass_report Tables Lack Tenant Scoping Column
**Description:** Both `equipment_bypass` and `equipment_bypass_report` tables are keyed solely by `veh_cd` (vehicle code) with no `cust_cd` or equivalent tenant discriminator column. In a multi-tenant system any application query over these tables that fails to join back to a customer-ownership table will expose cross-customer bypass data. The schema provides no structural guarantee of tenant isolation; correct scoping depends entirely on every application query being written correctly.

### S02-F05: ilike with Caller-Supplied Card ID Allows Wildcard Matching in Driver Lookup
**Description:** Both `sp_driver_shock_message` and `sp_eos_message` use `ilike` (case-insensitive pattern match) instead of `=` (exact equality) when looking up a driver by card ID. `ilike` honours SQL wildcard characters `%` (any sequence) and `_` (any single character). If a device or upstream caller can supply a card ID containing these characters, the lookup may match unintended driver records. In `sp_driver_shock_message` (else branch, line 306) the technician subquery is also less tightly scoped than the if-branch equivalent, compounding the risk by potentially matching a technician user from any customer when a wildcard card ID is supplied.
**Severity:** MEDIUM
**Checklist item:** 3. Injection Vulnerabilities — SQL injection / unexpected query broadening
**Evidence:**
```sql
-- sp_driver_shock_message.sql line 306 (else branch — no dealer-level scoping)
SELECT INTO diver_cd "USER_CD" from "FMS_USR_MST" where "CARD_ID" ilike cardID and ("USER_CD" in (select "USER_CD" from "FMS_USER_DEPT_REL" where "CUST_CD" = c_cd and "LOC_CD" = s_cd) OR "USER_CD" in (select "USER_CD" from "FMS_USR_MST" where "is_technician" is true));

-- sp_eos_message.sql lines 189-191 (if branch)
SELECT INTO diver_cd "USER_CD" from "FMS_USR_MST" where "CARD_ID" ilike cardID and "ACTIVE" is true and ("USER_CD" in (select "USER_CD" from "FMS_USER_DEPT_REL" where "CUST_CD" = c_cd and "LOC_CD" = s_cd) OR "USER_CD" in (SELECT "FMS_USR_MST"."USER_CD" FROM "FMS_USR_MST", "FMS_USER_DEPT_REL" ...));
```
**File:** sql/sp_driver_shock_message.sql (line 306); sql/sp_eos_message.sql (lines 189-191, 213-215)

### S02-F06: Inconsistent Technician Scoping Between Code Paths in sp_driver_shock_message
**Description:** In `sp_driver_shock_message`, the driver lookup query differs between the `if` branch (card ID contains a comma) and the `else` branch (no comma). The `else` branch at line 306 omits this dealer-level scoping entirely and instead allows any user marked `is_technician` to match, regardless of dealer or customer association. A device message that does not contain a comma separator in the driver field takes the less-scoped code path, potentially allowing a technician from a different customer's dealer to be recorded as the driver.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization — multi-tenancy / IDOR

### S02-F07: Caller-Supplied Delimiters Used in Data Parsing in sp_eos_message
**Description:** `sp_eos_message` accepts `$6` (delimiter1) and `$7` (delimiter2) as caller-supplied strings and uses them directly in `string_to_array($5, $6)` and `position($7 in data_component)`. While this is PL/pgSQL string handling (not dynamic SQL), it could cause `fld_nm` to be derived incorrectly, resulting in mismatched field code lookups and corrupted telemetry records.
