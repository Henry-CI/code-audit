### J61-F01: No customer/tenant scoping field in report beans — multi-tenant data isolation absent at bean level
**Description:** All three beans (`PreOpCheckFailReportBean`, `PreOpCheckReportBean`, `SeatHourUtilBean`) are data-transfer objects used to carry report data into Excel report generation. None of them contain a `customer_id`, `org_id`, `tenant_id`, or any equivalent tenant-scoping field. This means the beans themselves carry no signal about which customer the data belongs to, and any code path that populates or consumes these beans must enforce tenant scoping independently. In a multi-tenant application on the `multi-customer-sync-to-master` branch, this design creates a structural risk: if the layer that populates these beans incorrectly fetches cross-customer data (e.g., via a SQL query missing a `customer_id` WHERE clause), there is no in-bean guard to detect or prevent it. The report beans also expose vehicle IDs (`po_veh_id`, `veh_cd`) and driver data (`po_dnm`, `po_did`) without any ownership context. This is not itself exploitable in the bean class alone, but it is a structural multi-tenancy design weakness that must be traced through to the SQL and servlet layers.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy ("verify that every data access is scoped to the authenticated user's customer/organisation")
**Evidence:**

### J61-F02: All fields are package-private (default access) instead of private — broken encapsulation enables uncontrolled mutation
**Description:** In all three bean classes, every field is declared with default (package-private) access — there is no `private` keyword on any field. The beans do provide public getter/setter methods, but because the fields themselves are also directly accessible from any class within the same package (`com.torrent.surat.fms6.excel.reports.beans`), any code in that package can bypass the setters and write directly to the fields. This bypasses any validation that might be added to a setter in the future and means the data in the bean cannot be trusted to have passed through the intended access path. In a security context, this matters for report beans: if any setter were ever hardened to validate customer ownership before accepting a vehicle ID list, code within the same package could silently bypass it.
**Severity:** LOW
**Checklist item:** Section 2 — Authorization and multi-tenancy (data integrity of multi-tenant report structures); Section 5 — Data Exposure and Privacy (vehicle and driver data accessible without encapsulation boundary)

### J61-F03: Pervasive use of raw ArrayList types — type-unsafe collections carrying driver and vehicle data
**Description:** Across all three beans, the vast majority of `ArrayList` fields are declared as raw types — no generic type parameter is specified (e.g., `ArrayList po_dnm` instead of `ArrayList<String> po_dnm`). This means the JVM performs no runtime type checking on the contents of these lists. Any caller can insert any object type. In a security context, this is relevant because: (1) it makes it impossible to enforce that vehicle IDs, driver IDs, and check results are of the expected type, preventing any type-level validation; (2) raw collections are a common source of `ClassCastException` at runtime when the wrong data type is inadvertently inserted, which can produce verbose error messages exposing internal implementation details if those exceptions propagate to error pages; and (3) it reflects a low code-quality baseline that correlates with other security weaknesses such as missing input validation.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (runtime exceptions from type mismatches may leak internal details via error pages); Section 3 — Injection Vulnerabilities (lack of type constraints on data flowing into report generation)

### J61-F04: Uninitialised String fields in SeatHourUtilBean — null values may produce NullPointerException in report rendering
**Description:** In `SeatHourUtilBean`, the nine `String` fields (`util1` through `util8` and `utilt`) are declared without field initialisers and therefore default to `null` at construction time. If this exception propagates to the user as an unhandled error, it may expose stack traces or internal class names (see checklist section 5 — error responses). The other two beans initialise their `String tot_dur` field to `""` (line 15 in both), indicating this was a known pattern that was not applied consistently to `SeatHourUtilBean`.

## Checklist Items With No Issues

These files are pure data-transfer objects (DTOs) with no logic beyond getters and setters. They contain no SQL, no HTTP handling, no authentication logic, no JSP output, no configuration, and no dependency declarations. The absence of findings below reflects the limited scope of these file types, not a broad clearance of those topics across the application.

- **Section 3 — Injection Vulnerabilities (XSS):** No JSP expressions, no HTML rendering, no output writing. Not applicable to these DTOs.
- **Section 3 — Injection Vulnerabilities (file inclusion):** No file path handling, no `include` directives. Not applicable.
- **Section 3 — Injection Vulnerabilities (command injection):** No `Runtime.exec()`, no `ProcessBuilder`. Not applicable.
- **Section 3 — Injection Vulnerabilities (SSRF):** No HTTP client calls, no URL construction. Not applicable.
- **Section 4 — Session and CSRF:** No session handling, no form rendering, no CSRF token logic. Not applicable to these DTOs.
- **Section 6 — Transport Security:** No network I/O, no connector configuration. Not applicable.
- **Section 7 — Dependency Vulnerabilities:** No `import` of third-party libraries; the only import in all three files is `java.util.ArrayList` from the standard library.
