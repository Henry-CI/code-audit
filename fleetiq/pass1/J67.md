### J67-F01: Pervasive SQL Injection via String Concatenation Throughout All Query Methods
**Description:** Every SQL query in both `Databean_cdisp.java` and `Databean_dyn_reports.java` is constructed by direct string concatenation of field values into the query string, and then executed via `Statement.executeQuery()`. No `PreparedStatement` with `?` placeholders is used anywhere in either file. The values concatenated include `set_cust_cd`, `set_loc_cd`, `set_dept_cd`, `set_gmtp_id`, `set_op_code`, `access_cust`, `access_site`, `access_dept`, `model_cd`, `set_sc`, `set_veh_typ_cd`, `st_dt`, `end_dt`, `set_ucd`, `checkCustCd`, and `checkDealerID`. These are the primary report-filtering parameters and would ordinarily derive from user-supplied HTTP request parameters passed in through the JSP layer. An attacker with any authenticated session can inject arbitrary SQL into any of these parameters.
**Severity:** CRITICAL
**Checklist item:** 3 — Injection Vulnerabilities (SQL injection)
**Evidence:**

### J67-F09: Path Traversal Risk in `getExportDir()` Using Multiple `../` Segments

This is a fragile path construction that is sensitive to deployment directory structure. If the application is ever deployed differently (e.g., classes in a different location in the WAR), the export path could resolve outside the expected directory. More critically, the result of `getExportDir()` is used as the base path for writing user-report PDF files (`result` field), and subclasses control the `result` field via the `setResult()` setter. If calling code ever passes a user-supplied filename to `setResult()`, this combines with the traversal-based directory to enable a path traversal write.
**Severity:** MEDIUM
**Checklist item:** 3 — Injection Vulnerabilities (file inclusion / path traversal)
