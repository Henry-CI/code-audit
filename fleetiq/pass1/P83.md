## Findings

### P83-F01: Cross-Site Scripting (XSS) via unescaped output expressions
**Description:** Multiple JSP files output user-controlled or database-sourced values directly to HTML without proper escaping using `<%= ... %>` expressions. This allows attackers to inject malicious JavaScript if they can control database values or request parameters. In a multi-tenant application, one customer could potentially inject scripts that execute when another customer views reports.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)
**Evidence:**
All reviewed files output database values without escaping:
- `rpt_seat_hour_util.jsp` lines 118-138: `<%=veh_nm.get(i) %>`, `<%=veh_typ_nm1.get(i) %>` - vehicle names from database
- `rpt_serv_maintenance.jsp` lines 99-107: `<%=veh_nm.get(i) %>` - vehicle names
- `rpt_subscription.jsp` line 404: `<option value="<%=Vuser_cd.get(i) %>"><%=Vuser_nm.get(i) %></option>` - customer names in HTML
- `rpt_unit_unlock.jsp` lines 94-107: `<%=vunit_name.get(i) %>`, `<%=vunit_loreason.get(i) %>` - unit names and lockout reasons
- `rpt_unit_unlock_impact.jsp` lines 103-116: Same pattern
- `rpt_unit_unlock_question.jsp` lines 104-117: Same pattern
- `rpt_user_email_summary.jsp` lines 126-136: `<%=user_nm.get(i) %>`, `<%=cust_nm.get(i) %>` - user and customer names
- `rpt_user_summary.jsp` lines 159-173: `<%=user_nm.get(i) %>`, `<%=user_fnm.get(i) %>`, `<%=user_lnm.get(i) %>` - user names

None of these use proper JSP escaping (`<c:out>` or `${fn:escapeXml()}`).
**File:** All files listed above
**Line(s):** See evidence above

### P83-F05: Insecure external script loading over HTTP
**Description:** The subscription report page loads jQuery libraries from code.jquery.com over HTTP (not HTTPS). This creates a man-in-the-middle vulnerability where an attacker on the network can inject malicious JavaScript by intercepting and modifying the HTTP response. Given this is a fleet management system handling sensitive vehicle and driver data, all resources must be loaded over HTTPS.
**Severity:** HIGH
**Checklist item:** 6. Transport Security
**Evidence:**
```html
<script language="javascript" src="http://code.jquery.com/jquery-1.7.2.min.js"></script>
<script language="javascript" src="http://code.jquery.com/ui/1.8.21/jquery-ui.min.js"></script>
```
**File:** reports/rpt_subscription.jsp
**Line(s):** 12-13

### P83-F08: Potential multi-tenant data isolation concern - insufficient customer ID validation visibility
**Description:** While all files retrieve session attributes for access control (access_level, access_cust, access_site, access_dept) and pass them to the filter bean, the JSP layer itself does not perform any validation that the data returned from the bean actually belongs to the authenticated user's customer. The bean methods (filter.getVmachine_cd(), filter.getVunit_name(), etc.) are trusted to enforce multi-tenant isolation, but this cannot be verified from the JSP code alone. This is a potential Insecure Direct Object Reference (IDOR) vulnerability if the bean layer has bugs.
**Severity:** MEDIUM
**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy
**Evidence:**
All files follow this pattern:
```jsp
String access_cust=(String)session.getAttribute("access_cust");
// ... set on filter
filter.setAccess_cust(access_cust);
filter.init();
// ... retrieve data with no further validation
ArrayList veh_nm = filter.getVmachine_nm();
```
There is no verification that returned vehicle/unit IDs match the customer's scope. The trust boundary is entirely in the bean layer, which is not visible in this audit scope.
**File:** All reviewed files
**Line(s):** Various lines in each file's scriptlet section
