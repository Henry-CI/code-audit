### P37-F01: Unescaped User-Controlled Request Parameters Written Directly Into HTML (Stored and Reflected XSS)
**Description:** In `add-general.jsp` and `edit-general.jsp`, multiple request parameters that are read directly from the URL/form submission are immediately written into HTML `<input value="...">` attributes using raw `<%= ... %>` expressions, with no HTML encoding applied. An attacker who can cause a victim to load a crafted URL can inject arbitrary HTML and JavaScript into the page. The parameters `usr_cd` / `user_cd`, `gp_cd`, `loc_cd`, `st_dt`, `end_dt`, `search_crit`, `form_cd`, and `module_cd` are all affected. In `add-general.jsp`, `module_cd` is also injected into a JavaScript context on line 361 (`MOD_CD="<%=module_cd %>"`), which is a direct script injection vector requiring no HTML decoding by the browser.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// add-general.jsp line 81 — usr_cd direct from request parameter into hidden input value
<input type="hidden" name="dealer_id" value="<%=usr_cd%>">

// add-general.jsp line 142 — search_crit direct from request parameter
<input type="hidden" name="search_crit" value="<%=search_crit %>">

// add-general.jsp lines 361–362 — module_cd injected into JavaScript context

MOD_CD="<%=module_cd %>";

// edit-general.jsp line 117 — search_crit direct from request parameter
<input type="hidden" name="search_crit" value="<%=search_crit %>">

// edit-general.jsp line 119 — module_cd direct from request parameter
<input type="hidden" name="module_cd" value="<%=module_cd %>"/>
```

**Description:** All four files render database-sourced values — customer name, email address, physical address, contract number, salesperson names, subdomain, website URL, phone number, timezone ID — using raw `<%= ... %>` expressions with no HTML encoding. ### P37-F06: IDOR — Dealer Identity Taken From User-Controlled Request Parameter Without Server-Side Ownership Verification
**Description:** In `add-general.jsp`, the dealer being edited/added is identified exclusively by the request parameter `user_cd` (line 12: `String usr_cd=request.getParameter("user_cd")==null?"":request.getParameter("user_cd")`). This value is passed directly to `filter.setSet_cust_cd(usr_cd)` (line 26) and then used as the `dealer_id` in hidden form fields submitted to `Frm_saveuser`. There is no visible server-side check in the JSP that `usr_cd` belongs to the authenticated user's scope — the Expire.jsp guard checks whether a `user_cd` parameter matches the session's allowed customers, but only when the request parameter name is `user_cd` (line 68 of Expire.jsp), which will be populated. However, the Expire.jsp check (for `adminAccessLevel > 1`) compares the supplied `user_cd` against `access_cust` from the session, which is a comma-delimited list. A non-superadmin user who can enumerate dealer IDs could attempt to access or modify a dealer record belonging to a different dealer hierarchy by supplying an arbitrary `user_cd`. The risk is heightened because `view.jsp` line 8 also takes `user_cd` from request parameters and passes it directly to `filter.setUser_cd(user_cd)` and `filter.query("view_dealer")` with no additional ownership check visible in the JSP layer.

**Checklist item:** Section 2 — Authorization and Multi-Tenancy (IDOR)

### P37-F07: Admin Functionality Gate Relies on `SessionVar.isSuperAdmin` — Thread-Safety Risk With Static Field
**Description:** In `edit-general.jsp`, the main form rendering is gated at line 101 on `SessionVar.isSuperAdmin`. Reading `Expire.jsp` (line 16) reveals that `SessionVar` is a class with a `static` field `isSuperAdmin`. In a multi-threaded servlet container, a static field on a non-thread-local class is shared across all concurrent requests. If two requests arrive simultaneously — one from a super-admin and one from a regular user — the static `SessionVar.isSuperAdmin` could be set by one thread and read incorrectly by the other. This means a regular user could be granted super-admin access to the dealer management form. This is a class-level design issue in `SessionVar`, but it manifests as a security vulnerability in any JSP that consults `SessionVar.isSuperAdmin` for authorization gating.
**Severity:** HIGH
**Checklist item:** Section 2 — Authentication and Authorization
**Evidence:**
```java
// sess/Expire.jsp line 16 — static field set per request
SessionVar.isSuperAdmin = session.getAttribute("admin_access_level") == null ? false :
    session.getAttribute("admin_access_level").toString().trim().equals("0") ? true : false;
```

### P37-F12: Wildcard Imports Used Across All Files
**Description:** Every JSP file in scope uses `<%@ page import="java.util.*,com.torrent.surat.fms6.util.*"%>` — wildcard imports for both `java.util` and the application's own utility package. Wildcard imports expand the namespace of classes available in the scriptlet context beyond what is explicitly needed and make it harder to identify which classes are actually in use. While this is a code quality issue, in JSP scriptlets it also means any public class in `com.torrent.surat.fms6.util` is silently available to the scriptlet, including potentially sensitive utility methods. This makes security review and future maintenance more difficult.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (code quality / attack surface)
**Evidence:**
```jsp
// Appears in all four files, e.g. add-general.jsp line 4
<%@ page import="java.util.*,com.torrent.surat.fms6.util.*"%>
```
