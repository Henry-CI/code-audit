### P60-F01: Cross-Site Scripting (XSS) - Unescaped Request Parameters in HTML Context
**Description:** Multiple JSP files output user-controlled request parameters directly into HTML without escaping. The `customer`, `location`, `department`, `module_cd`, `form_cd`, and `vehicle_cd` parameters are retrieved from the request and rendered in HTML attributes and content using `<%= ... %>` expressions without sanitization. An attacker can inject malicious JavaScript by manipulating these URL parameters.

**Severity:** HIGH

**Checklist item:** 3. Injection Vulnerabilities - XSS (Cross-Site Scripting)

**Evidence:**
```jsp
// import-vehicle.jsp lines 18-21
<li class="active" style="width:25% !important" ><a data-cust-cd="<%=customer %>" href="vehicles/import-vehicle"><fmt:message key="global.vehicle"/></a></li>
```

### P60-F05: Sensitive Data Exposure - WiFi Passwords in Plain Text HTML
**Description:** WiFi passwords retrieved from the database are rendered in HTML password input fields with the value attribute set. While the input type is "password" (masking display in the browser), the actual password value is visible in the HTML source code, browser developer tools, and will be logged in proxy servers, browser history, and any monitoring tools. This exposes WiFi credentials to anyone with access to view the page source.

**Severity:** HIGH

**Checklist item:** 5. Data Exposure and Privacy

### P60-F06: Potential Insecure Direct Object Reference (IDOR) - Missing Customer Scoping Verification
**Description:** The JSP files retrieve vehicle_cd, customer, location, and department parameters from the request and use them to query vehicle and driver data. While session attributes (access_cust, access_site, access_dept, access_level) are retrieved, there is no visible verification in the JSP layer that the requested vehicle_cd or customer actually belongs to the authenticated user's customer scope. The application relies entirely on server-side servlet/bean logic for authorization. If the servlet/bean layer has any gaps, an attacker could access another customer's vehicles by manipulating the vehicle_cd or customer parameters.

**Severity:** MEDIUM

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
