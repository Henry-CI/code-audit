### J78-F01: SSRF — Unvalidated URL in GetHtml.getHTML() and getHTML1()
**Description:** All three `getHTML`/`getHTML1` methods in `GetHtml.java` accept a URL string that is passed directly to `new URL(...)` and `HttpURLConnection.openConnection()` without any whitelist validation, scheme enforcement, or hostname restriction. If any caller constructs this URL from user-supplied input (request parameters, database values entered by users), an attacker can direct the server to make HTTP requests to arbitrary internal or external endpoints (SSRF). The 600–900 second read timeouts also make this an effective vector for server-side denial-of-service.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — SSRF
**Evidence:**
```java
// GetHtml.java line 20-43
public String getHTML(String urlToRead, String param) {
    ...
    url = new URL(urlToRead+param);          // line 28 — no whitelist
    conn = (HttpURLConnection) url.openConnection();
    conn.setReadTimeout(600000);             // line 31 — 10 min timeout
    conn.setRequestMethod("GET");
    ...
}

// GetHtml.java line 45-77

    ...
    urlToRead += "&sess_lang="+ prefLang;
    url = new URL(urlToRead);               // line 56 — no whitelist

    conn.setReadTimeout(900000);            // line 60 — 15 min timeout
    ...
}

// GetHtml.java line 79-113
```

### J78-F04: Tenant (customer) identity derived from request parameter — IDOR / multi-tenancy bypass
**Description:** The `customer` value used for all import operations is read directly from `request.getParameter("customer")` (line 57). It is then passed into every DAO call as the tenant scoping key (`custCd`). There is no check that this value matches the authenticated user's actual customer in the session. An authenticated user of customer A can supply `customer=B` in the POST body and import drivers, vehicles, or questions into customer B's account. This is a classic IDOR/tenant-isolation failure in a multi-tenant system.

### J78-F05: Path traversal via unsanitised filename in file upload
**Description:** The uploaded file name is extracted from the multipart `content-disposition` header with only quote-stripping applied (line 2126). The resulting filename is concatenated directly into the server filesystem path (line 84): `appPath + RuntimeConf.UPLOAD_FOLDER + File.separator + fileName`. A `content-disposition: form-data; name="file"; filename="../../../conf/server.xml"` value would write the uploaded file outside the intended upload directory. There is no path component stripping (`File.getName()`), no extension whitelist, and no canonicalisation check.
**Severity:** HIGH
**Checklist item:** 3. Injection Vulnerabilities — File inclusion / path traversal
**Evidence:**

```java
            // Only removes quotes — does NOT strip directory components
        }
    }
    return null;
}

// ImportFiles.java line 84 — filename used directly in path
out = new FileOutputStream(new File(path + File.separator + fileName));
```

### J78-F07: Thread-safety — instance field `dHolder` written per-request on a singleton servlet
**Description:** `dHolder` is declared as an instance field (line 45) on `ImportFiles`, which is a servlet and therefore a singleton in Tomcat — one instance shared across all concurrent requests. `dHolder` is written inside `doPost` during CSV header parsing (line 137) and read during row validation in `validateCSV` (line 2176). ## Checklist Items With No Issues

The following checklist sections were reviewed against the three files assigned and had no findings within the scope of these files:

- **1. Secrets and Credentials:** No hardcoded database passwords, API keys, SMTP credentials, or encryption keys found in any of the three files. `ImpactUtil` constants are physics coefficients, not credentials.
- **3. Injection Vulnerabilities — SQL injection (direct):** No JDBC `Statement` or `PreparedStatement` calls appear in `GetHtml.java` or `ImpactUtil.java`. `ImportFiles.java` delegates all DB calls to DAO classes (`ImportDAO`, `DriverImportDAO`) which are not in scope for this agent; the `service` branch risk is reported under J78-F08 as a potential path. No direct SQL string construction was observed in the three assigned files.
- **3. Injection Vulnerabilities — XSS:** None of the three files are JSP files or produce HTML output via `<%= %>` expressions. `ImportFiles` produces JSON strings as responses; the message values are constructed from bundle strings and CSV field contents, but are not rendered as HTML by the servlet itself.
- **3. Injection Vulnerabilities — Command injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls in any of the three files.
- **4. Session and CSRF:** `ImportFiles` accepts only POST requests via `doPost`. No CSRF token mechanism is visible within these three files; CSRF protection (or its absence) would need to be assessed at the form/JSP layer and filter level, which are outside these three files.
- **7. Dependency Vulnerabilities:** No dependency version declarations appear in these source files. Log4j 2 (`org.apache.logging.log4j`) is used in `ImportFiles` (line 27, 42); version is not determinable from source alone — a separate inventory check of committed JARs is required.
- **ImpactUtil.java — all sections:** This is a pure mathematical utility class with no I/O, no user input, no database access, no network access, and no authentication logic. No security findings apply to this file.
