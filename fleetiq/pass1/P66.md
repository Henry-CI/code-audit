### P66-F01: Hardcoded Access Level Bypasses Authentication
**Description:** Seven JSP files hardcode the access_level variable to "1" instead of retrieving it from the authenticated user's session. This completely bypasses any role-based access control that should be enforced based on the user's actual access level. An attacker could access reports they should not have permission to view.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
```jsp
// email_preop_chk_detailed.jsp:25
String access_level = "1";

// email_preop_chk_fail.jsp:26
String access_level = "1";

String access_level = "1";

// email_unit_unlock_impact.jsp:27
```

**File:** Multiple files (7 of 10 files reviewed)

**Line(s):** See evidence above

### P66-F05: Missing Multi-Tenant Data Isolation
**Description:** None of the JSP files validate that the requested data belongs to the authenticated user's customer/tenant. Request parameters like user_cd, gp_cd, loc_cd, and dept_cd are accepted from the request and used directly in database queries without verifying they match the authenticated user's customer context. This is an Insecure Direct Object Reference (IDOR) vulnerability that allows users to access other customers' fleet data by manipulating request parameters.

**Severity:** CRITICAL

**Checklist item:** 2. Authentication and Authorization - Authorization and multi-tenancy

**Evidence:**
```jsp
// email_preop_chk_detailed.jsp:12-19

// This pattern repeats in all files - user_cd from request is never validated against session user
```

**File:** All 10 files reviewed

**Line(s):** Various - parameter retrieval at start of each file's scriptlet
