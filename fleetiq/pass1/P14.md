### P14-F01: Unescaped Output of Database-Sourced Values in Report Headers — Stored XSS
**Description:** Across all ten report include files, database-sourced values — customer/user display names (`get_user`), location names (`get_loc`), department names (`get_dep`), report form names (`form_nm`), start/end date strings (`st_dt`, `to_dt`), and month strings (`month`) — are rendered into HTML using raw `<%= ... %>` scriptlet expressions without any HTML escaping. There is no use of `<c:out>` or `${fn:escapeXml(...)}`. If any of these values contain characters such as `<`, `>`, `"`, `&`, or `'`, they will be injected verbatim into the HTML. Because these values are drawn from the database (customer names, site names, department names, report names), an attacker who controls stored data — for example, a customer account name or site name containing a script tag — would achieve stored XSS that fires when any user views any report that includes these headers or footers.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)

### P14-F03: User-Supplied `sess_lang` Parameter Written Directly to Session Without Validation
**Description:** In `report_email_header.jsp` (line 20), the value of the `sess_lang` request parameter is retrieved and stored directly into the HTTP session under the key `"lang"` without any validation, sanitization, or whitelisting against known locale values. ### P14-F04: User-Supplied `user_cd` / `cust_cd` / `usr_cd` Parameters Used as Database Lookup Key Without Visible Authentication Check

### P14-F05: `localeKey` Injected Into JSTL `<fmt:message>` Tag Attribute Without Escaping
**Description:** In six of the ten files, `localeKey` (derived from `filter.getLocaleKey()`) is used as the key argument to `<fmt:message key="<%=localeKey %>" />`. The `localeKey` value is drawn from the `filter` object passed into this include — which is ultimately assembled from report parameters. Although JSTL will perform a bundle lookup with this key (so arbitrary HTML is not directly rendered), supplying a crafted key string could cause unexpected lookup failures or, depending on the resource bundle implementation, expose bundle metadata. More critically, if `localeKey` is not validated to be a known bundle key, this represents an uncontrolled parameter flowing into a JSTL directive. This is also present in the `<fmt:setLocale>` tag in `report_email_header.jsp` at line 11 with the raw `sess_lang` parameter.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (XSS / tag injection)

### P14-F07: Mixed Content — Hardcoded HTTP Image URLs on Report Pages Served Over HTTPS
**Description:** Multiple include files contain hardcoded `http://` image URLs referencing `fms.fleetiq360.com`. If the parent application is served over HTTPS (which it should be for a fleet management system), these HTTP image references constitute mixed content. ### P14-F12: Report Include Files Use Raw Untyped `ArrayList` (No Generics) — Unchecked Cast Risk
