## Findings

### P32-F01: Unescaped User-Controlled Request Parameter Reflected into HTML — Reflected XSS
**Description:** In `summary.jsp`, the URL request parameter `module_cd` is read at line 58 directly from the HTTP request with no sanitization or encoding, then reflected raw into the HTML at line 66 inside an anchor `href` attribute using `<%=module_cd %>`. No JSTL `<c:out value="..."/>` or `${fn:escapeXml(...)}` escaping is applied. An attacker can craft a URL with a `module_cd` value containing HTML or JavaScript (e.g., `"><script>alert(document.cookie)</script>`), causing arbitrary script execution in the browser of any user who follows the malicious link. This is a classic reflected XSS vulnerability on an admin page.
**Severity:** HIGH
**Checklist item:** Section 3 — XSS (Cross-Site Scripting)
**Evidence:**
```jsp
// Line 58 — raw request parameter, no sanitization:
String module_cd = request.getParameter("module_cd") == null ? "" : request.getParameter("module_cd");

// Line 66 — reflected directly into href without any escaping:
<li><a href="../pages/admin.jsp?mnm=summary&module_cd=<%=module_cd %>"><fmt:message key="vehicle.summary.summary"/></a></li>
```
**File:** `pages/admin/summary.jsp`
**Line(s):** 58, 66

### P32-F02: Hardcoded User IDs Used as Access-Control Allowlist — Broken Authorization Pattern
**Description:** In `tab_heads.jsp`, conditional access to an additional administrative navigation tab is granted by comparing `session.getAttribute("user_cd")` against a hardcoded list of 8 specific numeric string IDs embedded directly in JSP source code (lines 5–16). This is an anti-pattern for access control: the allowlist is invisible to any role management system, cannot be modified without a code deployment, and creates a covert privilege escalation path. Two of the hardcoded IDs are annotated with full employee names in comments (see P32-F03). If a listed user leaves the organization or their account is compromised, access cannot be revoked without a source code change and redeployment. New administrators cannot be added without the same process. The correct approach is to store privileged role membership in the database and check it via a proper permission/role lookup.

**Checklist item:** Section 2 — Authorization and multi-tenancy

### P32-F04: `request.getSession(true)` Called After Session Attributes Already Read — Ineffective Session Protection
**Description:** Line 62 of `summary.jsp` calls `request.getSession(true)` as a standalone mid-page scriptlet. By this point, the page has already read all security-critical session attributes (`user_cd`, `access_level`, `access_cust`, `access_site`, `access_dept`) at lines 30–34, set them on the report bean at lines 36–40, and completed the data retrieval call `report.init()` at line 43. The `getSession(true)` call at line 62 will return the existing session unchanged (or create a new empty session if, somehow, none existed), and does not constitute session fixation protection. To protect against session fixation, the session must be invalidated and a new session ID issued immediately upon successful authentication — not mid-page after all protected operations have already run. The placement suggests a misunderstanding of what `getSession(true)` does in this context.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Session management (session fixation)
**Evidence:**
```jsp
// Lines 30–34: all security-sensitive session attributes already consumed before line 62:
String access_user = (String)session.getAttribute("user_cd");
String access_level = (String)session.getAttribute("access_level");
String access_cust=(String)session.getAttribute("access_cust");
String access_site=(String)session.getAttribute("access_site");
String access_dept=(String)session.getAttribute("access_dept");

// Line 43: data already retrieved using those values:
report.init();

// Line 62: getSession(true) called too late, after all protected operations:
<% request.getSession(true); %>
```
**File:** `pages/admin/summary.jsp`
**Line(s):** 30–34, 43, 62

**Description:** Multiple `Integer.parseInt()` calls are made without try-catch blocks:
- `summary.jsp` lines 100 and 163: `Integer.parseInt(no_units)` and `Integer.parseInt(no_units_hire)` — values from the database via the report bean. ### P32-F07: Site-Name Branching for Feature and Permission Gating — Multi-Tenant Risk if Misconfigured
**Description:** Both files use `LindeConfig.siteName` to branch behavior: `summary.jsp` line 74 shows a different data table layout for "AU" vs non-AU deployments, including additional columns for "Units Not on Hire" and lockout-disabled units only in the AU branch. `tab_heads.jsp` lines 5–6 restrict the extra tab (and indirectly the firmware admin check) to non-AU, non-MLA deployments. `LindeConfig.siteName` is a static server-side configuration value. On the `multi-customer-sync-to-master` branch — explicitly designed for multi-customer operation — any per-site feature gating via a single static config value is architecturally fragile. ### P32-F08: Wildcard Imports in JSP Page Directive
**Description:** `summary.jsp` uses two wildcard imports (`java.util.*` and `com.torrent.surat.fms6.util.*`) in the `<%@ page import="..." %>` directive. Wildcard imports make it impossible for a code reviewer or static analysis tool to determine exactly which classes are in scope. If the `com.torrent.surat.fms6.util` package contains classes with similar names to standard library classes, import ambiguity can cause subtle bugs. More importantly for an audit, wildcards obscure the attack surface — it is not possible from this file alone to determine which utility classes from the package are used and whether any are security-sensitive.
**Severity:** LOW
**Checklist item:** Section 3 — Injection Vulnerabilities (code quality / auditability)
**Evidence:**
```jsp
<%@ page import="java.util.*,com.torrent.surat.fms6.util.*"%>
```
**File:** `pages/admin/summary.jsp`
**Line(s):** 2
