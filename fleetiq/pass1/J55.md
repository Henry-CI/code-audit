### J55-F01: Hardcoded Plaintext Credentials in RuntimeConf.java
**Description:** `RuntimeConf.java` contains multiple plaintext credentials committed directly to source control. This includes firmware server passwords, an FTP account password, a generic test/admin password, and full Clickatell SMS API credentials (username, password, API ID). Any developer with read access to the repository has these credentials. If the repository is ever leaked or cloned by an unauthorised party, all of these accounts are immediately compromised.
**Severity:** CRITICAL
**Checklist item:** Section 1 — Secrets and Credentials
**Evidence:**

### J55-F02: SQL Injection in mail.Ename() — String Concatenation with Unsanitised User Input
**Description:** The `Ename()` method in `mail.java` builds a SQL query by directly concatenating the `userid` parameter into the query string, then executes it with a raw `Statement` (not a `PreparedStatement`). Any caller that passes an attacker-controlled `userid` value can manipulate the query, potentially dumping arbitrary data from `FMS_USR_MST` or executing stacked queries depending on the database driver. This file is directly in the call chain invoked from `MailExcelReports`.
**Severity:** HIGH
**Checklist item:** Section 3 — SQL Injection
**Evidence:**
```java
// mail.java line 54-55
query="select \"EMAIL_ADDR\",\"CONTACT_FIRST_NAME\",\"CONTACT_LAST_NAME\" from \"FMS_USR_MST\" where \"USER_CD\"='"+userid+"'";
rs = stmt.executeQuery(query);
```
`stmt` is created as `dbcon.createStatement()` (line 51), not a `PreparedStatement`. The `userid` parameter has no validation or escaping applied before insertion into the query string.
**File:** `WEB-INF/src/com/torrent/surat/fms6/util/mail.java`
**Line(s):** 51, 54, 55

### J55-F03: Email Header Injection via Unsanitised `subject` and `email_id` Parameters
**Description:** In both overloads of `sendExcelReport()`, the `subject` and `email_id` parameters are passed directly to `mail.sendMail()` and ultimately to `message.setSubject(subject)` (mail.java line 121) and `InternetAddress.parse(rEmail, false)` (mail.java line 115). The `false` flag to `InternetAddress.parse()` disables strict RFC 822 parsing, allowing multi-address strings. If either `subject` or `email_id` can contain newline characters (`\r\n`) from user-supplied or database-sourced input, an attacker can inject additional email headers (CC, BCC, To) and redirect or copy report emails to arbitrary addresses. There is no sanitisation of these parameters anywhere in the call chain.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities
**Evidence:**
```java
// MailExcelReports.java line 20
isMailSent = mail.sendMail(subject, mBody, "unknown", email_id, "unknown", LindeConfig.mail_from, attachment, attachmentName);

// mail.java line 115 — false disables strict address parsing
message.setRecipients(Message.RecipientType.TO, InternetAddress.parse(rEmail, false));

// mail.java line 121
message.setSubject(subject);
```
Neither `subject` nor `email_id` is stripped of CR/LF characters or validated before use.
**File:** `WEB-INF/src/com/torrent/surat/fms6/excel/reports/MailExcelReports.java`
**Line(s):** 13, 20, 32, 36

### J55-F04: Uncontrolled File Attachment Path — Potential Path Traversal
**Description:** The `attachment` parameter passed into both `sendExcelReport()` overloads is forwarded directly to `new FileDataSource(attachment)` in `mail.java` (line 175). This means any file path the `attachment` string resolves to on the server filesystem will be read and attached to the outgoing email. If the caller of `sendExcelReport()` accepts this path from user input without validation, an attacker could supply a path such as `/etc/passwd` or `/home/gmtp/linde_config/settings_multi.xml` and have sensitive server files emailed to an arbitrary address. The `ExcelUtil.sendEmail()` caller (ExcelUtil.java line 131) passes `attachment_data` which itself is built from a path derived from class location arithmetic — this specific caller path is not directly user-controlled, but the method's signature accepts an arbitrary String with no restriction.
**Severity:** HIGH
**Checklist item:** Section 3 — Injection Vulnerabilities / Section 5 — Data Exposure and Privacy
**Evidence:**
```java

public boolean sendExcelReport(String subject, String email_id, String attachment, String attachmentName){

// mail.java line 172-178 — attachment path used directly as FileDataSource
if(!attachment.equalsIgnoreCase("")){
    messageBodyPart = new MimeBodyPart();
    FileDataSource source = new FileDataSource(attachment);   // line 175
    messageBodyPart.setDataHandler(new DataHandler(source));

    multipart.addBodyPart(messageBodyPart);

```
No path canonicalisation, no allowlist, no containment check.
**File:** `WEB-INF/src/com/torrent/surat/fms6/excel/reports/MailExcelReports.java`

### J55-F05: Exception Stack Traces Written to System.out / e.printStackTrace()
**Description:** All caught exceptions in `MailExcelReports.java` are handled solely via `e.printStackTrace()`. In a Tomcat deployment, `System.out` and `printStackTrace()` output goes to `catalina.out` or the server console — neither of which is an appropriate structured log channel, and both may be exposed to operators or monitoring systems in plain text. More critically, if Tomcat is misconfigured to redirect console output to a JSP error page or if any caller re-throws and a default error page renders the exception, stack traces including internal class names, SQL queries, and file paths will be visible to end users. This is noted in conjunction with F02 (SQL injection) because an error triggered during SQL execution would expose the raw SQL string in the stack trace.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**
```java
// MailExcelReports.java lines 22-26
} catch (AddressException e) {
    // TODO Auto-generated catch block
    e.printStackTrace();
} catch (MessagingException e) {
    // TODO Auto-generated catch block
    e.printStackTrace();
}
```
The `// TODO Auto-generated catch block` comments confirm these are unimplemented stubs — the exception handling was never properly designed.
**File:** `WEB-INF/src/com/torrent/surat/fms6/excel/reports/MailExcelReports.java`
**Line(s):** 22–26, 38–43

### J55-F08: Raw (Non-Generic) ArrayList Usage in Bean Classes — Type Safety Absent
**Description:** Both `BatteryReportBean` and `CimplicityShockReportBean` declare all list fields and method parameters using raw `ArrayList` without type parameters (e.g., `ArrayList` rather than `ArrayList<String>`). While this is not directly an exploitable security vulnerability, the absence of type safety means the compiler cannot enforce what data is stored in or read from these lists. In the context of these beans being populated from database queries and then serialised into Excel reports, a type confusion at runtime could result in unintended data being included in an emailed report — a data exposure risk. This is flagged as an informational finding.
**Severity:** INFO
**Checklist item:** Section 5 — Data Exposure and Privacy

### J55-F09: No Customer/Tenant Scoping in Bean Classes
**Description:** `BatteryReportBean` and `CimplicityShockReportBean` contain no `customer_id`, `tenant_id`, or equivalent field to associate the report data with a specific customer. On a multi-customer branch (`multi-customer-sync-to-master`), beans that carry report data without a tenant discriminator risk being populated with data from the wrong customer if calling code mixes up the population step or reuses beans across requests. This is a structural observation — the actual isolation enforcement (or lack thereof) depends on callers not visible in these files.

## Checklist Items With No Issues

The following checklist items were reviewed against the three assigned files and the directly referenced utility classes and found to have no issues attributable to these specific files:

- **Section 3 — File Inclusion:** No JSP `<jsp:include>` or `<%@ include %>` directives exist in these Java files. Not applicable to `.java` bean/service classes.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage found in any of the three assigned files or in `mail.java`, `LindeConfig.java`, or `RuntimeConf.java`.
- **Section 3 — SSRF:** No `HttpURLConnection` or HTTP client usage in the three assigned files. `mail.java` uses JNDI to obtain a JavaMail `Session` from the container rather than constructing an SMTP connection from a user-supplied URL — no SSRF vector identified in these files.
- **Section 4 — CSRF:** These are email-sending and data bean classes with no HTTP request handling and no form processing; CSRF is not applicable at this layer.
- **Section 6 — Transport Security (SMTP):** The mail session is obtained via JNDI (`mail/Session`) and therefore TLS configuration for SMTP is a container/JNDI configuration concern, not controlled by these source files. No explicit plaintext SMTP socket construction found.
- **Section 7 — Dependency Vulnerabilities:** No third-party library versions are specified within these three files. Dependency version concerns are addressed at the `pom.xml`/`build.xml` and committed JAR level and are out of scope for this specific file set.
