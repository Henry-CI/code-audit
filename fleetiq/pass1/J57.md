### J57-F01: Raw (unchecked) ArrayList usage in DailyVehSummaryReportBean
**Description:** Every `ArrayList` field in `DailyVehSummaryReportBean` is declared as a raw type (no generic type parameter). Raw collections suppress compile-time type checking, allowing any object type to be inserted. If the data held in these lists originates from database queries or user input and is subsequently rendered to report output (Excel or JSP), the absence of type constraints means there is no compile-time guarantee about the data type being handled, increasing the risk of unexpected coercions, ClassCastExceptions at runtime, and difficulty auditing what data actually flows into these fields.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (fleet data in report beans must be correctly typed and bounded); also relevant to Section 3 — Injection Vulnerabilities (data flowing from DB into untyped lists and on to report output without type safety)
**Evidence:**

### J57-F02: All fields in DailyVehSummaryReportBean are package-private (no access modifier)
**Description:** Every field in `DailyVehSummaryReportBean` — including fields that hold driver names (`dsum_driver_nm`), vehicle codes (`dsum_veh_cd`), vehicle names (`dsum_veh_nm`), and the customer type (`cust_type`) — is declared without an access modifier, making it package-private. Any class within the same package (`com.torrent.surat.fms6.excel.reports.beans`) can directly read or overwrite these fields without going through the getter/setter pair, bypassing any future validation logic that might be added to the setters. This is especially relevant for `cust_type` (line 33), which appears to be a customer-scoping discriminator; direct package-level write access to this field could allow intra-package code to silently change the customer context of a report bean without going through a controlled setter.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy: customer data scoping); Section 5 — Data Exposure and Privacy
**Evidence:**

### J57-F03: No customer/tenant ID field in DailyVehSummaryReportBean — multi-tenancy scoping gap
**Description:** `DailyVehSummaryReportBean` is a report data transfer object that aggregates daily vehicle summary data including driver codes, driver names, vehicle codes, vehicle names, and vehicle type names. The bean contains a `cust_type` field (line 33) but no field for a customer identifier (e.g., `customerId`, `custId`, `customer_id`, or equivalent). The `cust_type` field appears to be a category or type classifier (e.g., "VEH" vs "DRIVER") rather than a tenant-scoping key. Without a customer ID bound to the bean, the bean itself carries no record of which customer the report data belongs to. The critical security question — whether the data loaded into this bean is actually scoped to the authenticated user's customer — cannot be answered from the bean alone and must be verified in the calling code (servlet or DAO layer). This is flagged here as a data model concern requiring follow-up in the report-generation servlet and DAO. If the calling layer does not enforce customer scoping, this bean will silently carry cross-customer data.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authorization and multi-tenancy ("verify that every data access is scoped to the authenticated user's customer/organisation"); Section 5 — Data Exposure and Privacy ("Check report generation and data export features for missing authorisation checks — can a user export another customer's vehicle data by manipulating a report parameter?")
**Evidence:**

### J57-F05: Driver name and vehicle data stored as plain String — no indication of output encoding at bean level

If any downstream layer renders these strings to HTML (e.g., a JSP report view), they must be escaped at the point of output. The bean provides no structural protection. This is noted as a data-flow concern: the risk is realized only if the caller renders the values unsanitized, but the bean offers no safety guardrail.

### J57-F06: DetailUnitReportBmwBean.items list is mutable and returned directly (no defensive copy)
**Description:** `DetailUnitReportBmwBean.getItems()` (line 46) returns a direct reference to the internal `items` list. No defensive copy is made. Any caller that obtains a reference via `getItems()` can mutate the internal list (add, remove, or replace `DetailUnitReportBmwItemBean` entries) without going through the bean's constructor or any controlled add method. If the same bean instance is shared across threads or accessed in a context where multiple callers interact with it, this exposes a data integrity risk. In a report-generation context, this could result in a report being modified mid-generation or data from one context leaking into another. There is no `addItem()` method, meaning callers must use the mutable list directly to populate the bean.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (data integrity of report output); Section 2 — Authorization and multi-tenancy (if items can be injected by a caller without validation, tenant isolation enforcement in the bean layer is impossible)
**Evidence:**
```java
// Lines 46–48 — direct reference returned, no defensive copy

    return items;
}
```

## Checklist Items With No Issues

The following checklist items were reviewed against the three files in scope and had no findings, because these bean/DTO files contain no SQL, no HTTP handling, no authentication logic, no session handling, no file I/O, no process execution, no HTTP client calls, no credentials, and no configuration:

- **Section 1 — Secrets and Credentials:** No hardcoded credentials, API keys, passwords, connection strings, or credential-bearing comments found in any of the three files.
- **Section 2 — Authentication and Authorization (container-managed auth and session management):** These are pure data beans; they contain no servlet code, no session access, no login logic, and no `web.xml`-level configuration.
- **Section 3 — SQL Injection:** No JDBC calls, no `Statement`, no `PreparedStatement`, no SQL strings of any kind in any of the three files.
- **Section 3 — File Inclusion:** No `jsp:include`, no `Runtime.exec()`, no file path handling in any of the three files.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage.
- **Section 3 — SSRF:** No `HttpURLConnection`, `HttpClient`, or URL construction from any source.
- **Section 4 — Session and CSRF:** No session access or form rendering in these bean files.
- **Section 6 — Transport Security:** No network or connector configuration in these files.
- **Section 7 — Dependency Vulnerabilities:** No third-party library imports; only `java.util.ArrayList` and `java.util.List` from the standard library are imported across all three files. No wildcard imports present.

**Note on scope limitation:** These three files are pure data transfer objects (DTOs / report beans). The higher-risk code paths — the servlets or DAOs that populate these beans with database data, the JSP views that render the bean data to HTML or Excel output, and the authorization checks (or lack thereof) that gate access to the report endpoints — are outside the scope of this agent's assigned files. The MEDIUM findings (F03, F04) are intentionally flagged as requiring follow-up in those layers. The absence of SQL injection findings here does not clear the report feature overall; it only clears these three bean files.
