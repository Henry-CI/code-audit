### J36-F02: Frm_unitSummary_rpt generates a cross-tenant report with no per-customer scoping
**Description:** `Frm_unitSummary_rpt.createExcel(String type)` calls `UnitDAO.getUnitSummary(type)` with no customer ID parameter. The underlying SQL query joins across all customers and filters only by dealer ID (hardcoded values 24 and 75 for `type="summary"`) or by timezone (for `type="summary_apac"`). Neither filter limits results to the requesting user's customer. The resulting Excel report exposes sensitive multi-customer data: customer name, site, department, truck model, serial number, fleet number, GMTP device ID, firmware version, SIM card CCID (current and previous), modem version, and service hours — for all customers matching the dealer filter. In a multi-tenant system this constitutes a data exposure vulnerability; a user at one customer could obtain another customer's full asset inventory.

### J36-F03: Race condition / thread-safety issue in SessionVar.EXCEL_RPT_PATH static field used for file output path
**Description:** `Frm_unitSummary_rpt.createExcel()` constructs the output file path using `SessionVar.EXCEL_RPT_PATH`, a public static (non-volatile, non-synchronized) `String` field. This field is written by `CustomReports.java` at request time from the servlet context real path. In a concurrent environment, one request could overwrite `SessionVar.EXCEL_RPT_PATH` while another request is using it to write an Excel file, causing reports to be written to an unexpected directory or to fail. The use of static mutable state for per-request path construction is a data integrity and potential information-disclosure issue in multi-user scenarios.
**Severity:** MEDIUM
**Checklist item:** Section 5 — Data Exposure and Privacy
**Evidence:**

### J36-F04: Path traversal pattern in getExportDir() in Frm_quater_rpt
**Description:** `Frm_quater_rpt.getExportDir()` derives the export directory by obtaining the classloader code source location, then appending `/../../../../../../../` — seven parent-directory traversal segments. This code is designed to escape the webapp directory structure and reach a system-level directory. The method is used as a base directory for writing chart image files. If the class source location changes (e.g., in a different deployment), the traversal could resolve to an unintended filesystem location. While the value is not derived from user input in this specific path, the pattern of constructing paths with multiple `../` segments is inherently fragile and dangerous; any future change that makes `chart_URL` user-influenced would create a full path traversal vulnerability.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities (File inclusion / path traversal)
**Evidence:**
```java
// Frm_quater_rpt.java lines 797–804
public String getExportDir() throws Exception {
    String dir = this.getClass().getProtectionDomain().getCodeSource()
            .getLocation().toString().substring(6);
    int charCount = dir.length() - dir.replaceAll("/", "").length();
    int index = DataUtil.nthOccurrence(dir, '/', charCount - 1);
    dir = dir.substring(0, index);
    dir += "/../../../../../../../";
    return dir;
}
```
**File:** `WEB-INF/src/com/torrent/surat/fms6/excel/Frm_quater_rpt.java`
**Line(s):** 797–804

## Checklist Items With No Issues

**Section 1 — Secrets and Credentials (within these three files directly):** No hardcoded database passwords, JDBC connection strings, or API keys appear directly in `Frm_national_rpt.java`, `Frm_quater_rpt.java`, or `Frm_unitSummary_rpt.java`. The credential risk is via the imported `RuntimeConf` class (finding J36-F01) and the imported `SessionVar` class.

**Section 3 — SQL Injection:** No SQL is constructed in any of the three reviewed files. All data access is delegated to DAO classes (`ImpactDAO`, `PreCheckDAO`, `LockOutDAO`, `UnitDAO`). SQL injection risk in those DAOs is out of scope for this agent but should be reviewed separately. No string concatenation into SQL was observed in these files.

**Section 3 — XSS:** These are Excel report generation classes (server-side POI), not JSP files. There are no `<%= ... %>` expressions, no HTML output, and no user-controlled values written to HTTP responses in these files. XSS is not applicable.

**Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` usage was found in any of the three files.

**Section 3 — SSRF:** No HTTP client calls (`HttpURLConnection`, `HttpClient`, etc.) were found in any of the three files.

**Section 4 — Session and CSRF:** These are report-generation classes invoked server-side; they do not handle HTTP requests, form submissions, or session management directly. CSRF is not applicable to this layer.

**Section 7 — Dependency Vulnerabilities:** Apache POI is used for Excel generation (via `org.apache.poi` imports). The specific version of POI in use is not determinable from these source files alone; it depends on the JAR committed to the repository, which is outside the scope of this agent's file set. No direct dependency version declarations appear in these files.
