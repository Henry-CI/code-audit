### J50-F01: Hardcoded Production Hostname in Source Code
**Description:** `ExcelImpactReport.java` contains a hardcoded HTTP URL pointing to the production host `fleetfocus.lindemh.com.au`. This string is baked into the compiled application and cannot be changed without a code change and redeployment. It also uses HTTP (not HTTPS), meaning any image fetch from this URL would occur over an unencrypted channel.
**Severity:** HIGH
**Checklist item:** Section 1 — Secrets and Credentials (hardcoded infrastructure); Section 6 — Transport Security (HTTP instead of HTTPS)
**Evidence:**
```java

```

### J50-F02: Image URL Constructed with Unvalidated Database-Sourced Value (Path Traversal / SSRF Risk)
**Description:** In `ExcelImpactReport.java`, the image URL embedded in the Excel report is constructed by concatenating the hardcoded base URL with `RuntimeConf.impactDir` and a value retrieved directly from the bean (`img_link.get(j)`). The `img_link` list is sourced from `impBean.getVimp_img()` which is populated from database query results. If an attacker can influence the stored photo filename (e.g., via a compromised device or a direct database manipulation), they could inject a path traversal sequence or a crafted URL into the link embedded in the Excel file. When opened by an end user, Excel may attempt to resolve that link, creating an SSRF or disclosure vector. The same pattern applies in `ExcelImpactReportBmw.java` where `itemBean.getPhoto()` is concatenated into the URL.

### J50-F03: No Customer/Tenant Scoping Enforced in Report Generation Layer
**Description:** All three report classes accept `cust`, `loc`, and `dep` as plain `String` parameters passed in by callers. None of the three classes perform any check that the `cust` value corresponds to the authenticated session's customer. The data in the beans (`impBean`, `bean`, `utilBean`) is passed in pre-populated from upstream code — but these classes themselves perform no guard. If the upstream caller does not correctly scope the query to the authenticated customer (e.g., by accepting a customer name or ID from a request parameter), any customer's impact or utilisation data could be rendered into a report and returned to a different customer. This is particularly relevant given the branch name (`multi-customer-sync-to-master`) indicating multi-tenant operation. The absence of any `customerId` or session validation within these classes means the report generation layer provides no defence-in-depth against IDOR or tenant data leakage.
**Severity:** HIGH

### J50-F04: GPS Coordinates Written Verbatim into Excel Report Without Sanitisation
**Description:** In `ExcelImpactReportBmw.java`, GPS latitude and longitude values from `itemBean` are written directly into the spreadsheet cell via string concatenation without any sanitisation or validation. While the immediate risk in a cell value context is lower than in HTML output, a formula-injection attack is possible: if any of the GPS-derived strings begins with `=`, `+`, `-`, or `@`, spreadsheet applications (Excel, LibreOffice Calc) may interpret the cell content as a formula. An attacker able to influence GPS data stored in the database could inject a formula that executes when the exported file is opened (CSV/Excel formula injection).
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities; Section 5 — Data Exposure and Privacy
**Evidence:**

### J50-F08: Driver Names, Card Numbers, and Vehicle Identifiers Written to Excel Without Sanitisation
**Description:** All three report classes write PII and operationally sensitive fleet data — driver names, card numbers, machine IDs, serial numbers, model types — directly into Excel cells from bean values sourced from the database. There is no evidence of any sanitisation, masking, or access-level check applied before writing these values. In the context of a multi-tenant system, if any upstream authorisation check fails (see J50-F03), PII for drivers and vehicles belonging to a different customer could be exported. Card numbers in particular (`Vcard_no` in ExcelImpactReport, derived from `impBean.getCardNo()`) may constitute sensitive access-control credentials.

## Checklist Items With No Issues

The following checklist sections were reviewed and no issues were found within the scope of these three files:

- **Section 1 — Secrets and Credentials:** No hardcoded passwords, API keys, JDBC connection strings, or SMTP credentials are present in these three files. The hardcoded hostname in J50-F01 is flagged separately as an infrastructure/transport issue. No `.env` or secrets files are introduced by these files.
- **Section 3 — SQL Injection:** None of these three files contain any JDBC calls, `Statement.executeQuery()`, `PreparedStatement`, or any SQL string construction. All data access is through pre-populated bean objects passed in by callers. SQL injection risk, if any, lies in the DAO/service layer upstream of these classes and is out of scope for these files.
- **Section 3 — XSS:** These are server-side Excel report generation classes. They produce `.xlsx` files, not HTML output. No JSP expressions, `<%= %>` outputs, or HTML rendering occur in these files.
- **Section 3 — File Inclusion:** No `jsp:include`, `<%@ include %>`, or dynamic file inclusion constructs are present.
- **Section 3 — Command Injection:** No `Runtime.getRuntime().exec()` or `ProcessBuilder` calls are present in any of the three files.
- **Section 4 — Session and CSRF:** These are pure report-generation utility classes with no servlet, session, or HTTP request handling. CSRF and session concerns apply at the servlet/JSP layer that invokes these classes, not within these classes themselves.
- **Section 6 — Transport Security:** Beyond the HTTP URL noted in J50-F01, these classes do not configure any TLS connectors, cookie flags, or HTTP headers.
- **Section 7 — Dependency Vulnerabilities:** These files use Apache POI (via `org.apache.poi.ss.usermodel.*` imports). The specific POI version cannot be determined from source code alone — version assessment requires inspection of the committed JAR files, which is out of scope for this agent's assigned files.
