## Findings

### J03-F01: CSV output built by direct string concatenation without encoding — potential data injection into downstream consumers
**Description:** In `DailyUsageHourBean.arrangeData()`, the method builds CSV-formatted strings by directly concatenating bean field values with no escaping or encoding applied. Values come from `DailyUsageDeptDataBean` instances (model name, location, department name, week, date range, unit total, and numeric data arrays). If any of these values originate from user-supplied input or database fields that contain commas, quotes, or control characters, the resulting CSV strings will be structurally malformed and could misdirect a downstream CSV consumer (spreadsheet parser, report engine). The same pattern appears in `getWeekList()` at line 105.
**Severity:** MEDIUM
**Checklist item:** Section 3 — Injection Vulnerabilities (data injection into structured output)
**Evidence:**
```java
// DailyUsageHourBean.java, lines 60–61

+Arrays.toString(usageBean.getData().get(j)).replace("[", "").replace("]", ""));

// DailyUsageHourBean.java, line 105

```

### J03-F02: Raw (unchecked) ArrayList instantiation — suppresses generic type safety
**Description:** All five `ArrayList` fields in `DailyUsageHourBean` are initialized using the raw type `new ArrayList()` rather than `new ArrayList<>()`. While this is a compile-time type-safety issue rather than a direct runtime security vulnerability, it suppresses unchecked-cast warnings that could mask incorrect object insertion at runtime. If the wrong type is inserted into one of these lists (e.g., via reflection or framework population), a `ClassCastException` could be thrown at an unexpected point, potentially causing application instability or information disclosure via stack traces.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (error responses that expose internal detail)
**Evidence:**

### J03-F03: DashboarSubscriptionBean carries email address with no access control enforcement at the bean level
**Description:** `DashboarSubscriptionBean` exposes a user's email address (`emailId`) via a plain getter with no access modifier restriction and no validation constraint on the setter. This bean also carries `custCd` and `userCd` as externally mutable integer fields. Because this is a plain JavaBean with public setters for all fields (including `custCd`), there is no structural barrier preventing a caller from setting `custCd` to an arbitrary value and retrieving subscription records belonging to a different customer. The security depends entirely on the code that populates and uses this bean — the bean itself offers no protection. Given the multi-tenant context of this branch, this design places full tenant-isolation responsibility on every call site.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy; IDOR risk); Section 5 — Data Exposure and Privacy (PII — email address)
**Evidence:**

### J03-F06: getModelList() and getWeekList() accumulate state across repeated calls — potential data leakage between invocations
**Description:** Both `getModelList()` and `getWeekList(String model)` in `DailyUsageHourBean` append to the instance-level fields `modelList` and `weekList` respectively on every invocation. There is no clearing of these lists at the start of each method call (unlike `arrangeData()` which does call `deptList.clear()` and `finalUtil.clear()` at lines 40–41). If the bean is reused across multiple requests (e.g., stored in session scope or used in a non-request-scoped context), calling `getModelList()` or `getWeekList()` a second time will return a list containing duplicated or stale data from a previous call. In a multi-tenant application this could cause data from one customer context to appear in the results of another customer's request.
**Severity:** MEDIUM
**Checklist item:** Section 2 — Authentication and Authorization (multi-tenancy data isolation)
**Evidence:**

## Checklist Items With No Issues

1. **Section 1 — Secrets and Credentials:** No hardcoded passwords, API keys, connection strings, or credentials in any of the three files. No JDBC calls, no `DriverManager` usage, no properties references.

2. **Section 3 — SQL Injection:** No JDBC calls, no `Statement`, no `PreparedStatement`, no SQL strings of any kind present in any of the three files. These are pure data-transfer beans with no database access logic.

3. **Section 3 — XSS:** These are Java bean classes, not JSP files. They contain no output-rendering logic, no `<%= %>` expressions, and no HTML generation. XSS risk from these files specifically is not applicable.

4. **Section 3 — Command Injection:** No `Runtime.exec()`, no `ProcessBuilder`, no external process invocation in any of the three files.

5. **Section 3 — SSRF:** No HTTP client calls, no URL construction, no network I/O in any of the three files.

6. **Section 4 — Session and CSRF:** These are plain JavaBeans with no session manipulation, no request handling, and no form processing. CSRF is not applicable at this layer.

7. **Section 6 — Transport Security:** No network or connector configuration in any of the three files.

8. **Section 7 — Dependency Vulnerabilities:** No library dependencies beyond the Java standard library (`java.io.Serializable`, `java.util.ArrayList`, `java.util.Arrays`) in these files.
