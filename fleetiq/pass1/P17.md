## Findings

### P17-F01: JavaScript redirect uses `location.href()` as a function call (broken redirect)
**Description:** The JavaScript `send()` function on line 10 calls `location.href(url)` using parentheses, treating `location.href` as a function. `location.href` is a property, not a function; calling it this way throws a `TypeError` in all modern browsers and the JavaScript redirect silently fails. The application relies on the meta-refresh fallback on line 4 to redirect to the login page. While this does not constitute a direct security vulnerability, it indicates the file is unmaintained dead code and the defensive redirect mechanism is broken. An attacker who can suppress or delay the meta-refresh (e.g., via a proxy stripping the meta tag or a slow network) would be presented with a blank page with an empty, unauthenticated POST form rather than being directed to login.
**Severity:** LOW
**Checklist item:** Section 2 — Authentication and Authorization (user not reliably redirected to login)
**Evidence:**
```javascript
// Line 7–11
function send()
{
  var url="./pages/login.jsp";
  location.href(url);   // BUG: location.href is a property, not a function
}
```
**File:** index.jsp
**Line(s):** 10

### P17-F02: Empty unauthenticated POST form
**Description:** Line 15 contains a `<form>` element with `method="post"` and an empty `action=""` attribute. The form submits a POST request to the same URL (`index.jsp`) with no destination and no content. Because the action is empty, a POST submission loops back to this page. This is a benign artefact in practice (the form has no inputs or submit button), but the presence of an unintended POST endpoint at the application root is a minor attack surface: certain CSRF-scanning tools or browser behaviours may interact with it unexpectedly, and it adds unnecessary noise to security testing.
**Severity:** INFO
**Checklist item:** Section 4 — Session and CSRF (unintended form endpoint)
**Evidence:**
```html

<form name="form1" method="post" action="">
```
**File:** index.jsp

### P17-F04: Legacy charset declaration (iso-8859-1)
**Description:** Line 13 declares the content type as `charset=iso-8859-1`. This is a legacy single-byte encoding. While this page contains no dynamic output or user-controlled values, the charset declaration applies to the entire page and could affect how a browser interprets any future content added to this file. UTF-8 is the required standard for web applications; iso-8859-1 has historically been used as a vector for certain multi-byte encoding attacks (e.g., charset sniffing exploits). The risk here is low given there is no dynamic output, but it should be updated for consistency with modern standards.
**Severity:** INFO
**Checklist item:** Section 5 — Data Exposure and Privacy (charset encoding hygiene)
**Evidence:**
```html
<!-- Line 13 -->
<meta http-equiv="Content-Type" content="text/html; charset=iso-8859-1">
```
**File:** index.jsp
**Line(s):** 13
