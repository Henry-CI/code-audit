### J76-F01: Broken/Trivially-Reversible Custom Encryption — EncryptTest
**Description:** The `EncryptTest` class implements a homegrown "encryption" scheme. It is not encryption: it applies a deterministic, position-based character shift (add or subtract the 1-based position index) and prepends a fixed prefix selected by password-length bucket. The algorithm has no key, no salt, no randomness, and no cryptographic strength. An attacker who observes any ciphertext can reverse it trivially using the published `decrypt()` method, or by inverting the arithmetic. If this class is used to protect passwords, session tokens, or other sensitive values stored in the database or transmitted over the network, those values offer no meaningful protection. The decompiler header also confirms the source was reconstructed from a `.class` file in 2006, indicating the original source may not exist.
**Severity:** CRITICAL
**Checklist item:** Section 1 — Secrets and Credentials (custom encryption used in place of standard cryptography)
**Evidence:**

### J76-F10: Incorrect Leap Year Algorithm — Dt_Checker
**Description:** The `isleap()` method at lines 141–150 implements an incorrect leap year algorithm. The correct Gregorian rule is: divisible by 4 AND (not divisible by 100 OR divisible by 400). The implementation checks `y % 1000 == 0` (returning true) before `y % 400 == 0` (returning false), which is inverted from the correct rule for millennium years: year 2000 is a leap year (divisible by 400), but the code would return `true` for `y % 1000 == 0` first, which accidentally produces the right answer for 2000, but returns `false` for years divisible by 400 but not 1000 (e.g., year 1600, 2400). Additionally, years divisible by 100 but not 400 are not excluded from the `y % 4 == 0` check, meaning years like 1900 would be incorrectly identified as leap years. While this is primarily a logic defect, if `days_Betn()` or `daysIn()` are used in billing, scheduling, access-period calculations, or audit log date ranges, incorrect date arithmetic could produce security-relevant miscalculations.
**Severity:** LOW
**Checklist item:** Section 5 — Data Exposure and Privacy (incorrect calculations affecting data integrity in reporting/scheduling)
**Evidence:**
