## Findings

### S03-F01: Missing formal documentation for sp_equipment_bypass_alert
**Severity:** HIGH
**Description:** The stored procedure `sp_equipment_bypass_alert` has no formal documentation (COMMENT ON FUNCTION). The function handles critical business logic for equipment bypass alerts and email notifications, processes impact data, and manages alert states. The only documentation present is SQL comments at lines 1-3 describing the function signature. There is no explanation of what the function does, what each parameter represents, or the business rules for alert thresholds (IMPACT_THRESHOLD=500000, ALERT_DURATION=15 minutes).
**Fix:** Add a COMMENT ON FUNCTION statement after line 176 that documents: (1) the function's purpose (monitoring equipment bypass conditions and sending email alerts), (2) all four parameters with their meanings (vehicleID=vehicle identifier, utcTime=timestamp of event, measures=duration in seconds, isEOS=flag indicating EOS message type), (3) the void return value, (4) the alert triggering conditions (duration >= 15 minutes AND impact count >= 2 with impacts >= 500000), (5) the email notification logic for subscribed users.
**File:** sql/sp_equipment_bypass_alert.sql
**Line(s):** 5-173
**Method:** sp_equipment_bypass_alert

### S03-F02: Missing parameter documentation for sp_equipment_bypass_alert
**Severity:** HIGH
**Description:** The function `sp_equipment_bypass_alert` accepts four unnamed parameters, but there is no formal documentation explaining what each parameter represents. While internal variable declarations (lines 18-21) map the parameters to named variables, this is insufficient for API consumers. Parameters are: $1 (integer), $2 (character varying), $3 (integer), $4 (integer). Based on code analysis, these represent vehicleID, utcTime, measures, and isEOS respectively, but this is not formally documented.
**Fix:** Add parameter documentation in the COMMENT ON FUNCTION statement. Document each parameter with its name, type, and purpose: Parameter 1 (integer) = vehicleID - the vehicle identifier from FMS_VEHICLE_MST.VEHICLE_CD; Parameter 2 (character varying) = utcTime - UTC timestamp when the bypass event occurred; Parameter 3 (integer) = measures - duration of bypass in seconds; Parameter 4 (integer) = isEOS - flag indicating if this is an EOS message type (1=yes, 0=no).
**File:** sql/sp_equipment_bypass_alert.sql
**Line(s):** 5-9
**Method:** sp_equipment_bypass_alert

### S03-F03: Incomplete parameter documentation for sp_opchks_response

The function signature uses unnamed parameters ($1-$7), making it difficult for callers to understand what values to pass without reading the implementation. ### S03-F04: Incomplete functional documentation for sp_opchks_response
**Severity:** MEDIUM
**Description:** The COMMENT ON FUNCTION for `sp_opchks_response` (line 669) provides only a high-level description: "This function is response for reading in any OPCHK messages that contains a RESPONSE message." This is insufficient given the function's complexity (667 lines) and three distinct execution paths: (1) survey initialization when $6=0 AND $7=0, (2) special handling when $6=0 only, (3) answer processing and survey completion for other cases. The documentation does not explain the state management, duplicate answer checking (lines 538-598), critical answer handling (lines 601-656), maintenance mode triggering, or email alert logic (line 583).
**Fix:** Expand the COMMENT ON FUNCTION to document: (1) the three execution modes based on $6 and $7 parameter values, (2) how the function manages checklist results in op_chk_checklistresult table, (3) how answers are stored in op_chk_checklistanswer or op_chk_checklistanswer_unsync tables, (4) the critical answer validation logic that can trigger maintenance mode, (5) the duplicate answer prevention mechanism added in 2012, (6) the email alert triggering logic when surveys complete (sp_email_checklist_alert call), (7) the session update logic via sp_update_session.
**File:** sql/sp_opchks_response.sql
**Line(s):** 669
**Method:** sp_opchks_response
